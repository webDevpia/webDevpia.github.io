## 프로젝트 A: 역할 전환 AI 챗봇 (`app_chatbot.py`)

### Part 1: 환경 설정 (1~16행)

```python
import streamlit as st          # ① Streamlit: 파이썬으로 웹 앱을 만드는 라이브러리
from openai import OpenAI        # ② OpenAI API 클라이언트 (v1+ 문법)
from dotenv import load_dotenv   # ③ .env 파일에서 환경변수를 읽어오는 도구
import os                        # ④ 운영체제 환경변수 접근용
```

| 라이브러리 | 역할 | 비유 |
|-----------|------|------|
| `streamlit` | 웹 화면(UI) 생성 | 인테리어 도구 |
| `openai` | ChatGPT API 호출 | AI에게 전화 거는 전화기 |
| `dotenv` | API 키를 안전하게 보관 | 금고에서 열쇠 꺼내기 |
| `os` | 환경변수 접근 | 금고 자체 |

```python
load_dotenv()                    # .env 파일의 내용을 환경변수로 불러옴
```

> **`load_dotenv()`는 왜 필요한가?**
> API 키를 코드에 직접 쓰면(`api_key = "sk-..."`) 보안 위험이 있습니다.
> `.env` 파일에 분리하고 `load_dotenv()`로 읽으면, 코드에는 키가 노출되지 않습니다.

```python
api_key = os.getenv("OPENAI_API_KEY")   # 환경변수에서 API 키 읽기
if not api_key:                          # 키가 없으면
    st.error("API 키가 설정되지 않았습니다...")  # 에러 메시지 표시
    st.stop()                            # 앱 실행 중단

client = OpenAI(api_key=api_key)         # OpenAI 클라이언트 생성 (이후 모든 API 호출에 사용)
```

> **왜 `st.stop()`을 쓰나?**
> API 키 없이 코드가 계속 실행되면 나중에 이해하기 어려운 에러가 발생합니다.
> 미리 차단하고 사용자에게 명확한 메시지를 보여주는 것이 좋습니다.

### Part 2: CRAFT 프레임워크 기반 역할 정의 (18~37행)

```python
ROLES = {                     # 딕셔너리: 역할 이름 → 프롬프트 문자열
    "파이썬 튜터": """...""",   # 키: 사이드바에 표시될 역할 이름
    "딥러닝 멘토": """...""",   # 값: system 메시지로 전달될 프롬프트
    "코드 리뷰어": """...""",
}
```

> **왜 딕셔너리인가?**
> 사용자가 "파이썬 튜터"를 선택하면 `ROLES["파이썬 튜터"]`로 해당 프롬프트를 바로 꺼낼 수 있습니다.
> 역할을 추가하려면 딕셔너리에 항목만 추가하면 됩니다.

#### "파이썬 튜터" 프롬프트 CRAFT 분석

```python
"""
[Context] 파이썬을 처음 배우는 비전공자 학생이 프로그래밍 기초를 질문합니다.
#  ↑ C: 누가 질문하는지 (학습자 수준/배경)

[Role] 당신은 3년차 백엔드 개발자이고 신입 교육을 담당합니다.
#  ↑ R: AI가 맡을 역할 (경력 + 전문분야)

[Action] 질문에 항상 실행 가능한 코드 예시를 포함해 답변합니다.
         잘못된 코드를 보여주면 어디가 잘못인지 짚고 수정된 코드를 제시합니다.
#  ↑ A: 구체적으로 어떻게 행동해야 하는지

[Format] 답변 구조: ① 핵심 설명 → ② 코드 예시 → ③ 한 줄 팁.
         마지막에 항상 '더 궁금한 점이 있나요?'라고 묻습니다.
#  ↑ F: 출력의 형태/구조

[Tone] 친절하고 격려하는 선배 개발자 말투. 전문 용어는 쉬운 비유와 함께 설명합니다.
#  ↑ T: 말투와 대상 수준
"""
```

> **핵심 포인트**: CRAFT 5요소가 빠짐없이 들어가야 일관된 품질의 답변이 나옵니다.
> Context나 Format이 빠지면 AI가 임의로 판단하여 답변 스타일이 들쭉날쭉해집니다.

```python
st.set_page_config(
    page_title="역할 전환 AI 챗봇",  # 브라우저 탭 제목

    layout="centered"                # 화면 가운데 정렬

st.title("💬 프로그래밍 학습 AI 챗봇")       # 페이지 큰 제목
st.caption("역할을 선택하면 해당 전문가로 대화합니다.")  # 작은 설명
```

```python
with st.sidebar:                     # 사이드바 영역 시작
    st.header("설정")
    선택_역할 = st.radio(             # 라디오 버튼으로 역할 선택
        "대화 상대 선택",
        list(ROLES.keys())           # ["파이썬 튜터", "딥러닝 멘토", "코드 리뷰어"]
    )
```

> **`st.radio()`란?**
> 여러 선택지 중 하나만 고를 수 있는 라디오 버튼입니다.
> `list(ROLES.keys())`로 딕셔너리의 키 목록을 선택지로 사용합니다.

#### 역할 변경 감지 로직

```python
    # 처음 실행 시: session_state에 현재 역할 저장
    if "현재_역할" not in st.session_state:
        st.session_state["현재_역할"] = 선택_역할

    # 역할이 바뀌었으면: 대화 초기화 + 페이지 새로고침
    if st.session_state["현재_역할"] != 선택_역할:
        st.session_state["현재_역할"] = 선택_역할
        st.session_state["messages"] = []   # 대화 기록 비우기
        st.rerun()                          # 페이지 새로고침
```

> **`st.session_state`란?**
> Streamlit은 사용자가 버튼을 누를 때마다 **전체 코드를 처음부터 다시 실행**합니다.
> 그래서 일반 변수는 매번 초기화됩니다. `session_state`는 새로고침해도 값이 유지되는 **특수 저장소**입니다.
>
> **비유**: 칠판(일반 변수)은 수업 끝나면 지워지지만, 공책(session_state)은 다음 수업에도 남아 있습니다.

```python
    st.divider()                     # 구분선
    st.markdown(f"**현재 역할**: {선택_역할}")
    st.markdown(f"**대화 수**: {len(st.session_state.get('messages', []))} 턴")
    #                                 .get('messages', [])는 키가 없으면 빈 리스트 반환

    if st.button("대화 초기화", use_container_width=True):
        st.session_state["messages"] = []
        st.rerun()
```

### Part 4: 대화 히스토리 표시 (64~72행)

```python
# session_state에 messages 키가 없으면 빈 리스트로 초기화
if "messages" not in st.session_state:
    st.session_state["messages"] = []

# 저장된 모든 대화를 화면에 표시
for msg in st.session_state["messages"]:
    with st.chat_message(msg["role"]):   # "user" 또는 "assistant" 말풍선
        st.write(msg["content"])          # 메시지 내용 표시
```

> **`st.chat_message("user")`란?**
> ChatGPT처럼 사용자와 AI의 말풍선을 자동으로 만들어줍니다.
> `"user"` → 오른쪽 말풍선, `"assistant"` → 왼쪽 말풍선

### Part 5: 사용자 입력 → API 호출 → 응답 표시 (74~97행)

```python
# 월러스 연산자(:=) — 입력값이 있으면 변수에 저장하고 True 반환
if user_input := st.chat_input(f"{선택_역할}에게 말을 걸어보세요..."):
```

> **`:=` (월러스 연산자)란?**
> `if x := 값:` 은 "값을 x에 넣고, x가 비어있지 않으면 실행"이라는 의미입니다.
> `st.chat_input()`은 사용자가 Enter를 누르면 입력한 텍스트를, 아직 안 눌렀으면 `None`을 반환합니다.

```python
    # 사용자 메시지를 화면에 표시하고 히스토리에 저장
    with st.chat_message("user"):
        st.write(user_input)
    st.session_state["messages"].append({"role": "user", "content": user_input})
```

```python
    # API에 보낼 메시지 구성: system(역할 프롬프트) + 전체 대화 히스토리
    api_messages = [
        {"role": "system", "content": ROLES[선택_역할]},  # CRAFT 프롬프트
    ] + st.session_state["messages"]                       # 이전 대화 전체
```

> **왜 전체 히스토리를 보내나?**
> OpenAI API는 **상태를 기억하지 않습니다**. 매번 새로운 요청입니다.
> "이전에 뭘 얘기했는지" 알려주려면 **전체 대화 기록을 매번 함께** 보내야 합니다.
>
> 이것이 바로 **멀티턴 대화**의 원리입니다. (LangChain의 "메모리"도 같은 원리)

```python
    with st.chat_message("assistant"):
        with st.spinner("생각 중..."):        # 로딩 스피너 표시
            response = client.chat.completions.create(
                model="gpt-4.1-nano",         # 사용할 모델
                messages=api_messages,         # system + 전체 대화 히스토리
                temperature=0.7,              # 창의성 수준 (0=정확, 1=창의적)
                stream=True,                  # 실시간 스트리밍 활성화
            )
            전체_답변 = st.write_stream(response)  # 스트리밍으로 한 글자씩 출력
```

> **`stream=True`란?**
> ChatGPT처럼 답변이 한 글자씩 나타나는 효과입니다.
> `False`면 전체 답변이 완성될 때까지 기다려야 합니다.
>
> **`temperature=0.7`이란?**
> - `0`: 매번 같은 답변 (정확성 중시)
> - `1`: 매번 다른 답변 (창의성 중시)
> - `0.7`: 적당히 창의적 (챗봇에 적합)

```python
    # AI 응답을 히스토리에 저장 (다음 대화 때 맥락으로 전달하기 위해)
    st.session_state["messages"].append({"role": "assistant", "content": 전체_답변})
```

```

### Part 2: Few-shot 예시 정의 (13~49행)

```python
EXAMPLES = {
    "마케팅": [                    # 키: 직무명
        {"role": "user", "content": "자소서: '...' \n직무: 마케팅"},     # 입력 예시
        {"role": "assistant", "content": """**강점**\n1. ..."""},       # 출력 예시
    ],
    "개발": [
        {"role": "user", "content": "자소서: '...' \n직무: 개발"},
        {"role": "assistant", "content": """**강점**\n1. ..."""},
    ],
}
```

> **Few-shot이란?**
> LLM에게 "이런 입력이 들어오면 이런 출력을 해줘"라는 **예시를 미리 보여주는** 기법입니다.
> 예시가 0개면 Zero-shot, 1~3개면 Few-shot입니다.
>
> **왜 직무별로 다른 예시를 쓰나?**
> 마케팅 자소서와 개발 자소서는 강조해야 할 역량이 다릅니다.
> 마케팅은 "팔로워 성장", 개발은 "API 설계, 성능 개선" 같은 사례가 필요합니다.

```python
few_shot = EXAMPLES.get(job, EXAMPLES["마케팅"])
#          .get(키, 기본값) → 해당 직무 예시가 없으면 마케팅 예시를 기본으로 사용
```

### Part 3: CO-STAR 프레임워크 기반 System 프롬프트 (51~61행)

```python
def review_resume(job: str, content: str) -> str:
    # job: 직무명 ("마케팅", "개발" 등)
    # content: 사용자가 입력한 자소서 텍스트
    # -> str: 첨삭 결과 문자열을 반환

    system = f"""
[Context]  취업 준비생이 {job} 직무에 지원하기 위해 자기소개서를 작성했습니다.
#  ↑ C: 상황 설명. f-string으로 직무명을 동적 삽입

[Objective] 자소서의 강점과 개선점을 분석하고, 합격 가능성을 높이는 수정안을 제시합니다.
#  ↑ O: 목표 (CRAFT의 Action과 유사하지만 "목표 지향적")

[Style] 채용 시장의 최신 트렌드와 직무별 핵심 역량을 기준으로 전문적으로 평가합니다.
#  ↑ S: 어떤 관점에서 평가할지 (문체/접근 방식)

[Tone] 격려하되 솔직한 멘토 어조. 강점을 먼저 언급한 뒤 개선점을 제시합니다.
#  ↑ T: 말투

[Audience] 자소서 작성 경험이 적은 대학생 또는 취업 준비생
#  ↑ A: 누가 읽는지 (CO-STAR의 핵심 차별점 — CRAFT에는 없는 요소)

[Response] 반드시 다음 형식으로 출력하세요:
**강점** (3개) → **개선점** (3개) → **수정된 자소서** (완성본)
#  ↑ R: 출력 형식 (CRAFT의 Format에 해당)

주의: 지원자가 경험하지 않은 허위 경험이나 성과를 날조하지 마세요.
#  ↑ 추가 제약 (RISEN의 Narrowing에 해당)
"""
```

> **CRAFT vs CO-STAR 비교**
>
> | | CRAFT | CO-STAR |
> |---|---|---|
> | 핵심 차이 | Role(역할) 중심 | Audience(독자) 중심 |
> | 적합한 상황 | 챗봇, 대화 | 문서 작성, 보고서 |
> | 이 프로젝트에서 | 프로젝트 A | 프로젝트 B |
>
> 자소서 첨삭은 **"읽는 사람(인사 담당자)"이 명확**하므로 CO-STAR가 적합합니다.

### Part 4: API 호출 구성 (63~78행)

```python
    few_shot = EXAMPLES.get(job, EXAMPLES["마케팅"])
    messages = [
        {"role": "system", "content": system},    # ① CO-STAR 프롬프트
        *few_shot,                                 # ② Few-shot 예시 (입력+출력 쌍)
        {"role": "user", "content": f"자소서: '{content}'\n직무: {job}"},  # ③ 실제 사용자 입력
    ]
```

> **`*few_shot`의 `*`는 무엇인가?**
> 리스트를 "풀어서" 넣는 문법(언패킹)입니다.
> ```python
> few_shot = [{"role": "user", ...}, {"role": "assistant", ...}]
> messages = [system, *few_shot, user_msg]
> # 결과: [system, {"role": "user", ...}, {"role": "assistant", ...}, user_msg]
> ```

```
API에 전달되는 messages 구조:

┌─────────────────────────────────────┐
│ system: CO-STAR 프롬프트              │ ← AI의 역할/규칙 설정
├─────────────────────────────────────┤
│ user: "자소서: '마케팅에 관심...'     │ ← Few-shot 예시 입력
│       직무: 마케팅"                  │
├─────────────────────────────────────┤
│ assistant: "**강점** 1. ..."         │ ← Few-shot 예시 출력 (이렇게 답하라는 본보기)
├─────────────────────────────────────┤
│ user: "자소서: '(실제 사용자 입력)'   │ ← 실제 첨삭 요청
│       직무: 개발"                    │
└─────────────────────────────────────┘
```

```python
    try:
        response = client.chat.completions.create(
            model="gpt-4.1-nano",
            messages=messages,
            temperature=0.5,        # 0.7보다 낮음 — 첨삭은 일관성이 더 중요
        )
        return response.choices[0].message.content or "(응답을 생성하지 못했습니다)"
        #      choices[0] → 첫 번째(유일한) 응답
        #      .message.content → 텍스트 내용
        #      or "..." → content가 None이면 대체 텍스트
    except Exception as e:
        return f"API 호출 중 오류가 발생했습니다: {e}"
        # 네트워크 오류, 잔액 부족 등 모든 예외를 잡아서 사용자에게 안내
```

> **왜 `temperature=0.5`인가?**
> 프로젝트 A(챗봇)는 `0.7`로 다양한 답변을 유도했지만,
> 프로젝트 B(첨삭)는 `0.5`로 **일관되고 정확한** 피드백을 우선합니다.

## 프로젝트 A vs B 핵심 비교

| | 프로젝트 A (챗봇) | 프로젝트 B (첨삭) |

| **핵심 Streamlit** | `st.chat_input`, `st.chat_message` | `st.text_area`, `st.button` |

## 자주 묻는 질문 (FAQ)

### Q: `session_state`가 뭔가요?
Streamlit은 코드를 매번 처음부터 다시 실행합니다. `session_state`는 재실행해도 값이 유지되는 저장소입니다. 대화 기록, 현재 역할 등을 여기에 저장합니다.

### Q: `stream=True`와 `stream=False`의 차이는?
- `True`: 답변이 생성되는 대로 한 글자씩 표시 (ChatGPT처럼)
- `False`: 전체 답변이 완성될 때까지 기다린 후 한 번에 표시

### Q: `temperature`를 어떻게 정하나요?
- `0`: 매번 동일한 답변 (수학 문제, 코드 생성에 적합)
- `0.3~0.5`: 약간의 변화 허용 (첨삭, 요약에 적합)
- `0.7~1.0`: 다양한 답변 (창작, 대화에 적합)

### Q: Few-shot 예시가 없는 직무를 선택하면?
`EXAMPLES.get(job, EXAMPLES["마케팅"])`에서 해당 직무의 예시가 없으면 마케팅 예시를 기본으로 사용합니다. 이상적으로는 모든 직무에 맞춤 예시를 추가해야 합니다.

### Q: `.env` 파일 없이 실행하면?
`os.getenv("OPENAI_API_KEY")`가 `None`을 반환하고, `st.error()`로 안내 메시지가 표시된 후 `st.stop()`으로 앱이 중단됩니다.
