# 05. Prompt Files 실습

## 목표

자주 반복되는 Copilot 요청을 Prompt Files로 만들어 재사용합니다.  
기초와 심화를 연속으로 진행합니다.

---

## 배경 지식

### 1) 프롬프트란 무엇인가

프롬프트는 모델에게 어떤 작업을 어떤 방식으로 수행할지 알려주는 작업 지시문입니다.
짧은 질문도 프롬프트지만, 실무에서는 보통 다음 요소를 함께 포함한 구조화된 지시문을 사용합니다.

- 목표: 무엇을 만들거나 분석할지
- 입력 맥락: 어떤 코드/문서/조건을 참고할지
- 제약 조건: 하지 말아야 할 것, 반드시 지킬 규칙
- 출력 형식: 표, JSON, 코드, 체크리스트 등

### 2) Prompt Files란 무엇인가

Prompt Files는 자주 반복하는 Copilot 요청을 `.prompt.md` 파일로 저장해, Chat에서 슬래시 명령처럼 다시 실행하는 기능입니다.

- 위치: `.github/prompts/*.prompt.md`
- 실행 방식: Chat 창에서 `/` 입력 후 프롬프트 선택
- 목적: 반복 작업을 팀이 공유 가능한 템플릿으로 표준화

Custom Instructions가 "항상 적용되는 규칙"이라면, Prompt Files는 "필요할 때 직접 실행하는 작업 템플릿"입니다.

#### 메모리 경로 오해 방지

실습 문서에서 `/memories/...` 같은 표기를 보더라도, 이는 VS Code 탐색기에서 보이는 폴더 경로가 아닙니다.

| 표기 | 의미 | VS Code 탐색기에서 폴더로 보이나? |
|------|------|-------------------------------|
| `/memories/...` | 에이전트 전용 메모리 네임스페이스(논리 경로) | 아니오 |
| `.github/...` | 실제 워크스페이스 파일 경로 | 예 |

확인 방법:

- `/memories/...`는 에이전트의 memory 조회 기능으로 확인
- `.github/...`는 VS Code 탐색기에서 파일로 확인

메모리 조회 요청 예시(복붙용):

1. "사용자 메모리 목록을 보여줘."
2. "세션 메모리 파일 목록을 보여줘."
3. "저장소 메모리(`/memories/repo/`)에 어떤 항목이 있는지 보여줘."

### 3) Prompt Files의 구조와 문법

프롬프트는 보통 frontmatter + 본문으로 구성합니다.

실습에서는 `description`, `name`, `agent` 같은 최소 필드부터 시작하고, 뒤 단계에서 `tools`와 `#` 참조를 추가합니다.

```yaml
---
description: "새 기능 구현과 테스트 추가를 요청할 때 사용"
name: "New Feature"
argument-hint: "예: 구현할 기능과 제약사항을 입력하세요"
agent: "agent"
model: "<선택한 모델 이름>"
---
프롬프트 본문...
기능명: ${input:featureName:구현할 기능명을 입력하세요}
```

핵심 필드:

- `description`: 어떤 상황에서 쓰는 프롬프트인지 검색 가능하게 설명
- `name`: Chat `/` 목록에 보이는 이름
- `argument-hint`(선택): 프롬프트 실행 전 입력창에 표시할 안내 문구
- `agent`: 실행에 사용할 에이전트
- `model`(선택): 프롬프트를 실행할 모델(생략 시 현재 선택 모델 사용)
- `tools`(선택): 이 프롬프트에서 사용할 도구/도구 세트 이름 목록

`tools` 필드 보충 설명:

- 도구 종류: 내장 도구, MCP 서버 도구, 확장(Extension) 제공 도구를 포함할 수 있음
- MCP 서버의 모든 도구 포함: `<server-name>/*` 형식 사용 (예: `my-mcp/*`)
- 우선순위: 프롬프트의 `tools` 지정값 → 프롬프트가 참조한 커스텀 에이전트 도구 → 현재 에이전트 기본 도구
- 주의: 프롬프트 실행 시 해당 도구를 사용할 수 없는 환경이면 그 도구는 무시됨

본문에서 자주 쓰는 문법:

- `${input:변수명}`: 실행 시 사용자 입력을 받는 변수
- `${input:변수명:힌트}`: 입력창에 안내 문구까지 함께 표시하는 변수
- `${selection}`: 에디터에서 선택한 코드/텍스트를 전달하는 변수
- `#` 참조: 추가 문맥을 붙이는 문법
	- `#codebase`: 워크스페이스 전체 코드에서 관련 파일/함수/호출부를 찾아 문맥으로 전달
	- `#problems`: VS Code Problems 패널의 오류/경고 정보를 문맥으로 전달
	- `#changes`: 현재 변경 사항을 문맥으로 전달
- `#tool:도구명`: 프롬프트 본문에서 특정 도구 사용을 명시하는 문법
- Markdown 상대 경로 링크: 관련 규칙/문서를 함께 참조하는 방법
- 추가 인자 전달: `/프롬프트이름 key=value`

한눈에 보는 선택 요약표:

| 상황 | 권장 문법 | 예시 |
|------|-----------|------|
| 실행할 때마다 값이 달라진다 | `${input:변수명}` 또는 `${input:변수명:힌트}` | 기능명, 범위, 옵션 |
| 현재 선택한 코드만 대상으로 삼는다 | `${selection}` | 함수 1개 리뷰, 선택 코드 테스트 작성 |
| 관련 파일/호출부/문제까지 함께 봐야 한다 | `#codebase`, `#problems`, `#changes` | 호출부 추적, 문제 목록 반영 |
| 특정 도구 사용을 본문에서 드러내고 싶다 | `#tool:도구명` | `#tool:browser` |
| 고정 문서나 규칙을 함께 참고하게 하고 싶다 | Markdown 상대 경로 링크 | 가이드 문서, 아키텍처 문서 |

혼합 사용 예시(주석 포함):

```md
---
description: "선택 코드 리뷰와 추가 맥락 검토를 함께 요청할 때 사용"
name: "Review With Context"
argument-hint: "예: 예외 처리, 타입 힌트, 테스트 누락"
agent: "agent"
model: "<선택한 모델 이름>"
tools: [read, search]
---

<!-- 실행할 때마다 달라지는 값은 input 변수로 받는다 -->
집중 리뷰 관점: ${input:focus:예: 예외 처리, 타입 힌트, 중복 로직}

<!-- 현재 에디터에서 선택한 코드는 selection으로 전달한다 -->
리뷰 대상 코드:
${selection}

<!-- 프롬프트 본문에서 특정 도구를 명시하려면 #tool:도구명 문법을 사용한다 -->
필요 시 도구 명시: #tool:browser

<!-- selection 만으로 부족하면 후속 메시지에서 # 참조를 함께 사용한다 -->
추가로 호출부와 관련 테스트까지 확인하려면 후속 입력에
`#codebase #problems`를 붙여 실행하세요.
```

위 예시는 문법을 한 번에 보여주기 위한 소개용 예시입니다. 실습 단계에서는 필요한 요소만 단계적으로 적용합니다.

본문 권장 구조:

- 작업 목표
- 입력 변수
- 제약 조건
- 출력 형식
- 완료 기준

### 4) 프롬프트의 특징

- 비결정성: 같은 프롬프트라도 결과가 조금씩 달라질 수 있음
- 문맥 의존성: 제공한 컨텍스트 품질이 결과 품질을 크게 좌우함
- 형식 민감성: 출력 형식을 구체적으로 지정할수록 일관성이 높아짐
- 모델 의존성: 모델 계열/버전에 따라 잘 먹히는 표현이 다름

### 5) 왜 프롬프트가 중요한가

- 품질: 모호한 요청보다 정확한 결과를 더 자주 얻음
- 재현성: 팀원 간 같은 결과 패턴을 유지하기 쉬움
- 속도: 반복 작업을 빠르게 자동화 가능
- 리뷰 용이성: 요청 자체를 코드처럼 검토하고 개선할 수 있음

### 6) 프롬프트 엔지니어링이란 무엇인가

프롬프트 엔지니어링은 원하는 결과를 안정적으로 얻기 위해 프롬프트를 설계, 실험, 평가, 개선하는 실무 방법론입니다.

일반적인 반복 사이클:

1. 목표/성공 기준 정의
2. 초안 프롬프트 작성
3. 테스트 데이터로 실행
4. 결과 평가(정확도, 일관성, 비용, 지연)
5. 프롬프트 수정 및 버전 관리

### 7) 프롬프트 엔지니어링이 왜 필요한가

- 모델 출력 품질 편차를 줄이기 위해
- 할루시네이션(근거 없는 생성) 가능성을 낮추기 위해
- 비용/지연과 정확도 사이의 균형을 맞추기 위해
- 모델/버전 변경 시 동작 변화를 관리하기 위해

### 8) 프롬프트 엔지니어링의 주요 특징

- 평가 중심: 감이 아니라 테스트와 지표로 개선
- 반복 최적화: 한 번에 완벽한 프롬프트보다 짧은 개선 루프가 효과적
- 구조화: 역할, 지시, 컨텍스트, 출력 형식을 분리
- 운영 관점: 버전 고정, 회귀 테스트, 변경 이력 관리가 중요

### 9) 반드시 알아둘 효과적인 기법

1. 명확한 목표와 완료 기준 명시
2. 출력 형식 강제(JSON, 표, 섹션 템플릿)
3. Few-shot 예시 제공(좋은 입력/출력 샘플 2~5개)
4. 긴 작업은 단계 분해(분석 → 초안 → 검증)
5. 근거 컨텍스트 제공(RAG, 문서 발췌, 코드 선택 영역)
6. 모를 때의 처리 규칙 지정("근거 없으면 모른다고 답하기")
7. 자기 검증 루프 추가("최종 답변 전 체크리스트로 검토")
8. 도구 사용 규칙 명시(언제 검색/파일 읽기/실행할지)
9. 파라미터 튜닝(temperature, max tokens 등)
10. 프롬프트 버전 고정 + eval 세트로 회귀 점검

### 10) 최신 기법 요약 (2026-03 조사 기준)

아래 항목은 2026년 3월 기준 주요 공식 문서(OpenAI, Anthropic, Google, Microsoft)에서 공통적으로 강조되는 최신 실무 패턴입니다.

1. Structured Outputs 우선 설계
	- 자연어 대신 JSON 스키마 기반 출력 강제
	- 후처리 오류 감소, 자동 파이프라인 연결 용이

2. 도구 호출 중심 프롬프팅
	- 모델 단독 추론보다 검색/파일/함수 호출을 결합
	- 최신 정보 반영과 검증 가능성 향상

3. 긴 컨텍스트 최적화
	- 대용량 문서 작업에서 문서 블록과 질의 위치를 전략적으로 배치
	- 문서 구조 태그(XML/Markdown)로 경계 명확화

4. 모델별 추론 강도 제어
	- 단순 작업은 저비용, 복잡 작업은 고추론 모드
	- 품질-지연-비용 균형을 상황별로 조절

5. Agentic 프롬프팅
	- 계획, 도구 실행, 중간 점검, 자기 교정까지 포함
	- 복합 작업에서 성능 향상, 단 과도한 실행은 제어 필요

6. Prompt Caching 및 고정 Prefix 전략
	- 재사용되는 상단 지시문을 안정적으로 유지
	- 지연/비용 절감 효과

7. Evals 기반 운영
	- 프롬프트 품질을 정량 평가하고 버전 교체 전 회귀 검증
	- "잘 되는 느낌"이 아닌 재현 가능한 품질 관리

### 11) 실습 관점에서 지금 꼭 기억할 것

- Prompt Files는 문장을 잘 쓰는 기술이 아니라, 재사용 가능한 작업 프로토콜을 만드는 일입니다.
- 좋은 프롬프트의 핵심은 길이가 아니라 구조(목표, 제약, 형식, 검증)입니다.
- 이 실습의 목적은 프롬프트를 저장하는 법이 아니라, 품질이 반복되는 요청 템플릿을 만드는 것입니다.

## 실습 단계

아래 순서로 진행하면 문법을 단계적으로 익힐 수 있습니다.

1. `input` 변수 중심 프롬프트 작성
2. `selection` 기반 프롬프트 작성
3. `tools`와 `#` 참조를 활용한 리뷰 프롬프트 작성

### 0단계: 사전 준비

1. 워크스페이스 루트가 프로젝트 폴더인지 확인합니다.
2. Copilot Chat 창을 열어 둡니다.
3. 실습 중 생성할 파일 경로를 미리 확인합니다.
	- `.github/prompts/new-feature.prompt.md`
	- `.github/prompts/write-tests.prompt.md`
	- `.github/prompts/code-review.prompt.md`

### 1단계: 폴더 생성

`.github/prompts/` 폴더를 만듭니다.

### 2단계: `input` 변수 중심 프롬프트 작성 - `new-feature.prompt.md`

#### 2-1. 템플릿 복사

아래 경로로 복사합니다.

```text
starter/new-feature.prompt.md.template  →  .github/prompts/new-feature.prompt.md
```

#### 2-2. 파일 내용 작성(복붙용)

아래 내용을 그대로 붙여 넣습니다.

```md
---
description: "Todo Manager에 새 기능을 구현하고 테스트를 추가할 때 사용"
name: "New Feature"
agent: "agent"
---

다음 기능을 Todo Manager에 추가해주세요.

기능명: ${input:featureName:구현할 기능명을 입력하세요}

요구사항:
${input:requirements:기능 요구사항을 입력하세요}

다음을 포함해주세요:
1. `src/todo/manager.py` 에 메서드 추가 (한국어 docstring 포함)
2. `tests/test_manager.py` 에 테스트 추가 (Arrange-Act-Assert 패턴)
3. 모든 기존 테스트가 계속 통과하는지 확인
4. 프로젝트 코딩 지침 준수
```

#### 2-3. 실행 및 확인

1. Chat 창에서 `/` 를 입력하고 `New Feature` 프롬프트를 선택합니다.
2. 예시 입력값을 넣습니다.
	- `featureName`: `완료된 할 일 개수 조회`
	- `requirements`: `완료 상태인 todo만 개수로 반환하고, 테스트 2개 이상 작성`

	- `src/todo/manager.py` 수정 제안 포함
	- `tests/test_manager.py` 테스트 제안 포함
	- AAA 패턴과 타입 힌트/네이밍 규칙 반영

### 4단계: `tools`와 `#` 참조를 활용한 리뷰 프롬프트 작성 - `code-review.prompt.md`

#### 4-1. 파일 생성

`.github/prompts/code-review.prompt.md` 파일을 직접 만듭니다.

#### 4-2. 파일 내용 작성(복붙용)

```md
---
description: "Python 코드의 품질, 네이밍, 구조를 검토하고 실행 가능한 피드백을 제공할 때 사용"
name: "Code Review"
agent: "agent"
tools: [read, search]
---

다음 코드를 리뷰해주세요:

${selection}

다음 관점으로 검토하고 개선 제안을 제공해주세요:

1. **명명 규칙**: 변수명, 함수명이 역할을 명확히 나타내는가?
2. **단일 책임**: 함수/클래스가 하나의 일만 하는가?
3. **타입 힌트**: 모든 시그니처에 타입이 명시되어 있는가?
4. **코드 중복**: 반복되는 로직이 있는가?
5. **테스트 가능성**: 단위 테스트하기 쉬운 구조인가?

## 출력 형식
각 항목에 대해 **현재 상태** → **개선 방향** 으로 작성해주세요.
코드를 직접 수정하지 말고, 피드백만 제공해주세요.
```

#### 4-2-1. `tools` 필드 자세히 이해하기

`tools`는 이 프롬프트가 실행될 때 Copilot이 사용할 도구 범위를 제한하는 설정입니다.

현재 예시의 `tools: [read, search]` 의미:

| 도구 | 역할 | 이 실습에서 필요한 이유 |
|------|------|-----------------------|
| `read` | 파일 내용을 읽어 문맥 파악 | 선택한 코드와 주변 구조를 정확히 이해하기 위해 |
| `search` | 워크스페이스에서 키워드/심볼 검색 | 관련 함수/테스트/호출부를 찾아 리뷰 정확도를 높이기 위해 |

왜 제한하는가?

- 이 프롬프트의 목적은 **수정이 아닌 리뷰 피드백**입니다.
- `tools`를 읽기 중심으로 제한하면, 의도치 않은 수정 제안이나 과도한 작업 흐름을 줄일 수 있습니다.

실무 팁:

1. 리뷰 전용 프롬프트는 `read`, `search`처럼 읽기 도구 위주로 시작합니다.
2. 실제 코드 수정까지 맡기고 싶을 때만 수정 가능한 도구를 별도 프롬프트로 분리합니다.
3. 프롬프트 목적(리뷰/수정/테스트 생성)을 섞지 않으면 결과 품질이 안정적입니다.

#### 4-2-2. Copilot에서 사용 가능한 `tools` 확인 방법

아래 두 가지 방법을 같이 쓰면 가장 빠르게 확인할 수 있습니다.

**방법 A: Tools Picker(도구 선택기)에서 확인**

1. Copilot Chat 창을 열고 Agent 모드로 둡니다.
2. 입력창의 **Configure Tools** 버튼(도구 아이콘)을 클릭합니다.
3. 현재 세션에서 사용 가능한 도구 목록을 확인합니다.
4. 검색창으로 도구명을 필터링해 필요한 도구만 선택/해제합니다.

**방법 B: `#` 자동완성으로 확인**

1. Chat 입력창에 `#`를 입력합니다.
2. 자동완성 목록에 현재 사용 가능한 도구/도구 세트가 표시됩니다.
3. 목록에서 도구를 선택하면 해당 도구를 명시적으로 참조할 수 있습니다.

실습에서 기억할 규칙:

1. 프롬프트 파일에 `tools`를 지정하면 그 목록이 1순위로 적용됩니다.
2. 프롬프트에 없는 도구는 자동으로 쓰이지 않을 수 있습니다.
3. 존재하지 않거나 비활성화된 도구를 `tools`에 적어도 실행 시 무시될 수 있습니다.

예시:

```yaml
tools: [read, search]
```

위 설정은 코드 읽기/검색 중심의 리뷰에 적합하며, 수정 중심 작업은 별도 프롬프트로 분리하는 것이 안전합니다.

#### 4-2-3. 리뷰 프롬프트에서 `${selection}`과 `#` 참조를 함께 쓰는 기준

4단계의 `code-review.prompt.md`는 입력 변수보다 `선택 코드 + 추가 문맥` 조합이 더 중요합니다.

| 리뷰 목표 | 권장 문법 | 예시 |
|-----------|-----------|------|
| 지금 선택한 코드 자체를 리뷰 | `${selection}` | 메서드 1개를 선택해 구조/네이밍 검토 |
| 선택 코드의 호출부나 연관 구현까지 함께 검토 | `#codebase` | `예외 처리 누락 가능성도 함께 봐줘 #codebase` |
| 현재 떠 있는 오류/경고까지 같이 검토 | `#problems` | `실제 문제 목록 기준으로도 봐줘 #problems` |
| 변경된 코드 중심으로 영향 범위 확인 | `#changes` | `최근 변경사항 기준으로 리뷰해줘 #changes` |

즉, 이 단계에서는 `${selection}`으로 대상을 좁히고, 필요할 때만 `#` 참조로 주변 문맥을 넓히는 방식이 가장 안정적입니다.

#### 4-3. 실행 및 확인

**① 코드 선택**

1. VS Code 에디터에서 리뷰할 파일을 엽니다.  
	예: `src/todo/manager.py`
2. 한 번에 너무 넓은 범위(파일 전체)보다, 우선 함수 1개 또는 클래스 일부를 선택합니다.
3. 선택 범위에는 "리뷰 대상 코드"가 명확히 포함되도록 합니다.

	> **권장**: 처음에는 `add`, `complete`, `delete` 중 메서드 1개만 선택해 실행하세요.

**② 프롬프트 실행**

4. 코드가 선택된 상태에서 Copilot Chat 창을 엽니다.
5. 입력창에 `/` 를 입력하고 `Code Review`를 선택합니다.
6. Enter 키로 실행합니다.

	> **팁**: 목록에 안 보이면 `/code` 또는 `/review`로 검색해 선택합니다.

**③ 결과 확인 (핵심 체크리스트)**

아래 기준으로 결과를 검토합니다.

| 확인 항목 | 기대 결과 | 실패 신호 |
|----------|----------|----------|
| 출력 형식 준수 | 각 항목이 **현재 상태 → 개선 방향** 순서로 제시됨 | 장단점만 나열하고 개선안이 없음 |
| 관점 커버리지 | 명명/단일책임/타입힌트/중복/테스트가능성 5개 관점이 모두 다뤄짐 | 일부 관점이 누락됨 |
| 피드백 전용 | 코드 수정 실행 없이 텍스트 피드백 중심 | "파일을 수정했습니다"처럼 수정 실행 흐름 포함 |
| 선택 코드 기반성 | 선택한 코드 기준으로 구체적인 지적 제공 | 일반론만 제시하고 선택 코드 맥락이 없음 |
| 실행 가능성 | 바로 적용 가능한 제안(예: 함수 분리, 시그니처 명확화) 포함 | 추상적 표현만 있고 실천 항목이 없음 |

**④ 품질 높이는 재실행 방법**

1. 결과가 너무 추상적이면 선택 범위를 더 좁혀 다시 실행합니다.
2. 결과가 너무 짧으면 관련 함수(호출부/테스트)까지 함께 선택해 재실행합니다.
3. 피드백만 받고 싶을 때는 후속 메시지로 아래처럼 고정합니다.

	- "코드 수정 없이 리뷰 의견만, 항목별로 현재 상태와 개선 방향을 작성해줘."

**③ Code Review 프롬프트 테스트**

1. VS Code 에디터에서 `src/todo/manager.py` 열기
2. `complete` 메서드 선택 (또는 다른 메서드)
3. Chat에서 `/Code Review` 선택, Enter
4. 확인 항목:

| 항목 | 기대 결과 | 실패 신호 |
|------|---------|----------|

| 피드백만 제공 | 텍스트 의견만 있음 | "파일을 수정했습니다" 같은 실행 흐름 포함 |
| 현재→개선 형식 | 각 항목이 "현재 상태 → 개선 방향" 구조 | 장단점만 나열 |
| 선택 코드 기반 | 선택한 메서드 기준으로 구체적 지적 | 추상적인 원칙만 설명 |

#### 5-4. 통합 동작 확인

**모든 조건을 만족하려면:**

| 확인 항목 | 조건 | 통과 기준 |
|---------|------|---------|
| 파일 생성 | 3개 파일 존재 | `.github/prompts/` 에 3개 모두 있음 |
| Chat 목록 | `/` 입력 시 3개 보임 | New Feature, Write Tests, Code Review 모두 검색 가능 |
| input 변수 | 프롬프트 실행 시 입력창 나타남 | New Feature 실행 시 `featureName`, `requirements` 입력 요청 |
| selection 전달 | 선택 코드가 프롬프트에 반영됨 | Write Tests에 선택 메서드 이름 포함 |
| 출력 차별화 | 프롬프트마다 다른 스타일 | New Feature: 코드 생성, Write Tests: 테스트 작성, Code Review: 피드백 |

**모두 통과하면 실습 완료입니다!**

#### 5-5. 문제 발생 시 빠른 대응

만약 위 검증 중 하나라도 실패하면:

1. 해당 프롬프트 파일을 다시 열어 frontmatter와 본문 확인
2. 파일명이 정확한지 확인
3. 6단계 "자주 발생하는 문제와 해결"을 참고해 문제 해결
4. 프롬프트 수정 후 Chat 창을 완전히 닫았다가 다시 오기

### 6단계: 자주 발생하는 문제와 해결

#### 1) `/` 목록에 프롬프트가 안 보일 때

**증상:** Chat 창에서 `/` 를 입력해도 `New Feature`, `Write Tests`, `Code Review` 중 일부나 전체가 안 나타남

**원인 확인 순서:**

1. 파일 이름 확인
   - 파일명이 `*.prompt.md` 확장자인가? (예: `new-feature.prompt.md` ✓, `new-feature.md` ✗)

2. frontmatter 확인
   - 파일 맨 첫 줄이 `---` 로 시작하는가?
   - frontmatter 끝에 `---` 가 있는가?
   
   ```md
   ---
   description: "..."
   name: "New Feature"
   agent: "agent"
   ---
   ← 빈 줄 또는 본문이 여기서 시작
   ```

3. YAML 구문 확인
   - `---` 다음과 frontmatter 끝 `---` 사이에 `key: value` 형식이 제대로 되어 있는가?
   - 콜론 뒤에 공백이 있는가? (예: `name: "New Feature"` ✓, `name:"New Feature"` ✗)

**해결책:**

```bash
# 파일 목록 확인
ls -la .github/prompts/
```

파일이 존재하면 VS Code에서 파일을 열어 frontmatter를 다시 확인합니다.

**재검증:** Chat 창을 완전히 닫았다가 다시 열고 `/` 를 입력해 확인합니다.

---

#### 2) `${selection}` 결과가 비어 있을 때

**증상:** 프롬프트를 실행했는데 선택한 코드 부분이 빈 값으로 전달됨

**원인:**

- 프롬프트 실행 전에 코드를 선택하지 않음
- 에디터에 포커스가 맞지 않음

**기대 동작:**

1. VS Code 에디터에서 코드를 **마우스로 드래그**해서 선택
2. 선택 상태를 유지한 채로 Copilot Chat 창을 열기
3. Chat에서 `/` 입력해 프롬프트 선택 및 실행

```python
# 예시: add 메서드를 선택 후 프롬프트 실행
def add(self, title: str) -> Todo:
    """새 Todo를 추가하고 반환한다"""
    raise NotImplementedError
```

**해결책:**

1. 에디터에서 코드 범위를 명확히 선택 (드래그는 느리면 Shift + 화살표로 가능)
2. Chat 창에서 프롬프트 실행 전 선택이 유지되었는지 시각적으로 확인
3. Selection 없이 프롬프트를 사용하려면 프롬프트 본문에 다른 방식(예: `#codebase`, 파일 경로)을 함께 사용

**재검증:** Chat 결과에서 "다음 코드에 대한" 같은 표현으로 코드가 전달되었는지 확인

---

#### 3) 변수 입력창이 안 뜰 때

**증상:** 프롬프트를 선택해서 Enter를 눌렀는데 변수 입력창이 나타나지 않음

**원인 - `${input:...}` 문법 확인:**

올바른 문법:

```markdown
${input:변수명:안내 문구}
```

세 부분이 정확히 필요합니다:
- `${ }` 괄호
- `input:` 키워드 (정확히 이 철자)
- `변수명` (직접 쓸 변수 이름, 한글/영문 모두 가능)
- `:` 구분자
- `안내 문구` (사용자에게 보일 설명)

**자주 하는 오타:**

| 오타 | 문제 | 올바른 형태 |
|------|------|-----------|
| `${input 변수명:힌트}` | 콜론 누락 | `${input:변수명:힌트}` |
| `${ input:변수명:힌트}` | 공백 추가 | `${input:변수명:힌트}` |
| `${Input:변수명:힌트}` | 대문자 I | `${input:변수명:힌트}` |
| `${input:변수명 힌트}` | 콜론 대신 공백 | `${input:변수명:힌트}` |
| `${input:변수명}` | 힌트 생략 (동작하지만 안내가 없음) | `${input:변수명:힌트}` 권장 |

**실제 예시:**

❌ 잘못된 예:
```markdown
기능명: ${input 기능명을 입력하세요}
```

✓ 올바른 예:
```markdown
기능명: ${input:featureName:구현할 기능명을 입력하세요}
```

**해결책:**

1. 프롬프트 파일을 VS Code에서 열기
2. `${input:` 로 검색 (Ctrl+F)
3. 각 항목을 위 문법과 비교:
   - `input:` 다음에 변수명이 있는가?
   - 변수명 다음에 `:` 가 있는가?
   - 그 다음에 안내 문구가 있는가?
4. 특수문자(`: {}`)가 빠지거나 추가되지 않았는가?

**재검증:**

프롬프트를 다시 실행해서:
- 입력창이 나타나는가?
- 입력창에 안내 문구가 보이는가?
- 텍스트를 입력할 수 있는가?

---

#### 4) 결과 형식이 들쭉날쭉할 때

**증상:** 같은 프롬프트를 여러 번 실행해도 결과 형식이 일관되지 않음 (표 vs 텍스트, 길이가 제각각 등)

**원인:**

프롬프트 본문의 "출력 형식" 지시가 모호하거나 없음

**해결책:**

프롬프트 본문의 요구사항에 출력 형식을 **구체적으로** 명시합니다.

❌ 상태 불명확:
```markdown
테스트를 작성해주세요.
```

✓ 형식 명확화:
```markdown
## 출력 형식

다음 형식으로 pytest 테스트를 작성해주세요:

1. 함수명: `test_메서드명_조건_기대결과`
2. 패턴: Arrange-Act-Assert (AAA) 패턴
3. 구조:
   ```python
   def test_add_정상입력_todo반환():
       # Arrange
       manager = TodoManager()
       
       # Act
       result = manager.add("새로운 할 일")
       
       # Assert
       assert isinstance(result, Todo)
   ```
4. 각 테스트 함수 앞에 한국어 주석으로 의도 설명
5. 정상 케이스 3개 + 예외 케이스 2개 이상 포함
```

프롬프트에서 사용하는 구체적 기법:

- **표**: `| 항목 | 설명 |` Markdown 표 형식 사용
- **JSON**: `{ "key": "value" }` 형식 강제
- **리스트**: `1.` `2.` 번호 매기거나 `-` `-` 글머리 강제
- **코드 블록**: 결과를 파이썬/테스트 코드로 강제
- **길이 제한**: "50자 이내", "3줄 정도" 같은 제약 명시

**재검증:**

같은 프롬프트를 2~3번 더 실행해서:
- 결과 형식이 일관되는가?
- 길이와 스타일이 비슷한가?
