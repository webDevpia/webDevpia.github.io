# 06. Custom Agents 실습

## 목표

특정 역할과 도구 제한을 가진 Custom Agent를 직접 만들고,
Copilot Chat에서 전용 에이전트로 실행해 결과 품질을 통제합니다.

이번 교시 완료 기준:

1. Planner 에이전트의 역할과 제약을 명확히 정의한다.
2. `tools`를 `read`, `search`로 제한해 계획 전용 모드로 동작시킨다.
3. 실행 결과가 "코드 작성"이 아니라 "구현 계획" 중심인지 검증한다.

---

## 배경 지식

### 1) Custom Agent란 무엇인가

Custom Agent는 특정 작업 목적에 맞춘 전용 AI 모드입니다.

- 위치: `.github/agents/*.agent.md`
- 실행 방식: Chat 드롭다운에서 선택 또는 `@에이전트명` 호출
- 효과: 같은 요청이라도 역할/도구/제약 설정에 따라 결과가 달라짐

핵심은 "질문을 잘 쓰는 것"을 넘어서,
"에이전트의 역할 자체를 설계"하는 데 있습니다.

### 2) 앞선 교시와 무엇이 다른가

| 구분 | 목적 | 실행 시점 | 대표 파일 |
|------|------|----------|-----------|
| Custom Instructions | 항상 적용되는 규칙 제공 | 자동 적용 | `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md` |
| Prompt Files | 반복 요청 템플릿 재사용 | `/프롬프트명` 실행 시 | `.github/prompts/*.prompt.md` |
| Custom Agents | 역할 + 도구 + 제약을 가진 전용 모드 | Agent 선택 시 | `.github/agents/*.agent.md` |

정리하면:

- Instructions는 "규칙"
- Prompt Files는 "요청 템플릿"
- Custom Agents는 "역할을 가진 전용 AI"

### 2-1) "Prompt Files도 `tools`를 제한할 수 있는데, 왜 Custom Agent가 따로 필요한가?"

Prompt Files의 `tools` 필드는 **그 프롬프트를 실행하는 동안만** 도구를 제한합니다.
프롬프트 실행이 끝나면 Copilot은 원래 상태로 돌아갑니다.

Custom Agent의 `tools` 제한은 **에이전트가 선택된 대화 전체**에서 유지됩니다.
"나는 Planner다"라는 정체성이 모든 대화에 유지되는 것입니다.

| 비교 항목 | Prompt Files + `tools` | Custom Agent + `tools` |
|----------|----------------------|----------------------|
| 도구 제한 유지 범위 | 해당 프롬프트 **1회 실행 동안만** | 에이전트 선택 후 **대화 전체** |
| 역할(페르소나) | 없음 — 요청 양식일 뿐 | 이름·역할·제약이 있는 **고정 정체성** |
| 이어지는 추가 질문 | 제한 없이 일반 Copilot으로 돌아감 | 여전히 에이전트 역할 안에서 답변 |
| 다른 파일과 조합 | — | Prompt File에서 `agent: "Planner"`로 참조 가능 |

**예시로 비교:**

```
# Prompt File 방식 (1회성)
/new-feature "로그인 기능"
→ 이 프롬프트 실행 중에만 tools 제한 적용
→ 다음 질문부터는 다시 일반 Copilot

# Custom Agent 방식 (지속)
[Planner 에이전트 선택]
"로그인 기능 구현 계획 잡아줘"
"그런데 인증 토큰은 어디에 저장하면 좋을까?"
"테스트도 어떻게 나눠야 할까?"
→ 세 질문 모두 Planner 역할과 도구 제한 안에서 일관되게 답변
```

Prompt File은 **"특정 요청 한 번을 잘 처리하는 양식"**,
Custom Agent는 **"특정 역할을 가진 AI와 대화하는 세션"** 입니다.

### 3) Agent 파일 구조와 핵심 필드

에이전트 파일은 frontmatter + 본문으로 구성합니다.

```yaml
---
description: "새 기능이나 변경 사항의 구현 계획이 필요할 때 사용한다. 코드를 작성하지 않고 단계별 구현 계획을 생성한다."
name: "Planner"
tools: [read, search]
---
에이전트 역할 설명...
```

핵심 필드:

- `description`: 언제 사용할 에이전트인지 설명
- `name`: Chat에서 보이는 에이전트 이름
- `tools`: 에이전트가 사용할 수 있는 도구 제한
- `handoffs`(선택): 다음 에이전트로 자동 전환할 때 사용

작성 규칙:

- `description`과 본문은 한글 작성
- `name`은 영문 작성
- 목적이 "계획"이면 `tools`에 `edit`, `execute`를 넣지 않음

### 4) 왜 `tools` 제한이 중요한가

Planner의 목적은 "코드 작성"이 아니라 "구현 계획 수립"입니다.

- 권장 도구: `read`, `search`
- 피해야 할 도구(기본 Planner 기준): `edit`, `execute`

도구 제한이 없으면 아래 문제가 자주 발생합니다.

1. 계획 대신 바로 코드 수정안을 제시함
2. 분석 없이 구현부터 시작함
3. 검증 항목(테스트 시나리오, 영향 범위)이 누락됨

### 5) 도구 별칭(Tool Aliases)

| 별칭 | 용도 |
|------|------|
| `read` | 파일 읽기 |
| `search` | 파일/텍스트 검색 |
| `edit` | 파일 편집 |
| `execute` | 터미널 명령 실행 |
| `web` | 웹 검색/URL 조회 |

### 6) 이번 실습에서 꼭 확인할 성공 기준

1. Chat에서 `Planner`가 선택 가능하다.
2. Planner 결과가 계획 중심으로 출력된다.
3. 영향 파일, 구현 단계, 테스트 케이스가 포함된다.
4. "코드를 수정했다" 같은 실행형 응답이 나오지 않는다.

## 실습 단계

아래 순서로 진행하면 가장 안정적으로 실습할 수 있습니다.

1. Agent 파일 준비
2. Planner 정의 작성
3. 실행 및 결과 검증
4. 재실행으로 품질 고정

### 0단계: 사전 준비

1. 워크스페이스 루트가 프로젝트 폴더인지 확인합니다.
2. Copilot Chat 창을 열고 Agent 모드를 켭니다.
3. 아래 경로를 미리 확인합니다.

- `starter/planner.agent.md.template`
- `solution/planner.agent.md`
- `.github/agents/planner.agent.md` (실습 생성 대상)

### 1단계: 폴더와 파일 준비

#### 1-1. 폴더 생성

`.github/agents/` 폴더를 만듭니다.

#### 1-2. 템플릿 복사

아래 파일을 복사합니다.

```text
starter/planner.agent.md.template  →  .github/agents/planner.agent.md
```

### 2단계: `planner.agent.md` 작성 (복붙용)

아래 내용을 `.github/agents/planner.agent.md`에 그대로 붙여 넣습니다.

`````md
---
description: "새 기능이나 변경 사항의 구현 계획이 필요할 때 사용한다. 코드를 작성하지 않고 단계별 구현 계획을 생성한다."
name: "Planner"
tools: [read, search]
---

당신은 소프트웨어 구현 계획 전문가입니다.

## 역할

- 주어진 기능 요구사항을 분석하여 단계별 구현 계획을 수립한다.
- 기존 코드베이스(`src/todo/`)를 파악하고 수정이 필요한 파일과 범위를 제시한다.
- 예상되는 테스트 케이스 목록을 작성한다.
- 구현 순서와 의존 관계를 명시한다.

## 제약

- 요청 없이는 코드를 직접 작성하거나 파일을 수정하지 않는다.
- 계획 수립에만 집중한다.

## 출력 형식

계획은 다음 구조의 마크다운으로 작성한다:

```text
## 구현 계획: [기능명]

### 분석
- 영향받는 파일:
- 추가/수정할 메서드:

### 구현 단계
- [ ] 단계 1
- [ ] 단계 2

### 예상 테스트 케이스
- test_...
```
`````

**① 에이전트 선택**

1. Copilot Chat의 Agent 드롭다운을 엽니다.
2. 목록에서 `Planner`를 선택합니다.

**② 실행 입력 (복붙용)**

아래 요청을 실행합니다.

```text
TodoManager에 "우선순위(priority)" 필드를 추가하는 기능을 구현해야 해.
코드 수정은 하지 말고, 영향 파일 분석과 단계별 구현 계획, 예상 테스트 케이스를 제시해줘.
```

**③ 결과 확인**

아래 표를 기준으로 결과를 점검합니다.

| 확인 항목 | 기대 결과 | 실패 신호 |
|----------|----------|----------|
| 출력 성격 | 계획 중심 응답 | 바로 코드 패치 제안 |
| 분석 내용 | 영향 파일, 구현 단계, 테스트 시나리오 포함 | 추상적 조언만 존재 |
| 제약 준수 | 코드 직접 수정 없이 계획만 제시 | "수정했습니다" 같은 실행 흐름 |
| 범위 통제 | 요청한 기능 범위 안에서 단계 제시 | 범위 밖 기능까지 확장 |

### 4단계: 품질 고정용 재실행

결과가 짧거나 모호하면 아래 재요청 문장을 사용합니다.

```text
코드 작성 없이 구현 계획만 다시 제시해줘.
아래 항목을 반드시 포함해줘:
1) 영향받는 파일
2) 단계별 구현 순서
3) 테스트 시나리오(정상/예외)
```

필요하면 제약을 한 번 더 고정합니다.

```text
파일 수정이나 코드 작성 제안은 제외하고, 계획 문서 형태로만 답변해줘.
```

### 5단계: 최종 동작 확인

아래 조건을 모두 만족하면 실습 완료입니다.

| 확인 항목 | 통과 기준 |
|----------|----------|
| 파일 생성 | `.github/agents/planner.agent.md` 존재 |
| 에이전트 선택 | Chat 드롭다운에서 `Planner` 표시 |
| 도구 제한 | `tools: [read, search]` 유지 |
| 출력 품질 | 분석 + 구현 단계 + 테스트 케이스 포함 |
| 제약 준수 | 코드 작성/수정 없이 계획만 제시 |

### 6단계: 자주 발생하는 문제와 해결

#### 1) Chat 드롭다운에 `Planner`가 안 보일 때

원인:

- 파일 경로가 잘못됨 (`.github/agents/` 아님)
- frontmatter `---` 누락
- `name` 필드 누락

해결:

1. 파일 경로 확인: `.github/agents/planner.agent.md`
2. frontmatter 시작/끝 `---` 확인
3. `name: "Planner"` 존재 확인
4. Chat 창을 닫았다가 다시 열기

#### 2) Planner가 코드를 직접 작성할 때

원인:

- `tools`에 `edit` 또는 `execute`가 포함됨
- 본문 제약에 "코드 작성 금지"가 약하게 쓰임

해결:

1. `tools: [read, search]`로 수정
2. 제약 문구를 아래처럼 명확히 고정

```text
요청 없이는 코드를 직접 작성하거나 파일을 수정하지 않는다.
```

3. 재실행 시 "코드 작성 없이 계획만"을 후속 메시지로 한 번 더 명시

#### 3) 결과가 너무 짧고 추상적일 때

원인:

- 입력 요청이 모호함
- 출력 형식 제약이 약함

해결:

- 요청에 필수 항목을 명시
  - 영향 파일
  - 단계별 구현 순서
  - 테스트 시나리오

#### 4) 분석 범위가 과하게 넓어질 때

원인:

- 기능 범위와 제외 범위를 명시하지 않음

해결:

요청에 범위를 같이 적습니다.

```text
이번 작업은 priority 필드 추가만 포함하고,
CLI 변경과 저장 방식 변경은 제외해줘.
```
