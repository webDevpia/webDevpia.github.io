# 하이브리드 검색

## 필수 라이브러리 설치

* Jupyter 노트북의 `%pip install` 매직 명령을 사용하여 이후 코드 실행에 필요한 파이썬 라이브러리들을 설치합니다.
* 설치되는 패키지들: `python-dotenv` (환경변수 설정 파일인 .env 로드용), `llama_index` (LLM 기반 정보 검색 및 인덱싱 라이브러리), `llama-index-vector-stores-qdrant` (LlamaIndex에서 **Qdrant** 벡터 데이터베이스를 연동하기 위한 모듈), `llama-index-retrievers-bm25` (LlamaIndex에서 **BM25** 알고리즘 기반 키워드 검색을 지원하는 모듈), `rank_bm25` (BM25 키워드 랭킹 알고리즘 구현 라이브러리).
* 하이브리드 검색 구현에 필요한 **벡터 임베딩 검색**과 **키워드 기반 검색** 도구들을 모두 설치하여 환경을 준비합니다.


```python
%pip install python-dotenv llama_index llama-index-vector-stores-qdrant llama-index-retrievers-bm25 rank_bm25
```

## 환경 변수 로드 및 API 키 설정

* `dotenv` 라이브러리의 `load_dotenv()` 함수를 이용해 `.env` 파일에 저장된 환경 변수들을 불러옵니다. 이 파일에는 API 키 등 외부 서비스 접속에 필요한 설정값들이 들어 있습니다.
* 불러온 환경 변수에서 **OpenAI API 키**, **Qdrant API 키**, **Qdrant 서버 URL** 값을 읽어와 각각 `OPENAI_API_KEY`, `QDRANT_API_KEY`, `QDRANT_URL` 파이썬 변수에 저장합니다. (`os.getenv('...')`를 통해 환경 변수 값을 가져옴)
* 이를 통해 OpenAI의 임베딩 서비스와 Qdrant 벡터 데이터베이스에 연결하기 위한 **인증 정보와 주소**가 준비됩니다. 이후 코드에서 이 변수들을 사용하여 외부 서비스에 안전하게 접속할 수 있습니다.


```python
from dotenv import load_dotenv
import os

load_dotenv()

OPENAI_API_KEY = os.getenv('OPENAI_API_KEY')
QDRANT_API_KEY = os.getenv('QDRANT_API_KEY')
QDRANT_URL = os.getenv('QDRANT_URL')
```

## Qdrant 클라이언트 초기화

* `qdrant_client` 패키지의 `QdrantClient` 클래스를 임포트하고, `QdrantClient(url=QDRANT_URL, api_key=QDRANT_API_KEY)`로 인스턴스를 생성합니다.
* 이 `client` 객체는 앞서 불러온 Qdrant 서버 URL과 API 키 정보를 사용하여 **Qdrant 벡터 데이터베이스 서버**에 연결합니다.
* 이제 이 클라이언트를 통해 Qdrant에 저장된 데이터에 접근하거나 질의할 수 있습니다. 예를 들어, 미리 저장된 **영화 데이터 임베딩** 컬렉션에서 벡터를 조회하거나 검색하는 작업을 수행할 준비가 된 것입니다.


```python
from qdrant_client import QdrantClient

client = QdrantClient(
    url=QDRANT_URL,
    api_key=QDRANT_API_KEY
)
```

## LlamaIndex 환경 설정 (LLM 및 임베딩 모델)

* LlamaIndex 라이브러리에서 OpenAI 기반 LLM과 임베딩 모델 클래스를 임포트하고 전역 설정을 지정합니다. (`OpenAI`, `OpenAIEmbedding` 클래스 등)
* `Settings.llm = OpenAI(temperature=0, model='gpt-4o-mini')`: LlamaIndex가 사용할 언어 모델(LLM)을 OpenAI의 GPT 계열로 설정합니다. 온도(temperature)를 0으로 하여 출력의 **일관성과 재현성을 높였고**, 모델로 `'gpt-4o-mini'`를 지정하여 실험에 적절한 GPT-4 모델 변형을 사용합니다. *(이 설정은 주로 LLM을 활용한 질문답변 생성 등에 필요하며, 본 검색 실습에서는 주로 임베딩 모델이 중요합니다.)*
* `Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")`: **텍스트 임베딩 모델**로 OpenAI의 `"text-embedding-3-small"` 모델을 사용하도록 설정합니다. 이를 통해 문서 내용과 질의를 벡터로 변환하여 의미 기반 비교가 가능해집니다.
* `Settings.chunk_size = 500` 및 `Settings.chunk_overlap = 50`: 문서가 길 경우 자동으로 청크(덩어리)로 분할하여 처리하는 크기와 중복 길이를 설정합니다. (한 번에 500자씩 처리하고, 청크 간 50자의 겹침을 둠으로써 문맥이 잘리지 않도록 함)
* 이 설정 단계를 통해 LlamaIndex가 OpenAI의 API를 활용할 준비를 마칩니다. 즉, **질의나 문서를 벡터화할 임베딩 모델**과 (필요 시) **응답 생성에 사용할 LLM**이 미리 구성됩니다.


```python
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.core import Settings

Settings.llm= OpenAI(temperature=0, model='gpt-4o-mini')
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.chunk_size = 500
Settings.chunk_overlap = 50
```

## Qdrant 벡터스토어 및 LlamaIndex 인덱스 연결

* `QdrantVectorStore` 클래스를 이용해 `vector_store = QdrantVectorStore(client=client, collection_name="movies")`로 벡터 스토어 객체를 생성합니다. 여기서 `client`는 앞서 만든 Qdrant 클라이언트이고, `"movies"`는 사용할 컬렉션 이름입니다. 이로써 Qdrant 데이터베이스 내 **"movies" 컬렉션에 저장된 영화 벡터들**을 가리키는 객체가 만들어집니다.
* `storage_context = StorageContext.from_defaults(vector_store=vector_store)`: LlamaIndex에서 기본 저장소 대신 방금 만든 Qdrant 벡터스토어를 사용하도록 **스토리지 컨텍스트**를 생성합니다.
* `index = VectorStoreIndex.from_vector_store(vector_store=vector_store, storage_context=storage_context)`: 주어진 벡터스토어로부터 LlamaIndex 인덱스 객체를 생성합니다. 이 인덱스는 외부 Qdrant에 저장된 벡터 임베딩들을 가리키며, LlamaIndex의 검색 인터페이스와 통합됩니다.
* 정리하면, 외부 벡터 DB(Qdrant)에 이미 **임베딩된 채 저장된 영화 데이터**를 LlamaIndex 라이브러리를 통해 활용할 수 있게 연결한 것입니다. 이제 이 `index` 객체를 통해 의미 기반의 벡터 검색을 수행할 수 있습니다.


```python
from llama_index.vector_stores.qdrant import QdrantVectorStore
from llama_index.core import VectorStoreIndex, StorageContext

# Qdrant 벡터스토어 객체 생성
vector_store = QdrantVectorStore(
    client=client,
    collection_name="movies"
)

# StorageContext 생성: 커스텀 벡터스토어를 연결
storage_context = StorageContext.from_defaults(vector_store=vector_store)

# 벡터 인덱스 조회
index = VectorStoreIndex.from_vector_store(vector_store=vector_store, storage_context=storage_context)
```

## 벡터 DB에서 문서 데이터 가져오기

* `client.scroll(collection_name="movies", limit=100)`을 호출하여 Qdrant의 "movies" 컬렉션에서 문서 레코드를 불러옵니다. `scroll` 메서드는 일종의 페이지네이션 기능으로, 최대 100개의 결과를 한 번에 가져오며 다음 페이지 커서도 함께 반환하지만 여기서는 처음 100개만 사용합니다.
* 이 호출 결과로 얻은 `records` 리스트에는 각 영화에 대한 저장된 벡터 포인트와 \*\*payload(부가 데이터)\*\*가 포함되어 있습니다. 즉, 벡터 DB에 입력했던 영화 정보들이 `records`에 담긴 것입니다.
* 이 단계는 벡터 검색과 별개로, **원본 문서 데이터를 확인 및 활용**하기 위한 것입니다. Qdrant에 어떤 영화 문서들이 들어있는지 확인하고, 다음 단계에서 BM25 키워드 검색에 사용할 **텍스트 데이터**를 미리 확보하려는 목적입니다.


```python
# 문서 데이터 조회
records, _ = client.scroll(collection_name="movies", limit=100)
records
```

## 문서의 텍스트 내용 추출

* Qdrant에서 불러온 `records`에서 각 문서의 텍스트 부분만 뽑아내어 `documents` 리스트를 생성합니다. 코드에서 리스트 내포 구문을 사용해 `record.payload.get('text')` 값을 모으고, 만약 해당 키가 없는 레코드는 건너뜁니다.
* 각 `record.payload["text"]`에는 영화의 **제목과 줄거리 등이 하나의 문자열**로 들어있습니다. 예를 들어 `'반도|좀비 바이러스로 황폐해진 한반도에서…'`처럼 제목과 설명을 '|'로 연결한 텍스트일 수 있습니다. 이렇게 추출된 문자열들만 모아 `documents`에 저장합니다.
* 그 결과 `documents`는 영화 소개 텍스트들로 이루어진 리스트가 됩니다. 이제 \*\*키워드 기반 검색(BM25)\*\*에 입력으로 사용할 **순수 텍스트 문서들**이 준비된 것입니다. (벡터 검색은 이미 벡터화된 데이터를 사용하지만, BM25는 텍스트 자체를 필요로 하므로 별도로 추출한 것입니다.)


```python
# 텍스트 추출
documents = [record.payload.get('text') for record in records if record.payload.get('text')]
documents
```

## Dense 및 BM25 검색기 초기화

* `dense_retriever = index.as_retriever(similarity_top_k=3)`: 앞서 만든 벡터 인덱스에서 **Dense Retriever** 객체를 얻어옵니다. `similarity_top_k=3`으로 설정하여 각 질의마다 **상위 3개의 유사한 문서**를 반환하도록 합니다. (벡터 임베딩 공간에서 질의와 가장 가까운 3개의 결과)
* `nodes = [TextNode(text=doc) for doc in documents]`: 추출한 문서 텍스트(`documents` 리스트)를 LlamaIndex의 `TextNode` 객체로 변환합니다. 각 영화 소개 글 문자열을 TextNode로 감싸 `nodes` 리스트를 만들었는데, 이는 LlamaIndex 내부에서 문서들을 다루기 위한 표준 구조입니다.
* `bm25_retriever = BM25Retriever.from_defaults(nodes=nodes, similarity_top_k=3)`: **BM25 Retriever** 객체를 생성합니다. 앞서 만든 `nodes` (영화 문서들)을 입력으로 주어 내부적으로 BM25 인덱스를 구축하고, 마찬가지로 상위 3개 결과를 반환하도록 설정합니다. BM25는 전통적인 **키워드 기반 문서 검색 알고리즘**으로, 단어 빈도와 역문헌 빈도를 고려해 문서의 점수를 매깁니다.
* 이제 동일한 영화 데이터에 대해 두 가지 검색 도구가 준비되었습니다: 하나는 **의미 벡터 유사도에 기반한 Dense Retriever**이고, 다른 하나는 **단어 매칭 점수에 기반한 BM25 Retriever**입니다. 두 검색기 모두 `top_k=3`으로 결과를 제한하여, 이후 각 방법의 상위 결과를 손쉽게 비교할 수 있게 합니다.
* Dense 검색은 **질문의 의미**를 파악해 관련 문서를 찾는 반면, BM25 검색은 **질문에 포함된 키워드**로 직접적으로 일치하는 문서를 찾습니다. 하이브리드 검색을 위해 이 둘을 모두 준비하는 단계입니다.


```python
from llama_index.core.schema import TextNode
from llama_index.retrievers.bm25 import BM25Retriever

dense_retriever = index.as_retriever(similarity_top_k=3)

# documents 리스트를 TextNode 객체로 변환
nodes = [TextNode(text=doc) for doc in documents]

# BM25Retriever 생성 시 nodes, 전달 메모리상에 저장해놓고 검색
bm25_retriever = BM25Retriever.from_defaults(
    nodes=nodes,
    similarity_top_k=3
)
```

## Dense 임베딩 검색 테스트 (질의: "좀비 영화")

* `dense_retriever.retrieve("좀비 영화")`를 호출하여 **의미 기반 벡터 검색**으로 질의 `"좀비 영화"`에 대한 검색을 수행합니다.
* `"좀비 영화"`라는 질의 문장도 앞서 설정된 임베딩 모델로 벡터화되며, 벡터 공간에서 이 질의와 **가장 가까운 영화들의 임베딩**을 찾아 상위 3개의 결과를 반환합니다.
* 예상되는 결과는 **좀비를 소재로 한 영화들**입니다. 예를 들어, 질의에 "좀비"라는 키워드 자체가 포함되어 있으므로 좀비 바이러스가 등장하는 \*\*<반도>\*\*나 **<부산행>** 같은 영화들이 높은 유사도로 검색될 가능성이 큽니다. (실제로 벡터 DB에 해당 영화들의 임베딩이 저장되어 있다면 상위 결과에 나타날 것입니다.)
* Dense Retriever의 특징은 **단어의 정확한 일치 여부와 무관하게** 문장의 의미가 비슷한지를 기준으로 결과를 찾는 것입니다. 따라서 질의와 내용이 의미적으로 연관된 영화라면, 설명에 "좀비"라는 단어가 있든 없든 유사한 주제의 영화를 식별해 낼 수 있습니다.


```python
dense_retriever.retrieve("좀비 영화")
```

## BM25 키워드 검색 테스트

* `bm25_retriever.retrieve("좀비 영화")`를 호출하여 **BM25 알고리즘**으로 동일한 질의를 검색합니다.
* BM25 검색기는 문서들의 텍스트에서 \*\*"좀비"\*\*와 **"영화"** 단어가 나타나는 빈도와 분포를 토대로 관련도를 계산합니다. 따라서 "좀비"라는 단어를 **명시적으로 포함한 영화 소개 글**이 높은 점수를 받아 상위에 랭크될 것입니다.
* 이 경우 **<반도>**, **<부산행>** 등 설명에 "좀비" 키워드가 들어간 영화들이 결과에 등장할 것으로 예상됩니다. Dense 검색과 대체로 비슷한 영화들이 반환되겠지만, **단어 매칭 횟수**에 따라 순위가 약간 다를 수 있습니다 (예를 들어, 좀비라는 단어를 많이 언급한 영화 설명이 1순위가 될 수 있습니다).
* BM25 방식은 **질의어와 정확히 일치하는 단어**에 의존하기 때문에, 만약 어떤 좀비 영화의 줄거리 설명에 "좀비"라는 단어가 전혀 없다면 (예: "언데드 괴물"처럼 다른 표현만 있을 경우) 그 영화는 검색 결과에 나타나지 않을 수 있습니다. 이는 의미를 파악하는 Dense 검색과 달리 BM25 검색의 한계점으로, 이후 하이브리드 검색에서 이러한 약점을 보완하게 됩니다.


```python
bm25_retriever.retrieve("좀비 영화")
```

## Hybrid Retriever 생성 (Dense + BM25 결합)

* `QueryFusionRetriever` 클래스를 이용하여 `hybrid_retriever = QueryFusionRetriever([bm25_retriever, dense_retriever], ...)` 형태로 **하이브리드 검색기**를 설정합니다. 두 개의 개별 검색기 객체를 리스트로 전달함으로써, Hybrid 검색기는 질의 시 이 둘을 **동시에 활용**하여 결과를 종합하게 됩니다.
* 매개변수 `similarity_top_k=3`으로 최종 반환할 결과 개수를 3개로 지정하였고, `num_queries=1`로 각 검색기당 한 번의 질의를 수행하게 설정했습니다. (`use_async=False`는 두 검색을 순차적으로 실행하도록 설정한 것이며, `verbose=True`는 검색 과정의 자세한 로그를 출력해줍니다.)
* QueryFusionRetriever는 내부적으로 **여러 검색기의 결과를 융합**하여 최종 랭킹을 산출합니다. 예를 들어, Dense와 BM25가 각자 찾아낸 문서 리스트를 받아 점수를 조정하거나 합산하여 가장 관련성이 높은 문서들을 선정합니다. (동일 문서를 두 검색기가 찾은 경우 가중치가 높아질 수 있습니다.)
* 이렇게 함으로써 **키워드 매칭의 정확도**와 **의미 검색의 포괄성**을 모두 활용하는 하이브리드 검색이 가능해집니다. 사용자 입장에서는 `hybrid_retriever.retrieve(질의)` 한 번으로 **두 가지 방법의 장점을 결합한 검색 결과**를 얻을 수 있습니다.
* 이제 실제 질의를 통해 하이브리드 검색이 어떻게 동작하는지 확인할 준비가 되었으며, 이어지는 실습에서 개별 Dense/BM25 결과와 하이브리드 결과를 비교해 볼 것입니다.


```python
from llama_index.core.retrievers import QueryFusionRetriever

# QueryFusionRetriever를 이용한 Hybrid 검색 (BM25 + Dense Embedding)
hybrid_retriever = QueryFusionRetriever(
    [bm25_retriever, dense_retriever],
    similarity_top_k=3,  # 최종 결과로 몇 개를 반환할지
    num_queries=1,       # 각 Retriever당 수행할 질의 개수
    use_async=False,      # 병렬처리 비활성화
    verbose=True         # 자세한 로깅 출력
)
```

## 하이브리드 검색 결과 확인 – 질의 1 ("좀비 영화")

* **테스트 질의 1:** `query1 = "좀비 영화"`로 설정하고, `results1 = hybrid_retriever.retrieve(query1)`을 실행하여 하이브리드 검색을 수행합니다. 사용자가 "좀비가 나오는 영화"를 찾는 시나리오입니다.
* 검색이 완료되면 `print`문을 통해 질의 내용과 결과 문서들의 제목을 출력합니다. `node.text.split('|')[0]` 구문은 각 결과 노드의 텍스트에서 제목 부분만 잘라오는 역할을 합니다. (데이터가 "제목|줄거리" 형태의 문자열이므로 '|' 앞의 제목을 추출)
* **예상 출력:** `질의: 좀비 영화` 라는 문구 아래에 `결과 문서 제목들: ['반도', '부산행', ...]` 처럼 **좀비 테마 영화 제목들**의 리스트가 화면에 표시됩니다. 실제로 Dense 검색과 BM25 검색 모두 좀비 키워드에 잘 대응하므로, 하이브리드 결과 역시 **좀비 소재의 대표 영화들**이 상위에 나올 것입니다.
* 이 결과는 앞서 개별 검색에서 얻은 결과와 큰 차이가 없을 수 있습니다. 두 방법 모두 "좀비"라는 명확한 키워드를 다뤘기 때문입니다. 하이브리드 검색은 이러한 경우 각 방법의 공통 결과를 보여주거나 약간의 순위 변동만 있을 것으로 예상됩니다. (예: BM25가 1위로 찾은 영화와 Dense가 1위로 찾은 영화가 다를 경우, 둘 다 상위에 포함되어 순위가 조정됨)
* **의미:** 첫 번째 테스트를 통해 하이브리드 검색기가 정상적으로 동작함을 확인하고, **키워드 매칭이 쉬운 질의**에서는 Dense와 BM25 양쪽 모두 무난하게 관련 결과를 찾을 수 있음을 보여줍니다.


```python
# 테스트 질의 1: "좀비 영화"
query1 = "좀비 영화"  # (사용자가 좀비가 등장하는 영화를 찾는 질의)
results1 = hybrid_retriever.retrieve(query1)
print(f"질의: {query1}")
print("결과 문서 제목들:", [node.text.split('|')[0].strip() for node in results1])
```

## Dense 임베딩 검색 테스트 – 질의 2 ("멜로 영화")

* **테스트 질의 2:** `dense_retriever.retrieve("멜로 영화")`를 호출하여 두 번째 예시 질의에 대해 **벡터 기반 검색**을 수행합니다. "멜로 영화"는 멜로(로맨스) 장르의 영화를 찾는 질문입니다.
* 질의어 "멜로 영화"도 임베딩 모델로 벡터화되어, 로맨틱한 내용이나 멜로 드라마적 요소가 있는 영화들의 벡터와 비교됩니다. Dense 검색은 **줄거리의 분위기나 주제**를 파악하여 관련성이 높은 영화를 선정하므로, 설명에 꼭 "멜로"라는 단어가 없어도 **사랑 이야기나 로맨스 테마의 영화들**을 상위 결과로 찾을 수 있습니다.
* 예를 들어, 어떤 영화 줄거리에 "연인"이나 "사랑" 같은 단어들이 있고 전반적인 내용이 로맨틱하다면, 해당 영화가 임베딩 공간에서 "멜로 영화" 질의와 가깝다고 판단되어 결과에 포함될 수 있습니다.
* 이 셀은 Dense Retriever의 **개념 파악 능력**을 보여주는 예시입니다. 사용자의 의도가 "멜로/로맨틱 영화"임을 임베딩이 이해하고, 꼭 같은 단어가 없더라도 유사한 감성의 영화를 찾아줄 수 있다는 점을 관찰합니다.


```python
dense_retriever.retrieve("멜로 영화")
```

## BM25 키워드 검색 테스트 – 질의 2 ("멜로 영화")

* `bm25_retriever.retrieve("멜로 영화")`를 실행하여 동일한 질의에 대해 **BM25 키워드 검색**을 수행합니다.
* BM25는 문서 텍스트 중 \*\*"멜로"\*\*라는 용어의 등장 여부와 빈도를 중요하게 평가합니다. 따라서 영화 소개 글에 "멜로" 혹은 유사 단어가 명시적으로 언급된 경우에만 해당 영화가 높은 점수를 받아 결과에 나타날 것입니다.
* 실제로 멜로/로맨스 장르의 영화 소개에는 "감성 멜로 드라마", "로맨스" 등의 단어가 포함되기도 하므로, 그런 단어가 있는 영화가 BM25 결과의 상위에 올 수 있습니다. 하지만 **만약 줄거리 텍스트에 '멜로'라는 단어가 전혀 없으면** BM25는 그 영화를 찾아내지 못합니다. (예: 내용은 매우 로맨틱하지만 "사랑"이나 다른 표현만 있고 "멜로"라는 단어가 없다면 키워드 매칭이 이루어지지 않음)
* 이 단계에서는 키워드 검색의 **한계와 특성**을 볼 수 있습니다. Dense 검색과 비교하여 BM25는 **정확한 용어 일치**에 의존하므로, 사용자가 선택한 단어에 따라 결과 범위가 제한될 수 있습니다. 로맨스 장르 영화 중에서도 표현에 따라 검색 결과에서 누락되는 사례가 있을 수 있음을 염두에 둡니다.


```python
bm25_retriever.retrieve("멜로 영화")
```

## 하이브리드 검색 결과 확인 – 질의 2 ("멜로 영화")

* 두 번째 테스트 질의 `"멜로 영화"`에 대해 `results2 = hybrid_retriever.retrieve(query2)`를 수행하고, 앞과 같이 질의 내용과 결과 문서들의 **제목 리스트**를 출력합니다.
* 출력된 **결과 영화 제목들**을 통해 하이브리드 검색의 효과를 확인할 수 있습니다. 멜로/로맨스 장르의 대표적인 영화들이 리스트에 포함될 것이며, 키워드 매칭과 의미 검색이 **보완적인 결과**를 제공함을 알 수 있습니다.

  * 예를 들어, BM25가 찾아낸 **직접 "멜로" 언급이 있는 영화**와 Dense 검색이 추가로 포착한 **로맨틱한 내용의 영화**들이 모두 결과에 섞여 나올 수 있습니다. 하이브리드 검색은 이들을 통합된 순위로 제공하기 때문에, 사용자는 로맨스 장르 영화에 대해 더 풍부한 추천을 얻게 됩니다.
* 개별 방법과 비교하면, BM25 단독으로는 놓쳤을 법한 영화(예: 설명에 "멜로"라는 단어가 없었던 영화)가 **Dense 검색을 통해 보완**되고, 반대로 Dense 검색이 잡아낸 결과 중 다소 엉뚱한 것이 있었다면 BM25가 **정확도 측면에서 거르는 효과**를 줄 수도 있습니다.
* 이로써 하이브리드 검색이 단순 키워드 검색보다 포괄적이고, 순수 벡터 검색보다 정확도 면에서 신뢰할 만한 절충안을 제공한다는 점을 보여줍니다.


```python
# 테스트 질의 2: "멜로 영화"
query2 = "멜로 영화"  # (멜로/로맨틱한 영화를 찾는 질의)
results2 = hybrid_retriever.retrieve(query2)
print(f"\n질의: {query2}")
print("결과 문서 제목들:", [node.text.split('|')[0].strip() for node in results2])
```
