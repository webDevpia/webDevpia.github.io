## ⭐ 핵심: 1. 왜 테스트가 필요한가? [↑](#toc)

### "내 코드가 맞는지 기계가 확인해준다"

지금까지 코드를 작성하고 `node 파일명.js`로 실행하면서 눈으로 확인했습니다.  
점점 코드가 복잡해지면 이 방법엔 한계가 생깁니다.

---

### 시험지 비유

학교에서 수학 문제를 풀면 선생님이 채점해줍니다.

```
문제: 3 + 5 = ?
내 답: 8
채점: ✓ 맞음
```

테스트는 바로 이겁니다.  
**"내 함수에게 문제를 주고, 기계가 자동으로 채점해준다."**

```javascript
// 내 코드 (답안)
function add(a, b) {
  return a + b;
}

// 테스트 (시험지) — 아래 expect 줄은 Vitest 환경에서 실행됩니다 (개념 설명용 예시)
expect(add(3, 5)).toBe(8);   // 3 + 5가 8이어야 한다
expect(add(-1, 1)).toBe(0);  // -1 + 1이 0이어야 한다
```

---

### 왜 특히 AI 시대에 중요한가?

앞으로 배울 Phase 2, Phase 3에서는 **AI(Copilot)가 코드를 만들어**줍니다.

AI는 "거의 맞는" 코드를 빠르게 만들지만 항상 완벽하지 않습니다.  
**테스트는 AI가 만든 코드가 실제로 올바른지 검증하는 유일한 도구**입니다.

```
AI가 코드를 생성한다
  ↓
테스트를 실행한다
  ↓
초록색: 코드가 정확함  ←→  빨간색: AI가 실수했음, 수정 필요
```

이 장에서 테스트 작성 방법을 배우는 이유가 바로 이것입니다.

---

### 테스트 없이는

```
새 기능 추가 → 다른 기능이 망가졌나? 모든 곳을 손으로 확인
버그 수정    → 수정하다가 다른 버그 생겼나? 또 손으로 확인
AI 코드 사용 → 이게 맞는 코드인가? 눈으로만 확인 → 배포 후 버그 발견...
```

### 테스트 있으면

```
새 기능 추가 → npm test → 빨간 줄 없으면 OK
버그 수정    → npm test → 이전에 통과하던 테스트도 모두 통과
AI 코드 사용 → npm test → 즉시 검증 완료
```

## ⭐ 핵심: 2. 프로젝트 초기화 — npm과 package.json [↑](#toc)

강사 시연을 보면서 따라하세요.

### 새 프로젝트 폴더 만들기

VS Code의 터미널을 열고 (Ctrl+\` 또는 Cmd+\`):

```bash
mkdir js-practice
cd js-practice
code .
```

---

### npm init — 프로젝트 초기화

```bash
npm init -y
```

`-y` 플래그는 모든 질문에 "yes"로 자동 답변합니다.

실행 후 `package.json` 파일이 생깁니다:

```json
{
  "name": "js-practice",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
```

`package.json`은 프로젝트의 **신분증**입니다.  
프로젝트 이름, 버전, 사용하는 라이브러리 목록이 여기에 저장됩니다.

---

### Vitest 설치

```bash
npm install -D vitest
```

- `install`: 라이브러리를 설치
- `-D`: 개발 환경에서만 사용 (배포 시 포함 안 됨)
- `vitest`: 설치할 라이브러리 이름

설치 후 `package.json`이 바뀝니다:

```json
{
  "devDependencies": {
    "vitest": "^3.0.0"
  }
}
```

`node_modules` 폴더도 생겼을 겁니다. 이 안에 vitest 코드가 있습니다.  
(이 폴더는 `.gitignore`에 넣어 git에 올리지 않습니다)

---

### package.json 수정 — test 스크립트와 ESM 설정

`package.json`을 열어서 아래처럼 수정하세요:

```json
{
  "name": "js-practice",
  "version": "1.0.0",
  "description": "",
  "type": "module",
  "scripts": {
    "test": "vitest"
  },
  "devDependencies": {
    "vitest": "^3.0.0"
  }
}
```

두 가지 변경사항:
1. `"type": "module"` 추가 — `import`/`export` 문법을 사용하기 위해 필요
2. `"test": "vitest"` — `npm test` 명령어가 vitest를 실행하도록 설정

> **`require` vs `import` — 왜 바뀌나요?**
> 이전 장에서 `const prompt = require('prompt-sync')();` 을 썼습니다. `require`는 Node.js의 **예전 방식(CommonJS)**입니다. `import`/`export`는 JavaScript의 **현재 표준(ESM)**입니다. 이 프로젝트부터는 현재 표준인 ESM을 사용합니다.
> 
> 이전 장에서 만든 `prompt-sync` 파일들(`javascript-study` 폴더)은 영향 없습니다 — 이 프로젝트(`js-practice`)는 별도 폴더입니다.

## ⭐ 핵심: 3. 첫 번째 테스트 — 초록색 체크 보기 [↑](#toc)

### 폴더 구조 만들기

```
js-practice/
├── src/
│   └── math.js
├── tests/
│   └── math.test.js
└── package.json
```

VS Code에서 `src` 폴더와 `tests` 폴더를 만드세요.

---

### src/math.js — 함수 작성

```javascript
// 더하기
export function add(a, b) {
  return a + b;
}

// 곱하기
export function multiply(a, b) {
  return a * b;
}
```

`export`는 이 함수를 다른 파일에서 사용할 수 있게 "내보내는" 것입니다.

---

### tests/math.test.js — 테스트 작성

```javascript
import { describe, it, expect } from 'vitest';
import { add, multiply } from '../src/math.js';

describe('더하기 함수', () => {
  it('3 + 5는 8이다', () => {
    expect(add(3, 5)).toBe(8);
  });

  it('음수도 더할 수 있다', () => {
    expect(add(-1, 1)).toBe(0);
  });

  it('소수도 더할 수 있다', () => {
    expect(add(0.1, 0.2)).toBeCloseTo(0.3); // 소수 비교는 toBeCloseTo
  });
});

describe('곱하기 함수', () => {
  it('4 × 5는 20이다', () => {
    expect(multiply(4, 5)).toBe(20);
  });

  it('0을 곱하면 0이다', () => {
    expect(multiply(100, 0)).toBe(0);
  });
});
```

---

### 테스트 코드 읽는 법

```javascript
describe('더하기 함수', () => {    // 테스트 그룹 이름
  it('3 + 5는 8이다', () => {     // 테스트 하나의 이름 (영어로 하면 더 좋음)
    expect(add(3, 5)).toBe(8);   // "add(3, 5)의 결과가 8이기를 기대한다"
  });
});
```

- `describe`: 관련 테스트를 그룹으로 묶음
- `it`: 하나의 테스트 케이스. "it should ..." 형태로 읽힘
- `expect`: "이것의 결과를 확인한다"
- `.toBe()`: "이 값이어야 한다"

## 4. 테스트 더 작성하기 — 빨강에서 초록으로 [↑](#toc)

### 테스트 케이스 생각하기

새 함수를 테스트할 때, 어떤 입력을 넣어봐야 할까요? 코드를 작성하기 **전에** 두 가지를 먼저 생각합니다:

1. **정상 입력** (happy path): 기대한 대로 동작하는가?
2. **경계값** (edge case): 이상한 입력이 들어와도 안전한가?

예를 들어, `greet(name)` 함수를 테스트한다면:

| 종류 | 입력 | 기대 결과 | 왜 테스트? |
|------|------|----------|-----------|
| 정상 입력 | `"홍길동"` | `"안녕하세요, 홍길동님!"` | 기본 동작 확인 |
| 경계값 | `""` (빈 문자열) | `"안녕하세요, 님!"` | 빈 입력에도 에러 없이 동작하는지 |

> 💡 테스트를 작성할 때는 항상 이 두 가지를 먼저 떠올리세요.
> "잘 되는 경우"와 "이상한 입력이 들어온 경우"를 모두 확인해야 합니다.
> 11장에서 AI가 생성한 테스트를 검토할 때 이 기준이 필요합니다.

이제 이 생각을 코드로 옮겨봅시다.

### greet 함수 테스트 추가

`src/math.js`에 인사 함수를 추가합니다:

```javascript
export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}

export function greet(name) {
  return `안녕하세요, ${name}님!`;
}
```

`tests/math.test.js`에 테스트를 추가합니다:

```javascript
import { describe, it, expect } from 'vitest';
import { add, multiply, greet } from '../src/math.js';

// ... 기존 테스트 ...

describe('인사 함수', () => {
  it('이름을 받아 인사 문자열을 반환한다', () => {
    expect(greet("홍길동")).toBe("안녕하세요, 홍길동님!");
  });

  it('이름이 빈 문자열이어도 동작한다', () => {
    expect(greet("")).toBe("안녕하세요, 님!");
  });
});
```

저장 후 `npm test`를 실행하세요. 초록색 ✓이 7개 나오면 성공입니다.

```
 ✓ tests/math.test.js (7)
   ✓ 더하기 함수 (3)
   ✓ 곱하기 함수 (2)
   ✓ 인사 함수 (2)

 Test Files  1 passed (1)
      Tests  7 passed (7)
```

> **문제가 생겼나요?** `greet is not defined` 또는 `does not provide an export named 'greet'` 에러가 나왔다면:  
> 1. `src/math.js`에 `export function greet(name)` 이 있는지 확인하세요 (저장했나요?)  
> 2. `tests/math.test.js`의 import 줄에 `greet`가 포함되어 있는지 확인하세요  

---

### 실패하는 테스트 만들어보기 (Red → Green)

일부러 틀린 함수를 만들어서 테스트가 실패하는 걸 봅니다.

`src/math.js`에 버그 있는 함수 추가:

```javascript
export function subtract(a, b) {
  return a + b;  // 버그! + 대신 - 써야 함
}
```

`tests/math.test.js`의 import 줄에 `subtract`를 추가하고, 테스트를 추가합니다:

```javascript
import { add, multiply, greet, subtract } from '../src/math.js';

// ... 기존 테스트 ...

describe('빼기 함수', () => {
  it('10 - 3은 7이다', () => {
    expect(subtract(10, 3)).toBe(7);
  });
});
```

`npm test` 실행:

```
 ✗ tests/math.test.js (8)
   ✓ 더하기 함수 (3)
   ✓ 곱하기 함수 (2)
   ✓ 인사 함수 (2)
   ✗ 빼기 함수 (1)
     ✗ 10 - 3은 7이다
       AssertionError: expected 13 to be 7

 Test Files  1 failed (1)
      Tests  1 failed | 7 passed (8)
```

빨간 `✗`와 에러 메시지가 보입니다.  
**"expected 13 to be 7"** — 13이 나왔는데 7이어야 한다는 뜻입니다.

이제 버그를 수정합니다:

```javascript
export function subtract(a, b) {
  return a - b;  // 수정!
}
```

다시 `npm test` → 초록색 ✓ 복귀.

이것이 **Red → Green** 사이클의 기본입니다.  
Phase 3에서 배울 TDD(테스트 주도 개발)의 핵심입니다.

### 자주 쓰는 매처(Matcher)

```javascript
expect(value).toBe(expected)          // 정확히 같음 (===)
expect(value).toEqual(expected)       // 깊은 비교 (객체/배열)
expect(value).toBeCloseTo(expected)   // 소수점 근사값
expect(value).toBeTruthy()            // truthy 값
expect(value).toBeFalsy()             // falsy 값
expect(value).toBeNull()              // null인지
expect(value).toBeUndefined()         // undefined인지
expect(value).toContain(item)         // 배열에 포함 여부
expect(fn).toThrow()                  // 에러를 던지는지
```

```javascript
// 객체/배열 비교는 toEqual 사용
expect({ name: "홍길동" }).toEqual({ name: "홍길동" }); // ✓
expect([1, 2, 3]).toEqual([1, 2, 3]);                   // ✓
```

### toThrow — 에러를 던지는지 확인하기

매처 표에서 `toThrow`는 사용법이 특별합니다. 함수 호출 결과가 아니라, **함수 자체를 화살표 함수로 감싸서** 전달해야 합니다.

```javascript
function divide(a, b) {
  if (b === 0) throw new Error('0으로 나눌 수 없습니다');
  return a / b;
}

// ✅ 올바른 사용법 — 함수를 감싸서 전달
expect(() => divide(10, 0)).toThrow('0으로 나눌 수 없습니다');

// ❌ 잘못된 사용법 — 함수를 바로 호출하면 에러가 그냥 터짐
// expect(divide(10, 0)).toThrow();  // 테스트 자체가 에러!
```

> 15장(TDD)에서 에러 처리 테스트를 자주 작성합니다. 이 패턴을 기억해두세요.

### Copilot이 뭔가요?

GitHub Copilot은 **코드 작성을 도와주는 AI 어시스턴트**입니다.  
코드를 입력하면 회색 글씨로 다음 코드를 제안합니다.

```
function calculateBMI(weight, height) {
  // ← 여기서 멈추면 Copilot이 나머지를 제안함
  return weight / (height * height);  ← 이게 회색으로 나타남
}
```

`Tab` 키를 누르면 제안을 수락, `Esc`를 누르면 거부합니다.

## 6. Copilot과 함께하는 첫 실습 [↑](#toc)

### 규칙: 참고만 하고 직접 입력

이 실습의 핵심은 **Copilot 제안을 보되, 손으로 직접 타이핑하기**입니다.

---

### 실습 1 — Copilot 제안 관찰하기

`src/strings.js` 파일을 만들고 아래 함수 이름만 입력해보세요:

```javascript
// 문자열 첫 글자를 대문자로 만드는 함수
export function capitalize(
```

Copilot이 어떤 제안을 하는지 봅니다. 그런 다음 **직접 타이핑**합니다:

```javascript
export function capitalize(str) {
  if (str.length === 0) return "";
  return str[0].toUpperCase() + str.slice(1);
}

// 단어들의 첫 글자를 모두 대문자로
export function titleCase(str) {
  return str.split(" ").map(word => capitalize(word)).join(" ");
}
```

---

### 실습 2 — 테스트 작성하기

`tests/strings.test.js` 파일을 만들고 직접 작성하세요:

```javascript
import { describe, it, expect } from 'vitest';
import { capitalize, titleCase } from '../src/strings.js';

describe('capitalize 함수', () => {
  it('첫 글자를 대문자로 만든다', () => {
    expect(capitalize("hello")).toBe("Hello");
  });

  it('빈 문자열은 그대로 반환한다', () => {
    expect(capitalize("")).toBe("");
  });

  it('이미 대문자면 그대로다', () => {
    expect(capitalize("Hello")).toBe("Hello");
  });
});

describe('titleCase 함수', () => {
  it('모든 단어의 첫 글자를 대문자로 만든다', () => {
    expect(titleCase("hello world")).toBe("Hello World");
  });
});
```

`npm test` 실행해서 모두 통과하는지 확인하세요.

## 8. 정리 [↑](#toc)

### 이 장에서 배운 것

| 개념 | 핵심 |
|------|------|
| 테스트의 필요성 | AI 코드를 검증하는 유일한 도구 |
| `npm init -y` | 프로젝트 초기화, package.json 생성 |

| `npm test` | 모든 테스트 실행 |
| GitHub Copilot | VS Code에 기본 내장된 AI 코드 어시스턴트 |
| Tab / Esc | Copilot 제안 수락 / 거부 |
| Copilot Chat | 한국어로 코드 질문 가능 |

### 테스트 작성 원칙

```
1. Arrange — 준비 (입력값 세팅)
2. Act     — 행동 (함수 호출)
3. Assert  — 검증 (결과 확인)
```

```javascript
it('더하기 함수는 두 수를 더한다', () => {
  // Arrange
  const a = 3;
  const b = 5;

  // Act
  const result = add(a, b);

  // Assert
  expect(result).toBe(8);
});
```
