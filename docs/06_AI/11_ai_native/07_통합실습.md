## 2️⃣ 프로젝트: "상태별 필터링" 기능 추가 [↑](#toc)

> 지금까지 만든 도구들을 총동원해서 첫 번째 실전 기능을 구현합니다.

이번 통합 실습에서는 Todo Manager에 **상태별 필터링** 기능을 추가합니다.

**기능 목표:**
- `pending` 상태인 할일만 조회할 수 있어야 합니다.
- `done` 상태인 할일만 조회할 수 있어야 합니다.
- 기존 기능(추가, 전체 조회, 완료 처리, 삭제)은 깨지지 않아야 합니다.

### 이번 실습에서 쓰는 도구들

| 단계 | 사용하는 것 | 해당 장 |
|------|-----------|--------|
| 계획 수립 | Planner 에이전트 | 05장 |
| 테스트 작성 | TDD Red 에이전트 | 06장 |
| 구현 | TDD Green 에이전트 | 06장 |
| 리팩터링 | TDD Refactor 에이전트 | 06장 |
| 코딩 규칙 적용 | Custom Instructions | 03장 |
| 테스트 규칙 적용 | testing.instructions.md | 03장 |

## 3️⃣ Step 1: Planner로 계획 수립 [↑](#toc)

### 1단계: Planner 에이전트 선택

Chat 드롭다운에서 **Planner**를 선택합니다.

### 2단계: 계획 요청

아래 요청을 입력합니다.

```text
TodoManager에 상태(pending/done)로 할일을 필터링하는 기능을 구현할 계획을 세워줘.

다음 항목을 포함해줘:
1. 수정할 파일 목록
2. TDD Red → Green → Refactor 순서
3. 각 단계의 완료 기준
4. 테스트해야 할 핵심 시나리오
```

### 3단계: Planner 결과 확인

| 확인 항목 | 기대 결과 |
|----------|----------|
| 파일 목록 | `src/todo/manager.py`, `tests/test_manager.py`, `docs/PRODUCT.md` 등이 언급됨 |
| TDD 순서 | Red → Green → Refactor 순서가 분리되어 제시됨 |
| 테스트 시나리오 | pending/done 필터링, 회귀 테스트 등이 포함됨 |
| 구현 범위 | 불필요한 대규모 리팩터링 없이 현재 기능에 필요한 범위로 제한됨 |

Planner 결과를 메모해두거나 화면에 띄워둡니다. TDD 단계에서 이 계획을 참조합니다.

> 🤔 **Planner가 코드를 직접 작성하려 하면?**
> "구현하지 말고 계획만 제시해줘"라고 재요청합니다. `planner.agent.md`의 `tools: [read, search]`가 올바른지도 확인하세요.

## 📝 컨텍스트 패킷 준비 (Step 2 전) [↑](#toc)

TDD 사이클을 시작하기 전에 05장에서 배운 컨텍스트 패킷을 준비합니다. Planner가 만든 계획과 결합하면 TDD 에이전트가 더 정확하게 동작합니다.

아래 패킷을 기반으로 작성합니다. Planner 계획 부분은 실제 결과로 교체합니다.

```text
[작업 목표]
- TodoManager에 상태(pending/done) 기준으로 태스크를 필터링하는 메서드를 추가합니다.

[참조 파일]
- docs/PRODUCT.md (기능 요구사항)
- src/todo/manager.py (구현 대상)
- tests/test_manager.py (테스트 기준)
- .github/instructions/testing.instructions.md (테스트 규칙)

[제약 조건]
- 공개 메서드에는 한국어 docstring을 반드시 포함합니다.
- 모든 파라미터와 반환값에 타입 힌트를 명시합니다.
- 예외 처리는 ValueError를 먼저 사용합니다.

[완료 기준]
1. 새로운 필터링 테스트가 추가되고 모두 통과합니다.
2. 기존 테스트에 회귀가 없습니다.
3. pending/done 모두의 필터링 동작이 검증됩니다.

[제외 범위]
- CLI 명령어 추가는 이번 작업에서 하지 않습니다.
- 데이터 저장 방식 변경은 하지 않습니다.
```

이 패킷을 TDD Red 에이전트에게 전달할 때 함께 붙여 넣으면, 에이전트가 작업 범위를 명확히 인식합니다.

## 4️⃣ Step 2: TDD 사이클 실행 [↑](#toc)

### TDD Red — 실패 테스트 작성

Chat 드롭다운에서 **TDD Red**를 선택합니다.

아래 요청을 입력합니다. `[Planner 계획]` 부분을 실제 Planner 출력으로 바꿉니다.

```text
상태별 필터링 기능에 대한 실패하는 pytest 테스트부터 작성해줘.

참조 문서:
- docs/PRODUCT.md

대상 파일:
- src/todo/manager.py
- tests/test_manager.py

Planner 계획:
[여기에 Planner가 만든 계획을 붙여넣기]
```

테스트 작성 후 확인합니다.

```bash
uv run pytest -v
```

기대 결과: 새로 추가된 테스트가 FAILED (구현이 아직 없으므로 정상)

### TDD Green — 최소 구현

자동 handoff를 클릭하거나, 드롭다운에서 **TDD Green**을 선택합니다.

필요하면 아래를 입력합니다.

```text
방금 작성한 실패 테스트를 통과시키는 최소 구현만 추가해줘.
불필요한 리팩터링은 하지 말아줘.
```

구현 후 확인합니다.

```bash
uv run pytest -v
```

기대 결과: 전체 PASSED

### TDD Refactor — 코드 품질 개선

자동 handoff를 클릭하거나, 드롭다운에서 **TDD Refactor**를 선택합니다.

```text
현재 테스트를 깨지 않는 범위에서만 코드 품질을 개선해줘.
중복 제거와 가독성 개선에 집중해줘.
```

리팩터링 후 확인합니다.

```bash
uv run pytest -v
```

기대 결과: 전체 PASSED (리팩터링 전과 동일)

### TDD 진행 요약

| 단계 | 확인 명령 | 기대 결과 |
|------|----------|-----------|
| Red 완료 | `uv run pytest -v` | 새 테스트 FAILED |
| Green 완료 | `uv run pytest -v` | 전체 PASSED |
| Refactor 완료 | `uv run pytest -v` | 전체 PASSED |

> 🤔 **Green 단계에서 과도하게 리팩터링하면?**
> "최소 구현만 추가해줘. 구조 개선은 Refactor 단계에서 할게."라고 재요청합니다.

> 🤔 **자동 handoff가 안 되면?**
> Chat 드롭다운에서 직접 다음 에이전트를 선택하고 "이전 단계에서 만든 테스트를 기반으로 진행해줘"라고 요청합니다.

## 5️⃣ Step 3: 최종 검증 [↑](#toc)

### 1단계: 전체 테스트 실행

```bash
uv run pytest -v
```

```text

```

모든 테스트가 PASSED이면 기능 구현이 완료된 것입니다.

### 2단계: Planner에게 코드 리뷰 요청

Planner 에이전트를 선택하고 아래를 입력합니다.

```text
방금 구현한 상태별 필터링 코드를 검토해줘.
- src/todo/manager.py의 새 메서드
- tests/test_manager.py의 새 테스트
코드 수정 없이 리뷰 의견만 제시해줘.
```

### 3단계: PRODUCT.md 업데이트

구현 완료 후 `docs/PRODUCT.md`의 상태를 갱신합니다.

```md
| 상태별 필터링 | 상태(pending/done)로 할일 목록을 필터링한다 | ✅ 구현 완료 |
```

## 🔎 자주 발생하는 문제와 해결 [↑](#toc)

통합 실습에서 가장 많이 막히는 지점과 해결 방법입니다.

### 1) Planner가 계획 대신 코드를 작성할 때

원인: `planner.agent.md`의 `tools` 설정이 잘못되어 있음

해결:
1. `planner.agent.md` 파일을 열어 `tools: [read, search]`인지 확인합니다.
2. `edit`이나 `execute`가 포함되어 있으면 제거합니다.
3. Chat 창을 재시작한 뒤 다시 시도합니다.
4. 요청에 "구현하지 말고 계획만 제시"를 명시합니다.

### 2) TDD Red 이후 자동 handoff가 안 될 때

원인: `handoffs[].agent` 값이 대상 에이전트 `name`과 정확히 일치하지 않음

해결:
1. `TDD-red.agent.md`를 열어 `handoffs[].agent` 값을 확인합니다.
2. `TDD-green.agent.md`의 `name` 값과 철자, 대소문자, 공백이 정확히 같은지 비교합니다.

```yaml
# TDD-red.agent.md
handoffs:
  - agent: "TDD Green"    # ← 이 값이

# TDD-green.agent.md
name: "TDD Green"         # ← 이 값과 정확히 일치해야 합니다
```

3. 불일치가 있으면 수정 후 Chat 세션을 재시작합니다.
4. 자동 handoff가 계속 안 되면 수동으로 다음 에이전트를 선택합니다.

### 3) `uv run pytest -v` 실행이 안 될 때

원인: 환경 설정 미완료 또는 경로 문제

해결 순서:
1. 프로젝트 루트 폴더에서 명령을 실행했는지 확인합니다.
2. `uv sync`를 먼저 실행합니다.
3. `uv run python -m pytest -v`로 대체해 시도합니다.

### 4) Green 단계에서 과도한 리팩터링이 일어날 때

원인: Green 에이전트가 "최소 구현" 원칙을 벗어남

해결:
1. "테스트를 통과시키는 데 필요한 코드만 추가해줘"라고 재요청합니다.
2. "중복 제거나 구조 개선은 다음 단계에서 할 거야"를 명시합니다.

### 5) `docs/PRODUCT.md`를 업데이트했는데 TDD 에이전트에 반영이 약할 때

해결:
1. TDD Red 요청 시 `docs/PRODUCT.md`를 명시적으로 참조로 포함합니다.
2. Planner 출력 계획을 함께 붙여넣어 더 구체적인 컨텍스트를 제공합니다.

```
.github/
├── copilot-instructions.md       # 프로젝트 공통 규칙 (항상 적용)
├── instructions/
│   ├── general.instructions.md   # Python 코딩 규칙
│   └── testing.instructions.md   # pytest + AAA 패턴 규칙
├── prompts/
│   ├── new-feature.prompt.md     # 새 기능 추가 템플릿
│   ├── write-tests.prompt.md     # 테스트 작성 템플릿
│   └── code-review.prompt.md     # 코드 리뷰 템플릿
└── agents/
    ├── planner.agent.md          # 계획 전용 읽기 에이전트
    ├── TDD-red.agent.md          # 실패 테스트 전담
    ├── TDD-green.agent.md        # 최소 구현 전담
    └── TDD-refactor.agent.md     # 품질 개선 전담
```

### AI-Native 개발의 핵심 원리

이 과정에서 반복적으로 등장한 3가지 원리입니다.

1. **역할 분리**: 하나의 AI에게 모든 것을 맡기지 않고, 목적에 맞게 에이전트를 나눕니다.
2. **컨텍스트 정밀성**: "많이 주기"가 아니라 "정확하게 주기"가 결과 품질을 결정합니다.
3. **검증 우선**: 기능이 동작하는지 테스트로 먼저 정의하고, 그것을 통과시키는 방향으로 구현합니다.

## 8️⃣ 다음 학습 안내 [↑](#toc)

이 과정은 GitHub Copilot을 기반으로 AI-Native 개발의 기초를 다루었습니다. 더 나아가면 이런 도구들이 있습니다.

| 도구 | 특징 |
