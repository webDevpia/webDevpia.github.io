# LlamaIndex를 활용한 RAG 구현

## 필요 라이브러리 설치

`%pip install`을 통해 환경을 준비합니다. 이전과 유사하게 `python-dotenv`, `qdrant-client`, `llama-index`, `llama-index-vector-stores-qdrant`를 설치하지만, 이번에는 추가로 `datasets` 라이브러리를 포함하고 있습니다. `datasets`는 Hugging Face의 데이터셋 라이브러리로 다양한 공개 데이터셋을 쉽게 불러올 수 있게 해줍니다. 이 명령을 실행하면 각 패키지가 설치되거나 이미 설치되어 있음을 나타내는 로그가 출력됩니다. 예를 들어, `datasets`가 새로 설치되면서 필요한 의존성들이 다운로드되는 모습이 보일 수 있습니다. 이 과정을 통해 LlamaIndex와 Qdrant 연동, Hugging Face 데이터셋 사용 등을 위한 환경이 모두 갖춰집니다.


```python
%pip install -U python-dotenv qdrant-client llama-index llama-index-vector-stores-qdrant datasets
```

## 환경변수 로드 (API 키 설정)

`.env` 파일에서 OpenAI API 키와 Qdrant 접속 키/URL을 불러오는 코드입니다. `load_dotenv()`로 환경변수를 로드한 뒤, `os.getenv`를 이용하여 `OPENAI_API_KEY`, `QDRANT_API_KEY`, `QDRANT_URL` 값을 가져와 변수로 저장합니다. 이로써 OpenAI의 임베딩/LLM 서비스와 Qdrant 데이터베이스를 사용할 수 있는 자격 증명이 설정됩니다. 출력은 없으며, 이러한 환경변수 방식은 **보안상 키를 코드에 직접 노출하지 않기 위한 표준적인 방법**임을 다시 한번 염두에 둡니다.


```python
from dotenv import load_dotenv
import os

load_dotenv()

OPENAI_API_KEY = os.getenv('OPENAI_API_KEY')
QDRANT_API_KEY = os.getenv('QDRANT_API_KEY')
QDRANT_URL = os.getenv('QDRANT_URL')
```

## Qdrant 클라이언트 연결 및 기존 컬렉션 확인

`QdrantClient`를 초기화하여 Qdrant 벡터 DB에 연결합니다 (`url`과 `api_key` 사용). 그런 다음 `client.get_collections()`로 현재 존재하는 컬렉션을 출력합니다. 만약 이 노트북을 첫 번째 노트북에 이어 실행했다면, 앞서 만든 `"news"` 컬렉션이 이미 존재할 것입니다. 실제 출력에서도 `CollectionDescription(name='news')`와 같이 **news 컬렉션**이 나왔는데, 이는 Qdrant 서버에 해당 컬렉션이 있다는 뜻입니다. (만약 별도의 새 환경이라면 빈 리스트가 나오겠지만, 여기서는 기존 컬렉션이 확인된 상황입니다.) 이 단계는 Qdrant 연결이 정상적으로 설정되었고, DB 상태를 파악하는 용도로 쓰입니다.


```python
from qdrant_client import QdrantClient

client = QdrantClient(
    url=QDRANT_URL,
    api_key=QDRANT_API_KEY,
)

print(client.get_collections())
```

## LlamaIndex 설정 구성

LlamaIndex (과거 명칭: GPT Index) 라이브러리의 글로벌 설정을 조정합니다. `Settings` 클래스를 이용하여 기본 LLM과 임베딩 모델, 문서 쪼개기 크기를 지정합니다:

   * `Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0)`: 기본 질문 답변에 사용할 LLM을 OpenAI의 `"gpt-4o-mini"` 모델로 설정하고, `temperature=0`으로 지정하여 출력을 **결정론적**(매우 일관되게) 나오도록 합니다.
   * `Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")`: 임베딩 모델로 OpenAI의 `"text-embedding-3-small"`을 사용하도록 설정합니다. 이는 OpenAI 임베딩 API를 활용하며, 앞서 사용한 것과 동일한 1536차원 임베딩 모델입니다.
   * `Settings.chunk_size = 500` 및 `Settings.chunk_overlap = 100`: LlamaIndex가 문서를 여러 조각(chunk)으로 나눌 때 한 조각의 최대 글자 수를 500자로 하고, 청크 간에 100자씩 겹치도록(overlap) 지정합니다. 이렇게 하면 긴 문서도 일정 크기로 나눠지며, 겹치는 부분 덕분에 인접 청크 간 문맥이 일부 유지됩니다.

   이 설정들은 LlamaIndex 전역으로 적용되어, 이후 문서를 인덱싱하거나 질문에 답변할 때 사용됩니다. 요약하면, **기본 LLM 모델**, **임베딩 모델**, **문서 청크 사이즈**를 미리 정의하여 일관된 환경을 만든 것입니다.


```python
from llama_index.core import Settings
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding

Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0)
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.chunk_size = 500
Settings.chunk_overlap = 100
```

## 예시 문서 생성

`llama_index.core.Document` 클래스를 사용해 간단한 문서 다섯 개를 만듭니다. 코드에서 `texts` 리스트에는 한국어로 된 짤막한 문장 다섯 개가 정의되어 있습니다:

   * 파이썬 언어에 대한 소개 문장
   * 딥러닝에 대한 설명 문장
   * GPT 모델에 대한 소개 문장
   * 랭체인(LangChain)에 대한 설명 문장
   * RAG(Retrieval-Augmented Generation)에 대한 개념 설명 문장

   그런 다음 리스트 컴프리헨션을 이용해 각 텍스트를 Document 객체로 감싼 `documents` 리스트를 만듭니다 (`Document(text=t)` 형태). 이렇게 생성된 Document는 LlamaIndex에서 사용하는 문서 자료구조로, 텍스트와 선택적으로 메타데이터를 포함할 수 있습니다. 마지막 줄의 `documents`를 통해 이 리스트를 출력하면 Document 객체들의 리스트가 나타납니다. (Document 객체를 출력하면 텍스트 내용이나 요약된 정보가 표시될 수 있습니다.) 이 단계까지는 학습용 예시로서 **5개의 간단한 텍스트 문서**를 준비한 것이고, 각 문서는 하나의 주제를 설명하는 내용으로 구성되어 있습니다.


```python
from llama_index.core import Document

texts = [
    "파이썬은 1991년 귀도 반 로섬이 발표한 인터프리터 방식의 프로그래밍 언어이다. 간결하고 쉬운 문법으로 인기가 많다.",
    "딥러닝은 기계학습의 한 분야로, 인공 신경망을 사용하여 복잡한 데이터로부터 패턴을 인식하고 예측하는 기술이다.",
    "GPT(Generative Pre-trained Transformer)는 OpenAI에서 개발한 대규모 언어 모델로, 자연어 처리 분야에서 뛰어난 성능을 보인다.",
    "랭체인은 LLM 기반의 애플리케이션 구축을 쉽게 해주는 프레임워크로, 다양한 AI 모델과 도구를 결합해 사용할 수 있다.",
    "RAG(Retrieval-Augmented Generation)는 검색과 생성 기술을 결합하여 보다 정확하고 최신의 정보를 생성할 수 있도록 하는 기법이다."
]

documents = [Document(text=t) for t in texts]
documents
```

## 벡터 인덱스 생성 (Qdrant 활용)

앞서 만든 5개 Document에 대해 벡터 인덱스를 구축합니다. 여기서는 LlamaIndex와 Qdrant를 통합하여 사용하므로 몇 가지 객체를 초기화합니다:

   * `QdrantVectorStore(client=client, collection_name="ai_tech")`: Qdrant를 LlamaIndex의 백엔드 벡터 저장소로 쓰기 위한 객체입니다. `client`로 Qdrant 연결을 주입하고, 새 컬렉션 이름을 `"ai_tech"`로 지정했습니다.
   * `StorageContext.from_defaults(vector_store=vector_store)`: 방금 만든 Qdrant 벡터 스토어를 StorageContext로 감싸 LlamaIndex에 전달합니다. StorageContext는 인덱스가 데이터를 저장/불러올 때 사용하는 추상 레이어입니다.
   * `VectorStoreIndex.from_documents(documents, storage_context=storage_context)`: 이제 LlamaIndex에게 Document 리스트를 넘겨주어 인덱스를 생성합니다. 내부적으로 일어나는 과정은 **(a) 각 Document를 설정된 chunk\_size(500) 기준으로 청크로 분할 -> (b) 각 청크에 대해 OpenAI 임베딩 생성 -> (c) 해당 임베딩들을 Qdrant `"ai_tech"` 컬렉션에 저장**입니다. 또한 필요한 인덱스 메타정보를 관리하여 추후 질의 시 어떤 문서를 검색할지 연결합니다.

   이처럼 **LlamaIndex**를 활용하면 외부 데이터(Document)를 LLM이 활용할 수 있는 인덱스로 손쉽게 구축할 수 있습니다. 결국 `"ai_tech"`라는 Qdrant 컬렉션에 5개 문서(혹은 청크된 결과)의 임베딩이 저장되었고, `index` 객체로 LlamaIndex 인덱스를 가리키게 됩니다. 별도 출력은 없지만, 이 단계를 통해 **"파이썬", "딥러닝", "GPT", "랭체인", "RAG"** 내용의 작은 지식베이스가 만들어졌습니다.


```python
from llama_index.vector_stores.qdrant import QdrantVectorStore
from llama_index.core import VectorStoreIndex, StorageContext

vector_store = QdrantVectorStore(client=client, collection_name="ai_tech")
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex.from_documents(documents, storage_context=storage_context)
```

## 위키피디아 데이터셋 불러오기

`datasets` 라이브러리를 사용하여 한글 위키피디아 데이터셋을 가져옵니다. `load_dataset('lcw99/wikipedia-korean-20240501', split='train')`은 2024년 5월 1일 기준 한국어 위키피디아 덤프를 가리키는 데이터셋의 훈련 분할을 로드합니다. 이 데이터셋은 방대한 문서를 포함하고 있지만, `ds.select(range(len(ds)-10, len(ds)))`를 통해 **마지막 10개 문서만 선택**했습니다. 이는 예시로 소규모 데이터를 다루기 위함이며, 이렇게 하면 `ds` 객체에는 10개의 위키 문서만 남게 됩니다. 이어서 `ds.to_pandas()`로 `pandas DataFrame`으로 변환하고, `'title'`과 `'text'` 열만 선택합니다. `df.drop_duplicates(subset='text', ...)`는 혹시 텍스트 내용이 중복된 행이 있다면 하나만 남기도록 합니다. 마지막으로 `df`를 출력하여 선택된 10개 위키 문서의 \*\*제목(title)\*\*과 \*\*내용(text)\*\*이 테이블로 표시됩니다. 이 단계에서는 한국어 위키피디아 문서 10건을 준비한 것으로, 각각 제목과 본문이 상당한 길이를 갖고 있을 것입니다. (출력된 데이터프레임을 통해 제목들을 확인할 수 있고, 텍스트는 길어 모두 표시되지 않을 수 있습니다.) 이렇게 `datasets`를 활용하면 공개된 대용량 데이터를 편리하게 가져와 실습해볼 수 있습니다.


```python
from datasets import load_dataset

ds = load_dataset('lcw99/wikipedia-korean-20240501', split='train',  token=os.getenv("HUGGINGFACE_TOKEN") )
ds = ds.select(range(len(ds)-10, len(ds)))
df = ds.to_pandas()[['title', 'text']]
df.drop_duplicates(subset='text', keep='first', inplace=True)
df
```

## 위키 문서들을 Document로 변환

방금 불러온 위키피디아 데이터 10개를 LlamaIndex의 Document 객체로 다시 만들어 줍니다. 빈 리스트 `documents = []`를 준비하고, `df.iterrows()`를 이용해 데이터프레임의 각 행에 접근하여 루프를 돌립니다. 각 행마다 `row['text']`가 문서 내용, `row['title']`이 문서 제목입니다. `Document(text=row['text'], metadata={"title": row['title']})` 형식으로 Document를 생성하는데, 여기서는 메타데이터로 제목을 함께 저장해 둡니다. 이렇게 하면 문서 내용과 함께 어떤 제목의 문서인지 추후 식별할 수 있습니다. 만들어진 Document를 `documents` 리스트에 추가하고, 반복을 마치면 10개의 Document가 리스트에 들어갑니다. 마지막에 `documents[0]`을 출력하여 첫 번째 Document 객체를 확인합니다. 출력 결과에는 Document의 내용 또는 요약이 표시될 텐데, 제목 메타데이터도 포함되어 있을 것입니다. 이를 통해 위키 문서들이 LlamaIndex Document로 잘 감싸졌는지, 메타데이터가 들어갔는지 확인할 수 있습니다.


```python
documents = []

for i, row, in df.iterrows():
  documents.append(Document(text=row['text'], metadata={"title": row['title']}))

documents[0]
```

## 위키 문서 벡터 인덱스 생성 

위 단계와 유사하게, 이번에는 위키피디아 Document 10개에 대한 벡터 인덱스를 만듭니다. `QdrantVectorStore(client=client, collection_name="wiki")`로 새로운 Qdrant 컬렉션 `"wiki"`를 지정하고, `StorageContext`를 생성한 뒤 `VectorStoreIndex.from_documents(documents, storage_context=...)`를 수행합니다. 이 과정에서도 LlamaIndex는 각 문서를 chunking -> 임베딩 -> Qdrant 저장을 자동으로 처리합니다. 10개의 문서지만 내용이 길기 때문에 여러 청크로 쪼개졌을 것입니다. (출력 설정이 없어서 중간 진행상황은 볼 수 없지만, 내부적으로 임베딩이 생성되어 저장되었을 것입니다.) 이 셀 실행 후 Qdrant에는 `"wiki"` 컬렉션이 생겼고 그 안에 10개 문서의 임베딩(청크 단위)이 보관되었습니다. 이제 `"wiki"` 데이터에 대해서도 질의 검색을 할 수 있는 상태가 됩니다. 이 과정에서 별도의 출력은 없지만, 데이터가 잘 저장되었다면 `client.get_collections()` 등으로 확인하면 "wiki" 컬렉션이 존재할 것입니다. (이 셀에서는 확인하지 않더라도, 추후 필요 시 사용 가능합니다.)


```python
vector_store = QdrantVectorStore(client=client, collection_name="wiki")
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex.from_documents(documents, storage_context=storage_context)
```

## 학사 정책 문서들 로드 (SimpleDirectoryReader)

이번에는 **학사 정책** 관련 문서들을 폴더에서 한꺼번에 읽어들입니다. `SimpleDirectoryReader`는 LlamaIndex에 내장된 편리한 리더기로, 특정 디렉토리의 모든 파일을 읽어 Document 리스트로 만들어 줍니다. `input_dir='./data'`로 지정하여 현재 노트북 경로의 `data/` 폴더를 읽도록 한 뒤, `.load_data()`를 호출합니다. 이 폴더에는 여러 개의 Markdown 형식(.md) 파일로 된 학사 정책 문서들이 들어있으며 (휴학, 학사경고, 장학금, 등록금 등 다양한 주제의 학사 규정 문서들), `load_data()`는 각 파일을 하나의 Document로 취급하여 내용을 읽어옵니다. 반환된 `documents` 리스트에는 폴더 내 파일 개수만큼 Document가 들어갑니다. (예를 들어 10개의 정책 문서 파일이 있었다면 리스트 길이가 10이 됩니다.) 곧바로 `documents[0]`을 출력하여 첫 번째 문서 객체를 확인합니다. 출력 내용에는 해당 정책 문서의 본문 일부 또는 메타데이터가 나타날 수 있습니다. 이 첫 번째 Document는 아마도 특정 학사 정책의 내용 (예: 학사경고나 유급 등에 관한 규정)일 것입니다. 이처럼 **SimpleDirectoryReader**를 사용하면 다수의 문서를 손쉽게 일괄 로딩할 수 있고, 개발자는 일일이 파일을 열어 읽는 수고를 덜 수 있습니다. 여기서는 학사 정책 Markdown 파일들이 `data/` 폴더에 준비되어 있고, 그 모든 내용이 `documents` 리스트에 담긴 것입니다.


```python
from llama_index.core.readers import SimpleDirectoryReader

documents = SimpleDirectoryReader(input_dir='./data').load_data()
documents[0]
```

## 청크 설정 조정

학사 정책 문서들을 본격적으로 인덱싱하기 전에, 문서 분할 크기 설정을 재조정합니다. `Settings.chunk_size = 500` 및 `Settings.chunk_overlap = 50`으로 청크 크기를 500자로 유지하되 **겹치는 부분을 50자로 감소**시켰습니다. 이전 설정에서 overlap이 100자였는데, 여기서는 50자로 줄인 것입니다. 이유는 학사 정책 문서들의 특성에 따라 적절한 겹침 길이를 조정하려는 것으로 추측됩니다. (예를 들어 질문-답변 형식의 문서가 많다면 겹침을 너무 길게 할 필요가 없다고 판단했을 수 있습니다.) 이 설정은 곧 진행할 인덱싱에 적용되어, 학사 정책 문서들이 약 500자씩 겹침 50자로 청크화됩니다. 설정을 변경해도 출력은 없지만, 이러한 파라미터 조정으로 **청크 분할 전략**을 다르게 할 수 있다는 점을 보여줍니다.


```python
Settings.chunk_size = 500
Settings.chunk_overlap = 50
```

## 학사 정책 문서 벡터 인덱스 생성

이제 `academic_policies`라는 새로운 Qdrant 컬렉션에 학사 정책 문서들을 색인합니다. `QdrantVectorStore(client=client, collection_name="academic_policies")`로 벡터 저장소를 설정하고, StorageContext를 만든 다음 `VectorStoreIndex.from_documents(documents, storage_context=..., show_progress=True)`로 인덱스를 구성합니다. `show_progress=True`로 지정했으므로 실행 중에 진행률이 출력되었습니다:

이로 미루어 볼 때, 학사 정책 10개 문서가 대략 합쳐서 85개 조각으로 나누어졌음을 알 수 있습니다. LlamaIndex는 각 청크에 대해 OpenAI 임베딩(1536차원)을 구해 Qdrant `"academic_policies"` 컬렉션에 저장했습니다. 인덱스 구축이 끝나면 마지막 줄에서 `index.storage_context.vector_store.collection_name`을 출력하여 어떤 컬렉션을 가리키고 있는지 확인합니다. 그 결과 `"academic_policies"`가 표시되어, 우리가 방금 만든 인덱스가 Qdrant의 academic\_policies 컬렉션에 연결되어 있음을 보여줍니다. 이제 이 `index` 객체를 사용하면 학사 정책 문서들을 대상으로 질문응답을 할 준비가 되었습니다. (참고: 이처럼 LlamaIndex와 Qdrant를 사용하여 대량의 정책 문서를 벡터화하고 저장함으로써, **대화형 질의응답 시스템**의 기반을 마련했습니다.)



```python
vector_store = QdrantVectorStore(client=client, collection_name="academic_policies")
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex.from_documents(documents, storage_context=storage_context, show_progress=True)
index.storage_context.vector_store.collection_name
```

## 자연어 질의에 대한 응답 생성

인덱스에 질의를 던지고 LLM을 통해 답변을 얻는 단계입니다. `index.as_query_engine(similarity_top_k=5)`를 호출하여 쿼리 엔진 객체를 만들었습니다. 이는 질문이 들어오면 우선 벡터 검색으로 가장 관련도 높은 5개의 청크를 찾도록 설정한 것입니다. 그런 다음 `query_engine.query("휴학은 최대 몇 학기까지 가능한가요?")`를 실행하는데, 이 한 줄에서 **질의응답 과정 전체**가 수행됩니다. 작동 원리를 풀어서 설명하면:

    * "휴학은 최대 몇 학기까지 가능한가요?"라는 자연어 질문을 임베딩 벡터로 변환합니다.
    * 이전에 구축한 `academic_policies` 벡터 인덱스에서 이 임베딩과 유사한 상위 5개 청크를 검색합니다 (휴학 관련 규정이 있는 청크들이 선택될 것입니다).
    * 선택된 청크들을 컨텍스트로 하여, 설정해둔 LLM(`gpt-4o-mini`)에 질문에 대한 답변 생성을 요청합니다.
    * LLM은 제공된 문서 조각들을 참고하여 질문에 맞는 답변을 생성합니다.

`print(response)`로 출력된 답변을 보면 **"휴학은 최대 6학기까지 가능합니다."** 라고 짤막하게 대답했습니다. 이는 학사 정책 중 **휴학 규정**에 대한 정확한 답변으로, 아마도 데이터에 "총 허용기간: 6학기"라는 내용이 있었기 때문입니다. 이렇게 LlamaIndex의 Query Engine을 활용하면 **RAG (Retrieval-Augmented Generation)**, 즉 **검색 기반 답변 생성**이 구현됩니다. 모델이 자체 학습 지식만 사용하는 것이 아니라, 우리가 제공한 최신 학사 정책 문서에서 정보를 찾아 근거 있는 답을 준 것입니다. 질문자가 원하는 "휴학 가능 최대 학기 수"에 대해 6학기라는 결과를 얻었고, 이는 실제 규정과 일치하는 답변입니다. 이 과정에서 사용자는 마치 ChatGPT에게 묻듯이 자연어로 질문했지만, 답변 내용은 연결된 사설 데이터(학사 정책)에서 추출한 최신 정보입니다.


```python
query_engine = index.as_query_engine(similarity_top_k=5)
response = query_engine.query("휴학은 최대 몇 학기까지 가능한가요?")
print(response)
```

## 근거 자료 추출 (Retriever 사용)

앞서는 LLM이 최종 답변만 주었는데, 이번에는 **어떤 근거로 그런 답이 나왔는지** 직접 확인해보겠습니다. `index.as_retriever(similarity_top_k=5)`를 통해 검색기(retriever)를 얻고, 같은 질문으로 `nodes = retriever.retrieve("휴학은 최대 몇 학기까지 가능한가요?")`를 호출했습니다. 이 경우 LLM을 쓰지 않고 단순히 상위 5개 관련 청크(`Node` 객체)만 반환받습니다. `print(len(nodes))`로 노드 개수를 출력하면 5개임을 확인했고, 이어서 각 `node.text`를 출력하여 내용들을 나열했습니다. 출력 결과를 보면 학사 정책 문서들 중 **휴학에 관한 부분**이 여러 조각 등장합니다. 예를 들어:

    * Q\&A 형식으로 된 휴학 관련 문의와 답변 일부 (복학 신청 방법, 군휴학 처리 등)
    * "총 허용기간: 6학기"와 같이 규정 조항을 나열한 부분
    * 휴학 종류와 요건을 설명하는 목록 등

이러한 **원문 조각들**을 통해 앞서 모델이 "6학기"라고 답한 근거를 찾을 수 있습니다. 실제로 출력된 텍스트 중 "*총 허용기간: 6학기*"라는 문구가 있으며, 모델은 이 정보를 사용해 질문에 답했음을 알 수 있습니다. 또 다른 조각들에는 휴학과 관련된 여러 세부 규정(Q1, Q2... 등 질의응답 예시)도 포함되어 있는데, 이는 모델이 답을 생성하는데 참고했을 배경 지식입니다. 이 retriever를 활용한 출력은 일종의 **근거 문서 표시** 기능으로, 최종 답변에 신뢰도를 더해줍니다. 즉, 사용자에게 "왜 6학기인지" 출처를 보여줄 수 있는 것이죠. 정리하면, 이 셀에서는 검색 단계에서 찾아낸 **원문 텍스트 노드**들을 직접 살펴봄으로써, LLM의 답변이 주어진 문서를 근거로 했음을 확인했습니다. 이는 RAG 시스템에서 투명성을 높이는 중요한 부분이며, 사용자는 필요시 이러한 원문을 검토해 추가 정보를 얻거나 답변의 정확성을 검증할 수 있습니다.


```python
retriever = index.as_retriever(similarity_top_k=5)
nodes = retriever.retrieve("휴학은 최대 몇 학기까지 가능한가요?")
print(len(nodes))
for node in nodes:
    print(node.text)
```
