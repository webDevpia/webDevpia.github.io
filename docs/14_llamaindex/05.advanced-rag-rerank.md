# RAG 검색 결과에 리랭커 적용

## 라이브러리 설치

* **목적:** RAG(Search Augmented Generation) 파이프라인과 리랭커 모델 사용을 위해 필요한 파이썬 패키지를 설치합니다. `llama_index` (LlamaIndex) 라이브러리와 Qdrant 벡터 스토어, 그리고 Cohere의 리랭커(postprocessor)를 사용하므로 관련 패키지를 미리 준비합니다.
* **사용 패키지:** `%pip install` 명령으로 `llama_index`, `llama-index-vector-stores-qdrant`, `llama-index-postprocessor-cohere-rerank` 등을 설치합니다. LlamaIndex는 문서 임베딩과 질의 응답을 수행하는 라이브러리이고, Qdrant는 벡터 DB(데이터베이스) 지원, Cohere rerank 패키지는 검색 결과 재정렬에 필요한 모듈입니다.
* **동작 방식:** 주피터 노트북의 매직 명령 `%pip install`을 사용하여 위 패키지들을 한 번에 설치합니다. 이미 설치되어 있다면 "Requirement already satisfied"와 같이 표시되고, 없는 패키지는 자동으로 다운로드 및 설치됩니다.
* **기대 결과:** 패키지 설치 로그가 출력되며, 추가로 Cohere rerank용 패키지가 다운로드됩니다. 이 셀이 성공적으로 실행되면 이후 코드에서 LlamaIndex, Qdrant 클라이언트, Cohere 리랭커 등을 임포트하여 사용할 수 있게 됩니다.


```python
%pip install llama_index llama-index-vector-stores-qdrant llama-index-postprocessor-cohere-rerank
```

## API 키 및 환경변수 로드

* **목적:** 외부 API(OpenAI, Cohere 등)와 데이터베이스(Qdrant) 접속에 필요한 **API 키**와 **URL** 정보를 안전하게 불러옵니다. 코드에 키를 직접 적지 않고 `.env` 파일에 저장해두고 사용하여 보안을 지키고, 협업 시에도 키 노출을 피합니다.
* **사용 메서드:** `python-dotenv`의 `load_dotenv()` 함수로 현재 작업 경로의 `.env` 파일을 불러옵니다. 그런 다음 `os.getenv('키_이름')`으로 환경변수에 저장된 값을 가져옵니다.
* **동작 방식:** `.env` 파일에는 `OPENAI_API_KEY`, `QDRANT_API_KEY`, `QDRANT_URL`, `COHERE_API_KEY` 등이 저장되어 있다고 가정합니다. `load_dotenv()`는 이 파일의 내용을 읽어 환경변수에 설정하고, `os.getenv`는 해당 변수값을 각 변수(`OPENAI_API_KEY`, `QDRANT_API_KEY` 등)에 할당합니다.
* **기대 결과:** 별도의 출력은 없지만, 이후 코드에서 사용할 API 키와 URL 정보들이 `OPENAI_API_KEY`, `QDRANT_API_KEY`, `QDRANT_URL`, `COHERE_API_KEY` 등의 파이썬 변수에 정상적으로 담깁니다. 이를 통해 OpenAI의 LLM, Qdrant 벡터스토어, Cohere 리랭커 API를 인증하여 사용할 준비를 마칩니다.


```python
from dotenv import load_dotenv
import os

load_dotenv()

OPENAI_API_KEY = os.getenv('OPENAI_API_KEY')
QDRANT_API_KEY = os.getenv('QDRANT_API_KEY')
QDRANT_URL = os.getenv('QDRANT_URL')
COHERE_API_KEY = os.getenv('COHERE_API_KEY')
```

## Qdrant 데이터베이스 클라이언트 연결

* **목적:** 벡터 데이터베이스인 **Qdrant**에 연결하여, 사전에 구축해둔 영화 데이터 임베딩 컬렉션에 접근할 수 있도록 합니다. 검색 쿼리에 대한 임베딩 유사도 탐색을 Qdrant에서 수행하므로, 클라이언트를 통해 해당 서비스에 접속해야 합니다.
* **사용 객체:** `qdrant_client.QdrantClient` 클래스를 이용해 클라이언트 인스턴스를 생성합니다. 초기화 시 `url=QDRANT_URL`과 `api_key=QDRANT_API_KEY`를 전달하여, 우리의 Qdrant 서버 주소와 인증키를 설정합니다.
* **동작 방식:** 이 코드가 실행되면 변수 `client`에 QdrantClient 객체가 만들어집니다. 내부적으로 해당 URL의 Qdrant 인스턴스에 연결하고 API 키로 인증을 수행합니다. 이후 이 `client`를 사용해 벡터 컬렉션에 질의하거나 데이터를 불러올 수 있습니다.
* **기대 결과:** 눈에 보이는 출력은 없지만, `client`가 성공적으로 생성되었다면 Qdrant 벡터스토어와 통신할 준비가 된 것입니다. 이제 데이터베이스 상의 "movies" 컬렉션을 활용하여 유사도 검색을 진행할 수 있습니다.


```python
from qdrant_client import QdrantClient

client = QdrantClient(
    url=QDRANT_URL,
    api_key=QDRANT_API_KEY
)
```

## LlamaIndex 환경 설정 (LLM 및 임베딩 모델)

* **목적:** LlamaIndex에서 사용할 **LLM**(대형 언어 모델)과 **임베딩 모델**, 그리고 문서 청크 설정을 미리 지정하여 일관된 환경을 설정합니다. 이를 통해 이후 인덱스 생성과 질의응답 시 동일한 모델과 파라미터가 사용됩니다.
* **사용 클래스:** `llama_index.llms.openai.OpenAI`와 `llama_index.embeddings.openai.OpenAIEmbedding` 클래스를 불러옵니다. 또한 `llama_index.core.Settings`를 가져와 전역 설정을 변경합니다.
* **설정 내용:** `Settings.llm`에 `OpenAI(temperature=0, model='gpt-4o-mini')`를 할당하여, 답변 생성에 OpenAI의 GPT-4 기반 모델(예: `gpt-4o-mini`)을 사용하도록 합니다. `temperature=0`으로 설정해 매번 동일하고 확정적인 응답을 얻도록 했습니다. `Settings.embed_model`에는 `OpenAIEmbedding(model="text-embedding-3-small")`을 지정하여 문서 임베딩에 사용할 모델을 설정합니다. 마지막으로 `Settings.chunk_size = 500`과 `Settings.chunk_overlap = 50`으로 문서 조각의 크기와 겹치는 부분 크기를 조정합니다 (문서를 500자 단위로 나누되 50자씩 겹치도록 함).
* **동작 방식:** 위 설정을 하면 LlamaIndex가 인덱스를 만들거나 쿼리를 수행할 때 자동으로 해당 설정을 참고합니다. 예컨대, 문서를 인덱싱할 때 500자씩 나누고, 질의시에는 `gpt-4o-mini` 모델로 응답 생성, `text-embedding-3-small` 모델로 임베딩 비교를 하게 됩니다.
* **기대 결과:** 이 셀도 출력은 없지만, 설정이 완료되어 이후 인덱스를 생성하고 질의응답을 할 때 일관된 모델들이 사용됩니다. 이를 통해 실습 환경에 맞게 경량화된 모델로 빠르게 테스트하고, 결과를 재현할 수 있습니다.


```python
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.core import Settings

Settings.llm= OpenAI(temperature=0, model='gpt-4o-mini')
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.chunk_size = 500
Settings.chunk_overlap = 50
```

## Qdrant 벡터스토어 및 인덱스 초기화

* **목적:** 앞서 연결한 Qdrant의 "movies" 컬렉션을 LlamaIndex의 벡터스토어로 지정하고, 이를 기반으로 질의에 활용할 **벡터 인덱스** 객체를 생성합니다. 이미 Qdrant에 삽입된 영화 데이터 임베딩을 그대로 활용하여 인덱스를 구성하는 단계입니다.
* **사용 클래스:** `llama_index.vector_stores.qdrant.QdrantVectorStore`와 `llama_index.core.VectorStoreIndex`, `StorageContext` 등을 사용합니다.
* **동작 순서:** 먼저 `QdrantVectorStore(client=client, collection_name="movies")`로 Qdrant 벡터스토어 객체를 만듭니다. 여기서 `client`는 이전 셀에서 생성한 Qdrant 연결이고, `"movies"`는 데이터가 저장된 컬렉션 이름입니다. 이 객체는 LlamaIndex가 벡터 임베딩을 Qdrant에서 검색하도록 해주는 인터페이스입니다.
  다음으로 `StorageContext.from_defaults(vector_store=vector_store)`를 호출해 **스토리지 컨텍스트**를 생성합니다. 이 컨텍스트는 LlamaIndex가 벡터스토어 등 외부 저장소를 사용할 수 있게 연결 정보를 담는 역할을 합니다.
  마지막으로 `VectorStoreIndex.from_vector_store(vector_store=vector_store, storage_context=storage_context)`를 통해 **인덱스** 객체를 얻습니다. 이 인덱스는 사실상 Qdrant의 해당 컬렉션에 저장된 임베딩들을 가리키는 래퍼로, 새로 데이터를 넣는 대신 기존의 "movies" 데이터를 조회(search)할 수 있게 해줍니다.
* **기대 결과:** 별도의 텍스트 출력은 없지만, 변수 `index`에 LlamaIndex의 벡터 인덱스가 생성됩니다. 이제 `index`를 사용해 자연어 질의를 던지면, Qdrant에 저장된 영화 데이터들 중 질문과 가장 유사한 내용들을 찾아주는 준비가 된 셈입니다.


```python
from llama_index.vector_stores.qdrant import QdrantVectorStore
from llama_index.core import VectorStoreIndex, StorageContext

# Qdrant 벡터스토어 객체 생성
vector_store = QdrantVectorStore(
    client=client,
    collection_name="movies"
)

# StorageContext 생성: 커스텀 벡터스토어를 연결
storage_context = StorageContext.from_defaults(vector_store=vector_store)

# 벡터 인덱스 조회
index = VectorStoreIndex.from_vector_store(vector_store=vector_store, storage_context=storage_context)
```

## 리랭커 적용 전 검색 질의 (baseline 결과)

* **목적:** **리랭커를 도입하기 전**에 기본 벡터 유사도 검색을 수행하여 결과를 확인합니다. 이를 통해 이후 리랭커 적용 후 어떻게 결과가 달라지는지 비교할 기준을 마련합니다.
* **질의 수행:** `index`로부터 기본 질의 엔진을 얻거나 바로 질의를 호출해, \*\*"좀비 영화"\*\*라는 키워드로 검색을 실행합니다. 여기서는 RAG의 기본 단계만 사용하므로, Qdrant에서 임베딩 유사도가 가장 높은 영화 줄거리들을 상위 몇 개(기본값 상위 2개 정도) 가져오고, LLM이 그들을 바탕으로 응답을 작성합니다.
* **동작 방식:** "좀비 영화"라는 질문에 대해 Qdrant 벡터스토어는 모든 영화의 임베딩을 확인하고, 좀비와 관련된 내용이 많이 등장하는 영화들을 추려냅니다. 예컨대 데이터에 "좀비 바이러스" 등의 내용이 있는 \*\*<부산행>\*\*이나 \*\*<반도>\*\*와 같은 영화가 상위 결과로 선택될 것입니다. LlamaIndex는 이렇게 검색된 노드들을 `source_nodes`로 받아들여, 설정된 LLM (`gpt-4o-mini`)을 통해 이들을 요약/종합하여 최종 답변을 만듭니다.
* **결과 출력:** 이 셀을 실행하면 `Response(...)` 객체 형태로 결과가 표시됩니다. 응답 문자열에는 **"부산행과 반도는 좀비 바이러스를 배경으로 한 액션/공포 영화입니다..."** 와 같이, 질의에 맞는 요약 답변이 들어 있습니다. 또한 `source_nodes` 리스트에는 해당 답변에 참고된 영화 노드 (부산행, 반도)의 세부 정보와 점수가 함께 포함됩니다. 이 단계에서는 벡터 유사도 점수가 높은 순서로 결과가 선택되었으며, 응답은 이 \*\*초기 검색 결과(top-k)\*\*에 기반하여 생성되었습니다.


## Cohere 리랭커 도입 및 재검색

* **목적:** 이전 단계의 **초기 검색 결과를 한층 향상**시키기 위해 **리랭커**(reranker) 모델을 적용합니다. 리랭커는 1차 검색에서 나온 후보들을 재평가하여 진짜 질문 의도에 가장 부합하는 결과를 상위에 오도록 재정렬합니다. 이를 통해 RAG 시스템의 응답 정확도를 높이고자 합니다.
* **리랭커 초기화:** `llama_index.postprocessor.cohere_rerank` 모듈에서 `CohereRerank` 클래스를 임포트하고, `cohere_rerank = CohereRerank(api_key=COHERE_API_KEY, top_n=2)`로 인스턴스를 생성합니다. 여기서 `api_key`는 Cohere 서비스의 API 키이며, `top_n=2`는 최종 상위 2개의 결과만 사용하도록 지정하는 인자입니다. (즉, 리랭커가 판단하기에 가장 관련성이 높은 문서 2개를 남기고 나머지는 버립니다.)
* **검색 엔진 재구성:** 리랭커를 적용하려면 질의 엔진에 후처리 단계로 추가해야 합니다. `index.as_query_engine(similarity_top_k=10, node_postprocessors=[cohere_rerank])`를 호출하여 새로운 `query_engine`을 얻습니다. 여기서 `similarity_top_k=10`으로 설정한 것은 **초기 검색 단계에서 상위 10개 결과**를 넉넉하게 가져오도록 한 것입니다. (어차피 두 번째 단계에서 재정렬할 것이므로, 더 많은 후보를 가져와도 최종 응답에 방해되지 않고, 오히려 최적의 답을 찾을 기회를 높입니다.) `node_postprocessors=[cohere_rerank]` 부분이 핵심으로, 검색된 노드들을 LLM에 보내기 전에 우리가 만든 Cohere 리랭커로 한 번 필터/정렬하는 과정을 삽입합니다.
* **리랭커 동작:** Cohere Rerank 모델은 주어진 질의와 후보 텍스트들에 대해 **자연어 이해 기반의 점수**를 매겨 순위를 매깁니다. 벡터 유사도와 달리, 질의와 문서 내용의 **맥락적 관련성**을 평가하므로, 단순 키워드 매칭 그 이상으로 "이 질문에 어느 문서가 답을 잘 줄 수 있는가?"를 판단합니다. 그 결과 상위 10개 후보 중 가장 관련 높은 2개(`top_n=2`)를 선택합니다. (예: 좀비 영화를 찾는다면 좀비 테마를 제대로 다룬 영화들이 선택되고, 다른 후보들은 탈락될 것입니다.)
* **재검색 실행:** 이제 `query_engine.query("좀비 영화")`로 동일한 질의를 다시 실행합니다. 이번에는 앞서 설명한 대로 10개의 후보를 가져온 뒤 Cohere 리랭커가 개입하여 최적의 2개를 선정했고, 그 2개 자료를 바탕으로 LLM이 응답을 생성합니다.
* **예상 응답 변화:** 출력 형식은 이전과 동일하게 `Response(response='...', source_nodes=[...])`로 나타나지만, 내용은 **리랭커가 선별한 자료**에 기반합니다. 만약 초기 벡터 검색이 다소 엉뚱한 결과를 상위에 올렸다면, 리랭커 적용 후에는 더 관련성 높은 답으로 교체되었을 것입니다. 이 경우 질의가 "좀비 영화"로 명확했기 때문에 리랭커 전후 결과가 비슷하게 보일 수 있지만, 리랭커는 여전히 **결과의 순위와 선택**을 최적화하는 역할을 합니다. 즉, 기존에도 부산행/반도가 나왔다면 동일한 두 영화가 선택되었겠지만, 리랭커를 쓰면 그 이유가 **질문과의 언어적 관련성 점수**에 근거하게 되어 더 신뢰도가 높아집니다.


```python
from llama_index.postprocessor.cohere_rerank import CohereRerank

#리랭커 top_2로 정의
cohere_rerank = CohereRerank(api_key=COHERE_API_KEY, top_n=2)

# 어차피 리랭크로 재소팅할거니까 넉넉한 top_k값 부여
query_engine = index.as_query_engine(
    similarity_top_k=10,
    node_postprocessors=[cohere_rerank], # 여러개 넣을수 있음.
)

response = query_engine.query("좀비 영화")
response
```

## 리랭커 적용 후 추가 질의

* **목적:** 리랭커의 효과를 보다 분명히 체감하기 위해 **다른 질의**를 던져봅니다. 예시로 \*\*"가족 영화"\*\*라는 질의를 실행하여, 리랭커 적용 시 어떤 응답이 도출되는지 확인합니다. 이 질의는 다의적으로 해석될 수 있어, 리랭커 없이 단순 벡터 검색만으로는 최적의 답을 찾기 어려울 수 있습니다.
* **질의 의미:** "가족 영화"라는 말은 가족과 관련된 내용을 가진 영화 또는 가족 모두 함께 볼 수 있는 영화 등으로 해석될 수 있습니다. 데이터셋에 여러 후보가 있을 것이며, 벡터 유사도만으로는 막연하게 '가족'이 언급된 영화를 고를 수 있습니다. 예를 들어, 줄거리에 가족이 등장하는 \*\*<기생충>\*\*이나 **<엑시트>** 등이 후보로 고려될 수 있습니다.
* **리랭커의 역할:** Cohere 리랭커는 이런 후보들 중에서 **질문의 의도에 가장 부합하는 영화**를 상위로 올려줍니다. "가족 영화"라는 질문에 대해 리랭커는 단순히 '가족' 단어 유무 이상으로, 영화가 가족을 주제로 하거나 가족을 구하는 내용인지 등을 평가합니다. 그 결과 \*\*<엑시트>\*\*처럼 "가족을 구하기 위해 노력하는 이야기"가 나오는 영화를 최우선으로 선택하고, 비록 '가족'을 언급하지만 맥락이 다른 기생충 등의 후보는 순위를 낮춥니다.
* **결과 생성:** `query_engine.query("가족 영화")` 실행 시 응답으로 **"<엑시트>는 가족을 구하기 위해 탈출을 시도하는 내용을 담고 있는 영화로, 재난과 코미디 장르에 속합니다."** 와 같은 문장이 출력됩니다. 이는 리랭커가 선정한 최상위 영화 `<엑시트>`의 줄거리를 근거로 LLM이 작성한 답변입니다. 함께 반환된 `source_nodes`를 보면 `<엑시트>` 노드가 높은 점수로 포함되고, `<기생충>` 노드도 후보로 있었지만 낮은 점수로 밀려난 것을 확인할 수 있습니다.
* **성능 개선:** 이 예시를 통해 리랭커 적용 전후의 차이를 이해할 수 있습니다. 만약 리랭커가 없었다면 단순 벡터 검색으로는 `<기생충>`처럼 가족이라는 단어만 등장하는 영화를 골랐을지 모르지만, 리랭커를 사용하면 **질문의 숨은 의도**(가족을 구하는 영웅적 내용의 영화)에 더 맞는 결과(`<엑시트>`)가 부각됩니다. 이런 방식으로 RAG 파이프라인에 리랭킹 단계를 추가하면 최종 답변의 정확성과 관련성이 크게 향상될 수 있습니다.


```python
response = query_engine.query("가족 영화")
response
```
