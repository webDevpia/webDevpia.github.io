## 1. 왜 프롬프트를 평가해야 하는가 [↑](#toc)

**학습목표**: 프롬프트 평가가 필요한 이유를 설명하고, 체계적인 개선 사이클을 이해한다.

"잘 만든 것 같은데 정말 잘 만든 건가?"

프롬프트를 작성하면 바로 사용하고 싶어집니다. 하지만 **느낌이 좋다**는 것과 **실제로 잘 동작한다**는 것은 다릅니다.

### 평가가 필요한 세 가지 이유

| 이유 | 설명 | 예시 |
|------|------|------|
| 모델·버전 의존성 | 동일 프롬프트도 GPT-3.5와 GPT-4에서 결과가 다름 | GPT-3.5에서 통하던 지시가 GPT-4에서 무시됨 |
| 온도(temperature) 영향 | temperature=0과 1에서 창의성·일관성이 다름 | 같은 요약 프롬프트, 매번 다른 길이 |
| 엣지 케이스 | 일반적인 입력에서는 좋지만 특이한 입력에서 실패 | 영어로 물어보면 갑자기 영어로 답변 |

### 프롬프트 개선 사이클

```
작성 → 실행 → 평가 → 분석 → 수정 → 다시 실행
```

평가 없이 수정하면 "나아졌는지"를 알 수 없습니다. 측정 가능한 기준을 세우는 것이 핵심입니다.

**핵심**: "프롬프트의 품질도 측정하고 개선할 수 있다."

> **확인**: 지금까지 작성한 프롬프트 중 하나를 골라 보세요. 어떤 기준으로 "잘 만들었다"고 판단했나요?

## 2. 4가지 평가 기준 [↑](#toc)

**학습목표**: 4가지 평가 기준을 설명하고, 각 기준에 맞는 루브릭으로 프롬프트 결과를 채점할 수 있다.

### 평가 기준 테이블

| 기준 | 의미 | 예시 |
|------|------|------|
| 정확성(Fact Consistency) | 사실에 맞는가 | 환각(hallucination) 여부: 없는 날짜나 출처를 만들어내는가 |
| 일관성(Role Consistency) | 역할·톤을 유지하는가 | 선생님 역할인데 갑자기 반말을 하거나 역할을 벗어나는 답변 |
| 톤 적절성(Tone Fitting) | 대상에 맞는 말투인가 | 초등학생에게 설명하는데 학술 용어를 남용하는가 |
| 포맷 적합성(Format) | 요청한 형식인가 | JSON으로 달라고 했는데 일반 텍스트로 답하는가 |

### 1~5점 루브릭

각 기준을 1점(매우 나쁨)부터 5점(매우 좋음)으로 채점합니다.

**정확성 루브릭**

| 점수 | 기준 |
|------|------|
| 5 | 모든 사실이 정확하고, 불확실한 내용은 명시적으로 표현 |
| 4 | 대부분 정확하며 사소한 오류 1~2개 |
| 3 | 핵심 사실은 맞지만 부수적 정보에 오류 있음 |
| 2 | 여러 사실 오류 또는 검증되지 않은 정보 포함 |
| 1 | 주요 사실이 틀렸거나 환각이 명백히 포함됨 |

**포맷 적합성 루브릭**

| 점수 | 기준 |
|------|------|
| 5 | 요청한 형식을 완벽히 준수 (JSON 키 이름까지 일치 등) |
| 4 | 형식 대체로 맞으나 사소한 차이 (공백, 순서 등) |
| 3 | 형식이 부분적으로 맞음 |
| 2 | 형식을 거의 따르지 않음 |
| 1 | 요청과 전혀 다른 형식 |

**핵심**: 루브릭을 미리 정해두면 팀원끼리 동일한 기준으로 평가할 수 있고, 개선 전후 점수를 비교할 수 있습니다.

> **확인**: 직전 파트에서 만든 역할 기반 프롬프트 결과를 위 4가지 기준으로 채점해 보세요.

## 3. 자동 평가 맛보기 (LLM-as-a-Judge) [↑](#toc)

**학습목표**: LLM에게 평가자 역할을 부여하여 다른 응답을 자동으로 평가하는 코드를 작성할 수 있다.

사람이 일일이 채점하는 것은 시간이 많이 걸립니다. **LLM을 평가자로 활용**하면 대량의 응답을 빠르게 채점할 수 있습니다. 이 방식을 **LLM-as-a-Judge**라고 부릅니다.

### 평가 프롬프트 작성 → 점수 받기 → DataFrame 정리

```python
# ===================================
# LLM-as-a-Judge: 자동 평가 구현
# ===================================
import os
import json
import pandas as pd
from openai import OpenAI

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# 평가할 응답 목록 (앞 파트에서 생성한 역할별 응답을 사용해도 좋음)
평가_대상 = [
    {
        "role": "신입 교육 담당 사수",
        "question": "마이크로서비스 아키텍처가 뭔가요?",
    },
    {
        "role": "시니어 아키텍트",
        "question": "마이크로서비스 아키텍처가 뭔가요?",
    },
]

# 먼저 각 역할로 응답을 생성합니다
roles = {
    "신입 교육 담당 사수": "당신은 IT 회사에서 신입 개발자를 교육하는 3년차 개발자입니다. 쉬운 비유와 그림을 활용해 설명해주세요.",
    "시니어 아키텍트": "당신은 대기업 소프트웨어 아키텍트입니다. 정확한 기술 용어와 트레이드오프를 포함하여 설명해주세요.",
}

for 항목 in 평가_대상:
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": roles[항목["role"]]},
            {"role": "user", "content": 항목["question"]},
        ],
        temperature=0.7,
    )
    항목["응답"] = response.choices[0].message.content

# 평가자 시스템 프롬프트 정의
평가자_system = """당신은 프롬프트 응답 품질을 평가하는 전문 평가자입니다.
주어진 응답을 다음 4가지 기준으로 1~5점 채점하고, 반드시 JSON 형식으로만 출력하세요.

출력 형식:
{
  "정확성": <1~5 정수>,
  "일관성": <1~5 정수>,
  "톤적절성": <1~5 정수>,
  "포맷적합성": <1~5 정수>,
  "총평": "<한 문장 평가>"
}

채점 기준:
- 정확성: 기술적 사실 오류나 환각이 없는가
- 일관성: 주어진 역할(사수/아키텍트)의 톤을 유지하는가
- 톤적절성: 신입 개발자 또는 기술 리더에 맞는 수준인가
- 포맷적합성: 요청된 형식을 따르는가"""

결과_목록 = []

for 항목 in 평가_대상:
    # 평가 요청 메시지 구성
    평가_user = f"""다음 응답을 평가해 주세요.

[역할]: {항목['role']}
[질문]: {항목['question']}
[응답]: {항목['응답']}"""

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": 평가자_system},
            {"role": "user", "content": 평가_user},
        ],
        temperature=0,  # 평가는 일관성이 중요하므로 0으로 설정
        response_format={"type": "json_object"},  # JSON 출력 강제
    )

    # JSON 파싱
    평가_결과 = json.loads(response.choices[0].message.content)
    평가_결과["역할"] = 항목["role"]
    결과_목록.append(평가_결과)

    print(f"[{항목['role']}] 평가 완료")
    print(json.dumps(평가_결과, ensure_ascii=False, indent=2))
    print()

# DataFrame으로 정리
df = pd.DataFrame(결과_목록)
열_순서 = ["역할", "정확성", "일관성", "톤적절성", "포맷적합성", "총평"]
df = df[열_순서]

# 평균 점수 계산
점수_열 = ["정확성", "일관성", "톤적절성", "포맷적합성"]
df["평균점수"] = df[점수_열].mean(axis=1).round(2)

print("=" * 60)
print("평가 결과 요약")
print("=" * 60)
print(df.to_string(index=False))
```

### LLM-as-a-Judge의 장단점

| 항목 | 장점 | 단점 |
|------|------|------|
| 속도 | 수백 건도 수 분 내 처리 | - |
| 일관성 | 동일 기준 반복 적용 | 평가자 LLM 자체의 편향 포함 |
| 비용 | 사람 평가 대비 저렴 | 평가 호출도 토큰 소비 |
| 신뢰도 | 대략적인 품질 추세 파악 가능 | 미묘한 품질 차이 감지 한계 |

**핵심**: LLM-as-a-Judge는 완벽하지 않지만, 대량 비교나 1차 필터링에 매우 유용합니다. 중요한 결정에는 사람 검토를 병행해야 합니다.

> **확인**: 평가자 LLM에게 "항상 5점을 주는 역할"을 부여하면 어떤 일이 생길까요? 이것이 LLM-as-a-Judge의 어떤 한계를 보여주나요?
