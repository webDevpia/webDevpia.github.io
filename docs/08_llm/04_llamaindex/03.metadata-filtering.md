# 메타데이터 활용 RAG 고급 검색

## 1. LlamaIndex 및 Qdrant 패키지 설치

* **목적:** 검색-생성(RAG) 실습에 필요한 라이브러리를 설치합니다. LlamaIndex 라이브러리와 Qdrant 벡터스토어 연동 패키지를 미리 준비합니다.
* **사용 라이브러리:** `%pip install ...` Jupyter 매직 명령어를 통해 Python 환경에 필요한 패키지들을 설치합니다. `llama_index`는 LlamaIndex 본체이고, `llama-index-vector-stores-qdrant`는 Qdrant 벡터 저장소를 지원하는 부가 패키지입니다.
* **동작 방식:** 셀 실행 시 pip가 인터넷에서 해당 패키지를 내려받아 설치하거나, 이미 설치된 경우 "Requirement already satisfied"라는 메시지를 출력합니다.
* **기대 결과:** 해당 라이브러리들이 최신 버전으로 설치되어 이후 코드에서 `llama_index`와 Qdrant 연동 기능을 문제없이 사용할 수 있게 됩니다. 설치가 완료되면 Jupyter Notebook에는 패키지 설치 로그가 출력됩니다.


```python
%pip install llama_index llama-index-vector-stores-qdrant
```

## 2. 환경 변수 로드 및 API 키 설정

* **목적:** OpenAI API와 Qdrant 서비스의 인증 정보를 불러옵니다. 미리 `.env` 파일 등에 저장된 비공개 키를 코드에서 사용할 수 있도록 환경 변수에서 읽어오는 단계입니다.
* **사용 함수:** `dotenv` 라이브러리의 `load_dotenv()`를 호출하여 현재 작업 경로의 `.env` 파일에 있는 환경 변수를 로드합니다. 그런 다음 `os.getenv('키이름')`을 사용해 필요한 키 값을 가져옵니다.
* **동작 방식:** `.env` 파일에 저장된 `OPENAI_API_KEY`, `QDRANT_API_KEY`, `QDRANT_URL` 값을 읽어 각각 Python 변수에 할당합니다. `OPENAI_API_KEY`는 OpenAI 모델을 사용할 때, `QDRANT_API_KEY`와 `QDRANT_URL`은 Qdrant 벡터 데이터베이스에 연결할 때 필요합니다.
* **기대 결과:** 변수 `OPENAI_API_KEY`, `QDRANT_API_KEY`, `QDRANT_URL`가 올바르게 설정됩니다. 이로써 이후 OpenAI LLM 호출이나 Qdrant 서버 접속에 필요한 인증 정보가 준비됩니다. (이 셀 자체는 출력이 없고 정상 실행 시 아무 변화 없이 넘어갑니다.)


```python
from dotenv import load_dotenv
import os

load_dotenv()

OPENAI_API_KEY = os.getenv('OPENAI_API_KEY')
QDRANT_API_KEY = os.getenv('QDRANT_API_KEY')
QDRANT_URL = os.getenv('QDRANT_URL')
```

## 3. Qdrant 벡터 데이터베이스 클라이언트 설정

* **목적:** Qdrant 벡터스토어에 연결하기 위한 클라이언트를 초기화합니다. LlamaIndex에서 벡터 검색을 활용하기 위해 외부 벡터 DB(Qdrant)를 사용하므로, 먼저 Qdrant 서버와 통신할 수 있는 객체를 만들어 둡니다.
* **사용 클래스:** `qdrant_client.QdrantClient` – Python에서 Qdrant에 연결하고 명령을 보낼 수 있는 클라이언트 객체입니다.
* **동작 방식:** `QdrantClient(url=..., api_key=...)` 생성자에 앞서 불러온 `QDRANT_URL`과 `QDRANT_API_KEY`를 전달합니다. 이로써 `client` 변수에 Qdrant 인스턴스가 만들어지며, 이후 이 객체를 통해 벡터를 저장하거나 메타데이터 필터링 쿼리를 전송할 수 있습니다.
* **기대 결과:** `client` 변수에 Qdrant와의 연결이 설정됩니다. 이 셀에는 별도의 출력이 없지만, 내부적으로 `client`가 유효하게 생성되면 다음 단계에서 오류 없이 Qdrant를 사용할 수 있게 됩니다.


```python
from qdrant_client import QdrantClient

client = QdrantClient(
    url=QDRANT_URL,
    api_key=QDRANT_API_KEY
)
```

## 5. LLM 및 임베딩 모델 설정 (OpenAI API)

* **목적:** LlamaIndex가 사용할 언어 모델(LLM)과 텍스트 임베딩 모델을 지정하고, 문서 chunking(청크 분할) 전략을 설정합니다. 작은 모델을 사용해 실습 환경에서 응답 속도를 높이고 비용을 낮춥니다.
* **사용 클래스:** `OpenAI` (llama\_index.llms.openai)와 `OpenAIEmbedding` (llama\_index.embeddings.openai) – OpenAI의 GPT 및 임베딩 API를 호출하는 래퍼 클래스입니다.
* **설정 내용:** `Settings.llm`에 `OpenAI(temperature=0, model='gpt-4o-mini')`를 할당하여, 질의응답 시 사용할 LLM을 지정합니다. 예시로 `'gpt-4o-mini'`라는 작은 GPT-4 기반 모델명을 사용하고 temperature=0으로 답변의 일관성을 높입니다. `Settings.embed_model`에는 `OpenAIEmbedding(model="text-embedding-3-small")`을 지정하여 문서 벡터화를 위한 임베딩 모델을 설정합니다. 또한 `Settings.chunk_size = 500`과 `Settings.chunk_overlap = 50`으로 문서를 벡터화할 때 한 청크의 토큰 길이를 500으로, 청크 간 중복을 50으로 정하여 긴 문서 분할 방식을 정의합니다.
* **동작 방식:** LlamaIndex의 `Settings`는 전역 설정으로, 이후 Index 생성이나 QueryEngine 동작 시 자동으로 이 설정을 참조합니다. 이 단계에서 지정한 LLM과 임베딩 모델은 OpenAI API 키(`OPENAI_API_KEY`)를 통해 호출되며, 모델 이름에 따라 OpenAI 서버에서 해당 모델이 응답하게 됩니다.
* **기대 결과:** 이후 쿼리를 수행하거나 임베딩을 계산할 때, 위에서 정의한 작은 모델들을 사용하므로 빠르고 효율적으로 동작합니다. (이 셀도 출력은 없으며, 설정이 정상적으로 적용되면 이후 단계에서 모델 호출이 잘 이루어집니다.)


```python
from llama_index.llms.openai import OpenAI
from llama_index.embeddings.openai import OpenAIEmbedding
from llama_index.core import Settings

Settings.llm= OpenAI(temperature=0, model='gpt-4o-mini')
Settings.embed_model = OpenAIEmbedding(model="text-embedding-3-small")
Settings.chunk_size = 500
Settings.chunk_overlap = 50
```

## 6. Qdrant 벡터스토어에 인덱스 생성

* **목적:** 준비된 `nodes` 데이터를 Qdrant 벡터 데이터베이스에 색인(index)합니다. 이를 통해 이후 비슷한 질문에 대한 벡터 유사도 검색을 빠르게 수행할 수 있습니다.
* **사용 클래스:** `QdrantVectorStore`와 `VectorStoreIndex` 등을 사용합니다. `QdrantVectorStore(client, collection_name)`는 주어진 Qdrant 클라이언트를 이용해 특정 컬렉션에 벡터 데이터를 저장/검색하는 객체이고, `VectorStoreIndex(nodes, storage_context=...)`는 주어진 노드 리스트를 벡터화하여 벡터스토어에 색인하는 LlamaIndex의 인덱스 클래스입니다.
* **동작 과정:** 먼저 `vector_store = QdrantVectorStore(client=client, collection_name="movies")`로 Qdrant 내 "movies"라는 컬렉션(테이블 비슷한 개념)에 연결되는 벡터스토어를 생성합니다. `StorageContext.from_defaults(vector_store=vector_store)`로 저장소 컨텍스트 객체를 만들고, `index = VectorStoreIndex(nodes, storage_context=storage_context)`를 실행합니다. 이 마지막 단계에서 `nodes`의 각 텍스트를 임베딩하고 Qdrant "movies" 컬렉션에 저장합니다.
* **기대 결과:** Qdrant 서버에 "movies" 컬렉션이 생성되고, 10개 영화 노드 각각이 벡터로 변환되어 저장됩니다. LlamaIndex의 `index` 객체에는 데이터에 대한 색인이 구축되어, 유사도 검색이나 질문 응답을 수행할 준비가 완료됩니다. (이 셀도 출력은 없지만, 내부적으로 임베딩 계산 및 데이터 적재가 수행됩니다.)


```python
from llama_index.vector_stores.qdrant import QdrantVectorStore
from llama_index.core import VectorStoreIndex, StorageContext

vector_store = QdrantVectorStore(client=client, collection_name="movies")
storage_context = StorageContext.from_defaults(vector_store=vector_store)
index = VectorStoreIndex(nodes, storage_context=storage_context)
```

## 7. 기본 Retriever로 유사도 검색 테스트

* **목적:** 메타데이터 필터링을 사용하지 않은 "순수 유사도 검색" 결과를 확인합니다. 사용자가 묻는 조건을 반영하지 않고 임베딩 유사도로만 관련 문서를 찾으면 어떤 한계가 있는지 살펴보기 위한 단계입니다. 예시 질문은 *“2018년 이전에 개봉한 봉준호 감독의 영화는?”* 입니다.
* **코드 설명:** `naive_retriever = index.as_retriever(similarity_top_k=3)`는 `index`로부터 단순 벡터 유사도 기반 Retriever를 생성합니다. `similarity_top_k=3`으로 가장 가까운 3개의 벡터(노드)를 가져오도록 설정했습니다. 그 다음 `nodes = naive_retriever.retrieve("2018년 이전에 개봉한 봉준호 감독의 영화는?")`로 질의를 던져 3개의 후보 노드를 받아옵니다. 마지막으로 `for node in nodes: print(node.text); print(node.metadata); print(node.score)` 부분에서 각 결과 노드의 내용을 출력합니다. `node.text`는 영화 시놉시스 텍스트, `node.metadata`는 해당 영화의 메타데이터 딕셔너리, `node.score`는 질의와 해당 노드 간 유사도 점수입니다.
* **동작 방식:** Retriever는 질의 문장을 임베딩 벡터로 변환한 후, 코사인 유사도 등을 사용해 모든 영화 노드 중 가장 비슷한 3개를 찾습니다. 이 방식은 질의에 언급된 조건("2018년 이전", "봉준호 감독") 등을 정확히 이해하지 않고, 질의 문장과 시놉시스 간의 단어/의미 유사성만으로 결과를 냅니다.
* **관찰 결과:** 출력된 3개의 영화는 **봉준호 감독의 2018년 이전 개봉작이 아닐 수 있습니다.** 실제로 결과를 보면 봉준호 감독과 무관한 영화나 2018년 이후의 영화도 섞여 있을 수 있습니다. 예를 들어 `'밀정'(2016, 김지운 감독)` 등이 출력될 수 있습니다. 이는 임베딩 유사도만으로 검색하면 질문의 구체 조건(감독, 년도)을 만족하는 결과를 보장하지 못함을 보여줍니다. 이러한 한계를 학생들이 인지하도록 돕는 예시입니다.


```python
naive_retriever = index.as_retriever(similarity_top_k=3)
nodes = naive_retriever.retrieve("2018년 이전에 개봉한 봉준호 감독의 영화는?")
for node in nodes:
    print(node.text)
    print(node.metadata)
    print(node.score)
```

## 8. 기본 Query Engine으로 LLM 응답 생성 테스트

* **목적:** 메타데이터 필터 없이 유사도 검색 + LLM 조합으로 질문에 답하게 해보고, 그 품질을 살펴봅니다. 바로 앞에서와 동일한 질문 \*“2018년 이전에 개봉한 봉준호 감독의 영화는?”\*에 대해 Query Engine을 사용해 봅니다.
* **코드 설명:** `naive_query_engine = index.as_query_engine(similarity_top_k=3)`는 LlamaIndex로부터 간단한 질의엔진을 생성합니다. 이 엔진은 내부적으로 위에서처럼 3개의 유사한 노드를 찾고, 그 내용을 바탕으로 LLM이 최종 답변을 만들도록 합니다. `naive_query_engine.query(질문)`을 호출하면 그 결과를 바로 출력하거나 `Response` 객체로 반환합니다.
* **동작 방식:** Query Engine은 먼저 Retriever와 동일하게 임베딩 유사도 검색으로 상위 3개 노드를 찾습니다. 이후, OpenAI LLM (`Settings.llm`에 설정된 모델)을 사용해 이 노드들의 내용을 참고하여 질문에 대한 답변을 생성합니다.
* **예상 동작:** 이전 셀에서 본 것처럼 Retriever 단계에서 이미 관련 없는 노드들이 선택되었으므로, LLM은 적절한 정보를 찾지 못합니다. **LLM의 반응은 질문에 답을 못 찾았다는 취지**가 될 수 있습니다. 실제 실행 시, 응답으로 *“해당 정보에는 봉준호 감독의 영화에 대한 내용이 포함되어 있지 않습니다.”* 와 같이, 주어진 자료에서 답을 찾을 수 없다는 결과를 내놓았습니다. 이는 LLM이 참고한 노드들에 봉준호 감독의 2018년 이전 영화 정보가 없기 때문입니다.
* **통찰:** 이 과정을 통해 학생들은 단순 벡터 검색으로는 질문의 제약조건이 반영되지 않아 엉뚱한 문서들이 선택될 수 있고, 아무리 LLM을 써도 적절한 답을 얻기 어렵다는 점을 확인하게 됩니다. 이는 다음 단계에서 **메타데이터 필터링**을 도입해야 할 필요성을 보여줍니다.


```python
naive_query_engine = index.as_query_engine(similarity_top_k=3)
naive_query_engine.query("2018년 이전에 개봉한 봉준호 감독의 영화는?")
#llamaindex에서 정보가 없을경우 없다고 답변하도록 프롬프트 템플릿에 정의되어 있음.
```

## 9. 메타데이터 정보 정의 및 AutoRetriever 생성

* **목적:** **메타데이터 필터링이 가능한 고급 Retriever**를 설정합니다. LLM이 질의를 해석해 자동으로 메타데이터 조건을 추출하도록 도와주는 정보를 제공하고, 이를 활용하는 `VectorIndexAutoRetriever`를 생성합니다.
* **메타데이터 스키마 정의:** `VectorStoreInfo` 객체를 만들어 인덱스의 내용과 메타데이터 구조를 설명합니다. `content_info="영화 제목과 간략한 시놉시스"`라고 적어둔 부분은 문서 본문의 내용이 무엇인지를 알려주는 설명입니다. 이어서 `metadata_info=[...]` 리스트에는 각 메타데이터 필드의 정보를 `MetadataInfo` 형태로 전달합니다. 예를 들어 `MetadataInfo(name="title", description="영화의 제목", type="string")`은 "title" 필드가 영화 제목이라는 문자열이라는 뜻이고, `MetadataInfo(name="year", description="영화가 개봉한 연도", type="integer")`는 "year" 필드가 개봉연도(정수)임을 나타냅니다. 이처럼 장르(genre), 감독(director), 평점(rating) 등 모든 메타데이터에 대해 이름, 설명, 타입을 명시했습니다. 이 설명들은 **질문을 해석하는 LLM에게 힌트 역할**을 합니다.
* **AutoRetriever 생성:** `auto_retriever = VectorIndexAutoRetriever(index, vector_store_info=vector_store_info, verbose=True)`로 자동 메타데이터 필터링을 지원하는 Retriever를 만듭니다. 기존 `index`와 우리가 정의한 `vector_store_info`를 연결하며, `verbose=True`로 설정하여 나중에 쿼리할 때 **LLM이 추출한 필터 조건을 콘솔에 출력**하도록 합니다.
* **동작 방식:** `VectorIndexAutoRetriever`는 내부적으로 프롬프트를 이용해 OpenAI LLM에게 질의를 해석시키는 과정을 거칩니다. 우리가 정의한 메타데이터 정보(`vector_store_info`)를 프롬프트에 포함시켜, LLM이 질문에서 "감독=봉준호", "year<2018" 등의 조건을 JSON 형태로 뽑아내게 합니다. 그런 다음, 이렇게 얻은 구조화된 질의(텍스트 질의 + 필터 조건)를 사용해 벡터스토어에서 검색을 수행합니다.
* **기대 효과:** 이제 사용자의 자연어 질문에 담긴 조건들이 무시되지 않고 반영될 준비가 되었습니다. 이 셀은 출력이 없지만, 다음 단계에서 `auto_retriever`를 사용하여 질의를 던지면 verbose 모드 덕분에 **LLM이 인식한 필터 조건**을 출력에서 확인할 수 있습니다. 학생들은 이로써 "자동 질의 분석 + 필터 적용"이라는 고급 RAG 검색 전략의 핵심 개념을 접하게 됩니다.


```python
from llama_index.core.vector_stores import MetadataInfo, VectorStoreInfo
from llama_index.core.retrievers import VectorIndexAutoRetriever

vector_store_info = VectorStoreInfo(
    content_info="영화 제목과 간략한 시놉시스",
    metadata_info=[
        MetadataInfo(
            name="title",
            description="영화의 제목",
            type="string",
        ),
        MetadataInfo(
            name="year",
            description="영화가 개봉한 연도",
            type="integer",
        ),
        MetadataInfo(
            name="genre",
            description="영화의 쟝르",
            type="list",
        ),
        MetadataInfo(
            name="director",
            description="영화의 감독 이름",
            type="string",
        ),
        MetadataInfo(
            name="rating",
            description="영화의 평점 (1-10)",
            type="float",
        ),
    ],
)
# 메타 정보를 이용해서 메타 검색이 가능하도록 만들어준다.
auto_retriever = VectorIndexAutoRetriever(
    index, vector_store_info=vector_store_info, verbose=True
)
```

## 10. Qdrant 컬렉션에 메타데이터 인덱스 생성

* **목적:** Qdrant 벡터 데이터베이스에서 메타데이터로 필터링 쿼리를 효율적으로 수행하려면, 각 필드에 인덱스를 생성해주는 것이 좋습니다. 이 셀에서는 Qdrant에 "director", "year", "genre", "rating", "title" 메타데이터 필드에 대한 인덱스를 만들어 검색 최적화를 합니다.
* **사용 코드:** `client.create_payload_index(collection_name, field_name, field_schema)` 메소드를 반복 호출합니다. `indexes_to_create` 딕셔너리에 필드 이름과 타입을 정의해 두고, 루프를 돌면서 Qdrant에 인덱스를 추가합니다. 여기서 `models.PayloadSchemaType.KEYWORD`는 문자열 검색을 위한 인덱스 타입(예: 감독, 제목 등), `INTEGER`, `FLOAT` 등은 숫자 필드를 위한 타입입니다. `genre`의 경우 여러 값이 들어갈 수 있으므로 KEYWORD 타입으로 지정했습니다.
* **동작 방식:** 코드가 실행되면 `for` 루프를 통해 다섯 개 필드 각각에 대해 Qdrant 서버에 인덱스 생성 요청이 갑니다. 성공적으로 생성될 때마다 `print(f"인덱스가 추가되었습니다: {field_name}")`가 호출되어 어떤 필드가 인덱싱되었는지 출력합니다.
* **기대 결과:** 출력으로 다섯 줄의 메시지(예: *"인덱스가 추가되었습니다: director"* 등)가 표시됩니다. 이로써 Qdrant 쿼리 시 `"director" == 봉준호`나 `"year" < 2018` 같은 필터 조건의 성능이 향상됩니다. (메타데이터 필터 자체는 인덱스 없이도 동작은 하지만, 인덱스가 있으면 대량 데이터에서도 검색이 빠르고 정확하게 이뤄집니다.) 학생들은 이 단계를 통해 외부 벡터DB에서도 메타데이터 필드를 따로 색인할 필요가 있음을 배울 수 있습니다.


```python
from qdrant_client.http import models

indexes_to_create = {
    "director": models.PayloadSchemaType.KEYWORD,
    "year": models.PayloadSchemaType.INTEGER,
    "genre": models.PayloadSchemaType.KEYWORD,
    "rating": models.PayloadSchemaType.FLOAT,
    "title": models.PayloadSchemaType.KEYWORD,
}

collection_name = "movies"

for field_name, schema_type in indexes_to_create.items():
    client.create_payload_index(
        collection_name=collection_name,
        field_name=field_name,
        field_schema=schema_type,
    )
    print(f"인덱스가 추가되었습니다: {field_name}")
```

## 11. 자동 Retriever로 필터링 검색 테스트

* **목적:** 메타데이터 기반 검색이 잘 이뤄지는지 확인하는 첫 질의 테스트입니다. 앞서 실패했던 질문 \*“2018년 이전에 개봉한 봉준호 감독의 영화는?”\*을 다시 시도합니다. 이번에는 `auto_retriever`를 사용하므로, **LLM이 질의를 구조화하여 적절한 필터를 적용**할 것입니다.
* **실행 코드:** `auto_retriever.retrieve("2018년 이전에 개봉한 봉준호 감독의 영화는?")`
* **예상 동작:** verbose=True 설정으로 인해, 콘솔에 LLM이 이해한 질의 구조를 출력합니다. 여기서는 `Using query str: 봉준호 감독의 영화`와 `Using filters: [('year', '<', 2018), ('director', '==', '봉준호')]` 같은 메시지가 나타납니다. 이는 LLM이 질문을 분석하여 **검색용 텍스트**(질문에서 키워드만 추린 내용)와 **필터 조건** 두 가지를 생성했음을 의미합니다. 즉, "봉준호 감독의 영화"를 키워드로 벡터 유사도 검색을 하고, 그 중에서 `year < 2018` 그리고 `director == 봉준호` 조건에 맞는 노드만 고려하도록 자동 설정된 것입니다.
* **검색 결과:** 필터가 적용되었으므로, **2018년 이전 개봉 & 감독이 봉준호**인 영화만 결과로 나오게 됩니다. 우리 예시 데이터에서 해당 조건에 부합하는 영화는 *옥자(2017년, 봉준호 감독)* 하나뿐입니다. 따라서 반환된 노드 리스트에는 옥자만 있고, `NodeWithScore` 형태로 출력됩니다. 옥자의 메타데이터(제목: 옥자, year: 2017, director: 봉준호 등)도 함께 확인할 수 있습니다.
* **비교 및 의의:** 이전의 기본 검색 셀과 달리, 이제는 정확하게 조건을 만족하는 영화만 찾아낸 것을 볼 수 있습니다. 학생들은 이를 통해 **메타데이터 필터링의 효과**를 직접 보게 됩니다. 즉, 질의의 의도를 이해한 LLM이 벡터 검색과 결합되어, 단순 유사도 검색보다 훨씬 관련성 높은 결과를 얻게 된 것입니다.


```python
auto_retriever.retrieve("2018년 이전에 개봉한 봉준호 감독의 영화는?")
```

## 12. 자동 Retriever로 복수 조건 검색

* **목적:** 자동 메타데이터 필터링이 여러 조건을 함께 처리할 수 있음을 보여줍니다. 이번 질의는 \*“2020년 이전에 개봉한 봉준호 감독의 영화는?”\*으로, 봉준호 감독 + 2020년 이전이라는 조건입니다. (2018년이 아니라 2020년으로 바꿔 본 것은 더 많은 결과를 얻어보기 위함입니다.)
* **실행 코드:** `auto_retriever.retrieve("2020년 이전에 개봉한 봉준호 감독의 영화는?")`
* **동작 방식:** 내부적으로 LLM이 `year < 2020`과 `director == 봉준호` 필터를 추출합니다. verbose 출력으로 `Using filters: [('year', '<', 2020), ('director', '==', '봉준호')]`가 나타나고, 검색 쿼리 문자열은 `"봉준호 감독의 영화"`로 나옵니다.
* **결과 확인:** 데이터상 봉준호 감독의 2020년 이전 개봉작은 두 편 있습니다: \*옥자(2017)\*와 *기생충(2019)*. AutoRetriever는 이 둘을 모두 찾아 리스트로 반환할 것입니다. 출력된 `NodeWithScore` 목록을 보면 기생충과 옥자의 메타데이터가 모두 포함되어 있을 것입니다. 두 노드 각각 유사도 점수와 함께 나열되며, 결과 개수가 최대 2개인 것을 확인할 수 있습니다.
* **의미:** 메타데이터 필터가 여러 개 동시에 잘 적용됨을 보여줍니다. 논리적으로 year<2020 AND director=Bong Joon-ho 조건이 만족되는 항목만 추려졌고, 그 외 봉준호 감독이 아닌 영화나 2020년 이후 영화는 배제되었습니다. 학생들은 이를 통해 **복합 조건 필터링도 자동화**되었음을 이해할 수 있습니다.


```python
auto_retriever.retrieve("2020년 이전에 개봉한 봉준호 감독의 영화는?")
```

## 13. 자동 Retriever로 다중 메타데이터 조건 검색 

* **목적:** 메타데이터 필터링이 문자열 뿐만 아니라 리스트 타입 필드(장르)도 처리할 수 있음을 보여줍니다. 이번 질의 \*“2020년 이후 개봉한 박찬욱 감독의 멜로 장르의 영화는?”\*에는 세 가지 조건이 포함되어 있습니다: (1) 감독=박찬욱, (2) 개봉년도 > 2020, (3) 장르=멜로.
* **실행 코드:** `auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 멜로 쟝르의 영화는?")`
* **LLM의 질의 해석:** verbose 로그로 `Using filters: [('year', '>', 2020), ('director', '==', '박찬욱'), ('genre', '==', '멜로')]`가 출력됩니다. LLM이 자연스럽게 "이후"를 `>` 연산자로, "박찬욱"을 director 필터로, "멜로"를 genre 필터로 인식한 것입니다. 검색용 query 문자열은 `"박찬욱 감독의 멜로 쟝르의 영화"`로 표시되어, 이 키워드들을 중심으로 유사도 검색이 이뤄집니다.
* **결과:** 예시 데이터에서 박찬욱 감독의 2020년 이후 멜로 장르 영화는 \*헤어질 결심(2022년, 장르: 멜로/스릴러)\*이 유일합니다. 따라서 결과로 해당 영화 노드 하나가 반환됩니다. Node 내용을 보면 title이 "헤어질 결심", year 2022, director 박찬욱, genre에 "멜로"가 포함된 것을 확인할 수 있습니다.
* **의의:** 세 가지의 복잡한 조건도 정확히 걸러낸다는 점을 보여줍니다. 특히 genre와 같은 리스트 필드도 `"멜로"`라는 값이 포함되어 있는지 필터링이 적용되었습니다. 학생들은 자연어 질문에 포함된 다양한 조건들이 LLM에 의해 정확히 추출되고, 원하는 결과만 얻는 흐름을 볼 수 있습니다. 이 셀까지의 실습을 통해 **메타데이터 기반 검색의 강력함**을 체감할 수 있습니다.


```python
auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 멜로 쟝르의 영화는?")
```

## 14. 자동 Retriever로 존재하지 않는 조건 검색

* **목적:** 데이터에 없는 조건의 경우 검색 결과가 어떻게 나타나는지 살펴봅니다. 질문은 \*“이병헌 감독의 코미디 장르의 영화는 어떤 것이 있나요?”\*로, 우리 데이터상 **존재하지 않는 감독 이름**(이병헌은 우리 예시에서 감독으로 등장하지 않음)을 묻고 있습니다.
* **실행 코드:** `auto_retriever.retrieve("이병헌 감독의 코미디 쟝르의 영화는 어떤 것이 있나요?")`
* **LLM의 필터 추출:** verbose 모드에 따르면 `Using filters: [('director', '==', '이병헌')]`가 출력되었습니다. 여기서 `"코미디 쟝르의 영화"`라는 부분은 query 문자열로만 취급되고, genre 필터가 추출되지 않았음을 알 수 있습니다. (참고: 메타데이터 설명에 "영화의 쟝르"라고 써 두었지만 LLM이 `"코미디"`를 genre 필터로 인식하지 못한 것으로 보입니다. 이는 '쟝르'라는 표기를 LLM이 혼동했거나, 추가 맥락이 필요해서인 듯 합니다. 다음 단계에서 이를 개선할 예정입니다.)
* **검색 결과:** director가 "이병헌"인 영화는 데이터에 없으므로, 필터를 적용한 후 결과 노드가 전혀 없게 됩니다. 따라서 반환값은 빈 리스트이며, Jupyter 출력에는 `[]` 또는 "Empty"와 같은 표시가 나옵니다. 이 경우 LLM을 사용한 QueryEngine이라면 "해당 조건의 영화를 찾을 수 없습니다."라는 식의 응답을 할 수도 있지만, 여기서는 Retriever 단계만 수행했으므로 단순 빈 결과만 확인됩니다.
* **의미:** 메타데이터 필터링 과정에서 **질문에 해당하는 데이터가 없을 경우**를 보여줍니다. 학생들은 RAG 파이프라인에서 이처럼 결과가 비어 있을 수 있고, 이를 코드에서 처리하거나 사용자에게 알릴 필요가 있음을 깨닫게 됩니다. 또한 이 예시를 통해 LLM의 질의 해석이 완벽하지 않을 수 있다는 점도 드러났습니다(genre 필터 누락). 다음 단계에서는 이러한 LLM 프롬πτ 해석을 개선하는 방법을 다룹니다.


```python
auto_retriever.retrieve("이병헌 감독의 코미디 쟝르의 영화는 어떤 것이 있나요?")
```

## 15. 자동 Retriever 필터 해석 한계 확인 

* **목적:** LLM의 질의 해석이 다소 불완전할 수 있음을 확인하는 추가 예시입니다. 질문 \*“2020년 이후 개봉한 박찬욱 감독의 액션 영화는?”\*을 테스트합니다. 우리 데이터에 **박찬욱 감독의 액션 장르 영화는 없습니다**. (`헤어질 결심`은 박찬욱 감독이지만 장르가 멜로/스릴러이고, 액션 장르는 아님.) 이 경우 AutoRetriever가 genre 필터를 어떻게 처리하는지 살펴봅니다.
* **실행 코드:** `auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 액션 영화는?")`
* **LLM의 필터 추출:** verbose 출력에 `Using filters: [('year', '>', 2020), ('director', '==', '박찬욱')]`라고 나타납니다. 주목할 부분은 genre 필터가 보이지 않는다는 것입니다. LLM이 `"액션 영화"`라는 표현을 단순 키워드로 간주하고, 필터로 추출하지 못했습니다. (아마 이전 셀의 "코미디"와 같은 이슈로, '액션'이라는 단어를 genre 필드로 연결 짓지 않은 듯 합니다.) 따라서 query 문자열로 "액션 영화"를 남겨두고, 실제 필터링 조건에는 감독과 연도만 사용되었습니다.
* **검색 및 결과:** 필터 조건상 2020년 이후 박찬욱 감독 영화면 `헤어질 결심(2022)`이 후보가 됩니다. AutoRetriever는 이를 찾아 반환할 것입니다. 그러나 이 영화는 액션 장르가 아님에도 결과에 포함되었죠. 즉, 사용자는 "액션 영화"를 물었지만 시스템은 이를 필터로 걸러내지 못해, 엉뚱하게도 멜로 영화인 헤어질 결심을 제공한 셈입니다.
* **교훈:** 이 예에서는 AutoRetriever가 genre 관련 단어를 필터로 인식하지 못한 한계를 보여줍니다. 학생들은 LLM 기반 질의 해석이 때로는 불완전할 수 있고, 이를 개선하기 위해 프롬프트를 다듬어야 함을 배웁니다. 다음 단계에서 우리는 **추가 지시문을 주어** 이런 상황을 바로잡는 방법을 실습합니다.


```python
auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 액션 영화는?") # 장르가 필터로 들어가지 않음.
auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 액션 장르의 영화는?")
```

## 16. AutoRetriever 내부 프롬프트 내용 확인

* **목적:** AutoRetriever가 질의를 메타데이터 필터로 변환하기 위해 사용하는 **프롬프트 템플릿**을 살펴봅니다. LLM이 어떤 지침을 받아 질의를 분석하는지 학생들에게 보여주어, 메타데이터 필터링이 작동하는 원리를 이해시키고자 합니다.
* **사용 함수:** `auto_retriever.get_prompts()` 메소드는 AutoRetriever가 내부적으로 사용하는 모든 프롬프트를 딕셔너리로 반환합니다. `display_prompt_dict(prompts_dict)`는 이러한 프롬프트를 보기 좋게 포맷팅하여 출력해 줍니다. (출력 형태는 Markdown으로 렌더링되며, Jupyter에서 rich output으로 나타납니다.)
* **동작 방식:** `prompts_dict = auto_retriever.get_prompts()`로 프롬프트들을 가져오고, `display_prompt_dict(prompts_dict)`가 이를 시각화합니다. 특히 우리의 관심은 `"prompt"` 키에 해당하는 **질의 구조화 프롬프트**입니다. 출력 결과에는 "**Prompt Key: prompt**"라는 제목 아래에 LLM에게 주어지는 지시문 텍스트가 표시됩니다.
* **프롬프트 내용:** 이 프롬프트에는 Structured Query의 스키마, 사용 예시 등이 들어 있습니다. 예를 들어,

  * *사용자 질문을 JSON 형태의 구조화된 요청으로 변환하라*,
  * *필터에 들어갈 조건들은 데이터 소스의 속성에 해당하는 것만 추출하라*,
  * *만약 필터로 뽑아낼 조건이 없으면 빈 리스트로 두라*,
  * *예시:* 노래 데이터, 책 데이터에 대한 질의 변환 샘플 등이 포함되어 있습니다.
    학생들은 출력된 긴 프롬프트 내용을 통해, AutoRetriever가 LLM에게 질의 해석을 부탁할 때 어떤 맥락과 규칙을 주는지 확인할 수 있습니다. 이러한 프롬프트 설계 덕분에 LLM이 질의에서 `year < 2020` 같은 조건을 알아낼 수 있었던 것입니다.
* **의의:** 이 셀을 통해 RAG의 **프롬프트 엔지니어링 측면**을 학습합니다. 메타데이터 필터링을 위해서는 LLM에게 단순히 질문만 던지는 것이 아니라, 위와 같은 복잡한 지시를 보내 구조화된 출력(필터 포함)을 얻어내고 있었음을 보여줍니다.


```python
from llama_index.core.prompts import display_prompt_dict

prompts_dict = auto_retriever.get_prompts()
display_prompt_dict(prompts_dict)
```

## 17. 프롬프트 템플릿 변수 확인

* **목적:** 앞서 확인한 프롬프트 템플릿에서 사용되는 **변수들**을 살펴봅니다. 프롬프트는 고정된 문자열이 아니라, 상황에 따라 내용이 채워지는 부분이 있는데, 그 자리표시자의 이름들을 확인하는 과정입니다.
* **실행 코드:** `prompts_dict["prompt"].template_vars`
* **동작 방식:** AutoRetriever의 prompt는 LlamaIndex 내부에서 `PromptTemplate` 클래스로 관리됩니다. `template_vars` 속성을 출력하면 그 프롬프트에서 자리표시자로 쓰이는 변수 이름들이 리스트로 나옵니다.
* **출력 결과:** `['schema_str', 'info_str', 'query_str']` 라는 리스트가 보입니다. 이는 `schema_str`, `info_str`, `query_str`라는 3개의 변수가 있음을 뜻합니다.

  * `schema_str` : JSON 구조의 스키마를 나타내는 문자열 (예: `{"query": ..., "filters": [...], "top_k": ...}` 형태 정의)
  * `info_str` : 우리가 앞에서 정의한 `VectorStoreInfo`의 내용, 즉 메타데이터 정보와 content 설명이 문자열로 들어갑니다.
  * `query_str` : 사용자가 던진 실제 질문 텍스트입니다.
* **활용:** 이 변수가 실제 프롬프트 출력 내용에 어딘가 `{schema_str}`, `{info_str}`, `{query_str}`로 삽입되며, LLM에게 맥락을 제공합니다. 학생들은 이 변수 목록을 통해 프롬프트가 **어떤 구성 요소들로 이루어지는지** 더 명확히 이해할 수 있습니다. (예: info\_str이 곧 우리가 메타데이터 설명으로 입력한 부분이구나 등을 깨닫게 됩니다.)
* **다음 단계:** 이제 우리는 이 프롬프트에 **custom\_instructions**라는 추가 변수를 넣어 LLM 지시를 보강하려 합니다. 현재 template\_vars에는 없지만, 사용자 정의 지시를 추가하면 변수 목록이 바뀔 것입니다.


```python
prompts_dict["prompt"].template_vars
```

## 18. 사용자 정의 프롬프트 템플릿 작성

* **목적:** LLM의 질의 해석 능력을 향상시키기 위해 프롬프트에 추가 지침을 넣습니다. 특히 이전에 genre 필터를 놓친 문제(예: "액션 영화"를 필터로 인식 못함)를 해결하기 위해, 그런 경우를 다뤄주는 문구를 추가합니다.

* **작성 방식:** 새로운 프롬프트 문자열 `custom_prompt_template_str`를 다중라인 문자열(`""" ... """`)로 정의합니다. 이 문자열은 기존 AutoRetriever 프롬프트 템플릿을 기반으로 하지만, 그 안에 `{custom_instructions}`라는 부분을 마련해두고, 우리가 원하는 추가 지침을 삽입할 수 있게 만들 것입니다.

* **프롬프트 내용 구성:** 작성한 문자열을 보면, 기본 구조는 LlamaIndex가 쓰던 것과 유사합니다:

  * 목표 설명 (“사용자의 질의를 구조화된 요청으로 변환하라”),
  * 구조화된 요청 스키마 안내,
  * 필터와 query\_str 생성 시의 유의사항 (예: 쓸데없는 조건 넣지 말 것, 데이터 속성에 있는 조건만 필터링할 것 등),
  * 여러 개의 **예시**(Example 1, 2, 3)로, 다양한 도메인에서 질문을 JSON으로 변환하는 샘플들이 포함되어 있습니다.
    마지막 부분에

  ```
  User Query:
  {query_str}

  Custom Instructions:
  {custom_instructions}

  Structured Request:
  ```

## 19. 사용자 정의 PromptTemplate 객체 생성

* **목적:** 앞서 문자열로 작성한 `custom_prompt_template_str`을 LlamaIndex가 이해할 수 있는 `PromptTemplate` 객체로 변환합니다.
* **사용 클래스:** `PromptTemplate` (llama\_index.core). 이 클래스는 내부에 프롬프트 문자열과 변수들을 관리하고, LLM 호출 시 적절히 채워진 프롬프트를 만들어주는 역할을 합니다.
* **코드 설명:** `prompt_template = PromptTemplate(custom_prompt_template_str)`
* **동작 방식:** `PromptTemplate` 생성자에 우리가 만든 멀티라인 문자열을 넘기면, 해당 문자열 내에 `{...}` 형태로 표기된 변수들을 인식하여 template 객체를 만듭니다. 현재 `custom_prompt_template_str`에는 `{schema_str}`, `{info_str}`, `{query_str}`, `{custom_instructions}` 등이 포함되어 있을 것입니다.
* **기대 결과:** 이 셀 자체는 출력이 없지만, `prompt_template` 변수에 우리가 정의한 맞춤 프롬프트가 담기게 됩니다. 이 상태에서는 아직 `custom_instructions` 변수가 비어 있는 템플릿입니다. 다음 단계에서 이 변수에 실제 값을 채울 예정입니다.
* **확인:** 제대로 생성됐는지 확인하려면 `prompt_template.template_vars`를 볼 수 있지만, 곧 다음 셀에서 부분적으로 포맷팅하면서 확인할 것이므로 여기서는 넘어갑니다. 학생들은 이 과정을 통해 LlamaIndex 프롬프트를 직접 생성/수정할 수 있음을 학습합니다.


```python
from llama_index.core import PromptTemplate

prompt_template = PromptTemplate(custom_prompt_template_str)
```

## 20. 프롬프트에 추가 지시사항 적용

* **목적:** 커스텀 프롬프트에 **장르 키워드 처리에 대한 특별 지시**를 삽입합니다. 그리고 해당 지시가 잘 들어갔는지 변수 목록을 다시 확인합니다.
* **추가 지시사항 작성:** `custom_instructions` 문자열에 한국어로 다음과 같은 내용을 넣습니다: *"‘장르’라는 단어 없이 액션, 멜로, 스릴러 등과 같은 장르에 대한 표현이 입력되면, genre로 필터링을 해야 합니다."*
  이 문장은 LLM에게, 사용자가 질문에 "액션 영화"처럼 말했더라도 '액션'을 genre 필터로 간주하라는 명시적인 지침입니다. 즉, 장르를 나타내는 단어가 나오면 query\_str로만 두지 말고, filters에 genre 조건을 추가하라는 뜻이죠.
* **PromptTemplate 부분 포맷:** `prompt_template = prompt_template.partial_format(custom_instructions=custom_instructions)`는 Template에 일부 변수를 채워넣는 기능입니다. 여기서는 `custom_instructions` 자리에 우리가 방금 작성한 문자열을 삽입합니다. 이로써 `prompt_template` 객체는 custom\_instructions 변수를 가진 버전에서 그 변수가 고정된 새로운 템플릿이 됩니다. (나머지 변수들은 아직 남아 있음.)
* **변수 목록 확인:** 마지막에 `prompt_template.template_vars`를 출력하여 현 시점의 템플릿에 어떤 변수가 남아있는지 봅니다. 출력으로 `['schema_str', 'info_str', 'query_str', 'custom_instructions']`가 보입니다. 이 리스트에는 `custom_instructions`가 포함되어 있는데, 이는 partial\_format을 잘못 이해할 수 있지만, 사실 `partial_format` 후에도 template\_vars에 custom\_instructions가 남아있는 모양입니다. (원래 기대는 custom\_instructions가 채워졌으니 template\_vars에서 사라질 것 같지만, 출력에 남아있어 다소 혼동될 수 있습니다.)
* **효과:** 중요한 것은 우리가 원하는 지시문이 프롬프트에 들어갔다는 점입니다. 이제 이 새로운 prompt\_template는 genre 언급을 제대로 필터로 바꾸도록 LLM에게 알려줄 것입니다.
* **요약:** 학생들은 이 단계를 통해 **프롬프트 변수 치환** 방법을 배웁니다. 또한 구체적인 문제 (장르 키워드 인식 실패)에 대해 프롬프트를 조정함으로써 해결하려는 실습을 경험합니다.


```python
# 사용자정의 지시어를 프롬프트에 포함
custom_instructions = """
쟝르라는 단어 없이 액션, 멜로, 스릴러 등와 같은 쟝르에 대한 표현이 입력되면, genre로 필터링을 해야 합니다.
"""

prompt_template = prompt_template.partial_format(custom_instructions=custom_instructions)
prompt_template.template_vars
```

## 21. AutoRetriever의 프롬프트 업데이트

* **목적:** 수정된 PromptTemplate를 AutoRetriever에 반영하여, 이후 질의부터는 새로운 지침이 사용되도록 합니다.
* **사용 메서드:** `auto_retriever.update_prompts({"prompt": prompt_template})` – AutoRetriever가 내부에서 사용하는 프롬프트 중 `"prompt"` 키에 해당하는 것을 우리가 만든 `prompt_template`로 교체합니다.
* **동작 방식:** AutoRetriever는 여러 프롬프트를 딕셔너리로 관리하고 있었는데, 그 중 질의 구조화에 핵심인 "prompt"를 새 템플릿으로 덮어씌우는 것입니다. 이 실행으로 AutoRetriever는 앞으로 질문을 해석할 때 방금 지정한 custom 지침이 포함된 프롬프트를 사용할 것입니다.
* **결과:** 이 셀은 별다른 출력이 없습니다. 하지만 성공적으로 실행되었다면, 다음에 `auto_retriever.retrieve()`를 호출할 때 genre 처리가 개선된 결과를 볼 수 있을 것입니다.
* **설명:** 학생들은 이 부분에서 LlamaIndex의 AutoRetriever가 프롬프트 교체를 허용한다는 것을 확인합니다. 즉, 시스템이 주는 LLM 지시를 사용자가 주도적으로 바꿀 수 있어, 도메인 특화나 오류 수정이 가능함을 알게 됩니다. 이러한 유연성은 RAG 시스템 튜닝에 중요합니다.


```python
auto_retriever.update_prompts({"prompt": prompt_template})
```

## 22. 수정된 프롬프트로 질의 재테스트

* **목적:** 프롬프트를 업데이트한 후에도 AutoRetriever가 잘 작동하고, 특히 이전과 동일한 질문에 대해서도 일관된 필터 추출을 하는지 확인합니다. (사실 "멜로 영화" 케이스는 업데이트 전에도 잘 되었지만, 여기서는 프롬프트 변경 후 기능 확인 차원에서 실행합니다.)
* **실행 코드:** `auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 멜로 영화는?")`
* **LLM 해석 (verbose 출력):** `Using filters: [('year', '>', '2020'), ('director', '==', '박찬욱'), ('genre', '==', '멜로')]`로 나와, 이전과 동일하게 필터를 잡아냅니다. Query str도 "박찬욱 감독의 멜로 영화"로 표시됩니다. 프롬프트를 바꿨지만 output 상으로는 큰 차이 없이 제대로 동작하는 모습입니다. (우리 custom 지시는 주로 "장르라는 말 없이 액션, 멜로 등이 나오면"에 대한 것이어서, 이 질문에서는 원래도 잘 되던 부분이라 변화가 눈에 띄지는 않습니다.)
* **검색 결과:** 여전히 조건을 만족하는 영화는 *헤어질 결심* 하나입니다. NodeWithScore로 반환되고, 메타데이터 내용 역시 이전과 동일합니다.
* **추가 확인:** 사실 진정한 테스트는 "액션 영화"처럼 이전에 문제 됐던 질의를 다시 해보는 것입니다. 곧 아래 QueryEngine 부분에서 비슷한 걸 시도할 것입니다.
* **의미:** 이 셀을 통해 프롬프트 업데이트로 인한 부작용 없이 AutoRetriever가 계속 잘 작동함을 보여줍니다. 학생들은 시스템 변경 후에도 기존 기능이 유지되는지 확인하는 과정도 중요함을 알게 됩니다. (테스트의 개념) 다음에는 QueryEngine을 통해 최종 응답 생성까지 해보겠습니다.


```python
auto_retriever.retrieve("2020년 이후 개봉한 박찬욱 감독의 멜로 영화는?")
```

## 23. AutoRetriever를 활용한 QueryEngine 설정

* **목적:** 지금까지 만든 AutoRetriever(고급 검색기)를 LLM 응답 생성과 결합하여 **최종 답변까지 출력하는 QueryEngine**을 구성합니다. 이를 통해 사용자의 복잡한 질문에 대해 관련 정보 검색 + 답변 생성까지 한 번에 수행할 수 있습니다.
* **사용 클래스:** `RetrieverQueryEngine` – 사용자 정의 Retriever와 응답 생성기를 조합하는 QueryEngine 클래스입니다. 또한 `get_response_synthesizer()`를 사용해 기본 응답 생성기(LLM을 이용해 결과를 가공하는 객체)를 얻습니다.
* **코드 설명:**

  * `response_synthesizer = get_response_synthesizer()` : LlamaIndex에서 기본 설정으로 LLM 응답 생성기를 받아옵니다. (이 생성기는 검색된 노드들을 입력으로 받아 최종 자연어 답변을 만들어주는 역할을 합니다.)
  * `retriever_query_engine = RetrieverQueryEngine(retriever=auto_retriever, response_synthesizer=response_synthesizer)` : 우리가 만든 `auto_retriever`를 Retriever로 쓰고 방금 얻은 `response_synthesizer`를 결합하여 QueryEngine 인스턴스를 생성합니다.
* **동작 방식:** 이제 `retriever_query_engine.query(질문)`을 호출하면, AutoRetriever가 질의를 해석해 필터링 검색을 수행하고, 찾은 노드들을 `response_synthesizer`에 넘겨 LLM이 멋진 답변 문장을 만들어줍니다.
* **출력:** 이 셀 자체는 객체 생성만 하므로 출력이 없습니다. 그러나 내부적으로 `retriever_query_engine`이 준비되었고, 다음 셀들에서 실제로 질문에 대한 응답을 해줄 것입니다.
* **이해 포인트:** 학생들은 이 단계를 통해 **전체 RAG 파이프라인 조립**을 보게 됩니다. 즉, 고급 Retriever + LLM 응답기의 조합으로 질의응답 시스템을 구성한 것입니다. 이는 LlamaIndex의 유연한 설계 덕분이며, 사용자가 원하는 형태로 검색기와 응답생성기를 맞춤 연결할 수 있다는 것을 보여줍니다.


```python
from llama_index.core import get_response_synthesizer
from llama_index.core.query_engine import RetrieverQueryEngine

response_synthesizer = get_response_synthesizer()

retriever_query_engine = RetrieverQueryEngine(retriever=auto_retriever, response_synthesizer=response_synthesizer)
```

## 24. 고급 QueryEngine으로 복합 질문 응답 

* **목적:** 최종 QueryEngine을 사용하여 복합적인 질문에 답변을 생성해봅니다. 이 질문은 단순 검색을 넘어 **계산/추론이 필요한 질문**입니다. 예시: *“2020년 이전에 개봉한 봉준호 감독의 영화는? 이 두 영화의 평균 평점은 얼마인가?”*
* **실행 코드:** `retriever_query_engine.query("2020년 이전에 개봉한 봉준호 감독의 영화 평균 평점은?")` (혹은 유사하게 두 영화를 묻고 평균 평점을 계산하게끔 묻는 문장)
* **처리 절차:**

  1. **AutoRetriever 단계:** LLM이 질문에서 필터를 추출합니다. "2020년 이전", "봉준호 감독"이 조건으로 인식되고, query string은 아마 "영화 제목과 간략한 시놉시스"처럼 기본 content\_info가 나와버릴 수도 있는데, (출력 로그를 보면 query str이 다소 이상하게 표시되지만) 중요한 건 필터가 제대로 적용되어 `옥자(2017)`, `기생충(2019)` 두 노드를 찾아냈다는 점입니다.
  2. **응답 생성 단계:** `response_synthesizer`가 선택된 두 노드의 내용(시놉시스와 메타데이터)을 참고하여 최종 답을 구성합니다. 여기에는 OpenAI LLM이 사용됩니다. 질문의 두 번째 부분 "평균 평점은 얼마인가?"는 단순 정보 나열을 넘어 계산을 요구하는 내용입니다. LLM은 각 영화의 평점을 메타데이터에서 확인하고, (8.6과 7.3), 이를 평균내어 7.95를 계산합니다.
* **출력 응답:** 예시로 LLM이 만들어낸 답변은 다음과 같습니다:
  *“봉준호 감독의 2020년 이전 개봉한 영화는 "기생충"과 "옥자"입니다. "기생충"의 평점은 8.6, "옥자"의 평점은 7.3입니다. 이 두 영화의 평균 평점은 (8.6 + 7.3) / 2 = 7.95입니다. 따라서 평균 평점은 7.95입니다.”*
  LLM이 두 영화 제목을 정확히 집어내고, 평점까지 제시한 후 계산까지 해주었습니다. 소스 노드 리스트도 출력에 함께 보이는데, 그건 내부 검증용 정보입니다.
* **의미:** 이 결과를 통해 학생들은 **RAG 시스템이 복잡한 질문에도 대응**할 수 있음을 봅니다. 검색 부분(auto\_retriever)이 적절한 영화들을 가져왔고, 생성 부분(Llm)이 그 정보를 토대로 추가적인 사고(평균 계산)까지 해내는 점이 흥미롭습니다. 특히, 단순한 DB 질의로는 얻기 힘든 요약/계산 통합 답변을 LLM이 해낼 수 있다는 것이 RAG의 강점임을 실감할 수 있습니다.


```python
retriever_query_engine.query("2020년 이전에 개봉한 봉준호 감독의 영화 평균 평점은?")
```

## 25. 고급 QueryEngine으로 조건부 목록 질문 응답

* **목적:** QueryEngine을 사용해 **특정 조건에 해당하는 항목들을 나열**하는 질문에 답변합니다. 예시: *“2015년 이후 개봉한 연상호 감독의 공포 영화는?”* – 연상호 감독의 2015년 이후 (즉 2016년부터) 공포 장르 작품들을 묻는 질문입니다.
* **실행 코드:** `retriever_query_engine.query("2015년 이후 개봉한 연상호 감독의 공포 영화는?")`
* **처리 흐름:**

  1. AutoRetriever 필터링: "2015년 이후" -> `year > 2015`, "연상호 감독" -> `director == 연상호`, "공포 영화" -> `genre == 공포`. 이번에는 우리의 custom 지시 덕분에 genre 필터도 제대로 잡힙니다. verbose 로그에 `Using filters: [('year', '>', '2015'), ('director', '==', '연상호'), ('genre', '==', '공포')]`가 보일 것입니다.
  2. Qdrant 검색: 해당 조건에 맞는 영화는 \*부산행(2016, 연상호, 장르 액션/공포)\*과 *반도(2020, 연상호, 장르 액션/공포)* 두 편입니다. AutoRetriever가 이 둘을 찾습니다.
  3. 응답 생성: LLM은 검색된 두 영화의 메타데이터/시놉시스를 참고하여 답변 문장을 만듭니다. 아마도 연도와 제목을 언급하면서 두 작품이 있다고 답변할 겁니다.
* **출력 응답:** 예를 들어 LLM이 *“2016년에 개봉한 "부산행"과 2020년에 개봉한 "반도"가 있습니다.”* 와 같이 답할 수 있습니다. 실제로 출력된 응답을 보면 위와 유사한 문장으로 두 영화를 언급했습니다. 이는 LLM이 두 노드의 정보를 자연스럽게 정리해 준 결과입니다.
* **의의:** 학생들은 이 예를 통해 **RAG를 사용한 조건부 목록 추출**이 가능함을 봅니다. 데이터베이스 조회만으로도 목록을 얻을 수 있지만, LLM이 관여함으로써 결과를 더 자연어로 서술해줄 수 있고, 필요하면 추가 설명도 붙일 수 있습니다. 또한 genre 필터 문제를 custom 지시로 해결한 것이 여기서 효과를 발휘하여, "공포 영화"를 정확히 인식하고 필터링 했다는 점도 짚고 넘어갑니다.


```python
retriever_query_engine.query("2015년 이후 개봉한 연상호 감독의 공포 영화는?")
```

## 26. 고급 QueryEngine으로 데이터 없음 시 응답 처리 

* **목적:** 존재하지 않는 조건에 대해 QueryEngine이 어떻게 응답하는지 확인합니다. 질문: *“이병헌 감독의 코미디 영화의 평균 평점은?”* – 앞서 AutoRetriever만으로 물었을 때 결과가 없었던 쿼리를, 이번엔 QueryEngine으로 전체 답변 생성까지 시켜봅니다.
* **실행 코드:** `retriever_query_engine.query("이병헌 감독의 코미디 영화의 평균 평점은?")`
* **과정:**

  1. AutoRetriever 필터링: custom 지시가 적용되어 `"코미디 영화"` -> genre 필터까지 추출될 것입니다. 로그에 `Using filters: [('director', '==', '이병헌'), ('genre', '==', '코미디')]`가 뜨고, query str은 "이병헌 감독의 코미디 영화" 정도로 나오겠죠.
  2. 검색 결과: 이 조건에 맞는 노드는 없으므로 빈 결과가 반환됩니다.
  3. 응답 생성: 검색된 노드가 하나도 없으면, `response_synthesizer`는 아마 이를 감지해 "Empty Response"를 생성하거나, 아무 정보 없음에 대응하는 기본 답변을 내놓습니다. 출력된 결과를 보면 `Response(response='Empty Response', source_nodes=[], ...)`와 같이 표시되었습니다. 즉 LLM이 줄 만한 답변이 없으니 빈 응답 객체를 반환한 것입니다. 실제 사용자 응답으로는 “해당 조건의 영화를 찾지 못했습니다.” 같은 문장을 내보낼 수도 있으나, 여기서는 Empty Response라고 표현된 걸로 보입니다.
* **출력 확인:** 콘솔 로그에는 `Empty Response`라고 보이고, source\_nodes도 빈 리스트로 나타납니다.
* **의미:** 이는 **질의에 해당하는 데이터가 전혀 없는 경우의 처리**를 나타냅니다. 학생들은 RAG 시스템에서도 이런 경우를 고려해야 함을 배웁니다. 즉, "죄송합니다, 해당 조건에 부합하는 영화가 없습니다."와 같은 사용자 메시지를 반환하도록 로직을 추가하거나, 기본 모델이 이런 문구를 생성하게 할 수 있습니다. 현재 결과는 단순 Empty지만, 교육적으로 이 경우를 논의해볼 수 있습니다. 또한 genre 필터가 이번에는 적용되었다는 점도 확인되어, custom 프롬프트의 유효성을 다시 한번 검증했습니다.


```python
retriever_query_engine.query("이병헌 감독의 코미디 영화의 평균 평점은?")
```

## 27. 고급 QueryEngine으로 추론형 질문 응답 

* **목적:** RAG를 사용하여 **데이터 기반 추론 질문**에 답해봅니다. 질문: *“연상호 감독은 영화를 몇 년 주기로 만드나요?”* – 이는 단순히 목록을 달라는 게 아니라, 데이터에 기반한 패턴(감독의 영화 제작 간격)을 묻는 질문입니다. LLM이 검색된 데이터로 추론해야 하는 흥미로운 과제입니다.
* **실행 코드:** `retriever_query_engine.query("연상호 감독은 영화를 몇 년 주기로 만드나요?")`
* **처리 흐름:**

  1. AutoRetriever 필터링: "연상호 감독" -> `director == 연상호`. 질문에 년도 조건은 없지만, LLM이 아마 모든 연상호 감독 영화를 다 보기 위해 `year < 2025` 같은 조건을 자동으로 넣은 듯 합니다. (출력 로그에 `('year', '<', '2025')`가 보입니다. 이는 현재 연도보다 작은 값, 즉 지금까지의 영화를 모두 대상으로 하려고 한 것으로 추측됩니다. AutoRetriever가 이런 컨텍스트까지 넣어준 점은 흥미롭습니다.)
  2. 검색 결과: 연상호 감독의 모든 영화가 반환될 것입니다. 우리 데이터에서는 부산행(2016)과 반도(2020) 두 편입니다.
  3. 응답 생성: LLM은 이 두 영화의 개봉연도를 보고 **연도 차이**를 계산합니다. 2016년과 2020년 사이 4년 간격이 있다는 것을 파악하고, 사용자 질문에 따라 “4년 주기”라는 결론을 답변합니다. 또한 답변에 근거로 두 작품과 연도를 함께 언급해 줄 것입니다.
* **출력 답변:** 예를 들어 LLM은 *“연상호 감독은 2016년에 "부산행"을, 2020년에 "반도"를 제작했습니다. 따라서 그는 4년 간격으로 영화를 만든 것으로 보입니다.”* 라고 답했습니다. 실제 출력에서도 이와 유사한 문장을 볼 수 있습니다. LLM이 데이터를 종합해 추론까지 해낸 것입니다.
* **학습 효과:** 이 마지막 예시는 RAG의 강점을 잘 보여줍니다. 단순 DB 조회가 아닌, **데이터 기반의 통찰**을 LLM이 도출해주고 있습니다. 학생들은 벡터DB + LLM 조합이 단순 정보 검색 이상으로 활용될 수 있음을 이해하게 됩니다. 연상호 감독의 영화들이 4년 간격으로 나왔다는 사실은 두 데이터 포인트로부터 사람이 생각해내듯 LLM도 도출해낸 것이죠.
* **마무리:** 이로써 일련의 코드 실습을 통해, **메타데이터 필터링을 활용한 고급 RAG** 기법을 모두 실험해보았습니다. 기본 검색과의 차이, AutoRetriever로 인한 향상, 프롬프트 수정에 의한 개선, 그리고 최종 LLM 응답 생성까지 단계별로 확인하였으며, 각 코드 셀의 동작과 의의를 모두 파악하게 되었습니다.


```python
retriever_query_engine.query("2025년 이전 연상호 감독은 몇 년에 한 번 영화를 만드나요?")
```
