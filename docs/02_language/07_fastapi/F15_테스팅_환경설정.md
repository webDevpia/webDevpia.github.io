# 15장. 테스팅 + 환경 설정

> "테스트 = 요리사의 시식 — 손님에게 내놓기 전에 직접 맛보고 확인합니다."

코드를 고칠 때마다 "혹시 다른 부분이 망가진 건 아닐까?" 라는 불안감이 드셨나요? 테스트가 그 불안을 해결합니다. 버튼 하나로 모든 기능이 여전히 잘 작동하는지 자동으로 확인할 수 있습니다.

## 1️⃣ 왜 테스트가 필요한가? [↑](#toc)

### 수동 테스트의 문제점

개발 중에 Swagger UI로 직접 테스트하는 방식은 다음과 같은 문제가 있습니다:

```
코드 수정 → Swagger 열기 → 로그인 → 토큰 복사 → Authorize 클릭
→ 엔드포인트 열기 → 파라미터 입력 → "Execute" → 결과 확인
→ 다른 엔드포인트 반복...
```

엔드포인트가 10개라면 수정할 때마다 이 과정을 10번 반복해야 합니다.

### 자동화 테스트의 장점

```bash
pytest
```

이 한 줄로:
- 모든 엔드포인트를 자동으로 테스트합니다
- 10초 만에 전체 결과를 확인합니다
- 어디가 망가졌는지 정확히 알려줍니다
- 언제든지 반복 실행할 수 있습니다

> 테스트는 "이미 잘 되는 것"을 증명하는 것이 아닙니다.
> "앞으로도 계속 잘 되어야 하는 것"을 보장하는 안전망입니다.

## 2️⃣ pytest 기초 [↑](#toc)

### 설치

```bash
uv add --dev pytest httpx
```

> `--dev`는 개발할 때만 필요한 패키지를 따로 관리하는 옵션입니다. 실제 서버에 배포할 때는 포함되지 않습니다.

### 첫 테스트 작성

`tests/test_sample.py` 파일을 만들어봅니다:

```python
def test_addition():
    result = 1 + 1
    assert result == 2


def test_string():
    greeting = "안녕하세요"
    assert "안녕" in greeting
    assert len(greeting) > 0
```

### 실행

```bash
pytest
```

실행 결과:
```
====================== test session starts ======================
collected 2 items

tests/test_sample.py ..                                   [100%]

======================= 2 passed in 0.12s =======================
```

### assert 키워드

`assert` 는 "이것이 참이어야 한다"는 선언입니다. 거짓이면 테스트가 실패합니다.

```python
assert 1 + 1 == 2          # 통과
assert "hello" in "hello world"  # 통과
assert response.status_code == 200  # 상태 코드 확인
assert response.json()["id"] == 1   # 응답 내용 확인
```

테스트 실패 시:

```
AssertionError: assert 404 == 200
```

어느 줄에서 무엇이 틀렸는지 정확히 알려줍니다.

## 3️⃣ TestClient 사용법 [↑](#toc)

### TestClient란?

FastAPI 앱에 실제 HTTP 요청을 보내는 것처럼 테스트할 수 있게 해주는 도구입니다. 실제로 서버를 실행하지 않아도 됩니다.

```python
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)
```

### GET 요청 테스트

```python
def test_read_root():
    response = client.get("/")
    assert response.status_code == 200
    assert response.json()["message"] == "안녕하세요"
```

### POST 요청 테스트

```python
def test_create_todo():
    response = client.post(
        "/todos/",
        json={"title": "FastAPI 공부", "completed": False},
    )
    assert response.status_code == 201
    data = response.json()
    assert data["title"] == "FastAPI 공부"
    assert data["id"] is not None
```

### 에러 케이스 테스트

```python
def test_get_nonexistent_todo():
    response = client.get("/todos/99999")
    assert response.status_code == 404
    assert "찾을 수 없습니다" in response.json()["detail"]
```

> 성공 케이스만 테스트하면 안 됩니다. 실패 케이스도 반드시 테스트해야 합니다.
> "잘못된 입력을 넣었을 때 올바른 에러가 나오는가"도 기능의 일부입니다.

## 4️⃣ 테스트 DB 분리 [↑](#toc)

### 왜 테스트 DB를 분리해야 하나?

# ❌ 이렇게 하면 안 됩니다
```python
# 테스트가 실제 database.db에 데이터를 만들고 지우면?
# 1. 테스트 실행 중 실제 데이터가 오염됩니다
# 2. 테스트 순서에 따라 결과가 달라질 수 있습니다
# 3. 개발 중인 데이터가 테스트에 영향을 줍니다
```

# ✅ 테스트 전용 인메모리 DB 사용
```python
# 메모리 안에서만 동작 — 테스트가 끝나면 모두 사라집니다
TEST_DATABASE_URL = "sqlite://"
```

### dependency_overrides 활용

FastAPI의 `app.dependency_overrides`를 사용하면 테스트할 때만 의존성 함수를 교체할 수 있습니다.

```python
from fastapi.testclient import TestClient
from sqlmodel import SQLModel, Session, create_engine
from sqlmodel.pool import StaticPool

from app.main import app
from app.database import get_session

# 테스트 전용 인메모리 DB 설정
TEST_DATABASE_URL = "sqlite://"

engine = create_engine(
    TEST_DATABASE_URL,
    connect_args={"check_same_thread": False},
    poolclass=StaticPool,  # 인메모리 DB에서 연결 공유
)


def override_get_session():
    SQLModel.metadata.create_all(engine)
    with Session(engine) as session:
        yield session


# 의존성 교체: 실제 get_session → 테스트용 override_get_session
app.dependency_overrides[get_session] = override_get_session

client = TestClient(app)
```

이제 `client`를 사용한 모든 테스트는 인메모리 SQLite DB를 사용합니다. 실제 `database.db`는 전혀 영향을 받지 않습니다.

## 5️⃣ 환경 설정 (pydantic-settings) [↑](#toc)

### 문제: 하드코딩된 설정값

지금까지 코드 여기저기에 이런 값들이 흩어져 있었습니다:

# ❌ 하드코딩된 설정 — 문제가 많습니다
```python
# auth.py
SECRET_KEY = "임시-비밀-키"

# database.py
DATABASE_URL = "sqlite:///database.db"
```

이 방식의 문제:
1. 코드가 GitHub에 올라가면 비밀 키가 노출됩니다
2. 로컬/개발/프로덕션 환경마다 값을 바꾸려면 코드를 직접 수정해야 합니다
3. 설정값이 여러 파일에 분산되어 관리가 어렵습니다

### pydantic-settings 설치

```bash
uv add pydantic-settings
```

### Settings 클래스 작성 (`app/config.py`)

```python
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    # 기본값이 있는 설정들
    database_url: str = "sqlite:///database.db"
    access_token_expire_minutes: int = 30
    algorithm: str = "HS256"

    # 기본값 없음 — .env 파일이나 환경변수에서 반드시 제공해야 합니다
    secret_key: str

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
    )


# 앱 전역에서 사용할 설정 인스턴스
settings = Settings()
```

### 설정 사용하기

```python
# app/routers/auth.py
from app.config import settings

def create_access_token(data: dict) -> str:
    to_encode = data.copy()
    expire = datetime.now(timezone.utc) + timedelta(
        minutes=settings.access_token_expire_minutes
    )
    to_encode.update({"exp": expire})
    return jwt.encode(to_encode, settings.secret_key, algorithm=settings.algorithm)
```

```python
# app/database.py
from app.config import settings
from sqlmodel import create_engine, Session, SQLModel

engine = create_engine(settings.database_url)
```

## 6️⃣ .env 파일과 보안 [↑](#toc)

### .env 파일 생성

프로젝트 루트에 `.env` 파일을 만듭니다:

```bash
# .env
DATABASE_URL=sqlite:///database.db
SECRET_KEY=a3f8c2e1d4b7a6f9e2c5d8b1a4f7c3e6d9b2a5f8c1e4d7b0a3f6c9e2d5b8a1f4
ACCESS_TOKEN_EXPIRE_MINUTES=30
ALGORITHM=HS256
```

SECRET_KEY는 앞에서 배운 명령어로 생성합니다:
```bash
openssl rand -hex 32
```

### .gitignore에 추가 — 필수

```bash
# .gitignore
.env
*.db
__pycache__/
.pytest_cache/
```

> `.env` 파일을 `.gitignore`에 추가하지 않으면 비밀 키가 GitHub에 공개됩니다.
> 이것은 실제로 매일 발생하는 보안 사고입니다.

### .env.example 파일 공유

실제 값 대신 형식을 보여주는 예시 파일은 공유해도 됩니다:

```bash
# .env.example  ← GitHub에 올려도 되는 파일
DATABASE_URL=sqlite:///database.db
SECRET_KEY=여기에-openssl-rand--hex-32로-생성한-값을-입력하세요
ACCESS_TOKEN_EXPIRE_MINUTES=30
ALGORITHM=HS256
```

팀원이 프로젝트를 받으면 `.env.example`을 복사해서 `.env`를 만들고 실제 값을 채웁니다.

## 7️⃣ 테스트 구조 — conftest.py [↑](#toc)

### 권장 테스트 구조

```
프로젝트 루트/
├── app/
│   ├── main.py
│   ├── config.py
│   ├── database.py
│   └── routers/
│       ├── auth.py
│       └── todos.py
└── tests/
    ├── conftest.py       ← 픽스처 정의 (테스트 DB, 클라이언트)
    ├── test_todos.py     ← ToDo 엔드포인트 테스트
    └── test_auth.py      ← 인증 엔드포인트 테스트
```

### conftest.py — 공유 픽스처

```python
# tests/conftest.py
import pytest
from fastapi.testclient import TestClient
from sqlmodel import SQLModel, Session, create_engine
from sqlmodel.pool import StaticPool

from app.main import app
from app.database import get_session


@pytest.fixture(name="session")
def session_fixture():
    engine = create_engine(
        "sqlite://",
        connect_args={"check_same_thread": False},
        poolclass=StaticPool,
    )
    SQLModel.metadata.create_all(engine)
    with Session(engine) as session:
        yield session


@pytest.fixture(name="client")
def client_fixture(session: Session):
    def override_get_session():
        yield session

    app.dependency_overrides[get_session] = override_get_session
    client = TestClient(app)
    yield client
    app.dependency_overrides.clear()
```

> `conftest.py`는 pytest가 자동으로 찾는 특별한 파일입니다.
> 여기서 정의한 픽스처(fixture)는 같은 폴더의 모든 테스트 파일에서 사용할 수 있습니다.

### test_todos.py — ToDo 테스트

```python
# tests/test_todos.py
from fastapi.testclient import TestClient


def test_create_todo(client: TestClient):
    response = client.post(
        "/todos/",
        json={"title": "pytest 공부", "completed": False},
    )
    assert response.status_code == 201
    data = response.json()
    assert data["title"] == "pytest 공부"
    assert data["completed"] is False
    assert "id" in data


def test_read_todos(client: TestClient):
    # 먼저 두 개 생성
    client.post("/todos/", json={"title": "첫 번째", "completed": False})
    client.post("/todos/", json={"title": "두 번째", "completed": True})

    response = client.get("/todos/")
    assert response.status_code == 200
    todos = response.json()
    assert len(todos) == 2


def test_update_todo(client: TestClient):
    # 생성
    create_response = client.post(
        "/todos/", json={"title": "원래 제목", "completed": False}
    )
    todo_id = create_response.json()["id"]

    # 수정
    update_response = client.patch(
        f"/todos/{todo_id}",
        json={"completed": True},
    )
    assert update_response.status_code == 200
    assert update_response.json()["completed"] is True


def test_delete_todo(client: TestClient):
    create_response = client.post(
        "/todos/", json={"title": "삭제할 항목", "completed": False}
    )
    todo_id = create_response.json()["id"]

    delete_response = client.delete(f"/todos/{todo_id}")
    assert delete_response.status_code == 204

    # 삭제 후 조회하면 404
    get_response = client.get(f"/todos/{todo_id}")
    assert get_response.status_code == 404


def test_get_nonexistent_todo(client: TestClient):
    response = client.get("/todos/99999")
    assert response.status_code == 404
```

## 8️⃣ 인증 테스트 [↑](#toc)

### test_auth.py — 인증 흐름 테스트

```python
# tests/test_auth.py
from fastapi.testclient import TestClient


def test_register(client: TestClient):
    response = client.post(
        "/auth/register",
        json={"email": "test@example.com", "password": "비밀번호123"},
    )
    assert response.status_code == 201
    data = response.json()
    assert data["email"] == "test@example.com"
    assert "hashed_password" not in data  # 민감 정보 노출 금지


def test_register_duplicate_email(client: TestClient):
    # 첫 번째 등록
    client.post(
        "/auth/register",
        json={"email": "dup@example.com", "password": "비밀번호123"},
    )
    # 같은 이메일로 재등록 시도
    response = client.post(
        "/auth/register",
        json={"email": "dup@example.com", "password": "다른비밀번호"},
    )
    assert response.status_code == 400
    assert "이미 등록된" in response.json()["detail"]


def test_login(client: TestClient):
    # 먼저 회원가입
    client.post(
        "/auth/register",
        json={"email": "login@example.com", "password": "비밀번호123"},
    )
    # 로그인
    response = client.post(
        "/auth/token",
        data={"username": "login@example.com", "password": "비밀번호123"},
    )
    assert response.status_code == 200
    data = response.json()
    assert "access_token" in data
    assert data["token_type"] == "bearer"


def test_login_wrong_password(client: TestClient):
    client.post(
        "/auth/register",
        json={"email": "wrongpw@example.com", "password": "올바른비밀번호"},
    )
    response = client.post(
        "/auth/token",
        data={"username": "wrongpw@example.com", "password": "틀린비밀번호"},
    )
    assert response.status_code == 401


def test_protected_route_without_token(client: TestClient):
    response = client.get("/todos/")
    assert response.status_code == 401


def test_protected_route_with_token(client: TestClient):
    # 회원가입 + 로그인
    client.post(
        "/auth/register",
        json={"email": "user@example.com", "password": "비밀번호123"},
    )
    login_response = client.post(
        "/auth/token",
        data={"username": "user@example.com", "password": "비밀번호123"},
    )
    token = login_response.json()["access_token"]

    # 토큰 포함 요청
    response = client.get(
        "/todos/",
        headers={"Authorization": f"Bearer {token}"},
    )
    assert response.status_code == 200
```

> `OAuth2PasswordRequestForm`은 JSON이 아니라 `data=` (form 데이터)로 전송합니다.
> `json=` 대신 `data=`를 사용하는 것에 주의하세요.

## 9️⃣ 테스트 실행과 커버리지 [↑](#toc)

pytest -v

### 테스트 작성 원칙

| 원칙 | 설명 |
|------|------|
| **AAA 패턴** | Arrange(준비) → Act(실행) → Assert(검증) |
| **독립성** | 각 테스트는 다른 테스트에 의존하지 않아야 합니다 |
| **격리** | 테스트 DB는 반드시 실제 DB와 분리합니다 |
| **명확한 이름** | `test_login_wrong_password` — 무엇을 테스트하는지 바로 알 수 있어야 합니다 |
| **실패 케이스 포함** | 성공 케이스뿐 아니라 에러 케이스도 테스트합니다 |
