# 08장. 의존성 주입 — Depends()
{: .no_toc }

> "의존성 주입 = 건물 관리실 — 열쇠(DB 세션, 인증 정보)가 필요하면 관리실에 요청합니다. 직접 만들 필요 없습니다."
> 아파트에서 운동실을 사용하려면 키카드가 필요합니다.
> 관리실에 요청하면 키카드를 빌려줍니다. 사용이 끝나면 반납합니다.
> 각 입주자가 직접 키카드를 만들 필요가 없습니다.

## 학습 목표
{: .no_toc }

- 코드 반복의 문제점을 이해하고 `Depends()`로 해결하는 방법을 배웁니다
- 의존성 함수를 작성하고 엔드포인트에 주입하는 패턴을 익힙니다
- `Annotated` 패턴으로 재사용 가능한 의존성 타입을 만들 수 있습니다
- 의존성 체이닝(의존성이 다른 의존성을 사용하는 패턴)을 이해합니다
- 라우터 레벨 의존성으로 그룹 전체에 인증을 적용하는 방법을 습득합니다
- `yield` 의존성으로 정리(cleanup) 로직을 작성하는 원리를 배웁니다

## 1️⃣ 문제: 코드 반복 [↑](#toc)

아래 코드를 보세요. 세 개의 엔드포인트가 있고, **각각 동일한 인증 확인 코드**를 가지고 있습니다.

```python
# main.py  ← 반복 코드 예시

from fastapi import FastAPI, Header, HTTPException
from typing import Optional

app = FastAPI()

# 엔드포인트 1
@app.get("/items")
def list_items(x_token: Optional[str] = Header(None)):
    # ── 인증 확인 (1번째 복사) ──
    if x_token is None:
        raise HTTPException(status_code=401, detail="토큰이 없습니다")
    if x_token != "secret-token":
        raise HTTPException(status_code=403, detail="유효하지 않은 토큰입니다")
    # ── 실제 로직 ──
    return [{"id": 1, "name": "아이템"}]


# 엔드포인트 2
@app.post("/items")
def create_item(x_token: Optional[str] = Header(None)):
    # ── 인증 확인 (2번째 복사) ──
    if x_token is None:
        raise HTTPException(status_code=401, detail="토큰이 없습니다")
    if x_token != "secret-token":
        raise HTTPException(status_code=403, detail="유효하지 않은 토큰입니다")
    # ── 실제 로직 ──
    return {"id": 2, "name": "새 아이템"}


# 엔드포인트 3
@app.delete("/items/{item_id}")
def delete_item(item_id: int, x_token: Optional[str] = Header(None)):
    # ── 인증 확인 (3번째 복사) ──
    if x_token is None:
        raise HTTPException(status_code=401, detail="토큰이 없습니다")
    if x_token != "secret-token":
        raise HTTPException(status_code=403, detail="유효하지 않은 토큰입니다")
    # ── 실제 로직 ──
    return {"deleted": item_id}
```

같은 코드가 **3번** 반복되고 있습니다.

이 코드의 문제는 무엇일까요?

| 상황 | 결과 |
|------|------|
| 토큰 확인 방식을 바꾸고 싶다 | 3곳을 모두 수정해야 합니다 |
| 한 군데 수정하고 나머지를 잊는다 | 보안 허점 발생 |
| 엔드포인트가 10개로 늘어난다 | 10곳에 복사, 10곳을 관리 |

> 소프트웨어 개발의 중요한 원칙 중 하나: **DRY (Don't Repeat Yourself)**
> "반복하지 마세요" — 같은 코드를 두 번 이상 쓰면 나중에 반드시 실수가 생깁니다.

## 2️⃣ 해결: Depends() [↑](#toc)

`Depends()`는 공통 로직을 함수로 추출하고, FastAPI가 엔드포인트 실행 **전에** 자동으로 호출하도록 합니다.

```python
# ✅ Depends()를 사용한 개선 버전

from fastapi import FastAPI, Header, HTTPException, Depends
from typing import Optional

app = FastAPI()


# ── 의존성 함수 정의 (공통 로직) ──────────────────────
def verify_token(x_token: Optional[str] = Header(None)) -> str:
    """토큰을 검증하고 유효하면 토큰 값을 반환합니다."""
    if x_token is None:
        raise HTTPException(status_code=401, detail="토큰이 없습니다")
    if x_token != "secret-token":
        raise HTTPException(status_code=403, detail="유효하지 않은 토큰입니다")
    return x_token   # 검증 통과 시 반환값이 엔드포인트 함수로 전달됨


# ── 각 엔드포인트에 Depends()로 주입 ────────────────
@app.get("/items")
def list_items(token: str = Depends(verify_token)):
    # verify_token이 성공해야 여기 도달합니다
    # token 변수에는 verify_token의 반환값이 담겨 있습니다
    return [{"id": 1, "name": "아이템"}]


@app.post("/items")
def create_item(token: str = Depends(verify_token)):
    return {"id": 2, "name": "새 아이템"}


@app.delete("/items/{item_id}")
def delete_item(item_id: int, token: str = Depends(verify_token)):
    return {"deleted": item_id}
```

인증 로직이 `verify_token` **한 곳**에만 있습니다. 변경이 필요하면 한 곳만 수정하면 됩니다.

### Depends() 동작 방식

```
클라이언트 요청
     ↓
FastAPI가 의존성 함수 목록 확인
     ↓
verify_token() 실행
  └─ 토큰 없음 → HTTPException(401) → 요청 종료
  └─ 토큰 불일치 → HTTPException(403) → 요청 종료
  └─ 토큰 유효 → 반환값을 엔드포인트로 전달
     ↓
엔드포인트 함수 실행 (token = "secret-token")
     ↓
응답 반환
```

## 3️⃣ 의존성이란? [↑](#toc)

의존성(Dependency)이란 **함수가 실행되기 전에 FastAPI가 먼저 준비해주는 것**입니다.

가장 실용적인 예시부터 보겠습니다. 여러 엔드포인트에서 공통 쿼리 파라미터(`skip`, `limit`)를 사용하는 경우입니다.

```python
# ❌ 반복 버전
@app.get("/items")
def list_items(skip: int = 0, limit: int = 10):
    return items_db[skip : skip + limit]

@app.get("/users")
def list_users(skip: int = 0, limit: int = 10):
    return users_db[skip : skip + limit]

@app.get("/categories")
def list_categories(skip: int = 0, limit: int = 10):
    return categories_db[skip : skip + limit]
```

```python
# ✅ Depends() 버전
from fastapi import Depends

def pagination(skip: int = 0, limit: int = 10):
    """페이지네이션 파라미터를 검증하고 반환합니다."""
    if limit > 100:
        raise HTTPException(status_code=400, detail="limit은 최대 100입니다")
    return {"skip": skip, "limit": limit}


@app.get("/items")
def list_items(page: dict = Depends(pagination)):
    return items_db[page["skip"] : page["skip"] + page["limit"]]

@app.get("/users")
def list_users(page: dict = Depends(pagination)):
    return users_db[page["skip"] : page["skip"] + page["limit"]]

@app.get("/categories")
def list_categories(page: dict = Depends(pagination)):
    return categories_db[page["skip"] : page["skip"] + page["limit"]]
```

Swagger UI에서 `/items` 엔드포인트를 열어보면 `skip`과 `limit` 쿼리 파라미터가 자동으로 문서화되어 있습니다. 의존성 함수의 파라미터도 문서에 반영됩니다.

> 의존성 함수는 **일반 Python 함수**입니다.
> FastAPI 특화 문법이 아닙니다. 단지 `Depends()`로 감싸서 "이걸 먼저 실행해줘"라고 알리는 것입니다.
> 의존성 함수 자체도 `Path()`, `Query()`, `Header()`, 다른 `Depends()`를 받을 수 있습니다.

## 4️⃣ Annotated 패턴 [↑](#toc)

Python 3.9+부터 `Annotated`를 활용해 의존성 타입을 **미리 정의**할 수 있습니다. 이 방식이 FastAPI 공식 문서에서 권장하는 현대적 스타일입니다.

```python
from typing import Annotated
from fastapi import Depends

# ── 의존성 함수 ──────────────────────────────────────
def get_pagination(skip: int = 0, limit: int = 10) -> dict:
    return {"skip": skip, "limit": limit}

def get_current_user(x_token: str = Header(...)) -> dict:
    if x_token != "secret-token":
        raise HTTPException(status_code=403, detail="유효하지 않은 토큰")
    return {"username": "alice", "role": "admin"}


# ── Annotated 타입 별칭 정의 ──────────────────────────
PaginationDep = Annotated[dict, Depends(get_pagination)]
CurrentUserDep = Annotated[dict, Depends(get_current_user)]
```

이제 엔드포인트 함수 시그니처가 훨씬 간결해집니다.

```python
# ❌ 기존 방식
@app.get("/items")
def list_items(page: dict = Depends(get_pagination)):
    ...

# ✅ Annotated 방식 (권장)
@app.get("/items")
def list_items(page: PaginationDep):
    return items_db[page["skip"] : page["skip"] + page["limit"]]


@app.get("/admin/items")
def admin_list_items(page: PaginationDep, user: CurrentUserDep):
    # user = {"username": "alice", "role": "admin"}
    return {"user": user, "items": items_db}
```

두 방식은 **동작이 완전히 동일**합니다. `Annotated` 방식이 타입 힌트가 명확하고 IDE 자동완성도 더 잘 됩니다.

### Annotated의 의미

```python
PaginationDep = Annotated[dict, Depends(get_pagination)]
#                          ↑           ↑
#               실제 타입(dict)   FastAPI에게 전달할 메타데이터
```

`Annotated[타입, 메타데이터]` — "이 값은 `dict` 타입인데, FastAPI는 `Depends(get_pagination)`을 실행해서 채워줘"라는 의미입니다.

## 5️⃣ 의존성 체이닝 [↑](#toc)

의존성 함수가 **다른 의존성 함수를 사용**할 수 있습니다. 관리실이 다른 관리실에 요청하는 것과 같습니다.

```python
from fastapi import Depends, Header, HTTPException
from typing import Annotated

# ── 1단계: 토큰 추출 ──────────────────────────────────
def get_token(x_token: str = Header(...)) -> str:
    """헤더에서 토큰을 가져옵니다."""
    return x_token


# ── 2단계: 토큰으로 사용자 조회 (1단계에 의존) ────────
def get_current_user(token: str = Depends(get_token)) -> dict:
    """토큰을 검증하고 사용자 정보를 반환합니다."""
    if token != "secret-token":
        raise HTTPException(status_code=403, detail="유효하지 않은 토큰")
    # 실제 서비스에서는 DB에서 사용자 조회
    return {"id": 1, "username": "alice", "role": "user"}


# ── 3단계: 관리자 권한 확인 (2단계에 의존) ───────────
def require_admin(user: dict = Depends(get_current_user)) -> dict:
    """관리자 권한을 확인합니다."""
    if user["role"] != "admin":
        raise HTTPException(status_code=403, detail="관리자 권한이 필요합니다")
    return user


# ── Annotated 타입 정의 ──────────────────────────────
CurrentUserDep = Annotated[dict, Depends(get_current_user)]
AdminUserDep = Annotated[dict, Depends(require_admin)]


# ── 엔드포인트 ───────────────────────────────────────
@app.get("/me")
def get_my_info(user: CurrentUserDep):
    return user   # 로그인한 사용자면 누구나 접근 가능


@app.delete("/admin/users/{user_id}")
def admin_delete_user(user_id: int, admin: AdminUserDep):
    return {"deleted_by": admin["username"], "user_id": user_id}
```

의존성 실행 순서:

```
DELETE /admin/users/5 요청
     ↓
1. get_token() 실행 → 헤더에서 토큰 추출
     ↓
2. get_current_user(token) 실행 → 사용자 조회
     ↓
3. require_admin(user) 실행 → 권한 확인
     ↓
4. admin_delete_user(user_id=5, admin={...}) 실행
```

FastAPI는 의존성 그래프를 분석해서 **올바른 순서**로 실행합니다. 개발자가 순서를 직접 관리할 필요가 없습니다.

## 6️⃣ 라우터 레벨 의존성 [↑](#toc)

특정 라우터의 **모든 엔드포인트**에 동일한 의존성을 적용하고 싶을 때 라우터 레벨 의존성을 사용합니다.

```python
# routers/admin.py

from fastapi import APIRouter, Depends, HTTPException, Header
from typing import Optional

def verify_admin_token(x_admin_token: Optional[str] = Header(None)):
    """관리자 토큰을 검증합니다."""
    if x_admin_token != "admin-secret":
        raise HTTPException(status_code=403, detail="관리자 권한이 필요합니다")


# ── 라우터에 의존성 등록 ──────────────────────────────
# 이 라우터의 모든 엔드포인트는 자동으로 verify_admin_token을 실행합니다
router = APIRouter(
    prefix="/admin",
    tags=["Admin"],
    dependencies=[Depends(verify_admin_token)],
)


@router.get("/stats")
def get_stats():
    # verify_admin_token을 직접 쓰지 않아도 이미 검증 완료
    return {"total_items": 100, "total_users": 50}


@router.delete("/users/{user_id}")
def delete_user(user_id: int):
    # 여기도 이미 검증 완료
    return {"deleted": user_id}


@router.get("/logs")
def get_logs():
    # 여기도 이미 검증 완료
    return [{"log": "..."}]
```

```python
# app/main.py
from app.routers import admin

app.include_router(admin.router)
```

```
# ❌ 엔드포인트마다 직접 검증
@router.get("/stats")
def get_stats(admin=Depends(verify_admin_token)):
    ...

@router.delete("/users/{user_id}")
def delete_user(user_id: int, admin=Depends(verify_admin_token)):
    ...

@router.get("/logs")
def get_logs(admin=Depends(verify_admin_token)):
    ...
```

```
# ✅ 라우터에 한 번만 등록
router = APIRouter(
    dependencies=[Depends(verify_admin_token)],
)
```

> 라우터 레벨 의존성은 엔드포인트 함수 파라미터에 값을 전달하지 않습니다.
> "이 함수가 성공해야 통과" 라는 **게이트** 역할만 합니다.
> 반환값이 필요하다면 각 엔드포인트에서 별도로 `Depends()`를 사용하세요.

## 7️⃣ yield 의존성 [↑](#toc)

일반 의존성 함수는 값을 `return`하면 끝납니다. 하지만 사용 후 **정리(cleanup)** 가 필요한 경우가 있습니다. 데이터베이스 연결이 대표적입니다.

> `yield` 의존성 = 렌탈 서비스
> 장비를 빌려주고(yield 이전), 사용이 끝나면 반납 처리를 합니다(yield 이후).
> 고객이 직접 반납 절차를 신경 쓰지 않아도 됩니다.

```python
# ── yield 의존성 패턴 ─────────────────────────────────

def get_db_connection():
    """DB 연결을 생성하고, 요청이 끝나면 자동으로 닫습니다."""
    # yield 이전: 준비 단계
    print("DB 연결 열기")
    db = {"connection": "postgresql://localhost/mydb", "open": True}

    try:
        yield db          # ← 엔드포인트 함수에 db가 전달됩니다
    finally:
        # yield 이후: 정리 단계 (요청 성공/실패 관계없이 항상 실행)
        print("DB 연결 닫기")
        db["open"] = False


DbDep = Annotated[dict, Depends(get_db_connection)]


@app.get("/items")
def list_items(db: DbDep):
    # 여기서 db를 사용
    print(f"DB 사용 중: {db['connection']}")
    return [{"id": 1}]
    # 함수가 끝나면 finally 블록이 실행 → "DB 연결 닫기" 출력
```

실행 결과:

```
DB 연결 열기
DB 사용 중: postgresql://localhost/mydb
DB 연결 닫기
```

요청이 성공하든, 에러가 나든 `finally` 블록은 항상 실행됩니다. 연결 누수(connection leak)가 발생하지 않습니다.

> 다음 Part(데이터베이스 연동)에서 SQLModel 세션 관리에 이 패턴을 정확히 이렇게 사용합니다.
> 지금은 "사용하고 나서 자동으로 정리되는 의존성"이 있다는 것만 기억해두세요.

## 8️⃣ 정리 [↑](#toc)

| 개념 | 설명 |
|------|------|
| `Depends(함수)` | 엔드포인트 실행 전에 해당 함수를 먼저 실행 |
| 의존성 함수 | 일반 Python 함수. `Query()`, `Header()`, `Depends()` 모두 사용 가능 |
| `Annotated[타입, Depends(...)]` | 재사용 가능한 의존성 타입 별칭 (권장 패턴) |
| 의존성 체이닝 | 의존성 함수가 다른 `Depends()`를 받는 것 |
| 라우터 레벨 의존성 | `APIRouter(dependencies=[...])` — 해당 라우터 전체에 적용 |
| `yield` 의존성 | 준비 → `yield` → 정리 패턴. DB 세션에서 주로 사용 |

### Depends() 사용 결정 기준

```
같은 코드가 2개 이상의 엔드포인트에 반복된다
     ↓
    YES → Depends()로 추출
     ↓
반환값이 필요한가?
  YES → 의존성 함수 파라미터로 받기
  NO  → 라우터 레벨 dependencies=[Depends(...)] 사용
```
