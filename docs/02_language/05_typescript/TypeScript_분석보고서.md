## 1. 분석 개요

### 1.1 배경

JavaScript 과정(14장)을 이수한 비전공자가 TypeScript를 배우는 40시간 과정을 설계한다.

**수강생이 이미 아는 것 (JS 과정에서 학습 완료):**

| JS 챕터 | 내용 | TS와의 연결 |
|---------|------|-----------|
| 02장 | 변수, 데이터 타입, typeof | → TS 기본 타입의 출발점 |
| 03장 | 연산자, 조건문 | → 타입 좁히기(narrowing)의 기초 |
| 04장 | 반복문, 배열 | → 배열 타입, 튜플 |
| 05장 | 함수 (선언, 화살표, 콜백) | → 함수 타입 시그니처 |
| 06장 | 스코프, 클로저 | → 타입 스코프 이해 |
| 07장 | 객체, 구조분해, 스프레드, 고차함수 | → 인터페이스, 타입 별칭 |
| 08장 | ES6+ (모듈, 옵셔널 체이닝) | → import type, 모듈 타입 |
| 09-10장 | DOM, 이벤트, ToDo 앱 | → DOM 타입, 이벤트 타입 |
| 11-12장 | 콜백, Promise, async/await, fetch | → Promise<T>, 제네릭 |
| 13장 | Node.js, npm | → tsx, @types 패키지 |
| 14장 | 통합 프로젝트 (날씨 앱) | → 타입 안전한 날씨 앱 리팩터링 |

## 2. 핵심 자료 요약

### 2.1 TypeScript 5.x 주요 변경사항

| 버전 | 핵심 기능 | 초보자 관련도 |
|------|----------|-------------|
| **5.5** (2024.06) | 추론된 타입 술어 — `.filter(Boolean)` 자동 타입 추론 | **높음** |
| **5.6** (2024.09) | 항상 참/null인 표현식 엄격 검사 | 중간 |
| **5.7** (2024.11) | 미초기화 변수 감지 개선 | 낮음 |
| **5.8** (2025.02) | 조건부 반환 타입 검사 개선, `--erasableSyntaxOnly` | 낮음 |
| **7.0** (2026 중반 예정) | Go로 재작성, **strict 기본값**, 10배 빌드 속도 | **높음** — strict를 처음부터 가르쳐야 함 |

> **교육적 함의**: TypeScript 7.0에서 `strict: true`가 기본값이 됩니다. 지금부터 strict 모드를 기본으로 가르쳐야 합니다.

### 2.2 교육 플랫폼 공통 커리큘럼 순서

Total TypeScript, Codecademy, Boot.dev, freeCodeCamp, 공식 문서 등 5개+ 플랫폼의 공통 순서:

```
기초 타입 → 함수 타입 → 유니언/리터럴 → 인터페이스/타입 별칭 
→ 타입 좁히기 → 제네릭 → 유틸리티 타입 → 클래스 → 모듈 → 통합 프로젝트
```

### 2.3 초보자 10대 실수 (교육 커뮤니티 합의)

| 순위 | 실수 | 해결 |
|------|------|------|
| 1 | **`any` 남용** — 타입 시스템 전체 무력화 | `unknown` 대안 가르치기, `noImplicitAny: true` |
| 2 | **strict 모드 미사용** | 첫날부터 `strict: true` |
| 3 | **함수 매개변수 타입 누락** | 추론은 변수, 함수는 명시 — 규칙 확립 |
| 4 | **null/undefined 미처리** | `strictNullChecks`로 컴파일 타임에 잡기 |
| 5 | **타입 단언(`as`) 오용** — 안전 우회 | 타입 가드를 우선, `as`는 최후 수단 |
| 6 | **interface vs type 혼동** | 객체 = interface, 유니언/원시 = type |
| 7 | **복잡한 추론에 명시적 타입 미작성** | "추론이 복잡하면 명시하라" 원칙 |
| 8 | **`unknown` 대신 `any` 사용** | 외부 데이터는 항상 `unknown` |
| 9 | **판별 유니언 미사용** | 선택적 속성 대신 판별 유니언 패턴 |
| 10 | **컴파일러와 싸움** — 에러를 `as`로 무시 | "에러는 코드 리뷰" 마인드셋 |

| 개념 | 비유 | 출처 |

| 제네릭 | **쿠키 틀** — 틀(구조)은 같고 반죽(타입)만 바꿀 수 있음 | 커뮤니티 |

## 3. 다중 페르소나 분석

### 3.1 Dr. 타입 — 교육공학 연구자의 분석

#### 핵심 주장: "JS 과정의 비유를 확장하고, 추론-우선으로 가르쳐야 한다"

**원칙 1: 추론 우선 (Inference First)**

비전공자에게 `const name: string = "홍길동"`부터 가르치면 "왜 뻔한 걸 적어야 해?"라는 저항이 발생한다.

```
❌ 전통적 순서: 타입 주석 → 추론 설명
✅ 추론 우선 순서: 
  1. const name = "홍길동"  // TS가 자동으로 string임을 안다! (추론)
  2. "그런데 함수 매개변수는 TS가 추론할 수 없습니다" → 타입 주석 필요
  3. 규칙: "변수는 추론에 맡기고, 함수 입출력은 명시하라"
```

이 순서는 "TypeScript = 추가 작업"이라는 인식을 "TypeScript = 자동 안전망 + 필요할 때만 명시"로 바꾼다.

**원칙 2: JS 비유 확장**

JS 과정의 비유를 그대로 이어받아 확장한다:

```
JS 02장: "변수 = 이름표가 붙은 상자, 아무거나 넣을 수 있다"
  ↓ 확장
TS 03장: "TS의 상자에는 라벨이 붙어 있어서 정해진 종류만 넣을 수 있다"
  ↓ 구체화
  let box: string = "사과";  // "과일" 라벨이 붙은 상자
  box = 42;                  // ❌ 에러! 숫자는 과일 상자에 못 넣습니다
```

```
JS 05장: "함수 = 자판기 (동전 넣으면 음료 나옴)"
  ↓ 확장
TS 04장: "TS 자판기에는 동전 투입구 크기가 정해져 있다"
  function vendingMachine(coin: number): string { ... }
  // 100원짜리만 들어가는 투입구 — 문자열 넣으면 걸림
```

**원칙 3: "고통 → 해결" 패턴 의무화**

매 챕터의 첫 예제는 **JS로 작성한 코드에서 발생하는 런타임 에러**를 보여준다. 그 다음 같은 코드를 TS로 작성하면 **컴파일 타임에 에러가 잡히는 것**을 보여준다.

```
고통 (JS):
  function getUser(id) {
    // 6개월 후 다른 개발자가 이 함수를 호출
    return fetch("/api/users/" + id);
  }
  getUser("abc");  // 런타임에서야 에러 발견 — 서비스 장애!

해결 (TS):
  function getUser(id: number): Promise<User> {
    return fetch("/api/users/" + id);
  }
  getUser("abc");  // ❌ 컴파일 에러! 코드 작성 시점에 바로 발견
```

#### 평가

| 항목 | 판정 |
|------|------|
| JS→TS 전환 전략 | **추론 우선 + 비유 확장** — 저항감 최소화 |
| 핵심 위험 | "왜 타입을 써야 해?" 저항 — 고통→해결 패턴으로 해소 |
| 인지 부하 관리 | 한 수업에 새 개념 1개, JS 비유 확장으로 친숙함 유지 |

### 3.2 김 시니어 — 시니어 개발자의 분석

#### 핵심 주장: "strict 모드와 실무 패턴을 처음부터 가르쳐야 한다"

**현업에서 TS를 쓰는 방식:**

1. `tsconfig.json`의 `strict: true`는 **비협상 사항**
2. `any`는 코드 리뷰에서 **즉각 리젝트**
3. API 응답은 항상 `unknown` → 타입 가드 → 안전한 타입
4. 컴포넌트 Props는 `interface`로, 유니언 값은 `type`으로

**40시간 과정의 기술 스택 권장:**

| 도구 | 역할 | 이유 |
|------|------|------|
| **tsx** | TS 파일 즉시 실행 | 제로 설정, 빠름, ts-node보다 간편 |
| **TypeScript Playground** | 첫날 실습 | 설치 없이 브라우저에서 즉시 |
| **VS Code** | 편집기 | TS 지원 내장, 자동완성 최고 |
| **Vitest** | 테스트 (선택적) | TS 네이티브 지원 |
| **Vite + react-ts** | React 연동 (선택적) | 프로젝트 챕터용 |

**40시간 범위 판정:**

```
✅ 포함:
  기본 타입, 함수 타입, 유니언, 인터페이스/타입 별칭, 타입 좁히기,
  enum, 클래스, 제네릭 기초, 유틸리티 타입, 모듈, 프로젝트 2개

❌ 제외 (40시간 초과):
  조건부 타입 (T extends U ? X : Y)
  매핑 타입 ([K in keyof T])
  템플릿 리터럴 타입
  데코레이터
  infer 키워드
  namespace (레거시)
  고급 제네릭 패턴
```

**interface vs type 명확한 규칙:**

> - **interface**: 객체 형태 정의, 확장(extends) 필요할 때, 공개 API
> - **type**: 유니언, 교차, 원시 타입 별칭, 튜플
> - TS 공식 가이드: "type이 필요한 기능을 쓸 때까지 interface를 사용하세요"

#### 평가

| 항목 | 판정 |
|------|------|
| 기술 정확성 | strict + tsx + unknown 우선 — **현업 표준 준수** |
| 40시간 범위 | 조건부/매핑 타입 제외는 **적절** |
| 도구 선택 | tsx는 ts-node보다 **확실히 우수** (제로 설정) |

### 3.3 이 비평가 — 교육 비평가의 분석

#### 핵심 주장: "제네릭은 40시간 과정의 최대 위험"

**위험 1: 제네릭의 인지 벽**

제네릭은 모든 TS 교육에서 가장 높은 이탈률을 보이는 주제다.

```
문제: <T> 문법이 수학적 추상 → 비전공자에게 공포 유발
  function identity<T>(arg: T): T { return arg; }
  // "T가 뭐야?" "왜 꺾쇠가 있어?" "이게 HTML 태그야?"
```

**완화 전략:**
1. **동기 먼저**: `any`를 써야 하는 상황을 먼저 보여주고, "이 `any`를 안전하게 바꿀 방법이 있습니다" → 제네릭
2. **읽기 먼저, 쓰기 나중**: `Array<string>`, `Promise<User>` 읽는 법을 먼저 가르치고, 직접 제네릭 함수를 작성하는 건 2번째 단계
3. **비유 집중**: "택배 서비스 — 내용물(T)이 뭔지 모르지만 안전 배달 보장"

**위험 2: "왜 타입을 써야 해?" 저항**

JS 과정을 마친 학습자는 "JS로도 잘 동작하는데 왜 추가 작업을?"이라고 느낀다.

연구 데이터:
> TypeScript는 일반적인 버그의 약 **15%를 컴파일 타임에 잡는다**. — freeCodeCamp

이 수치만으로는 동기부여가 약하다. 더 효과적인 접근:

```
[첫 수업 시작]
강사: "여기 JavaScript 코드가 있습니다. 버그를 찾아보세요."
  
  function calculateTotal(items) {
    return items.reduce((sum, item) => sum + item.price, 0);
  }
  calculateTotal("not an array");  // 런타임 에러!
  calculateTotal([{price: "100"}]); // NaN! 문자열 + 문자열

학생: (3분간 고민)

강사: "TypeScript에서는 이 코드를 저장하는 순간 빨간 줄이 뜹니다."
  → 즉각적 체감 → "아, 이래서 타입이 필요하구나"
```

**위험 3: tsconfig.json 설정 지옥**

tsconfig.json에는 100개 이상의 옵션이 있다. 비전공자가 이 파일을 열면 **즉시 공포**를 느낀다.

**완화**: 첫날은 tsx로 tsconfig 없이 실행. tsconfig은 프로젝트 챕터에서 "필요한 8개 옵션만" 설명하고, 나머지는 "추천 설정 파일"로 제공.

**위험 4: 에러 메시지 해독 불가**

TypeScript 에러 메시지는 **악명 높게 장황**하다:

```
Type '{ name: string; age: string; }' is not assignable to type 'User'.
  Types of property 'age' are incompatible.
    Type 'string' is not assignable to type 'number'.
```

비전공자는 이 메시지를 보고 "무슨 소리야?"라고 반응한다.

**완화**: "TS 에러 읽는 법" 전용 섹션을 초반에 배치. **아래에서 위로** 읽는 규칙 가르치기:
1. 마지막 줄: "string은 number에 할당 불가" ← **핵심 원인**
2. 중간 줄: "age 속성이 호환 안 됨" ← **어디서**
3. 첫째 줄: "이 객체가 User 타입에 안 맞음" ← **전체 맥락**

#### 평가

| 항목 | 판정 |
|------|------|
| 40시간 범위 | **타이트하지만 가능** — 제네릭을 "읽기+기초 쓰기"로 제한 시 |
| 최대 위험 | 제네릭 + 타입 저항 + 에러 메시지 = **3중 벽** |
| 필수 완화 | 고통→해결, 에러 읽기 교육, 제네릭 동기-먼저 |

## 4. 쟁점별 페르소나 간 교차 분석

### 4.1 "React + TypeScript를 포함하는가?"

| 페르소나 | 입장 | 근거 |
|---------|------|------|
| Dr. 타입 | 제외 — 40시간에 React까지 넣으면 인지 과부하 | 한 과정에 두 기술 동시 학습 위험 |
| 김 시니어 | 포함하고 싶으나 시간 부족 | 현업에서 TS+React가 표준 |
| 이 비평가 | **제외** — TS 기초도 빠듯한 40시간에 React는 무리 | 범위 초과 위험 |
| 정 설계자 | 제외, 단 15장 "실전 패턴"에서 React 타입 맛보기(1시간) 소개 | 절충 |

**합의**: TS+React는 **별도 과정**으로 분리. 15장에서 "다음으로 배우면 좋은 것"으로 React+TS를 소개만 한다.

### 4.2 "제네릭을 어디까지 가르치는가?"

| 페르소나 | 입장 | 근거 |
|---------|------|------|
| Dr. 타입 | **읽기+기초 쓰기**만 | 비전공자에게 고급 제네릭은 인지 벽 |
| 김 시니어 | 기초 + 제약(`extends`) + 내장 타입 이해 | 현업 최소 역량 |
| 이 비평가 | 기초만 — 제약은 선택 | 이탈 위험 최소화 |
| 정 설계자 | **3단계**: 읽기 → 기초 쓰기 → 제약 | 점진적 도입 |

**합의**: 제네릭은 **3단계**로 가르친다:
1. `Array<string>`, `Promise<User>` **읽기** (30분)
2. `function identity<T>(arg: T): T` **기초 쓰기** (1시간)
3. `<T extends HasId>` **제약** (1시간)

조건부 타입, 매핑 타입, infer는 **제외**.

### 4.3 "tsx vs tsc + node, 어떤 실행 방식?"

| 도구 | 장점 | 단점 |
|------|------|------|
| **tsx** | 제로 설정, 빠름, ESM/CJS 자동 | 컴파일 과정이 숨겨짐 |
| **tsc + node** | 컴파일 과정 가시적, .js 출력 확인 | 설정 복잡, 느림 |

**합의**: 
- **01-02장**: TypeScript Playground (설치 없이 시작)
- **03-13장**: **tsx** (제로 설정, 코드에 집중)
- **14-16장**: 프로젝트에서 **tsc** 빌드 경험 (컴파일 과정 이해)

### 4.4 "enum을 가르치는가, 유니언 리터럴로 대체하는가?"

| 페르소나 | 입장 | 근거 |
|---------|------|------|
| 김 시니어 | 둘 다 가르치되, 유니언 리터럴 우선 | 모던 TS 트렌드 |
| 이 비평가 | enum 과감히 제거, 유니언만 | 개념 최소화 |
| 정 설계자 | **둘 다 가르치고 비교** | 실무에서 둘 다 만남 |

**합의**: 둘 다 가르치되, **"유니언 리터럴이 더 간단하고, enum은 특별한 경우에 사용"**으로 프레이밍.

## 5. 비전공자 특화 설계 원칙

### 5.1 매 챕터 구조 (JS 과정과 동일)

```
┌─ 학습 목표 (2-3개)
├─ <a id="toc"></a> + 진행 순서 (앵커 링크)
├─ 1️⃣ 비유로 시작 — JS 과정 비유 확장 [↑](#toc)
├─ 2️⃣ 고통 (JS 코드의 문제) → 해결 (TS의 답) [↑](#toc)
├─ 3️⃣ 핵심 문법 + 다수의 예제 [↑](#toc)
├─ ...
├─ N️⃣ 정리 [↑](#toc)
│   ├── 핵심 개념 요약 (표: 개념 | 설명 | 비유)
│   ├── 다음 장 미리보기
│   └── 실습 과제 (기본 / 중급 / 심화)
└─ JS 과정 역참조: "JS N장 → TS M장" 연결
```

### 5.2 TypeScript Playground 활용

- **01-02장**: 설치 없이 Playground에서 즉시 시작
- **모든 챕터**: Playground 링크로 예제 공유 (URL에 코드 포함)
- **에러 교육**: Playground의 Errors 탭으로 에러 메시지 읽기 연습

### 5.3 "JS 코드 → TS 코드" Before/After 패턴

매 챕터에 최소 1개의 Before(JS)/After(TS) 비교:

```
// Before (JavaScript) — JS 5장에서 작성한 코드
function greet(name) {
  return "안녕, " + name;
}
greet(42);  // 런타임: "안녕, 42" — 의도치 않은 동작

// After (TypeScript) — 타입 추가
function greet(name: string): string {
  return "안녕, " + name;
}
greet(42);  // ❌ 컴파일 에러! number는 string이 아닙니다
```

### 5.4 TS 에러 읽기 교육 (02장에서 도입)

```
에러 읽기 3단계 규칙:

  Type '{ name: string; age: string; }' is not assignable 
  to type 'User'.                    ← ③ 전체 맥락
    Types of property 'age' are 
    incompatible.                    ← ② 어디서
      Type 'string' is not assignable 
      to type 'number'.             ← ① 핵심 원인 (여기부터!)

규칙: "아래에서 위로 읽으세요!"
```

## 7. 결론 — 페르소나 간 최종 합의

### 4인이 동의하는 것

1. **JS 과정과 동일한 스타일**(비유, 구조, 실습)로 통일해야 한다
2. **추론 우선** — 타입 주석보다 타입 추론을 먼저 가르쳐야 한다
3. **고통→해결 패턴** — JS 버그를 먼저 보여주고 TS가 잡는 것을 시연
4. **strict 모드 첫날부터** — TS 7.0에서 기본값이 될 예정
5. **any 금지 정책** — 나쁜 예시로만 사용, `unknown`을 대안으로
6. **제네릭은 읽기→쓰기→제약** 3단계로 점진적 도입
7. **에러 읽기 교육**을 초반에 배치해야 한다
8. **프로젝트는 JS 과정의 ToDo/날씨 앱을 TS로 리팩터링** — Before/After 극대화
9. **React+TS는 별도 과정**으로 분리

### 4인이 동의하지 않는 것

| 쟁점 | 범위 |
|------|------|
| enum 교육 깊이 | 김 시니어(상세히) vs 이 비평가(최소한) |
| 클래스 교육 범위 | Dr. 타입(기초만) vs 김 시니어(abstract까지) |
| 15장 React 맛보기 포함 여부 | 김 시니어(포함) vs 이 비평가(완전 제외) |
