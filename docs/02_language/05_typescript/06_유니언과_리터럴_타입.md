# 05장. 유니언 타입과 리터럴 타입

<a id="part1"></a>

## 1️⃣ 유니언 타입 [↑](#toc)

### 신분증 비유

> 공항 입국 심사대에서는 **여권 또는 신분증**, 둘 다 받습니다.
> 하지만 어떤 것을 제출했는지 확인한 뒤에만 해당 정보를 사용합니다.
> TypeScript의 유니언 타입도 같습니다 — 여러 타입을 허용하지만, 사용 전에 어떤 타입인지 확인이 필요합니다.

---

### 유니언 타입 기본

`|` 기호로 여러 타입을 연결합니다. "A 타입이거나 B 타입"을 의미합니다.

```typescript
// string 또는 number 타입
let id: string | number;

id = "user-123";  // ✅ string
id = 12345;       // ✅ number
// id = true;     // ❌ boolean은 허용되지 않음
```

---

### 유니언 타입 함수

```typescript
// id가 string일 수도 number일 수도 있는 함수
function printId(id: string | number): void {
  console.log(`ID: ${id}`);
}

printId("user-abc");  // ID: user-abc
printId(99);          // ID: 99
```

실행 결과:
```
ID: user-abc
ID: 99
```

---

### 유니언 타입에서 공통 메서드만 사용 가능

```typescript
function formatId(id: string | number): string {
  // id가 string | number 이므로 두 타입 모두에 있는 메서드만 사용 가능
  return id.toString(); // ✅ number와 string 모두 toString()이 있음

  // ❌ string에만 있는 메서드는 직접 사용 불가
  // return id.toUpperCase(); // Property 'toUpperCase' does not exist on type 'number'.
}

console.log(formatId("hello")); // hello
console.log(formatId(42));      // 42
```

실행 결과:
```
hello
42
```

## 2️⃣ 타입 좁히기 맛보기 [↑](#toc)

### 공항 보안 게이트 비유

> 공항 보안 게이트를 통과하면 "이 승객은 탑승권이 있음"이 확인됩니다.
> TypeScript에서 `typeof` 체크를 통과하면, 그 블록 안에서는 타입이 확정됩니다.

유니언 타입의 변수에서 특정 타입의 메서드를 사용하려면 **타입 좁히기(Type Narrowing)**가 필요합니다.

```typescript
function printId(id: string | number): void {
  // typeof로 타입 확인 — 타입 좁히기
  if (typeof id === "string") {
    // 이 블록 안에서 id는 string으로 확정됨
    console.log(`문자열 ID: ${id.toUpperCase()}`); // ✅ toUpperCase 사용 가능
  } else {
    // 이 블록 안에서 id는 number로 확정됨
    console.log(`숫자 ID: ${id.toFixed(0)}`); // ✅ toFixed 사용 가능
  }
}

printId("user-abc"); // 문자열 ID: USER-ABC
printId(99);         // 숫자 ID: 99
```

실행 결과:
```
문자열 ID: USER-ABC
숫자 ID: 99
```

타입 좁히기의 전체 내용은 7장에서 자세히 다룹니다. 여기서는 "유니언 타입에서 특정 타입을 사용하려면 확인이 필요하다"는 개념만 이해하면 됩니다.

## 3️⃣ 리터럴 타입 [↑](#toc)

### 특정 값만 허용

> 리터럴 타입은 **특정 값 자체를 타입으로 사용**합니다.
> `string` 대신 `"서울"`, `number` 대신 `42`처럼 구체적인 값만 허용합니다.

```typescript
// 리터럴 타입: 특정 값만 허용
let direction: "left" | "right" | "up" | "down";

direction = "left";  // ✅
direction = "right"; // ✅
// direction = "forward"; // ❌ Type '"forward"' is not assignable to type '"left" | "right" | "up" | "down"'.
```

---

### type 키워드로 리터럴 타입 이름 붙이기

```typescript
// type 별칭으로 이름 붙이기 (6장에서 자세히)
type Direction = "up" | "down" | "left" | "right";

function move(direction: Direction): void {
  console.log(`${direction} 방향으로 이동`);
}

move("up");    // up 방향으로 이동
move("left");  // left 방향으로 이동
// move("diagonal"); // ❌ 에러
```

실행 결과:
```
up 방향으로 이동
left 방향으로 이동
```

---

### 숫자 리터럴 타입

```typescript
// 숫자 리터럴 타입
type DiceValue = 1 | 2 | 3 | 4 | 5 | 6;

function rollDice(): DiceValue {
  return (Math.floor(Math.random() * 6) + 1) as DiceValue;
}

const result: DiceValue = rollDice();
console.log(`주사위: ${result}`);
```

---

### const vs let의 리터럴 타입

```typescript
// const — 값이 바뀌지 않으므로 리터럴 타입으로 추론
const direction = "north"; // 타입: "north" (리터럴 타입!)

// let — 값이 바뀔 수 있으므로 넓은 타입으로 추론
let status = "active";     // 타입: string (리터럴 타입 아님)
```

이것이 `const`를 기본으로 사용하는 또 다른 이유입니다.

## 4️⃣ 리터럴 + 유니언 실전 [↑](#toc)

### API 요청 상태 — 프론트엔드 필수 패턴

```typescript
// API 요청 상태를 타입으로 정의
type RequestStatus = "idle" | "loading" | "success" | "error";

interface ApiState {
  status: RequestStatus;
  data: string | null;
  errorMessage: string | null;
}

function renderUI(state: ApiState): string {
  switch (state.status) {
    case "idle":
      return "대기 중...";
    case "loading":
      return "로딩 중...";
    case "success":
      return `데이터: ${state.data}`;
    case "error":
      return `에러: ${state.errorMessage}`;
  }
  // TypeScript가 모든 케이스를 다뤘는지 확인함 (완전성 검사)
}

const loadingState: ApiState = { status: "loading", data: null, errorMessage: null };
const successState: ApiState = { status: "success", data: "홍길동", errorMessage: null };

console.log(renderUI(loadingState)); // 로딩 중...
console.log(renderUI(successState)); // 데이터: 홍길동
```

실행 결과:
```
로딩 중...
데이터: 홍길동
```

### 신호등 — 리터럴 타입 활용

```typescript
type TrafficLight = "red" | "yellow" | "green";

function getTrafficAction(light: TrafficLight): string {
  switch (light) {
    case "red":    return "정지하세요";
    case "yellow": return "감속하세요";
    case "green":  return "출발하세요";
  }
}

console.log(getTrafficAction("red"));    // 정지하세요
console.log(getTrafficAction("green"));  // 출발하세요
// console.log(getTrafficAction("blue")); // ❌ 컴파일 에러
```

실행 결과:
```
정지하세요
출발하세요
```

---

### 리터럴 타입의 완전성 검사

TypeScript는 `switch`에서 모든 케이스를 다루는지 검사할 수 있습니다.

```typescript
type Season = "spring" | "summer" | "fall" | "winter";

function getWeather(season: Season): string {
  switch (season) {
    case "spring": return "따뜻함";
    case "summer": return "더움";
    case "fall":   return "시원함";
    // ❌ "winter"를 빠뜨리면 TypeScript가 반환 타입에서 경고
    // Function lacks ending return statement and return type does not include 'undefined'.
  }
}
```

이런 경고는 실수로 케이스를 빠뜨리는 버그를 방지해줍니다.

## 5️⃣ null과 유니언 [↑](#toc)

### "값이 없을 수도 있는" 패턴

실무에서 가장 많이 사용하는 유니언 타입이 `T | null`입니다.

```typescript
// 사용자 정보가 있을 수도, 없을 수도 있는 상황
type UserOrNull = { name: string; age: number } | null;

function displayUser(user: UserOrNull): void {
  if (user === null) {
    console.log("로그인이 필요합니다.");
    return;
  }
  // 여기서 user는 null이 아님이 보장됨
  console.log(`${user.name}님 (${user.age}세)`);
}

displayUser({ name: "홍길동", age: 25 }); // 홍길동님 (25세)
displayUser(null);                         // 로그인이 필요합니다.
```

실행 결과:
```
홍길동님 (25세)
로그인이 필요합니다.
```

---

### 옵셔널 체이닝과 조합

JS 07장에서 배운 `?.`과 유니언 타입을 함께 사용합니다.

```typescript
type Config = {
  theme: string;
  language: string;
} | null;

// ?. 사용: null이면 undefined 반환, null이 아니면 속성 접근
const userConfig: Config = null;
const theme = userConfig?.theme ?? "dark"; // null이면 "dark" 기본값
console.log(`테마: ${theme}`); // 테마: dark

const activeConfig: Config = { theme: "light", language: "ko" };
const lang = activeConfig?.language ?? "en";
console.log(`언어: ${lang}`); // 언어: ko
```

실행 결과:
```
테마: dark
언어: ko
```

---

### string | undefined vs string | null

| | `string \| null` | `string \| undefined` |
|---|---|---|
| 명시적 "없음" | ✅ null = 의도적으로 비어있음 | ❌ |
| 선택적 속성 | ❌ | ✅ `prop?: string`과 동일 |
| API 응답 | ✅ 자주 사용 | △ |
| 처리 방법 | `if (x !== null)` | `if (x !== undefined)` 또는 `if (x)` |

## 6️⃣ 정리 [↑](#toc)

### 핵심 개념 요약

| 개념 | 설명 | 비유 |
|------|------|------|
| 유니언 타입 | `A \| B` — A이거나 B | 여권 또는 신분증 둘 다 허용 |

| 완전성 검사 | `switch`에서 모든 케이스 처리 확인 | 모든 시나리오 대비 |

  errorCode: number;
  message: string;

**심화**: 리터럴 타입을 활용해 간단한 상태 머신을 구현하세요.

```typescript
// 주문 상태를 나타내는 타입
type OrderStatus = "pending" | "paid" | "shipping" | "delivered" | "cancelled";

// 유효한 상태 전이를 확인하는 함수
// 예: pending → paid ✅, pending → delivered ❌
function canTransition(from: OrderStatus, to: OrderStatus): boolean {
  const validTransitions: Record<OrderStatus, OrderStatus[]> = {
    pending: ["paid", "cancelled"],
    paid: ["shipping", "cancelled"],
    shipping: ["delivered"],
    delivered: [],
    cancelled: []
  };
  return validTransitions[from].includes(to);
}

console.log(canTransition("pending", "paid"));      // true
console.log(canTransition("pending", "delivered")); // false
console.log(canTransition("shipping", "delivered")); // true
```
