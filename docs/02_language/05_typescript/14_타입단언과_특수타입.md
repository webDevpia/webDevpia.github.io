# 13장. 타입 단언과 특수 타입

<a id="part1"></a>

## 1️⃣ 타입 단언 (as) [↑](#toc)

### "컴파일러에게 '나를 믿어'라고 말하기"

TypeScript 컴파일러가 타입을 알지 못하는 상황에서, 개발자가 직접 "이 값은 이 타입이야"라고 주장하는 것이 **타입 단언(Type Assertion)**입니다.

> 친구에게 "저 사람이 교수님이야, 믿어"라고 말하는 것과 같습니다.
> 친구(컴파일러)는 확인하지 않고 그 말을 믿습니다. 당신이 틀렸다면 나중에 문제가 생깁니다.

```typescript
// DOM 요소를 가져올 때 — TypeScript는 null일 수도 있다고 생각
const input = document.getElementById("username");
// input의 타입: HTMLElement | null

// ❌ 단언 없이 value 접근 — 에러
// input.value; // 에러: 'HTMLElement | null'에는 'value' 속성이 없음

// ✅ as로 타입 단언 — "이 요소는 HTMLInputElement야"
const inputElement = document.getElementById("username") as HTMLInputElement;
inputElement.value = "김철수"; // 이제 가능
```

### 타입 단언의 두 가지 문법

```typescript
// 문법 1: as 키워드 (권장 — JSX에서도 사용 가능)
const el = document.querySelector(".title") as HTMLHeadingElement;

// 문법 2: 꺾쇠 괄호 (React/JSX 파일에서는 사용 불가)
const el2 = <HTMLHeadingElement>document.querySelector(".title");
```

### 타입 단언의 한계와 위험

```typescript
// ⚠️ 타입 단언은 컴파일 타임 검사만 우회합니다
// 실제 값이 해당 타입이 아니면 런타임 에러가 발생합니다

const el = document.getElementById("not-exist") as HTMLInputElement;
// el은 null이지만 컴파일러는 모름

// 실제로는 null이므로 런타임 에러 발생
// el.value; // TypeError: Cannot read properties of null

// ✅ 안전하게 사용하려면 null 체크와 함께
const safeEl = document.getElementById("username") as HTMLInputElement | null;
if (safeEl) {
  safeEl.value = "안전"; // null 체크 후 사용
}
```

### 언제 as를 써도 괜찮은가?

```typescript
// 1. DOM 요소 — getElementById는 null | HTMLElement 반환이지만
//    특정 id가 항상 존재함을 알 때
const form = document.getElementById("main-form") as HTMLFormElement;

// 2. JSON.parse 결과 — 구조를 확실히 알 때 (되도록 타입 가드 권장)
const config = JSON.parse(rawJson) as AppConfig;

// 3. 타입 간 중간 단계 변환 (double assertion)
// ❌ 완전히 관련 없는 타입 사이 강제 변환은 피하세요
const weird = "hello" as unknown as number; // 컴파일은 되지만 위험
```

## 2️⃣ non-null 단언 (!) [↑](#toc)

### "null이 아니라고 확신할 때"

`!` 연산자는 값 뒤에 붙여서 "이 값은 null도 아니고 undefined도 아니야"라고 컴파일러에게 알립니다.

```typescript
interface User {
  name: string;
  email?: string; // 선택적 속성
}

const user: User = { name: "김철수" };

// ❌ 선택적 속성 — undefined일 수 있음
// console.log(user.email.toUpperCase()); // 에러: 'undefined'일 수 있음

// ✅ non-null 단언 — "email이 반드시 있음을 개발자가 보장"
console.log(user.email!.toUpperCase());
// 주의: user.email이 undefined면 런타임 에러 발생!
```

실행 결과:
```
// undefined의 toUpperCase() 호출 — 런타임 에러
```

### 옵셔널 체이닝(`?.`)이 더 안전합니다

```typescript
// ❌ non-null 단언 — 잘못되면 런타임 에러
console.log(user.email!.toUpperCase());

// ✅ 옵셔널 체이닝 — undefined면 조용히 undefined 반환
console.log(user.email?.toUpperCase()); // undefined (에러 없음)

// ✅ ?? 와 함께 기본값 제공
console.log(user.email?.toUpperCase() ?? "이메일 없음"); // "이메일 없음"
```

실행 결과:
```
undefined
이메일 없음
```

### ! 사용이 허용되는 드문 경우

```typescript
// 클래스에서 나중에 반드시 초기화되는 필드
class UserService {
  // 생성자가 아니라 init() 메서드로 초기화됨을 개발자가 알 때
  private db!: Database; // ! 없으면 "초기화 전에 사용됨" 에러

  async init(): Promise<void> {
    this.db = await Database.connect();
  }

  async findUser(id: number) {
    return this.db.query(`SELECT * FROM users WHERE id = ${id}`);
  }
}
```

## 3️⃣ unknown — 미개봉 택배 [↑](#toc)

### any보다 안전한 불확실성 표현

> `unknown`은 미개봉 택배 상자와 같습니다. 안에 무엇이 있는지 모르므로, 열어보기(타입 좁히기) 전까지는 사용할 수 없습니다.
> `any`는 검사를 완전히 포기하는 것입니다. `unknown`은 "모르지만, 반드시 확인 후 사용"을 강제합니다.

```typescript
// any — 모든 연산 허용, 타입 안전성 포기
let anyValue: any = "안녕";
anyValue.toFixed(2);      // 컴파일 통과 (런타임 에러 가능)
anyValue.nonExistMethod(); // 컴파일 통과 (런타임 에러 가능)

// unknown — 좁히기 전에 사용하면 컴파일 에러
let unknownValue: unknown = "안녕";
// unknownValue.toUpperCase(); // ❌ 에러: 'unknown'에는 이 연산 불가

// ✅ typeof로 좁힌 후 사용
if (typeof unknownValue === "string") {
  console.log(unknownValue.toUpperCase()); // 이제 string으로 확정
}
```

실행 결과:
```
안녕
```

### 실전 패턴: API 응답 처리

```typescript
// fetch는 응답 본문의 구조를 알 수 없음 → unknown 사용

interface UserData {
  id: number;
  name: string;
  email: string;
}

// 타입 가드 함수 (7장 타입 좁히기 복습)
function isUserData(data: unknown): data is UserData {
  return (
    typeof data === "object" &&
    data !== null &&
    "id" in data &&
    "name" in data &&
    "email" in data &&
    typeof (data as UserData).id === "number" &&
    typeof (data as UserData).name === "string"
  );
}

async function fetchUser(id: number): Promise<UserData> {
  const response = await fetch(`https://api.example.com/users/${id}`);
  const data: unknown = await response.json(); // ✅ unknown으로 받기

  // ❌ any로 바로 사용 — 런타임에 구조가 달라도 에러 없음
  // const data: any = await response.json();

  // ✅ 타입 가드로 검증 후 사용
  if (!isUserData(data)) {
    throw new Error("API 응답 형식이 올바르지 않습니다");
  }

  return data; // 이제 UserData 타입으로 확정
}
```

### any vs unknown 비교

| 특성 | `any` | `unknown` |
|------|-------|---------|
| 할당 | 모든 타입에서 가능 | 모든 타입에서 가능 |
| 사용 | 즉시 모든 연산 허용 | 타입 좁히기 후에만 허용 |
| 타입 검사 | 비활성화 | 강제 |
| 권장도 | 피하는 것 권장 | 불확실한 값에 권장 |
| 비유 | 검사 없이 사용 | 열어보고 확인 후 사용 |

## 4️⃣ never — 절대 발생하지 않는 타입 [↑](#toc)

### "이 코드에는 절대 도달하면 안 됩니다"

`never`는 **절대 값이 될 수 없는 타입**입니다. 세 가지 상황에서 등장합니다.

```typescript
// 상황 1: 항상 예외를 던지는 함수
function throwError(message: string): never {
  throw new Error(message);
  // 이 함수는 정상적으로 반환되지 않음
}

// 상황 2: 무한 루프 함수
function infiniteLoop(): never {
  while (true) {
    // 절대 종료되지 않음
  }
}

// 상황 3: switch 문의 완전성 검사 (가장 실용적!)
```

### switch 완전성 검사 — never의 진가

```typescript
type Shape = "circle" | "square" | "triangle";

function getArea(shape: Shape, size: number): number {
  switch (shape) {
    case "circle":
      return Math.PI * size * size;
    case "square":
      return size * size;
    case "triangle":
      return (size * size) / 2;
    default:
      // ✅ 모든 case를 처리했다면 shape는 never 타입이어야 함
      const exhaustiveCheck: never = shape;
      throw new Error(`처리되지 않은 도형: ${exhaustiveCheck}`);
  }
}

console.log(getArea("circle", 5));   // 78.53...
console.log(getArea("square", 4));   // 16
console.log(getArea("triangle", 3)); // 4.5
```

실행 결과:
```
78.53981633974483
16
4.5
```

### never의 강점: 새 케이스 추가 시 컴파일 에러

```typescript
// 나중에 "hexagon"이 Shape에 추가되면?
type Shape = "circle" | "square" | "triangle" | "hexagon"; // 추가!

function getArea(shape: Shape, size: number): number {
  switch (shape) {
    case "circle":   return Math.PI * size * size;
    case "square":   return size * size;
    case "triangle": return (size * size) / 2;
    // "hexagon" case가 없음!
    default:
      const exhaustiveCheck: never = shape;
      // ❌ 컴파일 에러: 'hexagon'은 'never'에 할당 불가
      // → TypeScript가 "hexagon case를 처리하지 않았어!"라고 알려줌
      throw new Error(`처리되지 않은 도형: ${exhaustiveCheck}`);
  }
}
```

### never 타입의 동작 원리

```typescript
// 유니언 타입에서 never는 제거됨
type Result = string | never; // string (never 제거됨)

// 두 타입의 교차가 불가능하면 never
type Impossible = string & number; // never
```

## 5️⃣ satisfies 연산자 [↑](#toc)

### TypeScript 4.9+ 신기능: 검증하되 추론은 유지

`satisfies`는 **타입을 검증하면서도 더 정확한 추론된 타입을 유지**합니다.
타입 어노테이션(`: Config`)은 타입을 해당 타입으로 넓히지만, `satisfies`는 좁은 추론 타입을 유지합니다.

> 고통: 객체에 타입을 명시하면 개별 속성의 구체적인 타입 정보를 잃습니다.
> 해결: `satisfies`로 타입 준수는 검증하고, 추론된 정확한 타입은 유지합니다.

```typescript
type Color = string | [number, number, number]; // hex 문자열 또는 RGB 배열

interface Palette {
  primary: Color;
  secondary: Color;
  accent: Color;
}

// ❌ 타입 어노테이션만 사용 — 개별 속성은 Color(넓은 타입)로 처리됨
const palette1: Palette = {
  primary: "#ff0000",       // string
  secondary: [0, 128, 255], // [number, number, number]
  accent: "#00ff00",        // string
};

// palette1.primary는 Color 타입 → string 메서드 못 씀
// palette1.primary.toUpperCase(); // ❌ 에러: Color에는 toUpperCase 없음

// ✅ satisfies — Palette 준수를 검증하면서 추론된 타입 유지
const palette2 = {
  primary: "#ff0000",       // string으로 추론 유지
  secondary: [0, 128, 255], // [number, number, number]으로 추론 유지
  accent: "#00ff00",        // string으로 추론 유지
} satisfies Palette;

// 추론된 구체 타입이 유지됨
console.log(palette2.primary.toUpperCase());    // ✅ string 메서드 사용 가능
console.log(palette2.secondary[0]);             // ✅ 0 (숫자로 접근 가능)
```

실행 결과:
```
#FF0000
0
```

### satisfies의 실용 패턴: 설정 객체

```typescript
interface RouteConfig {
  path: string;
  method: "GET" | "POST" | "PUT" | "DELETE";
  requiresAuth: boolean;
}

// satisfies — 설정 구조 검증 + 리터럴 타입 추론 유지
const routes = {
  getUser:    { path: "/users/:id", method: "GET",    requiresAuth: true  },
  createUser: { path: "/users",     method: "POST",   requiresAuth: false },
  deleteUser: { path: "/users/:id", method: "DELETE", requiresAuth: true  },
} satisfies Record<string, RouteConfig>;

// ✅ method는 "GET" | "POST" | ... (구체 타입) — 리터럴 타입 유지
const method = routes.getUser.method;
// type: "GET" (넓은 "GET" | "POST" | "PUT" | "DELETE"가 아님)
console.log(method); // "GET"

// ❌ 잘못된 설정은 즉시 컴파일 에러
// const badRoutes = {
//   login: { path: "/login", method: "PATCH", requiresAuth: false }
//   //                                ^^^^^^ 에러: "PATCH"는 RouteConfig에 없음
// } satisfies Record<string, RouteConfig>;
```

실행 결과:
```
GET
```

### as vs satisfies 비교

| 특성 | `as Type` | `satisfies Type` |
|------|-----------|-----------------|
| 타입 검증 | 없음 (강제 변환) | 있음 (불일치 시 에러) |
| 추론된 타입 | Type으로 넓혀짐 | 원래 추론 타입 유지 |
| 안전성 | 낮음 | 높음 |
| 사용 목적 | "이 타입이야" 주장 | "이 타입을 준수해" 검증 |

## 6️⃣ 타입 단언을 줄이는 방법 [↑](#toc)

### "as를 쓰고 싶을 때, 먼저 타입 가드를 고려하세요"

타입 단언(`as`)은 컴파일러 검사를 우회하므로, 가능하면 타입 가드로 대체하는 것이 더 안전합니다.

```typescript
// ❌ as 사용 — 구조 검증 없이 강제 변환
async function loadConfig(): Promise<AppConfig> {
  const raw = await fetch("/config.json").then(r => r.json());
  return raw as AppConfig; // 실제 구조와 달라도 컴파일 통과
}

// ✅ 타입 가드로 리팩터링 — 구조 검증 포함
interface AppConfig {
  apiUrl: string;
  timeout: number;
  debug: boolean;
}

function isAppConfig(data: unknown): data is AppConfig {
  return (
    typeof data === "object" &&
    data !== null &&
    typeof (data as AppConfig).apiUrl === "string" &&
    typeof (data as AppConfig).timeout === "number" &&
    typeof (data as AppConfig).debug === "boolean"
  );
}

async function loadConfig(): Promise<AppConfig> {
  const raw: unknown = await fetch("/config.json").then(r => r.json());

  if (!isAppConfig(raw)) {
    throw new Error("설정 파일 형식 오류: apiUrl, timeout, debug 필드가 필요합니다");
  }

  return raw; // 이제 AppConfig 타입으로 안전하게 반환
}
```

### DOM 요소 접근 — 더 안전한 패턴

```typescript
// ❌ as로 강제 변환 — null 체크 없음
const input = document.getElementById("name") as HTMLInputElement;
input.value = "김철수"; // getElementById가 null 반환하면 런타임 에러

// ✅ 타입 가드(instanceof)로 안전하게
function getInputElement(id: string): HTMLInputElement {
  const el = document.getElementById(id);

  if (!(el instanceof HTMLInputElement)) {
    throw new Error(`#${id} 요소를 찾을 수 없거나 input 요소가 아닙니다`);
  }

  return el; // HTMLInputElement 타입으로 안전하게 좁혀짐
}

const nameInput = getInputElement("name");
nameInput.value = "김철수"; // ✅ 안전
```

### 타입 단언 최소화 체크리스트

```typescript
// as 쓰기 전에 먼저 고려할 것:

// 1. typeof 타입 가드로 해결 가능?
if (typeof value === "string") { /* string 타입 확정 */ }

// 2. instanceof 타입 가드로 해결 가능?
if (el instanceof HTMLInputElement) { /* 타입 확정 */ }

// 3. 사용자 정의 타입 가드로 해결 가능?
if (isUserData(data)) { /* UserData 타입 확정 */ }

// 4. 옵셔널 체이닝으로 해결 가능? (non-null 단언 대신)
user?.email?.toUpperCase();

// 5. 위 모두 안 될 때만 as 사용
const el = document.getElementById("name") as HTMLInputElement;
```

## 7️⃣ 정리 [↑](#toc)

### 핵심 개념 요약

| 개념 | 문법 | 핵심 특성 | 주의 사항 |
|------|------|----------|----------|
| 타입 단언 | `value as Type` | 컴파일러 검사 우회 | 검증 없음, 최소화 권장 |
| non-null 단언 | `value!` | null/undefined 아님 주장 | 런타임 에러 가능, `?.` 선호 |
| `unknown` | `let x: unknown` | 사용 전 좁히기 강제 | `any`보다 안전한 대안 |
| `never` | 함수 반환, switch default | 절대 발생하지 않는 타입 | 완전성 검사에 활용 |
| `satisfies` | `obj satisfies Type` | 검증 + 추론 타입 유지 | TS 4.9+, 설정 객체에 유용 |

### unknown vs any vs never — 세 가지 특수 타입

| 타입 | 의미 | 할당 가능 방향 | 사용 시점 |
|------|------|-------------|----------|
| `any` | "모르고, 신경 안 씀" | 양방향 자유 | 마이그레이션, 임시 처리 |
| `unknown` | "모르지만, 확인 필요" | 받기는 가능, 사용은 불가 | API 응답, 외부 입력 |
| `never` | "절대 불가능" | 어디에도 할당 불가 | 완전성 검사, 불가능 분기 |

### 단언 vs 가드 — 언제 무엇을?

```typescript
// 빠르지만 위험
const el = document.getElementById("id") as HTMLInputElement;

// 느리지만 안전 (null 가능성 처리 포함)
const el = document.getElementById("id");
if (el instanceof HTMLInputElement) {
  el.value = "안전";
}
```

**심화** — `satisfies`를 사용해 내비게이션 메뉴 설정을 타입 안전하게 만드세요.

```typescript
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";
type AuthLevel = "public" | "user" | "admin";

interface MenuItem {
  label: string;
  path: string;
  method: HttpMethod;
  auth: AuthLevel;
  icon?: string;
}

// satisfies를 사용해 아래 menu 객체를 선언하세요
// 조건: MenuItem 타입을 준수하면서, method의 리터럴 타입은 유지되어야 함
const menu = {
  home:    { label: "홈",      path: "/",       method: "GET",  auth: "public" },
  profile: { label: "프로필",  path: "/profile", method: "GET",  auth: "user"   },
  post:    { label: "글 작성", path: "/posts",   method: "POST", auth: "user"   },
  admin:   { label: "관리자",  path: "/admin",   method: "GET",  auth: "admin"  },
} /* 여기에 satisfies 추가 */;

// 타입 추론 확인: menu.home.method의 타입이 "GET"이어야 함
const homeMethod = menu.home.method;
console.log(homeMethod); // "GET"

// 인증 레벨별 메뉴 필터링
const publicMenus = Object.entries(menu)
  .filter(([_, item]) => item.auth === "public")
  .map(([_, item]) => item.label);

console.log(publicMenus); // ["홈"]
```
