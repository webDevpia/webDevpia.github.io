# 12장. 모듈과 타입 선언

> JS 08장(ES 모듈 시스템)과 JS 13장(Node.js, npm)을 먼저 보고 오셨다면 이 장의 내용이 훨씬 쉽게 이해됩니다.
> TypeScript에서 모듈 시스템은 JavaScript와 동일하되, 타입 정보를 어떻게 가져오고 내보내는지가 추가됩니다.

<a id="part1"></a>

## 1️⃣ import type / export type [↑](#toc)

### 타입만 가져오기

JavaScript의 `import`는 런타임에도 동작합니다. 하지만 타입 정보는 컴파일 후 JavaScript 코드에서 완전히 사라집니다.
`import type`은 이 사실을 명시적으로 표현합니다: "이것은 타입 정보만 가져오며, 런타임에는 필요 없습니다."

> 고통: 일반 `import`로 타입을 가져오면 번들러가 해당 파일을 런타임 의존성으로 포함시킬 수 있습니다.
> 해결: `import type`을 사용하면 컴파일 타임에만 사용되고, 컴파일된 JS에서는 완전히 제거됩니다.

```typescript
// ❌ 일반 import — 런타임에도 파일 참조가 남을 수 있음
import { User } from "./types";

// ✅ import type — 타입 정보만, 컴파일 후 완전히 사라짐
import type { User } from "./types";

// 여러 타입을 한번에 가져오기
import type { User, Product, Order } from "./types";

// 기본 내보내기 타입
import type ApiClient from "./api-client";
```

### export type — 타입만 내보내기

```typescript
// types.ts

interface User {
  id: number;
  name: string;
  email: string;
}

interface Product {
  id: number;
  name: string;
  price: number;
}

// export type — 타입만 내보냄 (런타임 값 아님을 명시)
export type { User, Product };

// 또는 인라인으로 선언과 동시에 내보내기
export interface Order {
  id: number;
  userId: number;
  products: Product[];
  total: number;
}
```

### import type vs import — 언제 무엇을?

| 대상 | 사용할 것 | 이유 |
|------|----------|------|
| 타입, 인터페이스만 | `import type` | 컴파일 후 자동 제거, 명시적 |
| 함수, 클래스, 상수 | `import` | 런타임에도 필요한 값 |
| 타입 + 값 혼합 | `import` 또는 분리 | 값이 포함되면 일반 import |

## 2️⃣ 타입 파일 분리 [↑](#toc)

### "공유 타입은 한 곳에 모아두기"

여러 파일에서 공통으로 사용하는 타입은 별도 파일로 분리합니다.
JS 13장(Node.js 프로젝트 구조)에서 배운 것처럼, 코드도 역할별로 나눠야 유지보수가 쉬워집니다.

```
src/
├── types/
│   ├── index.ts    ← 공통 타입 진입점 (재내보내기)
│   ├── user.ts     ← 사용자 관련 타입
│   ├── product.ts  ← 상품 관련 타입
│   └── api.ts      ← API 응답/요청 관련 타입
├── utils/
│   └── format.ts
└── main.ts
```

```typescript
// src/types/user.ts
export interface User {
  id: number;
  name: string;
  email: string;
  role: "admin" | "user";
}

export type CreateUserInput = Omit<User, "id">;
export type UpdateUserInput = Partial<Omit<User, "id">>;
```

```typescript
// src/types/api.ts
export interface ApiResponse<T> {
  success: boolean;
  data: T;
  message: string;
  timestamp: string;
}

export interface PaginatedResponse<T> extends ApiResponse<T[]> {
  page: number;
  totalPages: number;
  totalCount: number;
}
```

```typescript
// src/types/index.ts — 모든 타입을 한 곳에서 재내보내기
export type { User, CreateUserInput, UpdateUserInput } from "./user";
export type { ApiResponse, PaginatedResponse } from "./api";
```

```typescript
// src/main.ts — index.ts 하나에서 모든 타입을 가져올 수 있음
import type { User, ApiResponse } from "./types";

const user: User = {
  id: 1,
  name: "김철수",
  email: "kim@example.com",
  role: "user",
};

const response: ApiResponse<User> = {
  success: true,
  data: user,
  message: "조회 성공",
  timestamp: new Date().toISOString(),
};

console.log(response.data.name); // "김철수"
```

실행 결과:
```
김철수
```

## 3️⃣ 선언 파일 (.d.ts) [↑](#toc)

### "타입의 설명서"

`.d.ts` 파일은 **타입 선언만** 들어있는 파일입니다. 실제 구현 코드는 없고, "이 파일/라이브러리는 이런 타입들을 제공합니다"라는 설명서 역할을 합니다.

> 책의 목차와 같습니다. 목차만 봐도 책에 무엇이 있는지 알 수 있습니다.
> TypeScript 컴파일러는 `.d.ts`를 읽어 해당 모듈의 타입을 이해합니다.

### .d.ts 파일은 언제 생성되나?

```
상황 1: TypeScript 프로젝트를 컴파일할 때 (declaration: true)
  myLib.ts → myLib.js (실행 파일)
            → myLib.d.ts (타입 선언 파일)

상황 2: JavaScript 라이브러리에 타입 정보를 추가할 때
  lodash/ (JS 소스)
  @types/lodash/ (별도 .d.ts 파일들)

상황 3: 직접 글로벌 타입을 선언할 때
  global.d.ts (프로젝트 전역 타입 확장)
```

### .d.ts 직접 작성 예시

```typescript
// math-utils.d.ts
// "이 라이브러리는 이런 함수들을 제공합니다" — 구현 없이 타입만

export declare function add(a: number, b: number): number;
export declare function subtract(a: number, b: number): number;
export declare function multiply(a: number, b: number): number;

export declare interface MathResult {
  value: number;
  operation: string;
}
```

### tsconfig에서 선언 파일 생성 설정

```json
{
  "compilerOptions": {
    "declaration": true,       // .d.ts 파일 자동 생성
    "declarationDir": "./dist/types", // 생성 위치 지정
    "emitDeclarationOnly": true // .d.ts만 생성 (JS 없이)
  }
}
```

## 4️⃣ @types 패키지 [↑](#toc)

### DefinitelyTyped — 커뮤니티가 만든 타입 설명서 저장소

JavaScript 라이브러리 대부분은 TypeScript로 작성되지 않았습니다.
이런 라이브러리들의 타입 선언 파일을 커뮤니티가 모아둔 저장소가 **DefinitelyTyped**입니다.
`@types/` 접두사가 붙은 npm 패키지가 이에 해당합니다.

```bash
# Node.js API 타입 (fs, path, http 등)
npm install -D @types/node

# Express 프레임워크 타입
npm install -D @types/express

# Lodash 유틸리티 라이브러리 타입
npm install -D @types/lodash

# Jest 테스트 프레임워크 타입
npm install -D @types/jest
```

### @types/node 설치 후 사용 예시

```typescript
// @types/node 없이는 path, fs 등의 타입을 알 수 없음
import path from "path";
import fs from "fs";

// @types/node 설치 후 — 자동완성과 타입 검사가 됨
const filePath: string = path.join(__dirname, "data.json");
const content: Buffer = fs.readFileSync(filePath);

console.log(filePath);
console.log(typeof content); // "object" (Buffer 타입)
```

### @types 패키지가 필요한지 확인하는 방법

```typescript
// 방법 1: 설치 시 에러 메시지 확인
import express from "express";
// 에러: Could not find a declaration file for module 'express'
//      Try `npm install -D @types/express`
// → TypeScript가 직접 안내해줍니다!

// 방법 2: 라이브러리 자체에 types 포함 여부 확인
// package.json에 "types" 또는 "typings" 필드가 있으면 별도 설치 불필요
// 예: axios, zod, prisma 등은 자체 타입 포함
```

### 최신 라이브러리는 @types 불필요

```typescript
// axios — 라이브러리 자체에 타입 포함 (별도 @types 설치 불필요)
import axios from "axios";

const response = await axios.get("https://api.example.com/users");
// response.data 타입, response.status 등 모두 자동완성됨
```

## 5️⃣ 타입이 없는 라이브러리 [↑](#toc)

### `declare module` — 임시 처방

`@types` 패키지도 없고, 라이브러리 자체에도 타입이 없는 경우,
`declare module`로 해당 모듈을 `any` 타입으로 임시 처리할 수 있습니다.

> 고통: 타입 없는 라이브러리를 import하면 TypeScript가 에러를 냅니다.
> 해결: `declare module`로 "이 모듈은 내가 나중에 타입을 추가할게"라고 알려줍니다.

```typescript
// ❌ 타입 없는 라이브러리 import — 컴파일 에러
import myLegacyLib from "my-legacy-lib";
// 에러: Could not find a declaration file for module 'my-legacy-lib'
```

```typescript
// src/types/declarations.d.ts — 타입 선언 파일 생성

// 방법 1: any로 임시 처리 (가장 빠른 해결책)
declare module "my-legacy-lib";

// 방법 2: 실제로 사용하는 기능만 타입 정의 (더 안전)
declare module "my-legacy-lib" {
  export function process(input: string): string;
  export function validate(data: unknown): boolean;
  export const version: string;
  export default function run(config: Record<string, unknown>): void;
}
```

```typescript
// 이제 에러 없이 import 가능
import myLegacyLib from "my-legacy-lib";
import { process, validate } from "my-legacy-lib";

const result = process("입력값");
const isValid = validate({ key: "value" });
```

### 글로벌 타입 선언 확장

```typescript
// src/types/global.d.ts

// 환경 변수 타입 정의
declare namespace NodeJS {
  interface ProcessEnv {
    NODE_ENV: "development" | "production" | "test";
    API_URL: string;
    PORT?: string;
  }
}

// Window 객체에 커스텀 속성 추가
interface Window {
  __APP_VERSION__: string;
  analytics: {
    track: (event: string, data?: Record<string, unknown>) => void;
  };
}
```

```typescript
// global.d.ts 덕분에 타입 안전하게 사용 가능
const env = process.env.NODE_ENV; // "development" | "production" | "test"
const apiUrl = process.env.API_URL; // string

// window.__APP_VERSION__ 접근 시 타입 오류 없음
console.log(window.__APP_VERSION__);
```

## 6️⃣ tsconfig의 moduleResolution [↑](#toc)

### "TypeScript가 import 경로를 어떻게 찾는가"

`moduleResolution`은 TypeScript가 `import` 구문에서 모듈을 찾는 방식을 결정합니다.

```json
// tsconfig.json
{
  "compilerOptions": {
    "moduleResolution": "node"    // Node.js 방식 (전통적)
    // 또는
    "moduleResolution": "bundler" // 번들러(Vite, webpack) 환경용 (권장)
  }
}
```

### node vs bundler — 차이점

| 설정 | 환경 | 특징 |
|------|------|------|
| `"node"` | Node.js 직접 실행 | `.js` 확장자 명시 필요, CJS/ESM 엄격 |
| `"bundler"` | Vite, webpack, esbuild | 확장자 생략 가능, 현대적 개발 환경에 최적 |

```typescript
// moduleResolution: "node" — 확장자 명시 필요 (ESM 모드)
import { User } from "./types.js"; // .js 필요 (실제로는 .ts 파일)

// moduleResolution: "bundler" — 확장자 생략 가능
import type { User } from "./types"; // 더 간결
import { formatDate } from "./utils"; // 번들러가 알아서 해석
```

### 실전 권장 설정

```json
// 2025년 기준 Vite/Next.js 프로젝트용 권장 설정
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "esModuleInterop": true,
    "allowImportingTsExtensions": true,
    "noEmit": true
  }
}
```

## 7️⃣ 정리 [↑](#toc)

### 핵심 개념 요약

| 개념 | 설명 | 사용 시점 |
|------|------|----------|
| `import type` | 타입 정보만 가져오기, 컴파일 후 제거 | 타입/인터페이스만 필요할 때 |
| `export type` | 타입 정보만 내보내기 | 타입/인터페이스를 외부에 공개할 때 |
| `types.ts` | 공유 타입을 모아두는 파일 | 여러 파일이 같은 타입을 쓸 때 |
| `.d.ts` | 타입만 있는 선언 파일, 구현 코드 없음 | 라이브러리 타입 제공, 전역 타입 선언 |
| `@types/*` | DefinitelyTyped 커뮤니티 타입 패키지 | JS 라이브러리의 타입이 필요할 때 |
| `declare module` | 타입 없는 모듈을 임시로 선언 | @types 패키지가 없는 라이브러리 |
| `moduleResolution` | TypeScript의 모듈 경로 해석 방식 | tsconfig 프로젝트 설정 |

### import type 사용 규칙

```typescript
// ✅ 올바른 사용 패턴

// 타입과 인터페이스 — import type 사용
import type { User, Product } from "./types";

// 함수, 클래스, 상수 — 일반 import 사용
import { formatDate, UserService } from "./utils";

// 같은 파일에서 타입과 값을 함께 가져올 때
import { UserService } from "./user-service";
import type { User } from "./user-service"; // 타입만 따로
```
