# 11장. 유틸리티 타입

<a id="part1"></a>

## 1️⃣ 유틸리티 타입이란? [↑](#toc)

### 사진 편집 필터 비유

> 사진 편집 앱의 필터를 떠올려보세요. 원본 사진에 흑백 필터를 적용하면 흑백 사진이 나옵니다.
> 원본은 그대로 유지되고, 필터가 적용된 복사본만 바뀝니다.
> 유틸리티 타입도 마찬가지입니다. 원본 타입(T)은 그대로 두고, 변형된 새 타입을 만듭니다.

TypeScript는 자주 필요한 타입 변환 패턴을 **유틸리티 타입**으로 기본 제공합니다.
10장에서 배운 제네릭(`<T>`)을 내부적으로 사용하므로, 제네릭 개념이 익숙해야 합니다.

```typescript
// 매번 직접 타입을 새로 만들 필요가 없습니다
// TypeScript가 이미 만들어 뒀습니다

interface User {
  id: number;
  name: string;
  email: string;
  createdAt: Date;
}

// 필터 하나로 새 타입 완성
type PartialUser  = Partial<User>;   // 모든 속성이 선택적
type ReadonlyUser = Readonly<User>;  // 모든 속성이 읽기 전용
type UserPreview  = Pick<User, "id" | "name">; // 일부 속성만 선택
```

## 2️⃣ Partial\<T\> [↑](#toc)

### "일부만 업데이트"할 때

`Partial<T>`는 타입 T의 **모든 속성을 선택적(`?`)**으로 만듭니다.

> 고통: 업데이트 API에서 name만 바꾸고 싶은데, User 타입을 그대로 쓰면 모든 필드를 채워야 합니다.
> 해결: `Partial<User>`를 쓰면 바꾸고 싶은 필드만 전달할 수 있습니다.

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  age: number;
}

// ❌ 일반 User 타입으로 업데이트 — 모든 필드 강제
function updateUserBad(id: number, updates: User): void {
  // updates에 id, name, email, age 모두 있어야 함 → 불편!
}

// ✅ Partial<User> — 바꾸고 싶은 필드만 전달
function updateUser(id: number, updates: Partial<User>): void {
  console.log(`사용자 ${id} 업데이트:`, updates);
}

updateUser(1, { name: "김영희" });           // name만
updateUser(2, { email: "new@example.com" }); // email만
updateUser(3, { name: "박민준", age: 30 });  // 복수 필드
```

실행 결과:
```
사용자 1 업데이트: { name: '김영희' }
사용자 2 업데이트: { email: 'new@example.com' }
사용자 3 업데이트: { name: '박민준', age: 30 }
```

### Partial이 내부적으로 하는 일

```typescript
// TypeScript가 생성하는 동등한 타입
type PartialUser = {
  id?: number;
  name?: string;
  email?: string;
  age?: number;
};
```

## 3️⃣ Required\<T\> [↑](#toc)

### Partial의 반대 — 모든 속성을 필수로

`Required<T>`는 타입 T의 **모든 선택적 속성을 필수**로 만듭니다.
`Partial`의 정반대 역할입니다.

```typescript
interface Config {
  host?: string;
  port?: number;
  debug?: boolean;
}

// ❌ Config 타입 — 모두 선택적이라 빈 객체도 허용됨
const badConfig: Config = {};  // 에러 없음, 하지만 실행하면 문제 발생

// ✅ Required<Config> — 모두 채워야 함
type StrictConfig = Required<Config>;

const goodConfig: StrictConfig = {
  host: "localhost",
  port: 3000,
  debug: false,
};

// Required<Config>에 빈 객체를 넣으면 컴파일 에러
// const bad: StrictConfig = {}; // ❌ 에러: host, port, debug 누락
```

### 실용 패턴: 검증 후 완성된 타입으로 변환

```typescript
interface UserForm {
  name?: string;
  email?: string;
}

// 폼 검증 통과 후 → 모든 필드가 채워진 완성 타입으로 좁히기
function validateForm(form: UserForm): Required<UserForm> {
  if (!form.name || !form.email) {
    throw new Error("이름과 이메일은 필수입니다");
  }
  return { name: form.name, email: form.email };
}
```

## 4️⃣ Readonly\<T\> [↑](#toc)

### "변경 불가 복사본"

`Readonly<T>`는 타입 T의 **모든 속성을 읽기 전용**으로 만듭니다.
변경 시도 시 컴파일 에러가 발생합니다.

> 박물관 전시물 비유: 보는 것은 허용, 손대는 것은 금지.

```typescript
interface Point {
  x: number;
  y: number;
}

const mutablePoint: Point = { x: 1, y: 2 };
mutablePoint.x = 10;  // 허용

const fixedPoint: Readonly<Point> = { x: 1, y: 2 };
// fixedPoint.x = 10; // ❌ 에러: 읽기 전용 속성에 할당할 수 없습니다

console.log(fixedPoint); // { x: 1, y: 2 }
```

실행 결과:
```
{ x: 1, y: 2 }
```

### 실용 패턴: 설정값과 상수 객체 보호

```typescript
interface AppConfig {
  apiUrl: string;
  maxRetries: number;
  timeout: number;
}

// 애플리케이션 설정 — 한번 정하면 바꾸면 안 됨
const CONFIG: Readonly<AppConfig> = {
  apiUrl: "https://api.example.com",
  maxRetries: 3,
  timeout: 5000,
};

// CONFIG.apiUrl = "다른 URL"; // ❌ 컴파일 에러로 실수 방지
console.log(`API URL: ${CONFIG.apiUrl}`);
```

실행 결과:
```
API URL: https://api.example.com
```

## 5️⃣ Pick\<T, Keys\> [↑](#toc)

### "필요한 것만 쏙 뽑기"

`Pick<T, Keys>`는 타입 T에서 **지정한 속성들만 선택**해 새 타입을 만듭니다.

> 뷔페에서 원하는 음식만 접시에 담는 것과 같습니다. 뷔페(전체 타입)는 그대로, 내 접시(Pick된 타입)만 달라집니다.

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
  createdAt: Date;
}

// 목록 화면 — id, name만 필요
type UserListItem = Pick<User, "id" | "name">;

// 프로필 카드 — id, name, email만 필요
type UserCard = Pick<User, "id" | "name" | "email">;

const listItem: UserListItem = { id: 1, name: "김철수" };
// listItem.email = "..."; // ❌ 에러: UserListItem에 email 없음

const card: UserCard = { id: 1, name: "김철수", email: "kim@example.com" };
console.log(listItem);
console.log(card);
```

실행 결과:
```
{ id: 1, name: '김철수' }
{ id: 1, name: '김철수', email: 'kim@example.com' }
```

### API 응답 최적화 패턴

```typescript
// API에서 미리보기 데이터만 받아올 때
async function fetchUserPreviews(): Promise<Pick<User, "id" | "name">[]> {
  // 서버는 id, name만 포함된 가벼운 응답 반환
  return [
    { id: 1, name: "김철수" },
    { id: 2, name: "이영희" },
  ];
}
```

## 6️⃣ Omit\<T, Keys\> [↑](#toc)

### "이것만 빼고 다"

`Omit<T, Keys>`는 타입 T에서 **지정한 속성들을 제외**하고 나머지로 새 타입을 만듭니다.
`Pick`의 반대 관점입니다.

> 고통: 새 사용자 생성 폼에서 id와 createdAt은 서버가 자동 생성해야 하는데, User 타입을 쓰면 프론트엔드가 이 값을 채워야 합니다.
> 해결: `Omit<User, "id" | "createdAt">`으로 서버 생성 필드를 제거합니다.

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
  createdAt: Date;
}

// 생성 API — id, createdAt 제외 (서버가 자동 생성)
type CreateUserInput = Omit<User, "id" | "createdAt">;

// 공개 프로필 — 비밀번호 제외
type PublicUser = Omit<User, "password">;

function createUser(input: CreateUserInput): User {
  return {
    id: Math.random(),
    createdAt: new Date(),
    ...input,
  };
}

const newUser = createUser({
  name: "박지민",
  email: "jimin@example.com",
  password: "hashed_pw",
});

console.log(newUser.id);        // 서버에서 생성된 id
console.log(newUser.createdAt); // 서버에서 생성된 날짜
console.log(newUser.name);      // 입력한 name
```

실행 결과:
```
0.7392847561...
2026-04-10T00:00:00.000Z
박지민
```

### Pick vs Omit — 언제 무엇을 쓸까?

| 상황 | 추천 | 이유 |
|------|------|------|
| 뽑을 속성이 적을 때 (1~3개) | `Pick` | 원하는 것을 명시적으로 나열 |
| 제거할 속성이 적을 때 (1~2개) | `Omit` | 빼는 것이 적으면 더 간결 |
| 전체 타입에서 민감 정보 제거 | `Omit` | "password를 뺀 나머지" 의도 명확 |

## 7️⃣ Record\<Keys, Type\> [↑](#toc)

### "키-값 쌍을 타입으로 표현하기"

`Record<Keys, Type>`은 **키의 집합과 값 타입**으로 객체 타입을 만듭니다.
`{ [key: string]: number }`와 동일하지만, 유니언 리터럴 타입으로 키를 제한할 때 강력합니다.

```typescript
// 기본 예시: 문자열 키 → 숫자 값
type ScoreMap = Record<string, number>;

const scores: ScoreMap = {
  국어: 85,
  영어: 92,
  수학: 78,
};

console.log(scores["국어"]); // 85
```

실행 결과:
```
85
```

### 유니언 타입으로 키 제한 — Record의 진짜 강점

```typescript
type Role = "admin" | "editor" | "viewer";

// 각 역할별 권한 목록을 타입 안전하게 표현
type RolePermissions = Record<Role, string[]>;

const permissions: RolePermissions = {
  admin:  ["read", "write", "delete"],
  editor: ["read", "write"],
  viewer: ["read"],
  // guest: ["read"], // ❌ 에러: 'guest'는 Role 타입에 없는 키
};

console.log(permissions["editor"]); // ['read', 'write']
```

실행 결과:
```
[ 'read', 'write' ]
```

### 실용 패턴: HTTP 상태 코드 메시지 맵

```typescript
type HttpStatus = 200 | 400 | 401 | 403 | 404 | 500;

const statusMessages: Record<HttpStatus, string> = {
  200: "성공",
  400: "잘못된 요청",
  401: "인증 필요",
  403: "권한 없음",
  404: "찾을 수 없음",
  500: "서버 오류",
};

function getStatusMessage(code: HttpStatus): string {
  return statusMessages[code];
}

console.log(getStatusMessage(404)); // "찾을 수 없음"
console.log(getStatusMessage(200)); // "성공"
```

실행 결과:
```
찾을 수 없음
성공
```

## 8️⃣ ReturnType\<T\> [↑](#toc)

### "함수의 반환 타입을 자동으로 추출하기"

`ReturnType<T>`는 함수 타입 T의 **반환 타입을 추출**합니다.
함수 시그니처가 바뀌면 반환 타입도 자동으로 업데이트됩니다.

> 고통: 함수의 반환 타입이 복잡한 객체일 때, 해당 타입을 다른 곳에서 다시 선언해야 합니다.
> 함수 반환 타입이 바뀌면, 다시 선언한 타입도 수동으로 업데이트해야 합니다.
> 해결: `ReturnType`으로 함수에서 직접 타입을 추출하면 자동으로 동기화됩니다.

```typescript
function getUser() {
  return {
    id: 1,
    name: "김철수",
    email: "kim@example.com",
    createdAt: new Date(),
  };
}

// ❌ 반환 객체 타입을 수동으로 다시 선언 — 함수 바뀌면 이것도 바꿔야 함
type UserManual = {
  id: number;
  name: string;
  email: string;
  createdAt: Date;
};

// ✅ ReturnType으로 자동 추출 — 함수 바뀌면 타입도 자동 업데이트
type User = ReturnType<typeof getUser>;

const user: User = getUser();
console.log(user.name); // "김철수"
```

실행 결과:
```
김철수
```

### 실용 패턴: API 핸들러 반환 타입 재사용

```typescript
function createApiResponse<T>(data: T, message: string) {
  return {
    success: true,
    data,
    message,
    timestamp: new Date().toISOString(),
  };
}

// 반환 타입을 한번만 추출해서 여러 곳에서 재사용
type ApiResponse = ReturnType<typeof createApiResponse<unknown>>;

// 함수 본체가 바뀌면 ApiResponse도 자동으로 업데이트됨
const response = createApiResponse({ userId: 1 }, "조회 성공");
console.log(response.success);   // true
console.log(response.message);   // "조회 성공"
console.log(response.timestamp); // ISO 날짜 문자열
```

실행 결과:
```
true
조회 성공
2026-04-10T00:00:00.000Z
```

## 9️⃣ 유틸리티 타입 조합 [↑](#toc)

### "필터를 여러 겹 적용하기"

유틸리티 타입을 중첩해서 더 정밀한 타입을 만들 수 있습니다.

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  password: string;
  bio: string;
  avatar: string;
  createdAt: Date;
}

// name과 email만 선택하고, 그것들을 모두 선택적으로
// "프로필 수정 폼 — name이나 email 중 하나만 바꿀 수도 있음"
type ProfileUpdateForm = Partial<Pick<User, "name" | "email" | "bio" | "avatar">>;

// Readonly<Omit<...>> — 비밀번호 없는 공개 프로필, 수정 불가
type ImmutablePublicProfile = Readonly<Omit<User, "password">>;
```

### 실전 예시: 폼 상태 관리

```typescript
interface Product {
  id: number;
  name: string;
  price: number;
  description: string;
  stock: number;
  category: string;
}

// 상품 생성 폼 — id는 서버 생성, 나머지는 모두 입력 필요
type CreateProductForm = Omit<Product, "id">;

// 상품 수정 폼 — 원하는 필드만 선택적으로 수정 가능
type UpdateProductForm = Partial<Omit<Product, "id">>;

// 상품 목록 카드 — 목록에서 필요한 정보만
type ProductCard = Pick<Product, "id" | "name" | "price" | "category">;

function createProduct(form: CreateProductForm): Product {
  return { id: Date.now(), ...form };
}

function updateProduct(id: number, form: UpdateProductForm): void {
  console.log(`상품 ${id} 수정:`, form);
}

function displayCard(card: ProductCard): void {
  console.log(`[${card.category}] ${card.name} — ${card.price.toLocaleString()}원`);
}

const product = createProduct({
  name: "TypeScript 교재",
  price: 28000,
  description: "비전공자를 위한 TypeScript 가이드",
  stock: 100,
  category: "도서",
});

updateProduct(product.id, { price: 25000, stock: 95 });
displayCard(product);
```

실행 결과:
```
상품 1712678400000 수정: { price: 25000, stock: 95 }
[도서] TypeScript 교재 — 28,000원
```

## 🔟 정리 [↑](#toc)

### 핵심 개념 요약

| 유틸리티 타입 | 동작 | 대표 사용 사례 |
|--------------|------|--------------|
| `Partial<T>` | 모든 속성을 선택적(`?`)으로 | 업데이트 API 파라미터 |
| `Required<T>` | 모든 선택적 속성을 필수로 | 검증 완료 후 완성 타입 |
| `Readonly<T>` | 모든 속성을 읽기 전용으로 | 설정 객체, 상수 |
| `Pick<T, K>` | 지정한 속성만 선택 | 미리보기 타입, 목록 아이템 |
| `Omit<T, K>` | 지정한 속성을 제거 | 생성 입력 타입, 공개 프로필 |
| `Record<K, V>` | 키 집합 + 값 타입으로 객체 타입 | 상태 코드 맵, 역할별 권한 |
| `ReturnType<T>` | 함수의 반환 타입 추출 | 함수 반환값 재사용 |

### Partial vs Required vs Readonly — 한눈에 비교

| 유틸리티 | 변경 전 | 변경 후 |
|---------|---------|---------|
| `Partial<T>` | `name: string` (필수) | `name?: string` (선택적) |
| `Required<T>` | `name?: string` (선택적) | `name: string` (필수) |
| `Readonly<T>` | `name: string` (수정 가능) | `readonly name: string` (읽기 전용) |
