# 16장. 통합 프로젝트 — 타입 안전한 날씨 앱

<a id="part1"></a>

## 1️⃣ 프로젝트 소개 [↑](#toc)

> JS 14장에서 만든 날씨 앱을 TypeScript로 전면 재작성합니다.
> 동작은 같지만, 코드 전체에 타입이 흐르게 됩니다.

### JS 14장과 이번 장의 차이

| 항목 | JS 14장 | TS 16장 (이번 장) |
|------|---------|-----------------|
| API 응답 처리 | 어떤 필드가 있는지 모름 | `WeatherData` 인터페이스로 명시 |
| fetch 함수 | 반환 타입이 `any` | 제네릭 `fetchData<T>` 래퍼 |
| 에러 처리 | 문자열 메시지 | 판별 유니언 (`NetworkError \| NotFoundError`) |
| DOM 접근 | `getElementById` → 타입 불명확 | `getElement<T>` 헬퍼 함수 |
| 상태 관리 | 변수에 아무 값이나 | `AppState` 타입으로 제한 |

### 추가되는 TypeScript 기능

```
JS 14장 기능                  TS 16장에서 추가
─────────────                 ────────────────────────
fetch로 날씨 데이터 가져오기   → 제네릭 fetchData<T> 래퍼
DOM에 날씨 카드 표시          → HTMLElement 타입 단언
에러 메시지 표시              → 판별 유니언 에러 타입
최근 검색어 관리              → Partial<WeatherData> 로딩 상태
도시명 유효성 검사            → 리터럴 유니언 상태 타입
```

### OpenWeatherMap API 응답 구조 확인

```json

  "name": "Seoul",
  "sys": { "country": "KR" },
  "weather": [{ "description": "clear sky", "icon": "01d" }],

    "temp": 289.15,

    "temp_min": 287.0,
```

```typescript
// [06장 interface] OpenWeatherMap API 응답 타입
// 실제 API 응답 JSON 구조를 그대로 반영합니다

export interface WeatherCondition {
  id: number;
  main: string;
  description: string;
  icon: string;
}

export interface MainWeatherData {
  temp: number;          // 켈빈(K) 단위
  feels_like: number;
  temp_min: number;
  temp_max: number;
  humidity: number;      // 퍼센트
  pressure: number;      // 헥토파스칼(hPa)
}

export interface WindData {
  speed: number;         // m/s
  deg?: number;          // 방향 (optional — 항상 오지 않음)
}

export interface CoordData {
  lat: number;
  lon: number;
}

export interface SysData {
  country: string;
  sunrise?: number;
  sunset?: number;
}

// 최상위 API 응답 타입 — 전체를 하나로 묶음
export interface WeatherData {
  name: string;              // 도시명
  sys: SysData;
  weather: WeatherCondition[];  // 배열 (보통 첫 번째 요소만 사용)
  main: MainWeatherData;
  wind: WindData;
  coord: CoordData;
  visibility?: number;       // 가시거리 (optional)
}

// 좌표 기반 검색용 응답 타입
export interface GeoLocation {
  name: string;
  lat: number;
  lon: number;
  country: string;
  state?: string;
}
```

## 4️⃣ Step 2: 제네릭 fetch 래퍼 [↑](#toc)

> JS 14장에서 `fetch`를 직접 호출했습니다. 매번 `response.ok` 체크를 반복했습니다.
> 제네릭 래퍼 함수로 이를 하나로 묶고, 어떤 타입의 데이터든 받을 수 있게 만듭니다.

### JS 14장 방식 (반복 코드)

```javascript
// JS 14장 — 매 API 호출마다 같은 코드 반복
async function getWeather(city) {
  try {
    const response = await fetch(`.../${city}&appid=...`);
    if (!response.ok) {
      throw new Error(`HTTP 에러: ${response.status}`);
    }
    const data = await response.json();  // any 타입 — 구조 모름
    return data;
  } catch (error) {
    throw error;
  }
}
```

### `src/api/weather.ts` — 제네릭 래퍼 포함

```typescript
// [10장 제네릭] T는 응답 데이터의 타입을 나타내는 타입 매개변수
// fetchData<WeatherData>(url) → Promise<WeatherData> 반환

import type { WeatherData, GeoLocation } from "../types/weather";
import type { AppError } from "../types/app";

// API 키는 환경 변수에서 가져옵니다 (.env 파일)
const API_KEY = process.env["WEATHER_API_KEY"] ?? "";
const BASE_URL = "https://api.openweathermap.org/data/2.5";

// ─────────────────────────────────────────────────────────────
// 제네릭 fetch 래퍼 — 모든 API 호출에서 재사용
// ─────────────────────────────────────────────────────────────

async function fetchData<T>(url: string): Promise<T> {
  // [12장 async/await] 비동기 함수
  const response = await fetch(url);

  if (!response.ok) {
    // HTTP 에러 상태 처리
    if (response.status === 404) {
      // [07장 판별 유니언] NotFoundError 생성
      const error: AppError = {
        kind: "not_found",
        city: new URL(url).searchParams.get("q") ?? "알 수 없음"
      };
      throw error;
    }

    // 그 외 네트워크 에러
    const error: AppError = {
      kind: "network",
      message: `HTTP ${response.status}: ${response.statusText}`
    };
    throw error;
  }

  // [13장 타입 단언] JSON 파싱 결과를 T 타입으로 단언
  return response.json() as Promise<T>;
}

// ─────────────────────────────────────────────────────────────
// API 함수들 — fetchData<T>를 활용
// ─────────────────────────────────────────────────────────────

// [10장 제네릭 활용] fetchData에 WeatherData 타입 전달
export async function getWeather(city: string): Promise<WeatherData> {
  const url = `${BASE_URL}/weather?q=${encodeURIComponent(city)}&appid=${API_KEY}&units=metric&lang=kr`;
  return fetchData<WeatherData>(url);
}

// 좌표로 날씨 가져오기 (위도/경도 기반)
export async function getWeatherByCoords(
  lat: number,
  lon: number
): Promise<WeatherData> {
  const url = `${BASE_URL}/weather?lat=${lat}&lon=${lon}&appid=${API_KEY}&units=metric&lang=kr`;
  return fetchData<WeatherData>(url);
}

// 도시명 → 좌표 변환 (Geocoding API)
export async function getCoordinates(city: string): Promise<GeoLocation> {
  const url = `https://api.openweathermap.org/geo/1.0/direct?q=${encodeURIComponent(city)}&limit=1&appid=${API_KEY}`;
  const results = await fetchData<GeoLocation[]>(url);

  if (results.length === 0) {
    const error: AppError = { kind: "not_found", city };
    throw error;
  }

  // 첫 번째 결과 반환 (가장 관련성 높은 도시)
  return results[0]!;
}
```

### 제네릭이 주는 이점

```typescript
// fetchData<WeatherData> 호출 시
const data = await fetchData<WeatherData>(url);
// data의 타입: WeatherData
// → data.main.temp 에 바로 접근 가능 (자동완성)
// → data.main.tempp 오타 시 즉시 에러

// fetchData<GeoLocation[]> 호출 시
const locations = await fetchData<GeoLocation[]>(url);
// locations의 타입: GeoLocation[]
// → locations[0].lat 에 바로 접근 가능
```

## 5️⃣ Step 3: API 함수 활용 [↑](#toc)

> 제네릭 래퍼를 만들었으니, 실제로 어떻게 사용하는지 봅니다.

```typescript
// index.ts에서 API 함수 사용 예

import { getWeather, getCoordinates } from "./api/weather";
import type { WeatherData, GeoLocation } from "./types/weather";
import type { AppError } from "./types/app";

async function searchWeather(city: string): Promise<void> {
  try {
    // getWeather 반환 타입이 Promise<WeatherData>이므로
    // data의 타입은 WeatherData — 자동으로 알 수 있음
    const data: WeatherData = await getWeather(city);

    console.log(`도시: ${data.name}, ${data.sys.country}`);
    console.log(`날씨: ${data.weather[0]?.description ?? "정보 없음"}`);
    console.log(`온도: ${data.main.temp}°C`);
    console.log(`습도: ${data.main.humidity}%`);

  } catch (error) {
    // [07장 타입 좁히기] error를 AppError로 좁혀서 처리
    handleError(error);
  }
}

function handleError(error: unknown): void {
  // [13장 타입 단언] error가 AppError 형태인지 확인 후 처리
  if (typeof error === "object" && error !== null && "kind" in error) {
    const appError = error as AppError;

    switch (appError.kind) {
      case "not_found":
        console.error(`"${appError.city}" 도시를 찾을 수 없습니다.`);
        break;
      case "network":
        console.error(`네트워크 오류: ${appError.message}`);
        break;
      case "parse":
        console.error("응답 데이터 파싱 오류");
        break;
    }
  } else {
    console.error("알 수 없는 오류:", error);
  }
}
```

실행 결과 (Seoul 검색 시):
```
도시: Seoul, KR
날씨: 맑음
온도: 16.3°C
습도: 55%
```

## 6️⃣ Step 4: DOM 유틸 [↑](#toc)

> DOM 접근 코드를 매번 작성하는 대신, 타입 안전한 헬퍼 함수로 만들어 재사용합니다.

### `src/dom/elements.ts`

```typescript
// [10장 제네릭] T extends HTMLElement — HTML 요소 타입만 허용
// querySelector 결과를 원하는 타입으로 안전하게 가져오기

export function getElement<T extends HTMLElement>(selector: string): T {
  const element = document.querySelector<T>(selector);

  if (element === null) {
    throw new Error(
      `DOM 요소를 찾을 수 없습니다: "${selector}"\n` +
      `HTML에 해당 선택자가 있는지 확인해주세요.`
    );
  }

  return element;
}

// 여러 요소를 한 번에 가져오는 유틸
export function getElements<T extends HTMLElement>(selector: string): T[] {
  return Array.from(document.querySelectorAll<T>(selector));
}

// 요소가 없어도 에러가 나지 않는 버전 (optional)
export function tryGetElement<T extends HTMLElement>(
  selector: string
): T | null {
  return document.querySelector<T>(selector);
}
```

## 7️⃣ Step 5: 데이터 변환 [↑](#toc)

> API에서 받은 원시 데이터를 화면에 표시할 형태로 변환합니다.
> 유틸리티 타입(11장)을 활용해 필요한 필드만 선택합니다.

### `src/utils/format.ts`

```typescript
import type { WeatherData, MainWeatherData } from "../types/weather";

// ─────────────────────────────────────────────────────────────
// 온도 변환 유틸
// ─────────────────────────────────────────────────────────────

// Kelvin → Celsius (API units=metric 사용 시 이미 섭씨이므로 아래는 참고용)
export function kelvinToCelsius(kelvin: number): number {
  return Math.round(kelvin - 273.15);
}

// 온도 표시 문자열
export function formatTemp(celsius: number): string {
  return `${celsius.toFixed(1)}°C`;
}

// ─────────────────────────────────────────────────────────────
// 날씨 데이터 변환
// ─────────────────────────────────────────────────────────────

// [11장 Pick] WeatherData에서 화면에 필요한 필드만 추출한 타입
type WeatherDisplay = Pick<MainWeatherData, "temp" | "feels_like" | "humidity">;

export function extractDisplayData(weather: WeatherData): WeatherDisplay {
  return {
    temp: weather.main.temp,
    feels_like: weather.main.feels_like,
    humidity: weather.main.humidity
  };
}

// ─────────────────────────────────────────────────────────────
// 로딩 상태 처리 — Partial 활용
// ─────────────────────────────────────────────────────────────

// [11장 Partial] 로딩 중에는 일부 필드만 있어도 됨
type LoadingWeather = Partial<WeatherData> & {
  name: string;  // 검색한 도시명은 항상 있음
};

export function createLoadingState(city: string): LoadingWeather {
  return { name: city };
}

// ─────────────────────────────────────────────────────────────
// 아이콘 URL 생성
// ─────────────────────────────────────────────────────────────

export function getWeatherIconUrl(iconCode: string): string {
  return `https://openweathermap.org/img/wn/${iconCode}@2x.png`;
}

// ─────────────────────────────────────────────────────────────
// 날짜/시간 포맷
// ─────────────────────────────────────────────────────────────

export function formatLocalTime(): string {
  return new Date().toLocaleString("ko-KR", {
    year: "numeric",
    month: "long",
    day: "numeric",
    hour: "2-digit",
    minute: "2-digit"
  });
}

// 검색 기록에 표시할 간단한 날짜
export function formatShortDate(date: Date): string {
  return date.toLocaleDateString("ko-KR", {
    month: "short",
    day: "numeric"
  });
}
```

## 8️⃣ Step 6: 에러 처리 [↑](#toc)

> JS 14장에서는 에러를 `catch (error) { ... }` 블록에서 문자열로 처리했습니다.
> TypeScript의 판별 유니언으로 에러 종류별로 다른 처리를 타입 안전하게 할 수 있습니다.

### 판별 유니언 복습 (07장)

```typescript
// AppError = NetworkError | NotFoundError | ParseError
```

### 에러 처리 함수 (`src/index.ts` 내)

```typescript
import type { AppError } from "./types/app";

// [07장 타입 좁히기] kind 필드로 에러 타입 좁히기
function handleWeatherError(error: AppError): void {
  switch (error.kind) {
    case "not_found":
      // 이 블록에서 error는 NotFoundError 타입
      // error.city 에 접근 가능
      showError(`"${error.city}" 도시를 찾을 수 없습니다. 영문 도시명을 입력해주세요.`);
      break;

    case "network":
      // 이 블록에서 error는 NetworkError 타입
      // error.message 에 접근 가능
      showError(`네트워크 오류가 발생했습니다: ${error.message}`);
      break;

    case "parse":
      // 이 블록에서 error는 ParseError 타입
      showError("응답 데이터를 처리할 수 없습니다. 잠시 후 다시 시도해주세요.");
      console.error("파싱 에러 원본:", error.raw);
      break;
  }
  // TypeScript는 모든 case를 처리했는지 자동으로 확인합니다.
  // 새로운 에러 타입이 추가되면 이 switch에 case를 추가하라고 알려줍니다.
}

// unknown 타입의 에러를 AppError로 변환
function toAppError(error: unknown): AppError {
  if (error instanceof Error) {
    return {
      kind: "network",
      message: error.message
    };
  }

  // fetch에서 던진 AppError 형태인지 확인
  if (
    typeof error === "object" &&
    error !== null &&
    "kind" in error
  ) {
    return error as AppError;
  }

  return {
    kind: "network",
    message: "알 수 없는 오류가 발생했습니다"
  };
}
```

### try-catch 패턴 전체

```typescript
async function searchAndDisplay(city: string): Promise<void> {
  setStatus("loading");

  try {
    const data = await getWeather(city);
    displayWeather(data);
    addToRecentCities(city);
    setStatus("success");
  } catch (error: unknown) {
    // catch에서 error는 항상 unknown 타입 (strict 모드)
    const appError = toAppError(error);
    handleWeatherError(appError);
    setStatus("error");
  }
}
```

> **왜 `catch (error: unknown)`인가?**
> TypeScript 4.0부터 `catch` 블록의 `error`는 `unknown` 타입입니다.
> `any`가 아니라 `unknown`이므로 타입 좁히기 없이는 접근할 수 없습니다.
> 이것이 바로 타입 안전한 에러 처리입니다.

```typescript
import type { WeatherData, GeoLocation } from "../types/weather";
import type { AppError } from "../types/app";

const API_KEY = process.env["WEATHER_API_KEY"] ?? "";
const BASE_URL = "https://api.openweathermap.org/data/2.5";

async function fetchData<T>(url: string): Promise<T> {
  const response = await fetch(url);

  if (!response.ok) {
    if (response.status === 404) {
      const error: AppError = {
        kind: "not_found",
        city: new URL(url).searchParams.get("q") ?? "알 수 없음"
      };
      throw error;
    }
    const error: AppError = {
      kind: "network",
      message: `HTTP ${response.status}`
    };
    throw error;
  }

  return response.json() as Promise<T>;
}

export async function getWeather(city: string): Promise<WeatherData> {
  const url =
    `${BASE_URL}/weather?q=${encodeURIComponent(city)}` +
    `&appid=${API_KEY}&units=metric&lang=kr`;
  return fetchData<WeatherData>(url);
}
```

### TypeScript는 적이 아니라 동료입니다

처음에는 빨간 줄이 많아 답답했을 것입니다. 하지만 생각해보면:

- 빨간 줄 하나 = 런타임 버그 하나 예방
- "Type 'X' is not assignable to type 'Y'" = "여기에 잘못된 값이 들어올 수 있어요"
- "Object is possibly 'null'" = "이 경우도 처리해주세요"

**에러 메시지는 코드 리뷰입니다.** TypeScript가 코드를 읽고 문제가 될 부분을 미리 알려주는 것입니다.
