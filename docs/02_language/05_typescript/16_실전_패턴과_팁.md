## 1️⃣ tsconfig.json 완전 이해 [↑](#toc)

> `tsconfig.json`은 TypeScript 컴파일러에게 내리는 명령서입니다.
> "어디까지 엄격하게 검사할까?"를 이 파일에서 결정합니다.

### strict 플래그란?

```json
{
  "compilerOptions": {
    "strict": true
  }
}
```

`"strict": true` 하나가 아래 옵션 **여섯 개를 동시에 켜는** 단축 설정입니다.

| 하위 옵션 | 기본값 | `strict: true` 시 |
|----------|--------|-------------------|
| `strictNullChecks` | false | true |
| `noImplicitAny` | false | true |
| `noImplicitReturns` | false | true |
| `strictFunctionTypes` | false | true |
| `strictBindCallApply` | false | true |
| `strictPropertyInitialization` | false | true |

---

### strictNullChecks — null/undefined를 별도 타입으로

```typescript
// strictNullChecks: false (기본값) — 위험한 상태
// null/undefined를 모든 타입에 대입 가능 → 런타임 에러 원천
let name: string = null;   // 에러 없음 (위험!)

// strictNullChecks: true — 안전한 상태
let name: string = null;
// ❌ Type 'null' is not assignable to type 'string'

// ✅ null을 허용하려면 유니언 타입으로 명시해야 함
let name: string | null = null;
if (name !== null) {
  console.log(name.toUpperCase()); // 여기선 null이 아님이 보장됨
}
```

실행 결과 (strictNullChecks: true, 에러 없는 코드):
```
// name이 null이면 아무것도 출력되지 않음
// name이 "홍길동"이면 "홍길동" 출력
```

---

### noImplicitAny — 암묵적 any 금지

```typescript
// noImplicitAny: false — 매개변수 타입을 쓰지 않아도 됨
function greet(name) {     // name은 암묵적으로 any
  return "안녕, " + name;  // any에 문자열을 더해도 에러 없음
}

// noImplicitAny: true — 타입을 반드시 명시해야 함
function greet(name) {
// ❌ Parameter 'name' implicitly has an 'any' type.

// ✅ 해결: 타입 명시
function greet(name: string): string {
  return "안녕, " + name;
}
```

`noImplicitAny`는 "TS를 쓰는 의미"를 보장합니다. 이 옵션 없이는 모든 곳이 몰래 `any`가 될 수 있습니다.

---

### noImplicitReturns — 모든 경로에서 반환값 보장

```typescript
// noImplicitReturns: true — 함수의 일부 경로에서 반환이 빠지면 에러

// ❌ 에러: "a > 0"이 false일 때 반환값이 없음
function getLabel(a: number): string {
  if (a > 0) {
    return "양수";
  }
  // ❌ Not all code paths return a value.
}

// ✅ 해결: 모든 경로에서 반환값 제공
function getLabel(a: number): string {
  if (a > 0) {
    return "양수";
  }
  return "0 또는 음수";
}
```

---

### strictFunctionTypes — 함수 타입 공변성 검사

```typescript
// strictFunctionTypes: true — 함수를 인자로 넘길 때 타입 불일치 감지

type Handler = (event: MouseEvent) => void;

// MouseEvent를 받는 자리에 더 일반적인 Event를 받는 함수를 넣으면?
const looseFn = (event: Event) => console.log(event.type);
const strictFn: Handler = looseFn;
// ❌ strictFunctionTypes: true이면 에러
// Types of parameters 'event' and 'event' are incompatible.

// 이것이 왜 위험한가:
// Handler는 MouseEvent를 넘기는데, looseFn은 Event를 기대 → 타입 불일치
```

> 이 옵션은 함수를 인자로 많이 쓰는 코드에서 숨어있는 버그를 잡아줍니다.
> 처음에는 이해하기 어려울 수 있습니다. `strict: true` 설정만 기억하고 넘어가도 됩니다.

---

### 권장 tsconfig.json (비전공자용 시작 설정)

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "CommonJS",
    "lib": ["ES2020", "DOM"],
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

| 옵션 | 의미 |
|------|------|
| `target: "ES2020"` | 컴파일 결과물의 JS 버전 |
| `module: "CommonJS"` | Node.js용 모듈 형식 |
| `lib: ["DOM"]` | 브라우저 DOM API 타입 포함 |
| `outDir: "./dist"` | 컴파일 결과물 저장 위치 |
| `rootDir: "./src"` | 소스 파일 위치 |
| `strict: true` | 전체 엄격 모드 활성화 |
| `esModuleInterop: true` | CommonJS 모듈을 ES6 방식으로 import 가능 |
| `skipLibCheck: true` | `.d.ts` 타입 선언 파일 검사 건너뜀 (빌드 속도 향상) |

## 2️⃣ 자주 만나는 TS 에러 10가지와 해결법 [↑](#toc)

> 에러 메시지는 적이 아닙니다. TypeScript가 우리에게 보내는 코드 리뷰입니다.
> 메시지를 읽는 법을 배우면 두렵지 않습니다.

### 에러 1 — Type 'X' is not assignable to type 'Y'

```
에러 메시지: Type 'string' is not assignable to type 'number'
```

```typescript
// 원인: 타입이 맞지 않는 값을 대입했음
let age: number = "스물다섯";
// ❌ Type 'string' is not assignable to type 'number'

// 해결: 올바른 타입의 값을 사용하거나 타입을 수정
let age: number = 25;          // ✅ 숫자로 수정
let age: string = "스물다섯";  // ✅ 타입을 string으로 수정
```

---

### 에러 2 — Property 'X' does not exist on type 'Y'

```
에러 메시지: Property 'username' does not exist on type 'User'
```

```typescript
// 원인: 인터페이스에 없는 속성에 접근했거나 오타
interface User {
  name: string;
  email: string;
}

const user: User = { name: "김철수", email: "kim@test.com" };
console.log(user.username);
// ❌ Property 'username' does not exist on type 'User'

// 해결 1: 올바른 속성 이름 사용
console.log(user.name);   // ✅

// 해결 2: 인터페이스에 속성 추가
interface User {
  name: string;
  email: string;
  username?: string;  // optional로 추가
}
```

---

### 에러 3 — Object is possibly 'null'

```
에러 메시지: Object is possibly 'null'
```

```typescript
// 원인: null일 수 있는 값을 확인 없이 사용
const input = document.querySelector("#myInput");
input.value = "hello";
// ❌ Object is possibly 'null'

// 해결 1: null 체크 후 사용
if (input !== null) {
  (input as HTMLInputElement).value = "hello";  // ✅
}

// 해결 2: non-null assertion (!)  — null이 절대 아님을 단언
const input = document.querySelector("#myInput")!;
(input as HTMLInputElement).value = "hello";  // ✅ (주의: 확실할 때만)

// 해결 3: 타입 단언 (as) 사용
const input = document.querySelector("#myInput") as HTMLInputElement;
input.value = "hello";  // ✅
```

> `!` (non-null assertion)은 "내가 책임지고 null이 아니라고 보장한다"는 뜻입니다.
> `null`이 들어오면 런타임 에러가 발생하니, 확실할 때만 사용하세요.

---

### 에러 4 — Argument of type 'X' is not assignable to parameter of type 'Y'

```
에러 메시지: Argument of type 'string | undefined' is not assignable to parameter of type 'string'
```

```typescript
// 원인: 함수 매개변수 타입과 전달 값의 타입 불일치
function greet(name: string): string {
  return `안녕, ${name}!`;
}

const name: string | undefined = undefined;
greet(name);
// ❌ Argument of type 'string | undefined' is not assignable
//    to parameter of type 'string'

// 해결 1: 기본값 사용
greet(name ?? "손님");  // ✅ undefined이면 "손님" 사용

// 해결 2: undefined 체크 후 호출
if (name !== undefined) {
  greet(name);  // ✅ 이 블록 안에서는 string으로 좁혀짐
}

// 해결 3: 함수 매개변수를 optional로
function greet(name?: string): string {
  return `안녕, ${name ?? "손님"}!`;
}
```

---

### 에러 5 — Cannot find module 'X' or its corresponding type declarations

```
에러 메시지: Cannot find module 'lodash' or its corresponding type declarations
```

```
원인: 패키지는 설치했지만 타입 선언 파일(@types/*)이 없음

해결:
```

```bash
# 1. @types 패키지 설치
npm install --save-dev @types/lodash

# 2. 일부 패키지는 자체적으로 타입 포함 (설치만 해도 됨)
npm install axios  # axios는 자체 타입 포함

# 3. @types가 없는 패키지는 직접 선언
# src/types/module.d.ts 파일 생성
declare module "오래된패키지" {
  export function doSomething(): void;
}
```

### 에러 6 — Variable 'X' is used before being assigned

```
에러 메시지: Variable 'result' is used before being assigned
```

```typescript
// 원인: 선언만 하고 초기화 전에 사용
let result: string;
console.log(result);
// ❌ Variable 'result' is used before being assigned.

// 해결 1: 선언과 동시에 초기화
let result: string = "";

// 해결 2: 나중에 할당할 것임을 명시 (단언적 할당)
let result!: string;  // 나중에 반드시 할당됨을 보장
```

---

### 에러 7 — Element implicitly has an 'any' type (인덱스 접근)

```
에러 메시지: Element implicitly has an 'any' type because expression of type 'string' can't be used to index type 'User'
```

```typescript
// 원인: 변수로 객체 속성에 접근할 때 인덱스 시그니처 없음
interface User {
  name: string;
  age: number;
}

const user: User = { name: "김철수", age: 30 };
const key = "name";
console.log(user[key]);
// ❌ Element implicitly has an 'any' type

// 해결 1: keyof를 사용해서 유효한 키임을 보장
const key: keyof User = "name";
console.log(user[key]);  // ✅

// 해결 2: 인덱스 시그니처 추가
interface User {
  name: string;
  age: number;
  [key: string]: string | number;  // 문자열 키로 접근 허용
}
```

---

### 에러 8 — 'X' is possibly 'undefined'

```
에러 메시지: 'todo' is possibly 'undefined'
```

```typescript
// 원인: find, filter 등의 결과가 undefined일 수 있음
const todos: string[] = ["공부", "운동"];
const found = todos.find(t => t === "운동");
console.log(found.toUpperCase());
// ❌ 'found' is possibly 'undefined'

// 해결 1: 옵셔널 체이닝
console.log(found?.toUpperCase());  // ✅ undefined면 그냥 undefined 반환

// 해결 2: undefined 체크
if (found !== undefined) {
  console.log(found.toUpperCase());  // ✅
}

// 해결 3: nullish coalescing
console.log((found ?? "없음").toUpperCase());  // ✅
```

---

### 에러 9 — Conversion of type 'X' to type 'Y' may be a mistake

```
에러 메시지: Conversion of type 'string' to type 'number' may be a mistake
```

```typescript
// 원인: 관련 없는 타입 간의 as 타입 단언
const value = "123" as number;
// ❌ Conversion of type 'string' to type 'number' may be a mistake

// 해결 1: 올바른 변환 방법 사용
const value = Number("123");  // ✅ 숫자 변환

// 해결 2: 정말 필요하다면 unknown을 거쳐서 단언
const value = "123" as unknown as number;  // ✅ (주의: 타입 안전성 포기)
```

---

### 에러 10 — Type '{}' is not assignable to type 'X'

```
에러 메시지: Type '{}' is not assignable to type 'Todo'
```

```typescript
// 원인: 빈 객체를 구체적인 타입 변수에 대입
interface Todo {
  id: number;
  text: string;
}

const todo: Todo = {};
// ❌ Type '{}' is not assignable to type 'Todo'
//    Property 'id' is missing in type '{}'

// 해결 1: 모든 필드를 채움
const todo: Todo = { id: 1, text: "공부" };  // ✅

// 해결 2: 일부만 있어도 되면 Partial 사용
const todo: Partial<Todo> = {};  // ✅ (11장 Partial)
```

## 3️⃣ 타입과 런타임의 차이 (타입 소거) [↑](#toc)

> TypeScript 타입은 컴파일 후 사라집니다.
> 브라우저나 Node.js는 `.ts` 파일을 실행하지 않습니다. 컴파일된 `.js`만 실행합니다.

### 타입 소거(Type Erasure) 실제 확인

```typescript
// TypeScript 코드 (컴파일 전)
interface User {
  name: string;
  age: number;
}

function greet(user: User): string {
  return `안녕하세요, ${user.name}님 (${user.age}세)`;
}

const person: User = { name: "김철수", age: 30 };
console.log(greet(person));
```

컴파일 후 JavaScript:

```javascript
// JavaScript 코드 (컴파일 후) — 타입 관련 코드가 모두 제거됨
function greet(user) {
  return `안녕하세요, ${user.name}님 (${user.age}세)`;
}

const person = { name: "김철수", age: 30 };
console.log(greet(person));
```

실행 결과:
```
안녕하세요, 김철수님 (30세)
```

`interface User`가 완전히 사라졌습니다. `: User`, `: string` 같은 타입 주석도 모두 제거됩니다.

---

### "런타임에서 typeof로 interface 타입을 확인할 수 없다"

```typescript
interface Dog {
  name: string;
  breed: string;
}

const dog: Dog = { name: "바둑이", breed: "진돗개" };

// ❌ interface는 런타임에 사라지므로 instanceof, typeof로 확인 불가
// if (dog instanceof Dog) { ... }  // 에러: 'Dog' only refers to a type

// ✅ 런타임 타입 확인은 속성 존재 여부로 해야 함 (07장 타입 가드)
function isDog(value: unknown): value is Dog {
  return (
    typeof value === "object" &&
    value !== null &&
    "name" in value &&
    "breed" in value
  );
}

if (isDog(dog)) {
  console.log(dog.name);  // ✅ 여기서는 Dog 타입으로 좁혀짐
}
```

실행 결과:
```
바둑이
```

---

### 타입 소거가 있어서 좋은 점

```typescript
// TypeScript 코드 — 타입 덕분에 개발 중 안전
function add(a: number, b: number): number {
  return a + b;
}

// 컴파일된 JavaScript — 타입 없이도 브라우저에서 실행됨
function add(a, b) {
  return a + b;
}
```

- 브라우저가 새로운 언어를 배울 필요 없음
- 번들 크기가 늘어나지 않음 (타입 정보가 없으니 용량 증가 없음)
- 기존 JS 생태계와 완벽 호환

> "TypeScript는 개발할 때만 존재하는 안전망입니다. 배포할 때는 깨끗이 사라집니다."

## 4️⃣ ESLint + TypeScript 설정 [↑](#toc)

> ESLint는 코드 스타일과 잠재적 버그를 잡아주는 도구입니다.
> TypeScript와 함께 쓰면 타입 정보를 활용한 더 강력한 규칙을 사용할 수 있습니다.

### 설치

```bash
npm install --save-dev eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin
```

| 패키지 | 역할 |
|--------|------|
| `eslint` | ESLint 본체 |
| `@typescript-eslint/parser` | ESLint가 TypeScript 코드를 이해하게 해주는 파서 |
| `@typescript-eslint/eslint-plugin` | TypeScript 전용 규칙 모음 |

### 기본 설정 파일 (`.eslintrc.json`)

```json
{
  "parser": "@typescript-eslint/parser",
  "parserOptions": {
    "project": "./tsconfig.json",
    "ecmaVersion": "latest",
    "sourceType": "module"
  },
  "plugins": ["@typescript-eslint"],
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended"
  ],
  "rules": {
    "@typescript-eslint/no-unused-vars": "error",
    "@typescript-eslint/no-explicit-any": "warn",
    "@typescript-eslint/explicit-function-return-type": "off",
    "no-console": "warn"
  }
}
```

### 자주 사용하는 규칙

| 규칙 | 설명 | 추천 설정 |
|------|------|-----------|
| `@typescript-eslint/no-unused-vars` | 사용하지 않는 변수 경고 | `"error"` |
| `@typescript-eslint/no-explicit-any` | `any` 사용 경고 | `"warn"` |
| `@typescript-eslint/no-non-null-assertion` | `!` 단언 경고 | `"warn"` |

| `@typescript-eslint/no-floating-promises` | await 없는 Promise 경고 | `"error"` |

## 5️⃣ 프로젝트 구조 패턴 [↑](#toc)

> 작은 프로젝트는 파일 하나로 충분하지만, 규모가 커지면 파일을 역할별로 나눠야 합니다.
> 아래 구조는 TypeScript 중소 규모 프로젝트에서 널리 사용되는 패턴입니다.

```
my-project/
├── src/
│   ├── types/          # 공유 타입 정의 (.ts 인터페이스, 열거형 등)
│   │   ├── user.ts
│   │   └── api.ts
│   ├── utils/          # 순수 유틸 함수 (부작용 없음)
│   │   ├── format.ts
│   │   └── validate.ts
│   ├── api/            # 외부 API 호출 함수
│   │   └── weather.ts
│   ├── dom/            # DOM 조작 관련 함수 (브라우저 프로젝트)
│   │   └── elements.ts
│   └── index.ts        # 진입점 — 모든 것을 연결하는 파일
├── dist/               # 컴파일 출력 (gitignore에 추가)
├── tsconfig.json
├── package.json
└── .gitignore
```

### 각 폴더의 역할

```typescript
// src/types/user.ts — 타입만 정의, 로직 없음
export interface User {
  id: number;
  name: string;
  email: string;
}

export type UserRole = "admin" | "editor" | "viewer";
```

```typescript
// src/utils/format.ts — 순수 함수, 부작용 없음
export function formatDate(date: Date): string {
  return date.toLocaleDateString("ko-KR", {
    year: "numeric",
    month: "long",
    day: "numeric"
  });
}

export function formatTemperature(kelvin: number): string {
  const celsius = kelvin - 273.15;
  return `${celsius.toFixed(1)}°C`;
}
```

```typescript
// src/index.ts — 진입점, 모듈 연결
import { User } from "./types/user";
import { formatDate } from "./utils/format";

const user: User = {
  id: 1,
  name: "김철수",
  email: "kim@example.com"
};

console.log(`사용자: ${user.name}`);
console.log(`오늘 날짜: ${formatDate(new Date())}`);
```

### .gitignore 필수 항목

```
node_modules/
dist/
.env
*.js.map
```

`dist/`는 컴파일 결과물이므로 git에 올리지 않습니다. 배포 시에는 빌드 후 `dist/`를 사용합니다.

## 6️⃣ TS로 Node.js 프로젝트 구성 [↑](#toc)

> TypeScript를 Node.js 프로젝트에서 쓰려면 실행과 빌드 도구가 필요합니다.
> 개발 중에는 `tsx`, 배포용 빌드는 `tsc`를 사용합니다.

### 초기 설정

```bash
# 1. 프로젝트 초기화
mkdir my-ts-project
cd my-ts-project
npm init -y

# 2. TypeScript 설치
npm install --save-dev typescript

# 3. 빠른 실행 도구 설치 (컴파일 없이 .ts 파일 직접 실행)
npm install --save-dev tsx

# 4. Node.js 타입 선언 설치
npm install --save-dev @types/node

# 5. tsconfig.json 생성 (권장 설정)
npx tsc --init
```

### package.json 스크립트

```json
{
  "name": "my-ts-project",
  "version": "1.0.0",
  "scripts": {
    "dev": "tsx watch src/index.ts",
    "build": "tsc",
    "start": "node dist/index.js",
    "lint": "eslint src --ext .ts",
    "type-check": "tsc --noEmit"
  },
  "devDependencies": {
    "@types/node": "^20.0.0",
    "tsx": "^4.0.0",
    "typescript": "^5.0.0"
  }
}
```

| 스크립트 | 명령어 | 사용 시점 |
|---------|--------|----------|
| `dev` | `tsx watch src/index.ts` | 개발 중 — 파일 저장 시 자동 재시작 |
| `build` | `tsc` | 배포 전 — `dist/`에 JS 파일 생성 |
| `start` | `node dist/index.js` | 배포 환경 — 컴파일된 JS 실행 |
| `type-check` | `tsc --noEmit` | CI/CD — 파일 생성 없이 타입만 검사 |

### 개발 워크플로우

```
개발 중:
npm run dev
→ tsx가 .ts 파일을 직접 실행
→ 파일 변경 감지 시 자동 재시작
→ 컴파일 과정 없이 즉시 실행

배포 시:
npm run build
→ tsc가 src/*.ts를 dist/*.js로 컴파일
→ 타입 에러가 있으면 빌드 실패
→ 성공하면 dist/ 폴더에 순수 JS 생성

npm run start
→ node dist/index.js 실행
→ 타입 없는 순수 JS 환경에서 실행
```

### tsx vs ts-node

```bash
# tsx (권장) — esbuild 기반, 빠름
npm install --save-dev tsx
npx tsx src/index.ts

# ts-node (구버전 방식) — tsc 기반, 느림
npm install --save-dev ts-node
npx ts-node src/index.ts
```

2024년 기준 `tsx`가 훨씬 빠르므로 신규 프로젝트에는 `tsx`를 권장합니다.

## 7️⃣ 다음으로 배우면 좋은 것 [↑](#toc)

> TypeScript 기초를 마쳤습니다. 이제 어디로 가면 좋을까요?
> 목적에 따라 다음 단계가 달라집니다.

### React + TypeScript — 프론트엔드 개발

```typescript
// React 컴포넌트에 Props 타입 정의
interface ButtonProps {
  label: string;
  onClick: () => void;
  disabled?: boolean;
}

function Button({ label, onClick, disabled = false }: ButtonProps) {
  return (
    <button onClick={onClick} disabled={disabled}>
      {label}
    </button>
  );
}

// useState에 타입 지정
const [count, setCount] = useState<number>(0);
const [user, setUser] = useState<User | null>(null);
```

TypeScript + React는 현재 프론트엔드 개발의 사실상 표준입니다. 이 과정에서 배운 `interface`, 유니언 타입, 제네릭이 모두 사용됩니다.

---

### Next.js — React 풀스택 프레임워크

```typescript
// pages/index.tsx — Next.js 페이지 컴포넌트
import type { NextPage, GetServerSideProps } from "next";

interface Props {
  message: string;
}

const Home: NextPage<Props> = ({ message }) => {
  return <h1>{message}</h1>;
};

export const getServerSideProps: GetServerSideProps = async () => {
  return {
    props: { message: "안녕하세요!" }
  };
};

export default Home;
```

서버사이드 렌더링(SSR), 파일 기반 라우팅, API 라우트를 제공합니다. TypeScript를 기본으로 지원합니다.

---

### NestJS — Node.js 백엔드 프레임워크

```typescript
// NestJS는 데코레이터와 TypeScript를 핵심으로 사용
import { Controller, Get, Param } from "@nestjs/common";

@Controller("users")
export class UserController {
  @Get(":id")
  findOne(@Param("id") id: string): string {
    return `사용자 ${id}`;
  }
}
```

Angular와 비슷한 구조로, 클래스와 데코레이터를 활용합니다. 이 과정의 클래스(09장) 내용이 바탕이 됩니다.

---

### Prisma — TypeScript ORM (데이터베이스)

```typescript
// Prisma는 데이터베이스 스키마에서 TypeScript 타입을 자동 생성
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

// User 타입이 자동으로 생성됨 — 오타 불가능
const user = await prisma.user.findUnique({
  where: { id: 1 }
});
// user의 타입: { id: number; name: string; email: string } | null
```

데이터베이스 쿼리에도 TypeScript 타입 안전성을 제공합니다.

---

### Zod — 런타임 유효성 검사 + 타입 추론

```typescript
// Zod: 런타임에서도 타입 검사 (API 응답 검증에 유용)
import { z } from "zod";

const UserSchema = z.object({
  id: z.number(),
  name: z.string(),
  email: z.string().email()
});

// UserSchema에서 TypeScript 타입 자동 추론
type User = z.infer<typeof UserSchema>;

// 런타임에서 API 응답 검증
const result = UserSchema.safeParse(apiResponse);
if (result.success) {
  console.log(result.data.name);  // ✅ 타입 안전
}
```

> 1장에서 배운 "타입 소거" 문제를 Zod가 해결합니다.
> 런타임에서도 타입을 검사할 수 있습니다.

## 8️⃣ 정리 [↑](#toc)

### 핵심 개념 요약

| 주제 | 핵심 내용 | 기억할 것 |
|------|-----------|---------|
| tsconfig strict | 6개 하위 옵션을 동시에 활성화 | `strict: true` 하나면 충분 |
| strictNullChecks | null/undefined를 별도 타입으로 | 유니언(`string \| null`)으로 명시 |
| noImplicitAny | 암묵적 any 금지 | 모든 변수/매개변수에 타입 명시 |
| 타입 소거 | 컴파일 후 타입은 사라짐 | 런타임에 interface를 확인할 수 없음 |
| 프로젝트 구조 | `src/types`, `utils`, `api` 분리 | 역할별로 폴더 나누기 |
| tsx | 개발용 즉시 실행 도구 | `npm run dev` = `tsx watch` |
| tsc | 배포용 컴파일 도구 | `npm run build` = `tsc` |
