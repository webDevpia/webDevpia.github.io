# 07장. 타입 좁히기 (Type Narrowing)

<a id="part1"></a>

## 1️⃣ 타입 좁히기란? [↑](#toc)

### 공항 보안 비유

> 공항 입구에서는 승객인지 직원인지 모릅니다.
> 보안 게이트를 통과하는 순간 — 탑승권을 보여주거나 직원증을 꺼내는 순간 —
> 시스템은 그 사람이 정확히 누구인지 확인합니다.
> **타입 좁히기도 같습니다.** 유니언으로 여러 가능성을 열었으니,
> 실제로 사용하기 전에 "지금 이 값이 어떤 타입인지" 확인해야 합니다.

6장(유니언 타입)에서 배운 것처럼, 유니언은 여러 타입의 가능성을 표현합니다.

```typescript
// ← JS 6장 복습: 조건문으로 타입을 확인
function process(value: string | number) {
  // 여기서 value는 string일 수도, number일 수도 있음
  // TypeScript는 아직 어느 쪽인지 모른다
  value.toUpperCase(); // ❌ 컴파일 오류: number에는 toUpperCase가 없음
}
```

타입 좁히기(Type Narrowing)는 조건문을 통해 유니언 타입을 더 구체적인 타입으로 좁히는 과정입니다.

```typescript
function process(value: string | number) {
  if (typeof value === "string") {
    // ✅ 이 블록 안에서 value는 string으로 확정
    return value.toUpperCase();
  }
  // ✅ 이 블록 안에서 value는 number로 확정
  return value.toFixed(2);
}
```

TypeScript는 조건문의 흐름을 추적합니다. 이것을 **제어 흐름 분석(Control Flow Analysis)**이라 합니다.

## 2️⃣ typeof 가드 [↑](#toc)

`typeof` 연산자는 JavaScript에도 있는 연산자입니다 (JS 2장 참고).
TypeScript는 `if (typeof x === "...")` 패턴을 인식하여 타입을 좁힙니다.

```typescript
function formatValue(value: string | number | boolean) {
  if (typeof value === "string") {
    // value: string
    return `"${value}"`;
  } else if (typeof value === "number") {
    // value: number
    return value.toLocaleString();
  } else {
    // value: boolean
    return value ? "예" : "아니오";
  }
}

console.log(formatValue("hello")); // "hello"
console.log(formatValue(12345));   // 12,345
console.log(formatValue(true));    // 예
```

실행 결과:
```
"hello"
12,345
예
```

### typeof가 반환하는 값

| typeof 결과 | 해당 타입 |
|-------------|-----------|
| `"string"` | string |
| `"number"` | number |
| `"boolean"` | boolean |
| `"undefined"` | undefined |
| `"object"` | object, null, 배열 |
| `"function"` | function |

> 주의: `typeof null === "object"` 입니다. null을 좁히려면 동등성 체크(`=== null`)를 사용하세요.

```typescript
function printLength(value: string | null) {
  if (typeof value === "string") {
    // ✅ null은 이 블록으로 들어오지 않음
    console.log(value.length);
  }
}
```

## 3️⃣ 진실성(Truthiness) 좁히기 [↑](#toc)

### 고통 → 해결

```typescript
// ❌ null/undefined를 확인하지 않으면 런타임 오류
function greet(name: string | null | undefined) {
  console.log(name.toUpperCase()); // 오류: name이 null이면 crash
}

// ✅ 진실성 검사로 null/undefined를 걷어냄
function greet(name: string | null | undefined) {
  if (name) {
    // name: string (null, undefined, "" 는 모두 제외됨)
    console.log(name.toUpperCase());
  } else {
    console.log("이름 없음");
  }
}
```

`if (value)` 조건은 `null`, `undefined`, `0`, `""`, `false`, `NaN`을 모두 걸러냅니다.
(JS 3장에서 배운 falsy 값 목록 참고)

```typescript
type UserInput = string | null | undefined;

function processInput(input: UserInput): string {
  if (!input) {
    return "입력값이 없습니다";
  }
  // 여기서 input: string (null, undefined 제거됨)
  return input.trim().toUpperCase();
}

console.log(processInput("  hello  ")); // HELLO
console.log(processInput(null));        // 입력값이 없습니다
console.log(processInput(undefined));   // 입력값이 없습니다
```

실행 결과:
```
HELLO
입력값이 없습니다
입력값이 없습니다
```

> 빈 문자열(`""`)도 falsy이므로, 빈 문자열과 null/undefined를 다르게 처리해야 한다면
> `if (input !== null && input !== undefined)` 또는 `if (input != null)`을 사용하세요.

## 4️⃣ 동등성 좁히기 [↑](#toc)

`===` 또는 `!==`으로 비교하면 리터럴 타입이 좁혀집니다.

```typescript
type Role = "admin" | "editor" | "viewer";

function getPermissions(role: Role): string[] {
  if (role === "admin") {
    // role: "admin"
    return ["read", "write", "delete", "manage"];
  } else if (role === "editor") {
    // role: "editor"
    return ["read", "write"];
  } else {
    // role: "viewer"
    return ["read"];
  }
}

console.log(getPermissions("admin"));  // ["read", "write", "delete", "manage"]
console.log(getPermissions("viewer")); // ["read"]
```

실행 결과:
```
[ 'read', 'write', 'delete', 'manage' ]
[ 'read' ]
```

### switch 문과 함께 — 가장 깔끔한 패턴

```typescript
type Direction = "north" | "south" | "east" | "west";

function move(direction: Direction): string {
  switch (direction) {
    case "north": return "위로 이동";
    case "south": return "아래로 이동";
    case "east":  return "오른쪽 이동";
    case "west":  return "왼쪽 이동";
  }
}
```

## 5️⃣ in 연산자 [↑](#toc)

`"속성명" in 객체` 표현식은 객체가 해당 속성을 가지고 있는지 확인합니다.
유니언 타입의 객체들이 서로 다른 속성을 가질 때 유용합니다.

```typescript
type Fish = { swim: () => void; name: string };
type Bird = { fly: () => void;  name: string };

function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    // animal: Fish
    animal.swim();
    console.log(`${animal.name}이(가) 수영합니다`);
  } else {
    // animal: Bird
    animal.fly();
    console.log(`${animal.name}이(가) 날아갑니다`);
  }
}

const fish: Fish = { swim: () => {}, name: "참치" };
const bird: Bird = { fly: () => {},  name: "독수리" };

move(fish); // 참치이(가) 수영합니다
move(bird); // 독수리이(가) 날아갑니다
```

실행 결과:
```
참치이(가) 수영합니다
독수리이(가) 날아갑니다
```

> `in` 연산자는 JS 객체에서도 동작합니다 (JS 7장 참고).
> TypeScript는 이 패턴을 인식해 타입을 좁혀줍니다.

## 6️⃣ instanceof 가드 [↑](#toc)

`instanceof`는 어떤 클래스의 인스턴스인지 확인합니다. 에러 처리에서 특히 자주 씁니다.

```typescript
function handleError(error: unknown): string {
  if (error instanceof TypeError) {
    // error: TypeError
    return `타입 오류: ${error.message}`;
  } else if (error instanceof RangeError) {
    // error: RangeError
    return `범위 오류: ${error.message}`;
  } else if (error instanceof Error) {
    // error: Error
    return `일반 오류: ${error.message}`;
  }
  return "알 수 없는 오류";
}

try {
  null!.toString(); // TypeError 발생
} catch (e) {
  console.log(handleError(e));
}
```

실행 결과:
```
타입 오류: Cannot read properties of null (reading 'toString')
```

### 사용자 정의 클래스에도 적용 가능

```typescript
class Dog {
  bark() { return "멍멍"; }
}

class Cat {
  meow() { return "야옹"; }
}

function makeSound(animal: Dog | Cat): string {
  if (animal instanceof Dog) {
    return animal.bark();
  }
  return animal.meow();
}
```

## 7️⃣ 판별 유니언 (Discriminated Unions) [↑](#toc)

> 이것이 가장 중요합니다. 실무 TypeScript 코드의 핵심 패턴입니다.

### 고통 → 해결

```typescript
// ❌ 이렇게 하면 매번 속성이 있는지 확인해야 함
type Shape =
  | { radius: number }        // Circle
  | { width: number; height: number }; // Rectangle

function area(shape: Shape): number {
  if ("radius" in shape) {
    return Math.PI * shape.radius ** 2;
  }
  return shape.width * shape.height; // TS가 자동으로 좁힘
}
```

```typescript
// ✅ 판별자(discriminant) 필드를 추가 — 훨씬 명확하고 안전
type Circle    = { kind: "circle";    radius: number };
type Rectangle = { kind: "rectangle"; width: number; height: number };
type Triangle  = { kind: "triangle";  base: number; height: number };

type Shape = Circle | Rectangle | Triangle;

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      // shape: Circle
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      // shape: Rectangle
      return shape.width * shape.height;
    case "triangle":
      // shape: Triangle
      return (shape.base * shape.height) / 2;
  }
}

const c: Circle    = { kind: "circle",    radius: 5 };
const r: Rectangle = { kind: "rectangle", width: 4, height: 6 };
const t: Triangle  = { kind: "triangle",  base: 3, height: 8 };

console.log(area(c).toFixed(2)); // 78.54
console.log(area(r));            // 24
console.log(area(t));            // 12
```

실행 결과:
```
78.54
24
12
```

### 왜 판별 유니언인가?

| 방식 | 장점 | 단점 |
|------|------|------|
| `in` 연산자 | 별도 필드 불필요 | 속성 이름 충돌 가능 |
| 판별 유니언 | 명확, exhaustive 체크 가능 | kind 필드 필수 |

판별 유니언은 **`kind`, `type`, `tag`** 같은 리터럴 타입 필드를 공통으로 가집니다.
그 필드 하나만 확인하면 TypeScript가 나머지 속성을 자동으로 좁혀줍니다.

### 실용 예제: API 응답 처리

```typescript
type ApiResponse =
  | { status: "loading" }
  | { status: "success"; data: string[] }
  | { status: "error";   message: string };

function renderResponse(res: ApiResponse): string {
  switch (res.status) {
    case "loading":
      return "로딩 중...";
    case "success":
      return `결과: ${res.data.join(", ")}`;
    case "error":
      return `오류: ${res.message}`;
  }
}

const loading: ApiResponse = { status: "loading" };
const success: ApiResponse = { status: "success", data: ["사과", "배", "포도"] };
const error:   ApiResponse = { status: "error",   message: "서버에 연결할 수 없음" };

console.log(renderResponse(loading)); // 로딩 중...
console.log(renderResponse(success)); // 결과: 사과, 배, 포도
console.log(renderResponse(error));   // 오류: 서버에 연결할 수 없음
```

실행 결과:
```
로딩 중...
결과: 사과, 배, 포도
오류: 서버에 연결할 수 없음
```

## 8️⃣ never를 이용한 완전성 체크 [↑](#toc)

판별 유니언에 새 변형을 추가했는데 처리 코드를 빠뜨리면 어떻게 될까요?
`never`를 이용하면 **컴파일러가 미처리 케이스를 알려줍니다.**

```typescript
type Shape =
  | { kind: "circle";    radius: number }
  | { kind: "rectangle"; width: number; height: number }
  | { kind: "triangle";  base: number; height: number };

function assertNever(value: never): never {
  throw new Error(`처리되지 않은 케이스: ${JSON.stringify(value)}`);
}

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "rectangle":
      return shape.width * shape.height;
    case "triangle":
      return (shape.base * shape.height) / 2;
    default:
      // 모든 케이스가 처리되었다면 shape는 never 타입
      return assertNever(shape); // ✅ 컴파일 타임에 미처리 케이스 감지
  }
}
```

### 새 타입을 추가했을 때의 동작

```typescript
// pentagon(오각형)을 추가하고 switch에 case를 안 넣으면?
type Shape =
  | { kind: "circle";    radius: number }
  | { kind: "rectangle"; width: number; height: number }
  | { kind: "triangle";  base: number; height: number }
  | { kind: "pentagon";  side: number }; // 새로 추가

// ❌ 컴파일 오류 발생:
// Argument of type '{ kind: "pentagon"; side: number; }' is not assignable to parameter of type 'never'.
// → assertNever(shape) 줄에서 오류가 나서 pentagon 케이스를 빠뜨렸음을 알 수 있음
```

> `never`는 "절대 도달할 수 없는 타입"입니다.
> 모든 가능성을 처리했다면 `default` 분기는 never가 되어 컴파일이 통과하고,
> 빠진 케이스가 있다면 컴파일 오류로 알려줍니다.

## 9️⃣ 정리 [↑](#toc)

### 핵심 개념 요약

| 좁히기 방법 | 문법 | 적합한 상황 |
|-------------|------|-------------|
| typeof 가드 | `typeof x === "string"` | 기본 타입 구별 (string, number, boolean) |
| 진실성 좁히기 | `if (value)` | null/undefined/falsy 제거 |
| 동등성 좁히기 | `x === "admin"` | 리터럴 타입 특정 |
| in 연산자 | `"prop" in obj` | 객체 타입 구별 (속성 유무) |
| instanceof | `x instanceof SomeClass` | 클래스 인스턴스 구별 |
| 판별 유니언 | `shape.kind === "circle"` | 복잡한 유니언 처리 (권장) |
| never 완전성 체크 | `assertNever(x)` | 미처리 케이스 컴파일 타임 감지 |
