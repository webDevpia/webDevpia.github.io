# 06장. 머지와 충돌 해결 — 두 평행 세계 합치기

> **머지 = 두 명이 각각 수정한 문서를 하나로 합치는 것.** 같은 줄을 다르게 고쳤으면 Git이 "어떤 걸 쓸까요?" 하고 물어봅니다.

---

> 💡 **충돌은 오류가 아닙니다.** 충돌은 Git이 "여기서 사람의 판단이 필요합니다"라고 알려주는 신호입니다. 충돌을 경험한 모든 개발자가 처음에는 당황했습니다. 이 장이 끝나면 충돌이 무섭지 않게 됩니다.

---

<a id="part1"></a>

## 1️⃣ 머지란 무엇인가? [↑](#toc)

`git merge`는 한 브랜치의 변경사항을 다른 브랜치에 통합합니다.

```bash
# 기본 사용법
# "지금 있는 브랜치에 대상 브랜치를 합칩니다"
git merge 합칠_브랜치명
```

머지는 크게 두 종류입니다:

| 종류 | 발생 조건 | 결과 |
|------|----------|------|
| **Fast-forward** | main이 멈춰 있을 때 | 포인터만 이동, 새 커밋 없음 |
| **3-way** | 양쪽이 모두 변경됐을 때 | 머지 커밋 생성 |

---

<a id="part2"></a>

## 2️⃣ Fast-forward 머지 [↑](#toc)

> "main이 멈춰 있었으면 그냥 앞으로 이동합니다."

### 상황 설명

```
A -- B (main)
      \
       C -- D (feature-login)
```

`feature-login` 브랜치가 만들어진 이후 **main에는 새 커밋이 없습니다.** main은 B에서 멈춰 있고, feature-login은 C, D로 앞서 나간 상태입니다.

---

### Fast-forward 머지 실행

```bash
# main으로 이동
git switch main

# feature-login을 main에 합치기
git merge feature-login
```

실행 결과:
```
Updating a1b2c3d..7e8f9g0
Fast-forward
 login.html | 1 +
 1 file changed, 1 insertion(+)
 create mode 100644 login.html
```

`Fast-forward` 라는 문구가 보입니다.

---

### 머지 후 상태

```
A -- B -- C -- D (main, feature-login)
```

main 포인터가 D로 이동했습니다. 새 커밋은 생기지 않았습니다.

```bash
git log --oneline --graph --all
```

실행 결과:
```
* 7e8f9g0 (HEAD -> main, feature-login) Add: 로그인 페이지 추가
* a1b2c3d Initial commit
```

> 💡 **Fast-forward는 새 커밋을 만들지 않습니다.** 포인터만 앞으로 이동합니다. 히스토리가 선형으로 유지되어 깔끔합니다.

---

<a id="part3"></a>

## 3️⃣ 3-way 머지 [↑](#toc)

> "양쪽이 모두 변경된 경우, Git은 세 지점을 비교해 머지 커밋을 만듭니다."

### 상황 설명

```
A -- B -- C (main)
      \
       D -- E (feature-signup)
```

feature-signup이 B에서 분기된 이후 **main에도 C라는 새 커밋이 생겼습니다.** 이제 두 브랜치가 모두 앞서 나간 상태입니다.

---

### 3-way 머지 실행

```bash
git switch main
git merge feature-signup
```

실행 결과:
```
Merge made by the 'ort' strategy.
 signup.html | 1 +
 1 file changed, 1 insertion(+)
 create mode 100644 signup.html
```

에디터가 열리며 머지 커밋 메시지를 입력하라고 합니다. 기본 메시지를 그대로 두고 저장하면 됩니다.

---

### 머지 후 상태

```
A -- B -- C ------- M (main, HEAD)
      \            /
       D -- E (feature-signup)
```

`M`은 **머지 커밋**입니다. 두 브랜치의 변경사항을 모두 담고 있으며, 부모가 두 개(`C`와 `E`)입니다.

```bash
git log --oneline --graph --all
```

실행 결과:
```
*   f1a2b3c (HEAD -> main) Merge branch 'feature-signup'
|\
| * 3d4e5f6 (feature-signup) Add: 회원가입 페이지 추가
* | 7g8h9i0 Update: README 수정
|/
* a1b2c3d Initial commit
```

> 💡 **3-way란?** Git은 세 지점을 비교합니다: (1) 공통 조상 커밋 B, (2) main의 최신 C, (3) feature-signup의 최신 E. 이 세 개를 비교해서 변경사항을 합칩니다.

---

<a id="part4"></a>

## 4️⃣ 충돌이란? [↑](#toc)

> "두 브랜치에서 **같은 파일의 같은 줄**을 다르게 수정했을 때."

3-way 머지는 대부분 자동으로 처리됩니다. 하지만 **같은 줄**을 두 브랜치가 모두 변경했다면 Git은 어느 것을 선택해야 할지 알 수 없습니다. 이것이 **충돌(Conflict)**입니다.

---

### 충돌 상황 직접 만들어보기

**준비:**

```bash
mkdir conflict-practice
cd conflict-practice
git init
echo "제목: 홈페이지" > index.html
git add index.html
git commit -m "Init: 첫 번째 커밋"
```

**main에서 수정:**

```bash
# main에서 index.html 수정
echo "제목: 메인 홈페이지" > index.html
git add index.html
git commit -m "Update: main에서 제목 변경"
```

**feature 브랜치에서도 같은 줄 수정:**

```bash
# 첫 번째 커밋으로 돌아가서 브랜치 만들기
git switch -c feature-title main~1
echo "제목: 새로운 홈페이지" > index.html
git add index.html
git commit -m "Update: feature에서 제목 변경"
```

**머지 시도:**

```bash
git switch main
git merge feature-title
```

실행 결과:
```
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.
```

> ⚠️ **당황하지 마세요.** 이 메시지는 "자동 합치기에 실패했습니다. 사람이 직접 결정해주세요"라는 뜻입니다. 리포지토리가 망가진 것이 아닙니다.

## 5️⃣ 충돌 마커 읽기 [↑](#toc)

충돌이 발생하면 Git은 해당 파일을 아래와 같이 수정합니다:

```
<<<<<<< HEAD
제목: 메인 홈페이지
=======
제목: 새로운 홈페이지
>>>>>>> feature-title
```

---

### 각 부분의 의미

| 마커 | 의미 |
|------|------|
| `<<<<<<< HEAD` | 충돌 구간 시작. 현재 브랜치(HEAD)의 내용이 시작됨 |
| `=======` | 구분선. 위쪽 = 현재 브랜치, 아래쪽 = 머지 대상 |
| `>>>>>>> feature-title` | 충돌 구간 끝. 머지 대상 브랜치의 내용이 끝남 |

정리하면:
- **`<<<` 와 `===` 사이**: 나의 수정 내용 (현재 브랜치)
- **`===` 와 `>>>` 사이**: 상대의 수정 내용 (머지 대상 브랜치)

> 💡 **팁**: 파일 안에 충돌 구간이 여러 개 있을 수 있습니다. 각각 따로 해결해야 합니다.

---

### 충돌 상태 확인

```bash
git status
```

실행 결과:
```
On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   index.html

no changes added to commit (use "git add" and/or "git commit -a")
```

`both modified` = "두 브랜치 모두 이 파일을 수정했습니다" → 충돌 파일 목록입니다.

## 6️⃣ 충돌 해결 step-by-step [↑](#toc)

충돌 해결은 6단계입니다. 단계를 순서대로 따르면 반드시 성공합니다.

---

### 1단계: 충돌 파일 열기

```bash
# 어떤 파일이 충돌났는지 확인
git status

# 파일 열기 (VS Code 사용 시)
code index.html
```

---

### 2단계: 마커와 내용 확인

파일을 열면 다음이 보입니다:

```
<<<<<<< HEAD
제목: 메인 홈페이지
=======
제목: 새로운 홈페이지
>>>>>>> feature-title
```

두 수정 내용을 읽고 어떤 것을 쓸지 결정합니다.

---

### 3단계: 원하는 내용만 남기기

상황에 따라 세 가지 선택이 있습니다:

**선택 A: 현재 브랜치(HEAD) 내용만 유지**
```
제목: 메인 홈페이지
```

**선택 B: 머지 대상 브랜치 내용만 유지**
```
제목: 새로운 홈페이지
```

**선택 C: 두 내용을 조합**
```
제목: 메인 홈페이지 (새로운 디자인)
```

---

### 4단계: 마커 삭제

> ⚠️ **가장 중요한 단계입니다.** `<<<<<<<`, `=======`, `>>>>>>>` 마커를 모두 삭제합니다. 마커가 파일에 남아 있으면 프로그램이 오작동합니다.

최종 파일 내용:
```
제목: 메인 홈페이지
```

---

### 5단계: git add

> ⚠️ **가장 흔한 실수: `git add`를 잊는 것!**

충돌을 해결했으면 Git에게 "이 파일의 충돌을 해결했습니다"라고 알려야 합니다.

```bash
git add index.html
```

```bash
# 확인
git status
```

실행 결과:
```
On branch main
All conflicts fixed but you are still merging.
  (use "git commit" to conclude merge)

Changes to be committed:
        modified:   index.html
```

`All conflicts fixed` 메시지가 보이면 5단계 완료입니다.

---

### 6단계: git commit

```bash
git commit
```

에디터가 열리며 기본 메시지가 입력되어 있습니다:

```
Merge branch 'feature-title'

# Conflicts:
#       index.html
```

메시지를 그대로 두고 저장하면 머지가 완료됩니다.

실행 결과:
```
[main 9k0l1m2] Merge branch 'feature-title'
```

## 7️⃣ VS Code에서 충돌 해결 [↑](#toc)

VS Code는 충돌 마커를 자동으로 감지하고 편리한 버튼을 제공합니다.

---

### 충돌 파일 열기

충돌 파일을 VS Code로 열면 마커 위에 4개의 버튼이 표시됩니다:

| 버튼 | 동작 | Git 동등 명령 |
|------|------|--------------|
| **Accept Current Change** | 현재 브랜치(HEAD) 내용만 유지 | `<<<` 위쪽 선택 |
| **Accept Incoming Change** | 머지 대상 브랜치 내용만 유지 | `>>>` 아래쪽 선택 |
| **Accept Both Changes** | 두 내용을 모두 유지 (순서대로) | 둘 다 합치기 |
| **Compare Changes** | 두 버전을 나란히 비교 | diff 뷰 |

---

### VS Code로 충돌 해결하는 순서

1. Source Control 패널에서 충돌 파일을 클릭합니다.
2. 버튼 중 하나를 클릭해 원하는 내용을 선택합니다.
3. 필요하면 내용을 추가로 편집합니다.
4. 파일을 저장합니다.
5. 터미널에서 `git add 파일명` → `git commit`을 실행합니다.

> 💡 **팁**: 버튼을 클릭한 후에도 파일을 직접 편집할 수 있습니다. "Accept Both Changes"를 누른 다음 내용을 다듬는 방식이 실무에서 자주 쓰입니다.

> ⚠️ **주의**: VS Code 버튼을 사용해도 `git add`와 `git commit`은 직접 실행해야 합니다. VS Code가 자동으로 커밋하지 않습니다.

## 8️⃣ 탈출구: git merge --abort [↑](#toc)

> "머지를 취소하고 이전 상태로 돌아갑니다. 당황하면 이 명령어!"

충돌이 발생했는데 해결 방법을 모르겠거나, 잘못 진행된 것 같다면 머지를 통째로 취소할 수 있습니다.

```bash
git merge --abort
```

실행 결과:
```
(nothing — 명령 성공 시 출력 없음)
```

```bash
# 상태 확인
git status
```

실행 결과:
```
On branch main
nothing to commit, working tree clean
```

머지 이전 상태로 완전히 돌아왔습니다.

---

> ⚠️ **`git merge --abort`는 충돌 해결 중에만 사용 가능합니다.** 머지가 이미 완료(커밋)되었다면 `git revert` 또는 `git reset`을 사용해야 합니다 (나중 장에서 다룹니다).

---

### 언제 --abort를 사용하나?

- 충돌이 너무 복잡해서 일단 취소하고 다시 생각하고 싶을 때
- 잘못된 브랜치를 머지하려고 했을 때
- 동료에게 먼저 물어보고 싶을 때

**취소 → 생각 → 다시 머지** 순서가 당황해서 잘못 고치는 것보다 훨씬 낫습니다.

## 9️⃣ 실습: 의도적 충돌 만들고 해결하기 [↑](#toc)

### 기본 실습

아래 단계를 따라 충돌을 만들고 해결합니다.

```bash
# 1. 새 폴더에서 시작
mkdir merge-practice && cd merge-practice
git init

# 2. 공통 파일 만들기
cat > story.txt << 'EOF'
첫 번째 줄: 모험이 시작됩니다.
두 번째 줄: 주인공이 길을 떠납니다.
세 번째 줄: 끝.
EOF
git add story.txt
git commit -m "Init: 이야기 시작"

# 3. feature 브랜치 만들기
git switch -c feature-ending

# 4. feature 브랜치에서 세 번째 줄 수정
cat > story.txt << 'EOF'
첫 번째 줄: 모험이 시작됩니다.
두 번째 줄: 주인공이 길을 떠납니다.
세 번째 줄: 해피엔딩.
EOF
git add story.txt
git commit -m "Update: feature에서 해피엔딩으로 변경"

# 5. main으로 돌아와서 같은 줄 다르게 수정
git switch main
cat > story.txt << 'EOF'
첫 번째 줄: 모험이 시작됩니다.
두 번째 줄: 주인공이 길을 떠납니다.
세 번째 줄: 새드엔딩.
EOF
git add story.txt
git commit -m "Update: main에서 새드엔딩으로 변경"

# 6. 머지 시도 → 충돌 발생!
git merge feature-ending
```

충돌 메시지를 확인하고, 6단계 해결 프로세스를 따라 해결합니다.

---

### 중급 실습

같은 파일에 충돌 구간이 **두 개 이상** 생기는 상황을 만들고 해결합니다.

1. `story.txt`에 줄을 추가해서 첫 번째 줄과 세 번째 줄을 각각 두 브랜치에서 다르게 수정합니다.
2. 머지하면 두 개의 충돌 구간이 생깁니다.
3. 각각 해결하고 커밋합니다.

---

### 심화 실습

1. 위 실습을 완료한 상태에서 `git merge --abort`가 작동하는지 확인합니다.
2. 새로운 충돌을 만들고 `git merge --abort`로 취소한 다음, 다시 머지해서 해결합니다.

## 🔟 정리 [↑](#toc)

| 명령어 | 역할 |
|--------|------|
| `git merge 브랜치명` | 현재 브랜치에 대상 브랜치 합치기 |
| `git merge --abort` | 진행 중인 머지 취소 |
| `git status` | 충돌 파일 목록 확인 |
| `git add 파일명` | 충돌 해결 완료 표시 |
| `git commit` | 머지 커밋 완성 |

---

### 충돌 해결 6단계 요약

```
1. git status          → 충돌 파일 확인
2. 파일 열기           → 마커 찾기
3. 내용 결정           → 어느 것을 쓸지 선택
4. 마커 삭제           → <<<, ===, >>> 모두 제거
5. git add 파일명      → 해결 완료 표시  ← 잊지 마세요!
6. git commit          → 머지 커밋 생성
```

---

### 충돌 마커 한눈에 보기

```
<<<<<<< HEAD          ← 현재 브랜치 시작
현재 브랜치 내용
=======               ← 구분선
머지 대상 브랜치 내용
>>>>>>> feature-name  ← 머지 대상 브랜치 끝
```
