# 04장. 변경 이력 관리 — 변경 일지 읽고 쓰기

> 의사가 환자의 차트를 보면 언제 어떤 처방이 있었는지 알 수 있습니다.
> 건축가가 설계 변경 일지를 보면 어떤 부분이 왜 바뀌었는지 알 수 있습니다.
> `git log`와 `git diff`는 코드의 "변경 일지"를 읽는 도구입니다.

---

<a id="part1"></a>

## 1️⃣ git log 심화 [↑](#toc)

> **Local Repo의 내용을 조회합니다.**

기본 `git log`는 정보가 많고 한 화면에 커밋 몇 개밖에 안 보입니다.
옵션을 활용하면 원하는 형태로 볼 수 있습니다.

---

### 3장에서 만든 커밋 기록

이어서 실습하기 위해 3장에서 만든 `my-first-git` 폴더를 사용합니다.

```bash
cd ~/my-first-git
git log
```

실행 결과:
```
commit d4e5f6a... (HEAD -> main)
Author: 홍길동 <honggildong@example.com>
Date:   Thu Apr 10 09:10:00 2026 +0900

    Docs: README 파일 추가

commit c3d4e5f...
...
```

---

### 옵션 1: `--oneline` — 한 줄 요약

가장 많이 쓰는 옵션입니다. 커밋을 한 줄로 압축해서 보여줍니다.

```bash
git log --oneline
```

실행 결과:
```
d4e5f6a (HEAD -> main) Docs: README 파일 추가
c3d4e5f Update: index.html에 CSS 링크 추가
b2c3d4e Add: 기본 CSS 스타일 추가
a1b2c3d 첫 번째 페이지 완성
```

각 줄의 구성: `짧은해시` `커밋메시지`

---

### 옵션 2: `--graph` — 브랜치 시각화

브랜치가 여러 개 있을 때 그래프로 보여줍니다. 지금은 브랜치가 하나뿐이므로 단순하게 나옵니다. 6장(브랜치)에서 진가를 발휘합니다.

```bash
git log --oneline --graph
```

실행 결과:
```
* d4e5f6a (HEAD -> main) Docs: README 파일 추가
* c3d4e5f Update: index.html에 CSS 링크 추가
* b2c3d4e Add: 기본 CSS 스타일 추가
* a1b2c3d 첫 번째 페이지 완성
```

---

### 옵션 3: `--all` — 모든 브랜치 포함

원격 브랜치를 포함해서 전체 커밋 이력을 보여줍니다.

```bash
git log --oneline --graph --all
```

5장(GitHub 연결) 이후에 원격 브랜치가 추가되면 이 옵션의 차이를 확인할 수 있습니다.

---

### 옵션 조합 — 실무 설정

실무에서 많이 쓰는 조합입니다.

```bash
git log --oneline --graph --all --decorate
```

실행 결과:
```
* d4e5f6a (HEAD -> main) Docs: README 파일 추가
* c3d4e5f Update: index.html에 CSS 링크 추가
* b2c3d4e Add: 기본 CSS 스타일 추가
* a1b2c3d 첫 번째 페이지 완성
```

> 💡 실무 팁: 긴 명령어를 매번 입력하기 불편하면 별칭(alias)을 등록하세요.
> `git config --global alias.lg "log --oneline --graph --all --decorate"`
> 이후 `git lg`만 입력해도 동일한 결과를 볼 수 있습니다.

---

### 특정 파일의 이력만 보기

```bash
git log --oneline -- index.html
```

실행 결과:
```
c3d4e5f Update: index.html에 CSS 링크 추가
a1b2c3d 첫 번째 페이지 완성
```

`index.html`이 변경된 커밋만 보여줍니다.

---

### 최근 N개만 보기

```bash
# 최근 2개 커밋만 보기
git log --oneline -2
```

실행 결과:
```
d4e5f6a (HEAD -> main) Docs: README 파일 추가
c3d4e5f Update: index.html에 CSS 링크 추가
```

## 2️⃣ git diff [↑](#toc)

> **Working Dir 또는 Staging Area의 변경 내용을 비교합니다.**

> 의사가 X-ray 사진 두 장을 나란히 놓고 비교하듯,
> `git diff`는 파일의 이전 상태와 현재 상태를 줄 단위로 비교해 보여줍니다.

---

### 실습 준비: 파일 수정

```bash
# index.html에 내용 추가 (VS Code로 직접 편집해도 됩니다)
echo "<footer>하단 영역</footer>" >> index.html
```

---

### `git diff` — Working Dir vs 마지막 커밋

`git add` 전에 실행하면 Working Dir의 변경 사항을 보여줍니다.

```bash
git diff
```

실행 결과:
```
diff --git a/index.html b/index.html
index 3a5b6c7..8d9e0f1 100644
--- a/index.html
+++ b/index.html
@@ -1 +1,2 @@
 <!DOCTYPE html><html><body><h1>안녕하세요!</h1></body></html>
+<footer>하단 영역</footer>
```

---

### 출력 해석

| 표시 | 의미 |
|------|------|
| `--- a/index.html` | 변경 전 (이전 버전) |
| `+++ b/index.html` | 변경 후 (현재 버전) |
| `-` (빨간색) | 삭제된 줄 |
| `+` (초록색) | 추가된 줄 |
| 아무 표시 없음 | 변경되지 않은 줄 (문맥) |

---

### `git diff --staged` — Staging Area vs 마지막 커밋

`git add` 후에 실행하면 스테이징된 변경 사항을 보여줍니다.

```bash
git add index.html
git diff --staged
```

실행 결과:
```
diff --git a/index.html b/index.html
index 3a5b6c7..8d9e0f1 100644
--- a/index.html
+++ b/index.html
@@ -1 +1,2 @@
 <!DOCTYPE html><html><body><h1>안녕하세요!</h1></body></html>
+<footer>하단 영역</footer>
```

> 💡 실무 팁: 커밋 전에 `git diff --staged`를 실행해서 정말 원하는 내용만 포함되었는지 확인하는 습관을 들이세요.
> 의도치 않은 변경이나 디버그 코드(`console.log`)가 포함되지 않도록 방지할 수 있습니다.

---

### 두 커밋 사이의 차이 비교

```bash
# 커밋 해시는 git log --oneline으로 확인
git diff a1b2c3d d4e5f6a
```

실행 결과 (a1b2c3d 커밋과 d4e5f6a 커밋 사이의 모든 변경사항):
```
diff --git a/README.md b/README.md
new file mode 100644
index 0000000..1234567
--- /dev/null
+++ b/README.md
@@ -0,0 +1 @@
+# 나의 첫 Git 프로젝트
...
```

---

### VS Code에서 확인

VS Code에서 확인: 소스 제어 패널에서 변경된 파일을 클릭하면 변경 전/후를 나란히 비교하는 diff 뷰가 열립니다. 빨간색(삭제), 초록색(추가)으로 직관적으로 볼 수 있습니다.

## 3️⃣ .gitignore [↑](#toc)

> **.gitignore는 "이 파일들은 택배 상자에 절대 넣지 마세요" 목록입니다.**

> 모든 파일을 커밋에 포함할 필요는 없습니다.
> 비밀번호 파일, 로그 파일, 자동 생성 파일은 Git이 추적하지 않아야 합니다.
> `.gitignore`는 이런 파일들을 미리 제외하는 설정 파일입니다.

---

### 추적하면 안 되는 파일들

| 분류 | 예시 | 이유 |
|------|------|------|
| 비밀 정보 | `.env`, `secrets.json` | 비밀번호, API 키가 GitHub에 공개됨 |
| 자동 생성 파일 | `node_modules/`, `__pycache__/` | 용량이 크고 `npm install`로 재생성 가능 |
| 운영체제 파일 | `.DS_Store` (Mac), `Thumbs.db` (Windows) | 불필요한 시스템 파일 |
| 빌드 결과물 | `dist/`, `build/` | 소스 코드에서 자동 생성됨 |
| 로그 파일 | `*.log`, `error.log` | 기록이지 소스 코드가 아님 |
| IDE 설정 | `.vscode/settings.json` (선택) | 개인 설정은 공유하지 않는 경우 |

---

### .gitignore 파일 만들기

```bash
# 프로젝트 루트(최상위 폴더)에 .gitignore 생성
touch .gitignore

# VS Code로 열기
code .gitignore
```

---

### .gitignore 문법

```gitignore
# 이것은 주석입니다 (샵으로 시작)

# 1. 특정 파일 제외
.env
secrets.json

# 2. 특정 확장자 전체 제외
*.log
*.tmp

# 3. 특정 폴더 전체 제외 (끝에 / 붙이기)
node_modules/
dist/
build/
__pycache__/

# 4. 특정 경로의 파일만 제외
/설정파일.json

# 5. 예외 — 위에서 제외한 것 중 이것만 포함
!important.log
!필수설정.json

# 6. 운영체제 자동 생성 파일
.DS_Store
Thumbs.db
```

---

### 실제 프로젝트별 권장 .gitignore

**웹 프론트엔드 (HTML/CSS/JS):**

```gitignore
# OS 생성 파일
.DS_Store
Thumbs.db

# 에디터 설정
.vscode/
*.swp

# 로그
*.log

# 비밀 파일
.env
.env.local
```

**Node.js 프로젝트:**

```gitignore
# 의존성 폴더 (npm install로 재생성 가능)
node_modules/

# 빌드 결과물
dist/
build/

# 환경 변수 (절대 커밋하면 안 됨)
.env
.env.local
.env.production

# 로그
*.log
npm-debug.log*

# OS 파일
.DS_Store
Thumbs.db
```

---

### .gitignore 적용 확인

```bash
# .gitignore 적용 후 상태 확인
echo "DB_PASSWORD=1234" > .env
git status
```

실행 결과 (`.env`가 목록에 없으면 정상):
```
On branch main
Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   .gitignore
```

`.env`는 `.gitignore`에 등록되어 있으므로 `git status`에 나타나지 않습니다.

---

### .gitignore 캐시 문제 해결

이미 추적 중인 파일을 나중에 `.gitignore`에 추가해도 Git이 계속 추적하는 경우가 있습니다. 이럴 때는 캐시를 제거합니다.

```bash
# 캐시 전체 삭제 후 재추가
git rm -r --cached .
git add .
git commit -m "Fix: .gitignore 적용 — 캐시 정리"
```

> ⚠️ 이 명령어는 **파일 자체를 삭제하지 않습니다**. Git의 추적 캐시만 삭제합니다.
> 실행 후 파일은 그대로 존재하고, `.gitignore`가 올바르게 적용됩니다.

---

### .gitignore 생성기 활용

프로젝트 유형에 맞는 `.gitignore`를 자동으로 생성해주는 사이트:

- [gitignore.io](https://www.toptal.com/developers/gitignore) — 언어/프레임워크 선택하면 생성
- GitHub 저장소 생성 시 — `.gitignore` 템플릿 선택 옵션 제공

> 💡 실무 팁: `.gitignore`는 **`git init` 직후, 첫 번째 커밋 전에** 만들어야 합니다.
> 이미 추적 중인 파일은 나중에 제외하기가 더 복잡해집니다.

## 4️⃣ git show [↑](#toc)

> **특정 커밋 하나의 변경 내용을 상세히 보여줍니다.**

> `git log`가 "앨범 목차"라면, `git show`는 "앨범의 특정 페이지를 펼쳐보는 것"입니다.

```bash
# 가장 최근 커밋 내용 보기 (HEAD)
git show
```

실행 결과:
```
commit d4e5f6a... (HEAD -> main)
Author: 홍길동 <honggildong@example.com>
Date:   Thu Apr 10 09:10:00 2026 +0900

    Docs: README 파일 추가

diff --git a/README.md b/README.md
new file mode 100644
index 0000000..1234567
--- /dev/null
+++ b/README.md
@@ -0,0 +1 @@
+# 나의 첫 Git 프로젝트
```

---

### 특정 커밋 내용 보기

```bash
# 커밋 해시 앞 7자리로 지정 (git log --oneline으로 확인)
git show a1b2c3d
```

실행 결과:
```
commit a1b2c3d...
Author: 홍길동 <honggildong@example.com>
Date:   Thu Apr 10 09:00:00 2026 +0900

    첫 번째 페이지 완성

diff --git a/index.html b/index.html
new file mode 100644
index 0000000..abcdef1
--- /dev/null
+++ b/index.html
@@ -0,0 +1 @@
+<!DOCTYPE html><html><body><h1>안녕하세요!</h1></body></html>
```

---

### 특정 커밋의 파일만 보기

```bash
# 특정 커밋 시점의 파일 내용 보기
git show a1b2c3d:index.html
```

실행 결과:
```
<!DOCTYPE html><html><body><h1>안녕하세요!</h1></body></html>
```

## 5️⃣ 커밋 메시지 규칙 [↑](#toc)

> 좋은 커밋 메시지는 "미래의 나와 팀원을 위한 편지"입니다.
> 6개월 후에 `git log`를 봤을 때, 무슨 작업을 했는지 알 수 있어야 합니다.

---

### Conventional Commits 간소화 스타일

이 강의에서 사용하는 커밋 메시지 형식입니다.

```
타입: 한 줄 요약 (50자 이내)

(선택) 더 자세한 설명이 필요하면 빈 줄 후에 작성합니다.
왜 이 변경이 필요했는지, 어떤 문제를 해결했는지 설명합니다.
```

---

### 타입 목록

| 타입 | 사용 상황 | 예시 |
|------|----------|------|
| `Add` | 새 기능, 새 파일 추가 | `Add: 사용자 로그인 기능` |
| `Fix` | 버그 수정 | `Fix: 결제 금액 계산 오류 수정` |
| `Update` | 기존 기능 개선 | `Update: 이미지 로딩 속도 개선` |
| `Remove` | 파일, 코드 삭제 | `Remove: 사용하지 않는 레거시 코드 삭제` |
| `Docs` | 문서, 주석 변경 | `Docs: API 사용 방법 주석 추가` |
| `Style` | 코드 스타일, 포맷 변경 (기능 변화 없음) | `Style: 들여쓰기 정리` |
| `Refactor` | 리팩토링 (기능 변화 없음) | `Refactor: 중복 함수 통합` |
| `Test` | 테스트 추가/수정 | `Test: 로그인 유효성 검사 테스트 추가` |

---

### 나쁜 vs 좋은 커밋 메시지 비교

| 나쁜 예 | 좋은 예 |
|---------|---------|
| `수정함` | `Fix: 모바일 화면에서 버튼이 겹치는 문제 수정` |
| `asdf` | `Add: 상품 상세 페이지 이미지 슬라이더 추가` |
| `작업중` | `Update: 회원가입 이메일 유효성 검사 강화` |
| `ㅋㅋ` | `Remove: 디버그용 console.log 코드 삭제` |
| `final` | `Docs: 배포 방법 README에 추가` |

---

### 커밋 메시지 작성 팁

1. **현재형 명령문**으로 작성합니다. "수정했다" → "Fix"
2. **50자 이내**로 한 줄 요약합니다.
3. 커밋 내용이 여러 주제라면 **커밋을 나눕니다**. ("Add 기능 + Fix 버그"는 두 개의 커밋으로)
4. "무엇을"보다 "**왜**"를 설명하면 더 좋습니다.

---

### 빈 줄 포함 커밋 메시지 작성

```bash
# -m 여러 번 사용
git commit -m "Fix: 로그인 후 리다이렉트 오류 수정" -m "iOS Safari에서 쿠키가 올바르게 저장되지 않아 로그인 후 메인 페이지로 이동하지 않는 문제를 수정했습니다."
```

또는 VS Code 편집기로 작성:

```bash
# 메시지 없이 commit → VS Code 편집기가 열림
git commit
```

VS Code가 열리면 첫 줄에 요약, 빈 줄, 세부 설명 순서로 작성 후 저장하고 닫습니다.

## 6️⃣ 정리 [↑](#toc)

### 핵심 개념 요약

| 개념 | 설명 | 비유 |
|------|------|------|
| `git log --oneline` | 커밋 이력을 한 줄로 압축해서 보기 | 앨범 목차 보기 |
| `git log --graph` | 브랜치 분기를 그래프로 시각화 | 가계도 그리기 |
| `git diff` | Working Dir의 변경 내용 확인 (add 전) | X-ray 비교 |
| `git diff --staged` | Staging Area의 변경 내용 확인 (add 후) | 포장 전 최종 검수 |
| `.gitignore` | Git이 추적하지 않을 파일 목록 | 비밀 서류함 |
| `git rm --cached` | 이미 추적 중인 파일을 추적 중단 | "이건 앨범에서 빼주세요" |
| `git show` | 특정 커밋의 상세 변경 내용 보기 | 앨범 특정 페이지 펼치기 |
| Conventional Commits | 타입: 메시지 형식의 커밋 메시지 규칙 | 표준화된 업무 일지 양식 |
