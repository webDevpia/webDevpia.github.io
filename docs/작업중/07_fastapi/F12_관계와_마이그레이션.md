# 12장. 관계와 마이그레이션 — 데이터 연결하기

<a id="onetomany"></a>

## 1️⃣ 1:N 관계란? [↑](#toc)

> "관계 = 식당의 주문서와 메뉴 연결.
> 하나의 주문서(Order)에는 여러 메뉴(Item)가 담길 수 있고,
> 각 메뉴 항목은 반드시 하나의 주문서에 속합니다."

실제 서비스에서 데이터는 항상 서로 연결되어 있습니다.

```
사용자 (User)        할 일 (Todo)
─────────────       ──────────────────────
홍길동         ──→  "FastAPI 공부하기"
               ──→  "운동하기"
               ──→  "책 읽기"

김철수         ──→  "장보기"
               ──→  "청소하기"
```

"한 명의 사용자가 여러 개의 할 일을 가진다" — 이것이 **1:N 관계**입니다.

| 관계 종류 | 예시 | 설명 |
|-----------|------|------|
| 1:N | 사용자 → 할 일들 | 한 쪽이 여러 쪽을 가짐 |
| 1:1 | 사용자 → 프로필 | 각각 하나씩 대응 |
| M:N | 게시글 ↔ 태그 | 양쪽 모두 여럿 가능 |

이번 장에서는 가장 흔한 **1:N 관계**를 구현합니다.

### 외래 키 (Foreign Key)

1:N 관계를 DB에 저장하려면 "어느 사용자의 것인지" 표시하는 열이 필요합니다.
이것이 **외래 키**입니다.

```
Todo 테이블
─────────────────────────────────────
id │ title           │ user_id  ← 외래 키
───┼─────────────────┼─────────
 1 │ FastAPI 공부하기  │   1     ← 사용자 1번(홍길동)의 것
 2 │ 운동하기          │   1     ← 사용자 1번(홍길동)의 것
 3 │ 장보기            │   2     ← 사용자 2번(김철수)의 것
```

> "외래 키 = 주문서에 적힌 테이블 번호.
> '이 주문은 3번 테이블 손님 것'이라는 표시입니다."

## 2️⃣ SQLModel 관계 정의 [↑](#toc)

User와 Todo 모델에 관계를 추가합니다.

```python
# models.py
from typing import Optional
from sqlmodel import Field, Relationship, SQLModel


# ── User 모델 ────────────────────────────────────────────

class UserBase(SQLModel):
    username: str = Field(index=True, unique=True)
    email: str


class User(UserBase, table=True):
    id: int | None = Field(default=None, primary_key=True)

    # User → Todo 관계: 한 명의 사용자가 여러 개의 할 일을 가집니다
    todos: list["Todo"] = Relationship(back_populates="owner")


class UserCreate(UserBase):
    pass


class UserPublic(UserBase):
    id: int


# ── Todo 모델 ────────────────────────────────────────────

class TodoBase(SQLModel):
    title: str
    done: bool = False


class Todo(TodoBase, table=True):
    id: int | None = Field(default=None, primary_key=True)

    # 외래 키: 이 Todo가 어느 User에 속하는지
    user_id: int | None = Field(default=None, foreign_key="user.id")

    # Todo → User 관계 (역방향)
    owner: Optional["User"] = Relationship(back_populates="todos")


class TodoCreate(TodoBase):
    user_id: int  # 생성 시 반드시 user_id 지정


class TodoPublic(TodoBase):
    id: int
    user_id: int
```

### 핵심 포인트

| 코드 | 설명 |
|------|------|
| `foreign_key="user.id"` | `user` 테이블의 `id` 열을 참조합니다 (소문자 주의) |
| `Relationship(back_populates="todos")` | 양방향 관계 설정 — User에서 `.todos`로 접근 가능 |
| `Relationship(back_populates="owner")` | Todo에서 `.owner`로 해당 User에 접근 가능 |
| `list["Todo"]` | 문자열 참조 — 아직 정의되지 않은 클래스를 참조할 때 |

### 관계 방향 그림

```
User                           Todo
──────────────                 ─────────────────────
id: int                        id: int
username: str                  title: str
todos: list[Todo]  ←──────→   user_id: int (FK)
                               owner: User
```

## 3️⃣ 관계 데이터 조회 [↑](#toc)

### 기본 조회 vs 관계 포함 조회

기본적으로 SQLModel은 관계 데이터를 **자동으로 불러오지 않습니다** (지연 로딩, lazy loading).
세션이 닫힌 뒤 `.todos`에 접근하면 오류가 발생합니다.

# ❌ 세션이 닫힌 뒤 관계 접근 (오류 발생)
```python
@router.get("/{user_id}")
def read_user(user_id: int, session: Session = Depends(get_session)):
    user = session.get(User, user_id)
    return user  # todos는 아직 로딩되지 않음
    # → response_model에서 user.todos 접근 시 DetachedInstanceError!
```

# ✅ selectinload로 관계 데이터 함께 불러오기
```python
from sqlmodel import select
from sqlalchemy.orm import selectinload


class UserWithTodos(UserPublic):
    todos: list[TodoPublic] = []


@router.get("/{user_id}", response_model=UserWithTodos)
def read_user_with_todos(user_id: int, session: Session = Depends(get_session)):
    statement = (
        select(User)
        .where(User.id == user_id)
        .options(selectinload(User.todos))  # ← todos도 함께 불러옵니다
    )
    user = session.exec(statement).first()
    if not user:
        raise HTTPException(status_code=404, detail="User not found")
    return user
```

실행 결과 (Swagger UI에서 `GET /users/1` 호출):
```json
{
  "id": 1,
  "username": "honggildong",
  "email": "hong@example.com",
  "todos": [
    { "id": 1, "title": "FastAPI 공부하기", "done": false, "user_id": 1 },
    { "id": 2, "title": "운동하기", "done": true, "user_id": 1 }
  ]
}
```

> "`selectinload`는 SQL을 두 번 실행합니다.
> 첫 번째로 User를 가져오고, 두 번째로 해당 User의 Todo들을 한꺼번에 가져옵니다.
> N+1 문제를 피할 수 있는 효율적인 방법입니다."

### 특정 사용자의 할 일 목록 조회

```python
@router.get("/{user_id}/todos", response_model=list[TodoPublic])
def read_user_todos(user_id: int, session: Session = Depends(get_session)):
    # user가 존재하는지 먼저 확인
    user = session.get(User, user_id)
    if not user:
        raise HTTPException(status_code=404, detail="User not found")

    statement = select(Todo).where(Todo.user_id == user_id)
    todos = session.exec(statement).all()
    return todos
```

## 4️⃣ 관계 데이터 생성 [↑](#toc)

### 사용자 생성

```python
@router.post("/", response_model=UserPublic, status_code=201)
def create_user(user_data: UserCreate, session: Session = Depends(get_session)):
    # username 중복 확인
    existing = session.exec(
        select(User).where(User.username == user_data.username)
    ).first()
    if existing:
        raise HTTPException(status_code=400, detail="이미 사용 중인 username입니다")

    db_user = User.model_validate(user_data)
    session.add(db_user)
    session.commit()
    session.refresh(db_user)
    return db_user
```

### 특정 사용자의 할 일 생성

```python
@router.post("/", response_model=TodoPublic, status_code=201)
def create_todo(todo_data: TodoCreate, session: Session = Depends(get_session)):
    # user_id가 실제로 존재하는지 확인
    user = session.get(User, todo_data.user_id)
    if not user:
        raise HTTPException(status_code=404, detail="해당 사용자가 존재하지 않습니다")

    db_todo = Todo.model_validate(todo_data)
    session.add(db_todo)
    session.commit()
    session.refresh(db_todo)
    return db_todo
```

실행 결과 (Swagger UI에서 `POST /todos` 호출):
```json
// 요청 Body
{
  "title": "SQLModel 공부하기",
  "done": false,
  "user_id": 1
}

// 응답
{
  "id": 3,
  "title": "SQLModel 공부하기",
  "done": false,
  "user_id": 1
}
```

### 존재하지 않는 user_id로 요청하면?

```json
// 요청 Body
{

  "done": false,
  "user_id": 999
}

// 응답 (404)
{
  "detail": "해당 사용자가 존재하지 않습니다"
}
```

## 5️⃣ Alembic 소개 [↑](#toc)

11장에서 배운 `SQLModel.metadata.create_all(engine)`은 테이블이 없을 때 새로 만들어줍니다.
하지만 이미 서비스 중인 DB에서 테이블 구조를 바꾸려면 어떻게 해야 할까요?

### 문제 상황

```
서비스 시작: Todo 테이블 = id, title, done
    ↓
서비스 3개월 운영 중 — 데이터 5만 건 존재
    ↓
기획: "Todo에 마감일(due_date)을 추가해주세요!"
    ↓
개발자: 코드에 due_date 필드 추가
    ↓
???  서버 재시작 시 create_all은 이미 있는 테이블을 건드리지 않습니다
     → due_date 열이 DB에 추가되지 않아서 오류 발생!
```

# ❌ 이렇게 하면 안 됩니다
```bash
# 기존 DB 삭제하고 다시 create_all
rm database.db
# → 5만 건의 데이터가 모두 사라집니다!
```

# ✅ Alembic으로 안전하게 변경
```bash
alembic revision --autogenerate -m "add due_date to todo"
alembic upgrade head
# → 기존 데이터 유지하면서 due_date 열만 추가합니다
```

> "Alembic = DB 스키마의 git.
> 코드 변경을 커밋하듯, DB 구조 변경을 '마이그레이션'으로 기록합니다.
> 언제 어떤 변경이 일어났는지 추적할 수 있고, 롤백도 가능합니다."

### Alembic의 역할

| 상황 | create_all | Alembic |
|------|-----------|---------|
| 테이블이 없을 때 | ✅ 생성 | ✅ 생성 |
| 열 추가 | ❌ 무시 | ✅ 추가 |
| 열 이름 변경 | ❌ 무시 | ✅ 변경 |
| 열 타입 변경 | ❌ 무시 | ✅ 변경 |
| 변경 이력 관리 | ❌ 없음 | ✅ 버전 관리 |
| 롤백 | ❌ 불가 | ✅ 가능 |

## 6️⃣ Alembic 기초 [↑](#toc)

### 설치

```bash
pip install alembic
```

### Step 1: 초기화

```bash
alembic init alembic
```

실행 결과:
```
  Creating directory /your-project/alembic ...  done
  Creating directory /your-project/alembic/versions ...  done
  Generating /your-project/alembic/env.py ...  done
  Generating /your-project/alembic/script.py.mako ...  done
  Generating /your-project/alembic.ini ...  done
```

생성되는 파일 구조:
```
프로젝트/
├── alembic/
│   ├── env.py          ← DB 연결 설정
│   ├── script.py.mako  ← 마이그레이션 파일 템플릿
│   └── versions/       ← 마이그레이션 파일들이 저장됩니다
├── alembic.ini         ← Alembic 설정 파일
└── app/
```

### Step 2: alembic.ini 설정

```ini
# alembic.ini
sqlalchemy.url = sqlite:///database.db
```

### Step 3: env.py 설정

`alembic/env.py`에서 SQLModel 모델을 인식하도록 설정합니다.

```python
# alembic/env.py (수정 부분)
from sqlmodel import SQLModel
from app.models import User, Todo  # 모든 모델을 import 해야 자동 감지됩니다

target_metadata = SQLModel.metadata  # SQLModel의 메타데이터 사용
```

### Step 4: 마이그레이션 파일 생성

```bash
alembic revision --autogenerate -m "add users and todos tables"
```

실행 결과:
```
INFO  [alembic.runtime.migration] Context impl SQLiteImpl.
INFO  [alembic.runtime.migration] Will assume non-transactional DDL.
INFO  [alembic.autogenerate.compare] Detected added table 'user'
INFO  [alembic.autogenerate.compare] Detected added table 'todo'
  Generating .../alembic/versions/a1b2c3d4e5f6_add_users_and_todos_tables.py ...  done
```

`--autogenerate`는 **현재 모델 코드**와 **현재 DB 상태**를 비교해서 차이를 자동으로 감지합니다.

### Step 5: 마이그레이션 실행

```bash
alembic upgrade head
```

실행 결과:
```
INFO  [alembic.runtime.migration] Context impl SQLiteImpl.
INFO  [alembic.runtime.migration] Will assume non-transactional DDL.
INFO  [alembic.runtime.migration] Running upgrade  -> a1b2c3d4e5f6, add users and todos tables
```

`head`는 "가장 최신 버전으로 업그레이드"를 의미합니다.

### 주요 Alembic 명령어

| 명령어 | 설명 |
|--------|------|
| `alembic init alembic` | 프로젝트 초기화 |
| `alembic revision --autogenerate -m "설명"` | 변경사항 자동 감지 후 마이그레이션 파일 생성 |
| `alembic upgrade head` | 최신 버전으로 DB 업데이트 |
| `alembic downgrade -1` | 한 버전 롤백 |
| `alembic current` | 현재 DB 버전 확인 |
| `alembic history` | 마이그레이션 히스토리 조회 |

## 7️⃣ 실전: 사용자 모델 추가 마이그레이션 [↑](#toc)

실제 시나리오: Todo만 있던 DB에 User 모델을 추가하는 과정을 따라가봅니다.

### 시작 상태

```
현재 DB: todo 테이블만 존재 (id, title, done)
목표: user 테이블 추가 + todo에 user_id 외래 키 추가
```

### Step 1: 모델 코드 수정

```python
# models.py에 User 클래스와 Todo.user_id 추가 (앞서 작성한 코드)
```

### Step 2: 마이그레이션 파일 생성

```bash
alembic revision --autogenerate -m "add user table and user_id to todo"
```

생성된 파일 (`alembic/versions/xxxx_add_user_table.py`):

```python
"""add user table and user_id to todo

Revision ID: xxxx
Revises:
Create Date: 2026-04-10 10:00:00

"""
from alembic import op
import sqlalchemy as sa


def upgrade() -> None:
    # user 테이블 생성
    op.create_table(
        'user',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('username', sa.String(), nullable=False),
        sa.Column('email', sa.String(), nullable=False),
        sa.PrimaryKeyConstraint('id'),
    )
    op.create_index(op.f('ix_user_username'), 'user', ['username'], unique=True)

    # todo 테이블에 user_id 열 추가
    with op.batch_alter_table('todo') as batch_op:
        batch_op.add_column(sa.Column('user_id', sa.Integer(), nullable=True))
        batch_op.create_foreign_key('fk_todo_user_id', 'user', ['user_id'], ['id'])


def downgrade() -> None:
    with op.batch_alter_table('todo') as batch_op:
        batch_op.drop_constraint('fk_todo_user_id', type_='foreignkey')
        batch_op.drop_column('user_id')

    op.drop_index(op.f('ix_user_username'), table_name='user')
    op.drop_table('user')
```

> "Alembic이 `upgrade()`와 `downgrade()` 양쪽을 모두 자동 생성합니다.
> `upgrade()`는 변경 적용, `downgrade()`는 롤백 코드입니다."

### Step 3: 마이그레이션 실행

```bash
alembic upgrade head
```

실행 결과:
```
INFO  [alembic.runtime.migration] Running upgrade  -> xxxx, add user table and user_id to todo
```

### Step 4: 결과 확인

```bash
alembic current
```

실행 결과:
```
xxxx (head)
```

### Step 5: 롤백이 필요하면

```bash
# 한 버전 되돌리기
alembic downgrade -1
```

실행 결과:
```
INFO  [alembic.runtime.migration] Running downgrade xxxx -> , add user table and user_id to todo
```

## 8️⃣ 정리 [↑](#toc)

### 핵심 개념 정리

| 개념 | 설명 | 코드 |
|------|------|------|
| 1:N 관계 | 한 쪽이 여러 쪽을 가짐 | User → Todos |
| `foreign_key` | 다른 테이블의 열 참조 | `Field(foreign_key="user.id")` |
| `Relationship()` | SQLModel 관계 정의 | `todos: list["Todo"] = Relationship(back_populates="todos")` |
| `back_populates` | 양방향 관계 연결 | 양쪽 모델에 같은 이름 지정 |
| `selectinload` | 관계 데이터 즉시 로딩 | `.options(selectinload(User.todos))` |
| Alembic | DB 스키마 버전 관리 | `alembic revision`, `alembic upgrade` |
| `--autogenerate` | 변경사항 자동 감지 | 모델 코드와 DB 비교 |
| `upgrade head` | 최신 버전으로 DB 업데이트 | `alembic upgrade head` |
| `downgrade -1` | 한 버전 롤백 | `alembic downgrade -1` |

### 관계 정의 체크리스트

```
✅ 1. "많은 쪽" 모델에 foreign_key 필드 추가
✅ 2. "많은 쪽" 모델에 Relationship(back_populates="소유자필드명") 추가
✅ 3. "하나 쪽" 모델에 Relationship(back_populates="목록필드명") 추가
✅ 4. Create 스키마에 user_id(또는 FK 필드) 포함
✅ 5. 생성 시 FK가 실제로 존재하는지 검증
✅ 6. 조회 시 selectinload로 관계 데이터 명시적으로 불러오기
```

### Alembic 워크플로우

```
모델 코드 수정
    ↓
alembic revision --autogenerate -m "변경 설명"
    ↓
생성된 마이그레이션 파일 검토 (자동 생성이라도 확인!)
    ↓
alembic upgrade head
    ↓
서버 재시작
```
