# 09장. 에러 처리 — 우아하게 실패하기
{: .no_toc }

> "에러 처리 = 식당의 위기 대응 매뉴얼 — 재료 소진 시(404), 주문서 오류 시(422), 주방 화재 시(500) 각각 다르게 대응합니다"
> 좋은 식당은 문제가 생겼을 때 당황하지 않습니다.
> "죄송합니다, 오늘 해당 메뉴는 재료 소진으로 제공이 어렵습니다"는 친절한 안내입니다.
> "ERROR"라고만 적힌 종이를 내미는 것은 서비스가 아닙니다.

## 학습 목표
{: .no_toc }

- `HTTPException`으로 표준 HTTP 에러 응답을 반환하는 방법을 이해합니다
- 커스텀 예외 클래스와 `@app.exception_handler()`를 활용해 일관된 에러 형식을 만듭니다
- Pydantic 검증 에러(`RequestValidationError`)를 가로채 친절한 메시지로 바꿉니다
- Swagger UI에서 에러 응답을 문서화하는 방법을 배웁니다
- 예상치 못한 에러를 잡아내는 전역 핸들러 패턴을 습득합니다

## 1️⃣ HTTPException — 가장 기본적인 에러 [↑](#toc)

`HTTPException`은 FastAPI에서 HTTP 에러를 반환하는 가장 표준적인 방법입니다. Part 1부터 계속 사용해왔지만, 이번 장에서 더 깊이 다루겠습니다.

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

items_db = {1: {"name": "사과", "price": 1000}}


@app.get("/items/{item_id}")
def get_item(item_id: int):
    if item_id not in items_db:
        raise HTTPException(
            status_code=404,
            detail="아이템을 찾을 수 없습니다",
        )
    return items_db[item_id]
```

`GET /items/999` 요청 시 응답:

```json
HTTP/1.1 404 Not Found
Content-Type: application/json

{
  "detail": "아이템을 찾을 수 없습니다"
}
```

`detail`에는 문자열 외에도 딕셔너리나 리스트를 넣을 수 있습니다.

```python
raise HTTPException(
    status_code=404,
    detail={
        "message": "아이템을 찾을 수 없습니다",
        "item_id": item_id,
        "hint": "올바른 item_id를 입력했는지 확인해주세요",
    },
)
```

응답:

```json
{
  "detail": {
    "message": "아이템을 찾을 수 없습니다",
    "item_id": 999,
    "hint": "올바른 item_id를 입력했는지 확인해주세요"
  }
}
```

### 자주 쓰는 상태 코드 정리

| 상태 코드 | 의미 | 사용 예시 |
|-----------|------|-----------|
| `400` | Bad Request | 요청 형식이 잘못됨 |
| `401` | Unauthorized | 인증 필요 (로그인 안 됨) |
| `403` | Forbidden | 권한 없음 (로그인은 됐지만 접근 불가) |
| `404` | Not Found | 리소스를 찾을 수 없음 |
| `409` | Conflict | 이미 존재 (이메일 중복 등) |
| `422` | Unprocessable Entity | Pydantic 검증 실패 (자동 생성) |
| `500` | Internal Server Error | 서버 내부 오류 |

```python
# 자주 쓰는 패턴 모음
from fastapi import status

# 404 — 존재하지 않음
raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail="아이템 없음")

# 409 — 중복
raise HTTPException(status_code=status.HTTP_409_CONFLICT, detail="이미 존재하는 이메일입니다")

# 401 — 인증 필요
raise HTTPException(
    status_code=status.HTTP_401_UNAUTHORIZED,
    detail="로그인이 필요합니다",
    headers={"WWW-Authenticate": "Bearer"},
)
```

## 2️⃣ 커스텀 에러 응답 [↑](#toc)

`HTTPException`의 `headers` 파라미터로 응답 헤더를 추가할 수 있습니다.

```python
@app.get("/protected")
def protected_route(x_token: str = Header(...)):
    if x_token != "valid-token":
        raise HTTPException(
            status_code=401,
            detail="유효하지 않은 토큰입니다",
            headers={
                "WWW-Authenticate": "Bearer",
                "X-Error-Code": "INVALID_TOKEN",
            },
        )
    return {"message": "보호된 리소스"}
```

응답 헤더:

```
HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer
X-Error-Code: INVALID_TOKEN
Content-Type: application/json
```

### 에러 코드 구조화

프론트엔드와 협업할 때는 에러 응답에 **에러 코드**를 포함하면 편리합니다. 에러 코드가 있으면 프론트엔드가 언어별로 다른 메시지를 보여줄 수 있습니다.

```python
# ❌ 문자열만 있는 에러
raise HTTPException(status_code=404, detail="아이템을 찾을 수 없습니다")

# ✅ 에러 코드 포함
raise HTTPException(
    status_code=404,
    detail={
        "code": "ITEM_NOT_FOUND",
        "message": "아이템을 찾을 수 없습니다",
    },
)
```

## 3️⃣ 커스텀 예외 클래스 [↑](#toc)

프로젝트가 커지면 도메인별 예외 클래스를 만들고, FastAPI에 핸들러를 등록하는 방식이 더 깔끔합니다.

```python
# exceptions.py

class ItemNotFoundError(Exception):
    """아이템을 찾을 수 없을 때 발생하는 예외"""
    def __init__(self, item_id: int):
        self.item_id = item_id
        super().__init__(f"Item {item_id} not found")


class DuplicateEmailError(Exception):
    """이미 등록된 이메일을 사용하려 할 때 발생하는 예외"""
    def __init__(self, email: str):
        self.email = email
        super().__init__(f"Email {email} already exists")
```

```python
# main.py

from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from exceptions import ItemNotFoundError, DuplicateEmailError

app = FastAPI()


# ── 예외 핸들러 등록 ──────────────────────────────────
@app.exception_handler(ItemNotFoundError)
def item_not_found_handler(request: Request, exc: ItemNotFoundError):
    return JSONResponse(
        status_code=404,
        content={
            "code": "ITEM_NOT_FOUND",
            "message": f"아이템 {exc.item_id}을(를) 찾을 수 없습니다",
        },
    )


@app.exception_handler(DuplicateEmailError)
def duplicate_email_handler(request: Request, exc: DuplicateEmailError):
    return JSONResponse(
        status_code=409,
        content={
            "code": "DUPLICATE_EMAIL",
            "message": f"이메일 '{exc.email}'은(는) 이미 사용 중입니다",
        },
    )


# ── 엔드포인트에서 예외 발생 ──────────────────────────
items_db = {1: {"id": 1, "name": "사과"}}

@app.get("/items/{item_id}")
def get_item(item_id: int):
    if item_id not in items_db:
        raise ItemNotFoundError(item_id)   # ← HTTPException 아님
    return items_db[item_id]
```

실행 결과 (`GET /items/99`):

```json
HTTP/1.1 404 Not Found

{
  "code": "ITEM_NOT_FOUND",
  "message": "아이템 99을(를) 찾을 수 없습니다"
}
```

> 왜 커스텀 예외 클래스를 사용할까요?
> `raise ItemNotFoundError(item_id)` 는 비즈니스 로직 파일에서도 자연스럽게 읽힙니다.
> HTTP 상태 코드와 JSON 형식은 에러 핸들러에서만 다루면 됩니다.
> 비즈니스 로직이 HTTP에 대해 알 필요가 없어집니다.

## 4️⃣ Pydantic 검증 에러 커스터마이징 [↑](#toc)

FastAPI는 요청 본문이 Pydantic 검증에 실패하면 자동으로 `422 Unprocessable Entity`를 반환합니다. 기본 에러 메시지는 영어로 길게 나옵니다.

기본 422 에러 응답 예시:

```json
{
  "detail": [
    {
      "type": "missing",
      "loc": ["body", "name"],
      "msg": "Field required",
      "input": {},
      "url": "https://errors.pydantic.dev/2.9/v/missing"
    }
  ]
}
```

이것을 더 친절한 형태로 바꿔보겠습니다.

```python
from fastapi import FastAPI, Request, status
from fastapi.exceptions import RequestValidationError
from fastapi.responses import JSONResponse

app = FastAPI()


@app.exception_handler(RequestValidationError)
def validation_exception_handler(request: Request, exc: RequestValidationError):
    """Pydantic 검증 에러를 친절한 한국어 메시지로 변환합니다."""
    errors = []
    for error in exc.errors():
        # loc: ('body', 'name') → "name"
        field = " → ".join(str(loc) for loc in error["loc"] if loc != "body")
        errors.append({
            "field": field,
            "message": error["msg"],
            "input": error.get("input"),
        })

    return JSONResponse(
        status_code=status.HTTP_422_UNPROCESSABLE_ENTITY,
        content={
            "code": "VALIDATION_ERROR",
            "message": "입력값을 확인해주세요",
            "errors": errors,
        },
    )
```

커스터마이징 후 422 에러:

```json
{
  "code": "VALIDATION_ERROR",
  "message": "입력값을 확인해주세요",
  "errors": [
    {
      "field": "name",
      "message": "Field required",
      "input": {}
    }
  ]
}
```

## 5️⃣ 에러 응답 스키마 [↑](#toc)

Swagger UI에서 에러 응답도 문서화할 수 있습니다. `responses` 파라미터를 사용합니다.

```python
from pydantic import BaseModel
from fastapi import FastAPI, HTTPException

app = FastAPI()


# ── 에러 응답 스키마 정의 ─────────────────────────────
class ErrorDetail(BaseModel):
    code: str
    message: str


class NotFoundResponse(BaseModel):
    detail: ErrorDetail


# ── 엔드포인트에 응답 스키마 등록 ────────────────────
@app.get(
    "/items/{item_id}",
    responses={
        404: {
            "model": NotFoundResponse,
            "description": "아이템을 찾을 수 없습니다",
            "content": {
                "application/json": {
                    "example": {
                        "detail": {
                            "code": "ITEM_NOT_FOUND",
                            "message": "아이템 1을(를) 찾을 수 없습니다",
                        }
                    }
                }
            },
        },
        403: {"description": "권한이 없습니다"},
    },
)
def get_item(item_id: int):
    """특정 아이템을 조회합니다."""
    if item_id not in items_db:
        raise HTTPException(
            status_code=404,
            detail={"code": "ITEM_NOT_FOUND", "message": f"아이템 {item_id}을(를) 찾을 수 없습니다"},
        )
    return items_db[item_id]
```

`http://localhost:8000/docs`에서 `GET /items/{item_id}`를 열면 **Responses** 섹션에 200, 404, 403 응답이 모두 문서화됩니다.

> API를 사용하는 프론트엔드 개발자나 다른 팀원이 "어떤 에러가 올 수 있는지" Swagger에서 바로 볼 수 있습니다.
> 에러 케이스를 문서화하는 것은 좋은 API 설계의 일부입니다.

## 6️⃣ 전역 에러 핸들러 [↑](#toc)

예상하지 못한 에러(서버 버그, 외부 서비스 장애 등)가 발생하면 FastAPI는 기본적으로 500 에러를 반환합니다. 이때 에러 세부 정보가 클라이언트에 노출되면 보안 문제가 생깁니다.

```python
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
import logging

app = FastAPI()

# 로거 설정
logger = logging.getLogger(__name__)


@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    """예상치 못한 모든 예외를 처리합니다."""
    # 에러 세부 정보는 서버 로그에만 기록
    logger.error(
        f"Unexpected error: {exc}",
        exc_info=True,
        extra={
            "method": request.method,
            "url": str(request.url),
        },
    )

    # 클라이언트에는 일반적인 메시지만 반환
    return JSONResponse(
        status_code=500,
        content={
            "code": "INTERNAL_ERROR",
            "message": "서버 내부 오류가 발생했습니다. 잠시 후 다시 시도해주세요.",
        },
    )


# 테스트용: 의도적으로 에러 발생
@app.get("/crash")
def crash():
    raise RuntimeError("의도적인 서버 에러!")
```

`GET /crash` 요청 시:

클라이언트 응답:

```json
HTTP/1.1 500 Internal Server Error

{
  "code": "INTERNAL_ERROR",
  "message": "서버 내부 오류가 발생했습니다. 잠시 후 다시 시도해주세요."
}
```

서버 로그:

```
ERROR: Unexpected error: 의도적인 서버 에러!
Traceback (most recent call last):
  File "...
RuntimeError: 의도적인 서버 에러!
```

> 클라이언트에는 "무슨 일이 있었는지" 가 아니라 "어떻게 대응해야 하는지"를 알려줍니다.
> 스택 트레이스를 클라이언트에 노출하면 서버 내부 구조가 드러나서 보안 위험이 생깁니다.
> 에러의 원인 파악은 서버 로그에서 합니다.

## 7️⃣ 정리 [↑](#toc)

| 도구 | 사용 상황 |
|------|-----------|
| `HTTPException` | 단순 HTTP 에러. 빠르게 쓸 때 |
| 커스텀 예외 클래스 + `@exception_handler` | 도메인별 에러를 일관되게 처리할 때 |
| `RequestValidationError` 핸들러 | 422 에러 메시지를 커스터마이징할 때 |
| `responses={}` 파라미터 | Swagger에 에러 케이스를 문서화할 때 |
| 전역 `Exception` 핸들러 | 예상 밖 500 에러를 안전하게 감출 때 |

### 핸들러 등록 우선순위

FastAPI는 가장 구체적인 예외 핸들러를 먼저 실행합니다.

```
raise ItemNotFoundError()
     ↓
ItemNotFoundError 핸들러 있음? → YES → 실행
     ↓ (없으면)
Exception 핸들러 있음? → YES → 실행 (전역 핸들러)
     ↓ (없으면)
FastAPI 기본 500 응답
```
