# 07. Context Engineering 원리

## 목표

AI에게 "무엇을, 어떤 순서로" 전달해야 품질이 좋아지는지 구조적으로 이해하고,
바로 다음 시간에 진행할 TDD 실습에 사용할 컨텍스트 패킷을 직접 작성합니다.

이번 시간 완료 기준:

1. 작업 목표를 한 문장으로 정의합니다.
2. 참조 파일을 3~5개로 제한합니다.
3. 완료 기준과 제외 범위를 명시합니다.

---

## 배경 지식

### 1) Context Engineering이란 무엇인가

Context Engineering은 모델이 작업을 올바르게 수행하도록
요구사항, 코드, 제약, 완료 기준을 "적정량 + 적정순서"로 전달하는 설계 활동입니다.
핵심은 "많이 넣기"가 아니라 "정확히 넣기"입니다.

### 1-1) 컨텍스트 패킷이란 무엇인가

컨텍스트 패킷은 AI에게 작업을 맡길 때 필요한 맥락을 한 번에 묶어 전달하는 구조화된 입력입니다.
쉽게 말해, "무엇을, 어떤 근거로, 어디까지" 수행할지 명확히 전달하는 작업 브리프입니다.

일반적으로 아래 5가지로 구성합니다.

1. 작업 목표: 이번 요청에서 바꿀 내용을 한 문장으로 정의
2. 참조 파일: 근거가 되는 파일 3~5개
3. 제약 조건: 반드시 지켜야 할 규칙(코딩 규칙, 예외 처리, 테스트 방식 등)
4. 완료 기준: 작업 완료 여부를 검증할 수 있는 조건
5. 제외 범위: 이번 요청에서 하지 않을 항목

컨텍스트 패킷을 쓰면 다음 효과가 있습니다.

- 요청 범위가 불필요하게 확장되는 것을 줄일 수 있습니다.
- 테스트와 검증 기준이 명확해집니다.
- 같은 요청을 다시 실행해도 결과 편차를 줄일 수 있습니다.

간단한 예시는 아래와 같습니다.

```text
[작업 목표]
- TodoManager에 상태별 필터링 메서드를 추가합니다.

[참조 파일]
- docs/PRODUCT.md
- src/todo/manager.py
- tests/test_manager.py

[제약 조건]
- 공개 메서드에 한국어 docstring을 작성합니다.
- 타입 힌트를 유지합니다.

[완료 기준]
- 신규 테스트가 통과합니다.
- 기존 테스트에 회귀가 없습니다.

[제외 범위]
- CLI 변경은 포함하지 않습니다.
```

### 2) Context Engineering이 중요한 이유

- 같은 요청이라도 어떤 파일을 같이 주는지에 따라 결과가 달라집니다.
- 컨텍스트가 과하면 노이즈가 늘고, 부족하면 추측(환각)이 증가합니다.
- 팀 실무에서는 컨텍스트 표준화가 결과 편차를 줄입니다.

### 3) Copilot 컨텍스트는 어떻게 배치되는가 (실무 관점)

Copilot 내부 시스템 프롬프트 전문은 공개되지 않았지만,
사용자가 제어 가능한 컨텍스트 레이어는 실무적으로 구분해 설계할 수 있습니다.

| 레이어 | 예시 | 정의 위치 | 배치 시점 |
|------|------|-----------|-----------|
| 시스템 레이어(비공개) | 안전 정책, 제품 기본 지시문 | Copilot 서비스 내부 | 모든 요청에 항상 포함 |
| 공통 지침 | `.github/copilot-instructions.md`, `AGENTS.md` | 저장소 루트/.github | 대체로 모든 채팅 요청 |
| 조건부 지침 | `.github/instructions/*.instructions.md` + `applyTo` | `.github/instructions/` | 현재 파일/작업이 패턴과 맞을 때 |
| 실행형 템플릿 | `.github/prompts/*.prompt.md` | `.github/prompts/` | Chat에서 `/프롬프트` 실행 시 |
| 역할/도구 제약 | `.github/agents/*.agent.md` | `.github/agents/` | 해당 에이전트 모드 선택 시 |
| 사용자 입력 | Chat 메시지, `${selection}`, `#codebase` 등 | Chat 입력창 | 메시지 전송 시 |
| 세션 기록 | 같은 대화의 이전 메시지 | Chat 세션 | 연속 대화 동안 누적 |

핵심 포인트:

- 시스템 레이어는 수정 불가, 나머지 레이어는 팀이 설계 가능
- 같은 규칙을 여러 파일에 중복 배치하면 충돌/편차가 커짐
- 이번 시간의 컨텍스트 패킷은 "사용자 입력 레이어" 품질을 높이는 작업입니다.

### 4) Prompt Files와 Chat 입력은 어떻게 결합되는가

`*.prompt.md`를 실행하면 보통 아래 순서로 컨텍스트가 합쳐집니다.

1. `/프롬프트명` 선택
2. frontmatter 파싱 (`name`, `description`, `agent`, `tools`)
3. `${input:...}` 변수 입력값 수집
4. `${selection}`, `#codebase`, `#problems`, `#changes` 등 동적 컨텍스트 결합
5. 프롬프트 본문 + 사용자 후속 입력을 함께 모델에 전달

즉, Prompt Files는 "고정 템플릿", Chat 입력은 "현재 작업 의도"를 담당합니다.

### 5) 컨텍스트 우선순위를 설계하는 기본 원칙

실무에서는 아래처럼 역할을 분리하면 가장 안정적입니다.

1. 공통 규칙: `.github/copilot-instructions.md`
2. 파일별 세부 규칙: `.github/instructions/*.instructions.md`
3. 반복 작업 템플릿: `.github/prompts/*.prompt.md`
4. 작업 순간의 의도/범위: Chat 입력(컨텍스트 패킷)

주의:

- 안전/정책 레이어는 항상 최상위 제약으로 작동합니다.
- 한 규칙은 가능한 한 "한 위치"에만 둡니다.
- `applyTo: "**"`는 필요한 경우가 아니면 피합니다(컨텍스트 과적재 위험).

### 6) Memory는 어디에 저장되는가 (이 실습 환경 기준)

이 실습 에이전트 환경에서는 메모리를 다음 경로로 구분해 사용합니다.

| 메모리 종류 | 경로 | 용도 |
|------------|------|------|
| 사용자 메모리 | `/memories/` | 대화 간 유지되는 개인 선호/규칙 |
| 세션 메모리 | `/memories/session/` | 현재 대화에서만 쓰는 작업 메모 |
| 저장소 메모리 | `/memories/repo/` | 현재 저장소에만 의미 있는 규칙/사실 |

오해 방지:

| 표기 | 의미 | VS Code 탐색기에서 폴더로 보이나? |
|------|------|-------------------------------|
| `/memories/...` | 에이전트 전용 메모리 네임스페이스(논리 경로) | 아니오 |
| `.github/...` | 실제 워크스페이스 파일 경로 | 예 |

반면, 프로젝트 규칙 자체는 버전 관리되는 파일에 둬야 합니다.

- `.github/copilot-instructions.md`
- `.github/instructions/*.instructions.md`
- `.github/prompts/*.prompt.md`
- `.github/agents/*.agent.md`

즉, 메모리는 "작업 보조", `.github/` 파일은 "팀 규칙의 정식 소스"입니다.

확인 방법(실습 환경):

- `/memories/...`는 에이전트의 memory 조회 기능으로 확인합니다.
- `.github/...`는 VS Code 탐색기에서 실제 파일로 확인합니다.

메모리 조회 요청 예시(복붙용):

1. "사용자 메모리 목록을 보여줘."
2. "세션 메모리 파일 목록을 보여줘."
3. "저장소 메모리(`/memories/repo/`)에 어떤 항목이 있는지 보여줘."

### 7) 컨텍스트 설계 5원칙

1. 목표 우선: 작업 목적과 완료 기준을 먼저 고정합니다.
2. 최소 충분성: 정답에 필요한 파일만 선택합니다.
3. 우선순위: 최신 요구사항과 제약을 먼저 제공합니다.
4. 근거 연결: 중요한 결정은 참조 파일 경로와 함께 제시합니다.
5. 오염 방지: 오래된 규칙과 문서를 명시적으로 제외합니다.

### 8) 자주 쓰는 컨텍스트 소스

| 경로 | 용도 |
|------|------|
| `docs/PRODUCT.md` | 기능 요구사항 |
| `docs/ARCHITECTURE.md` | 구조/설계 원칙 |
| `docs/CONTRIBUTING.md` | 코드 스타일/협업 규칙 |
| `.github/copilot-instructions.md` | 프로젝트 공통 규칙 |
| `.github/instructions/` | 파일 유형별 세부 규칙 |
| `tests/` | 기대 동작(정답 조건) |
| `src/todo/` | 실제 구현 대상 |

### 9) 왜 이번 시간에는 별도 실습 파일이 없는가

이번 시간은 앞선 시간처럼 새로운 `.github/...` 설정 파일을 만드는 단계가 아닙니다.
핵심은 이미 있는 규칙, 프롬프트, 에이전트, 코드 파일을 어떤 순서와 구조로 Chat에 넣어야 결과 품질이 좋아지는지 실험해 보는 것입니다.

정리하면:

- 앞선 시간은 설정 파일을 만드는 시간입니다.
- 이번 시간은 그 설정과 참조 파일을 조합해 "컨텍스트 패킷"을 설계하는 시간입니다.
- 따라서 `starter/`, `solution/` 같은 별도 실습 파일 없이 README의 복붙용 패킷을 바로 Chat에 입력해 실습합니다.

권장 진행 방식:

- 가능하면 앞선 시간의 산출물이 이미 있는 상태에서 진행합니다.
- 하지만 산출물이 없더라도 `docs/`, `src/`, `tests/`를 참조해 컨텍스트 패킷 작성 연습은 가능합니다.
- 실습 산출물은 새 파일이 아니라, Chat에 입력할 컨텍스트 패킷 초안입니다.

## 실습 단계

아래 순서로 진행하면 Context Engineering의 효과를 직접 경험할 수 있습니다.

> 참고: 이번 시간은 별도 `.md` 실습 파일을 새로 만드는 단계가 아니라, Chat 입력 품질을 설계하는 단계입니다.

1. 컨텍스트 패킷 템플릿 이해
2. 실습용 컨텍스트 패킷 작성
3. 패킷 실행 및 구현 진행
4. 결과 점검 및 개선
5. 자주 발생하는 문제 해결 (선택)

### 0단계: 사전 준비

**① 참조 파일 확인**

- `docs/PRODUCT.md` — 기능 요구사항
- `docs/ARCHITECTURE.md` — 구조/설계 원칙
- `docs/CONTRIBUTING.md` — 코드 규칙
- `src/todo/manager.py` — 대상 코드
- `tests/test_manager.py` — 테스트 기준

**② Chat 준비**

1. Copilot Chat 창을 엽니다.
2. 새 대화 창을 시작합니다 (이전 대화와 별도로).
3. 기본 에이전트가 선택되어 있는지 확인합니다.

이번 실습은 **0단계에서 시작한 새 대화 1개를 끝까지 유지**합니다.
즉, 1단계부터 마지막 단계까지는 같은 대화 창에서 이어서 진행합니다.

**③ 앞선 시간 산출물 확인 (권장)**

앞선 시간을 순서대로 진행하셨다면 아래 항목이 이미 존재할 수 있습니다.

- `.github/copilot-instructions.md`
- `.github/instructions/*.instructions.md`
- `.github/prompts/*.prompt.md`
- `.github/agents/*.agent.md`

이 파일들이 있으면, 이번 시간에 설계하는 컨텍스트 패킷이 실제로 어떤 레이어 위에서 동작하는지 더 쉽게 체감하실 수 있습니다.

### 1단계: 컨텍스트 패킷 템플릿 이해

#### 1-1. 템플릿 구조 확인

아래 구조를 이해합니다. 이것이 컨텍스트를 구조화하는 기본 형식입니다.

`````text
[작업 목표]
- 변경할 내용을 한 문장으로 명확히 정의

[참조 파일]
- 핵심 파일 3~5개 (순서는 우선순위순)

[제약 조건]
- 반드시 지켜야 할 스타일/성능/보안 규칙

[완료 기준]
- 검증 가능한 테스트/동작 조건

[제외 범위]
- 명시적으로 "이번 작업에서 하지 않을 것들"
`````

#### 1-2. 템플릿 작성 기준 빠르게 체크

- 작업 목표는 한 문장으로 작성합니다.
- 참조 파일은 3~5개로 제한합니다.
- 제약 조건은 프로젝트 규칙 중심으로 3~5개를 작성합니다.
- 완료 기준은 검증 가능한 조건으로 3~4개를 작성합니다.
- 제외 범위는 하지 않을 작업을 명시적으로 작성합니다.

#### 1-3. 컨텍스트 패킷 사용 시 장점

컨텍스트 패킷을 사용하면 아래 장점을 바로 체감할 수 있습니다.

- 수정 범위를 작업 목표에 맞게 좁혀 과도한 변경을 줄일 수 있습니다.
- 테스트와 완료 기준이 명확해져 검증 누락 가능성이 줄어듭니다.
- 프로젝트 규칙(docstring, 타입 힌트, 예외 처리) 반영률이 높아집니다.
- 제외 범위를 명시해 스코프 확장을 통제하기 쉬워집니다.

### 2단계: 실습용 컨텍스트 패킷 작성

#### 2-1. 예시 패킷 (복붙용)

```text
[작업 목표]
- TodoManager에 상태(pending/done) 기준으로 태스크를 필터링하는 메서드를 추가합니다.

[참조 파일]
- docs/PRODUCT.md (기능 요구사항)
- docs/CONTRIBUTING.md (코드 스타일)
- src/todo/manager.py (대상 클래스)
- tests/test_manager.py (테스트 기준)

[제약 조건]
- 공개 메서드에는 한국어 docstring을 반드시 포함합니다.
- 모든 파라미터와 반환값에 타입 힌트를 명시합니다.
- 예외 처리는 ValueError를 먼저 사용합니다.

[완료 기준]
1. 새로운 테스트가 추가되고 모두 통과합니다.
2. 기존 테스트에 회귀가 없습니다.
3. pending/done 모두의 필터링 동작이 검증됩니다.
4. docs/CONTRIBUTING.md의 모든 규칙을 따릅니다.

[제외 범위]
- CLI 명령어 추가는 이번 작업에서 하지 않습니다.
- 데이터 저장 방식 변경은 하지 않습니다.
- UI와 시각화는 포함하지 않습니다.
```

#### 2-2. 각 섹션 작성 팁

**[작업 목표]**
- 한 문장, 구체적인 동사와 객체 포함
- "나쁜 예": 상태 필터링을 추가해주세요 (동작이 애매합니다)
- "좋은 예": pending/done 상태로 필터링하는 메서드를 추가합니다 (구체적입니다)

**[참조 파일]**
- 3~5개로 제한합니다 (너무 많으면 방해가 됩니다)
- 우선순위 순서: 요구사항 → 규칙 → 구현 대상 → 테스트
- 최신 파일을 우선으로

**[제약 조건]**
- 프로젝트의 기술 규칙과 코드 스타일을 명시합니다.
- `.github/copilot-instructions.md`나 `.github/instructions/*.instructions.md`에서 발췌 가능
- 3~5개 정도가 적당합니다.

**[완료 기준]**
- "검증 가능해야 함" (테스트 통과, 파일 확인 가능 등)
- "느낌으로만" 작성하면 효과가 떨어집니다.
- 최소 3~4개 항목

**[제외 범위]**
- 이번 작업과 헷갈릴 수 있는 관련 작업들을 명시합니다.
- 스코프 크리프(Scope Creep)를 방지합니다.

### 3단계: 패킷 실행 및 구현 진행

2단계에서 작성한 패킷을 **같은 대화 창**에 붙여 넣고 실행합니다.

```text
[작업 목표]
TodoManager에 상태(pending/done) 기준으로 태스크를 필터링하는 메서드를 추가합니다.

[참조 파일]
docs/PRODUCT.md (기능 요구사항)
docs/CONTRIBUTING.md (코드 스타일)
src/todo/manager.py (대상 클래스)
tests/test_manager.py (테스트 기준)

[제약 조건]
- 공개 메서드에는 한국어 docstring을 반드시 포함합니다.
- 모든 파라미터와 반환값에 타입 힌트를 명시합니다.
- 예외 처리는 ValueError를 먼저 사용합니다.

[완료 기준]
1. 새로운 테스트가 추가되고 모두 통과합니다.
2. 기존 테스트에 회귀가 없습니다.
3. pending/done 모두의 필터링 동작이 검증됩니다.
4. docs/CONTRIBUTING.md의 모든 규칙을 따릅니다.

[제외 범위]
- CLI 명령어 추가는 이번 작업에서 하지 않습니다.
- 데이터 저장 방식 변경은 하지 않습니다.
- UI와 시각화는 포함하지 않습니다.

위 컨텍스트를 기준으로 구현 계획을 먼저 제시해주세요.
```

1. 위 패킷을 그대로 복사합니다.
2. 0단계에서 연 대화 창에 붙여 넣고 실행합니다.
3. 계획이 제시되면 "구현해주세요"로 다음 단계를 진행합니다.

### 4단계: 결과 점검 및 개선

실행 결과를 아래 점검표로 확인합니다.

| 점검 항목 | 확인 포인트 |
|----------|-----------|
| 수정 범위 적절성 | 작업 목표와 직접 관련된 파일만 수정되었는가? |
| 테스트 충실도 | 완료 기준을 검증하는 테스트가 포함되었는가? |
| 규칙 준수 | docstring, 타입 힌트, 예외 처리 규칙을 지켰는가? |
| 제외 범위 준수 | 제외한 항목이 실제로 변경되지 않았는가? |
| 후속 조치 | 바로 마무리 가능한가, 추가 보강이 필요한가? |

**개선 포인트:**

- 수정 범위가 넓으면 `[참조 파일]`과 `[제외 범위]`를 더 구체화합니다.
- 테스트가 약하면 `[완료 기준]`에 테스트 항목을 명시적으로 추가합니다.
- 규칙 누락이 있으면 `[제약 조건]`에 규칙 문구를 직접 넣어 강화합니다.

### 5단계: 자주 발생하는 문제와 해결

#### 1) 패킷 사용 후에도 변화가 거의 없을 때

원인:

- 참조 파일 경로가 존재하지 않거나 내용이 비어 있음
- 제약 조건이 너무 일반적임 ("좋은 코드 작성" 같은 추상적 표현)
- 완료 기준이 검증 불가능한 형태

해결:

1. 참조 파일이 실제로 존재하고 내용이 있는지 확인합니다.
2. 제약 조건을 구체적으로 다시 작성합니다.
   ```
   나쁜 예: "좋은 코드 작성"
   좋은 예: "모든 공개 메서드에 한국어 docstring 포함, 타입 힌트 필수"
   ```
3. 완료 기준에 "테스트 통과", "파일 변경 확인" 등 검증 가능한 표현을 추가합니다.

#### 2) 패킷의 제외 범위를 무시하고 과도하게 수정할 때

원인:

- 제외 범위를 명시했지만 본문 요청에서 명확하지 않음
- 컨텍스트 패킷 이후에 추가 요청을 할 때 제외 범위를 다시 언급하지 않음

해결:

1. 패킷 맨 끝에 제외 범위를 다시 명시합니다.
   ```
   위 제외 범위를 반드시 지키고, 이 범위 밖의 파일이나 기능은 수정하지 마세요.
   ```
2. 실제 요청할 때도 한 줄을 추가합니다.
   ```
   ... (위 패킷 내용) ...
   
   다시 강조: CLI 추가, 저장 방식 변경은 절대 하지 마세요.
   ```

#### 3) 참조 파일을 너무 많이 추가했을 때

원인:

- "혹시 도움될까봐" 파일을 계속 추가
- 참조 파일이 6개 이상이 됨

해결:

1. 핵심 파일만 3~5개로 제한합니다.
2. 각 파일이 정말 필요한지 다시 확인합니다.
   - 요구사항: `docs/PRODUCT.md`
   - 기술 규칙: `docs/CONTRIBUTING.md`
   - 구현 대상: `src/todo/manager.py`
   - 테스트 예시: `tests/test_manager.py`
   - 선택: 추가 설계 문서 1개

#### 4) 완료 기준이 너무 모호할 때

원인:

- "좋은 결과", "충분한 테스트" 같은 추상적 표현 사용
- 검증 방법이 명확하지 않음

해결:

구체적으로 다시 작성:

```
나쁜 예:
- 충분한 테스트가 추가됩니다.
- 모든 예외 케이스가 처리됩니다.

좋은 예:
- test_filter_by_pending, test_filter_by_done, test_empty_state 3개의 테스트가 추가되고 통과합니다.
- 입력 상태 값이 잘못되었을 때 ValueError를 발생시킵니다 (예: invalid_status).
- 기존 20개의 테스트가 모두 여전히 통과합니다.
```
