# 05장. Custom Agents와 Context Engineering

<a id="part1"></a>

## 1️⃣ Custom Agent란? [↑](#toc)

### 전문 컨설턴트 고용 비유

> 마케팅 전문가에게 코딩을 시키지 않듯,
> AI에게도 **역할과 권한을 명확히 지정**하면 더 정확한 결과를 얻습니다.

04장에서 만든 Prompt Files는 "특정 요청 한 번을 잘 처리하는 양식"이었습니다. Custom Agent는 다릅니다. "Planner입니다"라는 정체성이 대화 전체에 걸쳐 유지되는, 역할을 가진 전용 AI 모드입니다.

### Prompt Files vs Custom Agents

| 비교 항목 | Prompt Files | Custom Agents |
|----------|-------------|---------------|
| 도구 제한 유지 범위 | 해당 프롬프트 **1회 실행 동안만** | 에이전트 선택 후 **대화 전체** |
| 역할(페르소나) | 없음 — 요청 양식일 뿐 | 이름·역할·제약이 있는 **고정 정체성** |
| 이어지는 추가 질문 | 제한 없이 일반 Copilot으로 돌아감 | 여전히 에이전트 역할 안에서 답변 |
| 실행 방법 | `/프롬프트명` 명령 | Chat 드롭다운에서 에이전트 선택 |
| 위치 | `.github/prompts/*.prompt.md` | `.github/agents/*.agent.md` |

실제 차이를 코드로 비교하면 이렇습니다.

```text
# Prompt File 방식 (1회성)
/new-feature "로그인 기능"
→ 이 프롬프트 실행 중에만 tools 제한 적용
→ 다음 질문부터는 다시 일반 Copilot

# Custom Agent 방식 (지속)
[Planner 에이전트 선택]
"로그인 기능 구현 계획 잡아줘"
"그런데 인증 토큰은 어디에 저장하면 좋을까?"
"테스트도 어떻게 나눠야 할까?"
→ 세 질문 모두 Planner 역할과 도구 제한 안에서 일관되게 답변
```

### 앞선 장과 Custom Agents의 위치

| 구분 | 목적 | 실행 시점 | 대표 파일 |
|------|------|----------|-----------|
| Custom Instructions | 항상 적용되는 규칙 제공 | 자동 적용 | `copilot-instructions.md`, `*.instructions.md` |
| Prompt Files | 반복 요청 템플릿 재사용 | `/프롬프트명` 실행 시 | `*.prompt.md` |
| Custom Agents | 역할 + 도구 + 제약을 가진 전용 모드 | Agent 선택 시 | `*.agent.md` |

정리하면, Instructions는 "규칙", Prompt Files는 "요청 템플릿", Custom Agents는 "역할을 가진 전용 AI"입니다.

## 2️⃣ Agent 파일 구조 [↑](#toc)

에이전트 파일은 frontmatter + 본문으로 구성합니다.

```yaml
---
description: "언제 이 에이전트를 사용하는지 설명 (한글)"
name: "Planner"
tools: [read, search]
---

당신은 ... 전문가입니다.

## 역할
...

## 제약
...

## 출력 형식
...
```

### 핵심 필드 설명

| 필드 | 필수 | 설명 |
|------|------|------|
| `description` | 필수 | Chat에서 이 에이전트를 언제 써야 하는지 설명. 한글로 작성 |
| `name` | 필수 | Chat 드롭다운에서 보이는 에이전트 이름. **영문**으로 작성 |
| `tools` | 필수 | 에이전트가 사용할 수 있는 도구 목록. 없는 도구는 사용 불가 |
| `handoffs` | 선택 | 다음 에이전트로 자동 전환할 때 사용 (06장에서 다룸) |

### 도구 별칭(Tool Aliases)

| 별칭 | 용도 |
|------|------|
| `read` | 파일 읽기 |
| `search` | 파일/텍스트 검색 |
| `edit` | 파일 편집 |
| `execute` | 터미널 명령 실행 |
| `web` | 웹 검색/URL 조회 |

### tools 제한이 핵심인 이유

`tools`에 `edit`이 없으면 에이전트는 코드를 수정할 수 없습니다. 이것이 역할 제한의 핵심입니다.

Planner의 목적은 "코드 작성"이 아니라 "구현 계획 수립"입니다. 도구 제한이 없으면 이런 문제가 자주 발생합니다.

1. 계획 대신 바로 코드 수정안을 제시함
2. 분석 없이 구현부터 시작함
3. 검증 항목(테스트 시나리오, 영향 범위)이 누락됨

## 3️⃣ 실습: Planner 에이전트 만들기 [↑](#toc)

읽기 전용 계획 전문가 에이전트를 만듭니다.

### 1단계: 폴더 생성

프로젝트 루트에서 `.github/agents/` 폴더를 만듭니다.

```text
.github/
└── agents/     ← 이 폴더를 만듭니다
```

VS Code 탐색기에서 `.github` 폴더를 우클릭 → "New Folder" → `agents` 입력.

### 2단계: 파일 생성

`.github/agents/planner.agent.md` 파일을 만듭니다.

### 3단계: 아래 내용을 그대로 붙여 넣기

```md
---
description: "새 기능이나 변경 사항의 구현 계획이 필요할 때 사용한다. 코드를 작성하지 않고 단계별 구현 계획을 생성한다."
name: "Planner"
tools: [read, search]
---

당신은 소프트웨어 구현 계획 전문가입니다.

## 역할

- 주어진 기능 요구사항을 분석하여 단계별 구현 계획을 수립한다.
- 기존 코드베이스(`src/todo/`)를 파악하고 수정이 필요한 파일과 범위를 제시한다.
- 예상되는 테스트 케이스 목록을 작성한다.
- 구현 순서와 의존 관계를 명시한다.

## 제약

- 요청 없이는 코드를 직접 작성하거나 파일을 수정하지 않는다.
- 계획 수립에만 집중한다.

## 출력 형식

계획은 다음 구조의 마크다운으로 작성한다:

## 구현 계획: [기능명]

### 분석
- 영향받는 파일:
- 추가/수정할 메서드:

### 구현 단계
- [ ] 단계 1
- [ ] 단계 2

### 예상 테스트 케이스
- test_...
```


### 4단계: Chat에서 Planner 선택

1. VS Code에서 Copilot Chat 창을 엽니다.
2. Chat 입력창 위쪽 드롭다운(기본값: "Ask")을 클릭합니다.
3. 목록에서 `Planner`를 선택합니다.

> 🤔 **Planner가 드롭다운에 보이지 않으면?**
> - 파일 경로가 `.github/agents/planner.agent.md`인지 확인하세요.
> - frontmatter의 시작/끝 `---`이 모두 있는지 확인하세요.
> - `name: "Planner"` 필드가 있는지 확인하세요.
> - Chat 창을 닫았다가 다시 열어보세요.

### 5단계: 테스트 실행

아래 요청을 입력합니다.

```text
TodoManager에 "우선순위(priority)" 필드를 추가하는 기능을 구현해야 해.
코드 수정은 하지 말고, 영향 파일 분석과 단계별 구현 계획, 예상 테스트 케이스를 제시해줘.
```

### 기대 결과 확인

| 확인 항목 | 기대 결과 | 실패 신호 |
|----------|----------|----------|
| 출력 성격 | 계획 중심 응답 | 바로 코드 패치 제안 |
| 분석 내용 | 영향 파일, 구현 단계, 테스트 시나리오 포함 | 추상적 조언만 존재 |
| 제약 준수 | 코드 직접 수정 없이 계획만 제시 | "수정했습니다" 같은 실행 흐름 |
| 범위 통제 | 요청한 기능 범위 안에서 단계 제시 | 범위 밖 기능까지 확장 |

> 🤔 **Planner가 코드를 직접 수정하면?**
> `tools`에 `edit`이 포함되어 있지 않은지 확인하세요. `tools: [read, search]`만 있어야 합니다. 파일을 수정하고 Chat 창을 재시작하세요.

결과가 짧거나 모호하면 아래 재요청 문장을 사용합니다.

```text
코드 작성 없이 구현 계획만 다시 제시해줘.
아래 항목을 반드시 포함해줘:
1) 영향받는 파일
2) 단계별 구현 순서
3) 테스트 시나리오(정상/예외)
```

## 4️⃣ 여기까지 되셨나요? [↑](#toc)

Planner 에이전트 체크리스트입니다.

- [ ] `.github/agents/planner.agent.md` 파일이 생성되었다
- [ ] `description`을 한글로 작성했다
- [ ] `name`을 영문(`Planner`)으로 작성했다
- [ ] `tools`를 `[read, search]`로 제한했다
- [ ] Chat 드롭다운에서 `Planner`가 선택 가능하다
- [ ] Planner 결과가 계획(분석 + 구현 단계 + 테스트 케이스) 중심으로 출력된다
- [ ] "수정했습니다" 같은 실행형 응답이 나오지 않는다

## 5️⃣ Context Engineering이란? [↑](#toc)

### 업무 지시서 작성법 비유

> 신입 직원에게 "알아서 해"라고 하면 엉뚱한 결과가 나옵니다.
> "목표, 참고 자료, 제약 조건, 완료 기준"을 명확히 적으면 정확한 결과가 나옵니다.

Context Engineering은 AI가 작업을 올바르게 수행하도록 요구사항, 코드, 제약, 완료 기준을 **적정량 + 적정순서**로 전달하는 설계 활동입니다.

핵심은 "많이 넣기"가 아니라 "정확히 넣기"입니다.

### 컨텍스트가 많으면 왜 문제가 될까?

AI 모델은 입력 길이에 제한이 있을 뿐 아니라, 관련 없는 정보가 많을수록 핵심을 찾아내는 정확도가 떨어집니다.

| 컨텍스트 상태 | 결과 |
|-------------|------|
| 너무 적음 | AI가 추측(환각)을 시작함 |
| 적정량 | 의도한 결과가 안정적으로 출력됨 |
| 너무 많음 | 노이즈가 증가하고 핵심 지시문이 묻힘 |

### Copilot 컨텍스트 레이어

사용자가 제어 가능한 컨텍스트 레이어는 아래와 같이 설계할 수 있습니다.

| 레이어 | 정의 위치 | 배치 시점 |
|------|-----------|-----------|
| 공통 지침 | `.github/copilot-instructions.md` | 대부분의 채팅 요청 |
| 조건부 지침 | `.github/instructions/*.instructions.md` | 현재 파일이 패턴과 맞을 때 |
| 실행형 템플릿 | `.github/prompts/*.prompt.md` | `/프롬프트` 실행 시 |
| 역할/도구 제약 | `.github/agents/*.agent.md` | 에이전트 선택 시 |
| **사용자 입력** | Chat 메시지 | 메시지 전송 시 |

이번 장에서 배우는 컨텍스트 패킷은 **사용자 입력 레이어**의 품질을 높이는 작업입니다.

## 6️⃣ 컨텍스트 패킷 5가지 요소 [↑](#toc)

컨텍스트 패킷은 AI에게 작업을 맡길 때 필요한 맥락을 한 번에 묶어 전달하는 구조화된 입력입니다. "무엇을, 어떤 근거로, 어디까지" 수행할지 명확히 전달하는 작업 브리프입니다.

```text
[작업 목표]
- 변경할 내용을 한 문장으로 명확히 정의

[참조 파일]
- 핵심 파일 3~5개 (순서는 우선순위순)

[제약 조건]
- 반드시 지켜야 할 스타일/성능/보안 규칙

[완료 기준]
- 검증 가능한 테스트/동작 조건

[제외 범위]
- 명시적으로 "이번 작업에서 하지 않을 것들"
```

### 각 섹션 작성 기준 (Todo Manager 예시)

**[작업 목표]** — 한 문장, 구체적인 동사와 객체 포함

```text
나쁜 예: 상태 필터링을 추가해주세요
좋은 예: TodoManager에 상태(pending/done) 기준으로 태스크를 필터링하는 메서드를 추가합니다
```

**[참조 파일]** — 3~5개로 제한, 우선순위 순서로

```text
- docs/PRODUCT.md (기능 요구사항)
- docs/CONTRIBUTING.md (코드 스타일)
- src/todo/manager.py (대상 클래스)
- tests/test_manager.py (테스트 기준)
```

| 경로 | 용도 |
|------|------|
| `docs/PRODUCT.md` | 기능 요구사항 |
| `docs/CONTRIBUTING.md` | 코드 스타일/협업 규칙 |
| `.github/copilot-instructions.md` | 프로젝트 공통 규칙 |
| `src/todo/` | 실제 구현 대상 |
| `tests/` | 기대 동작(정답 조건) |

**[제약 조건]** — 프로젝트 기술 규칙과 코드 스타일

```text
- 공개 메서드에는 한국어 docstring을 반드시 포함합니다.
- 모든 파라미터와 반환값에 타입 힌트를 명시합니다.
- 예외 처리는 ValueError를 먼저 사용합니다.
```

**[완료 기준]** — "검증 가능"해야 합니다

```text
나쁜 예: 충분한 테스트가 추가됩니다.
좋은 예: test_filter_by_pending, test_filter_by_done 테스트가 추가되고 통과합니다.
```

**[제외 범위]** — 헷갈릴 수 있는 관련 작업들을 명시

```text
- CLI 명령어 추가는 이번 작업에서 하지 않습니다.
- 데이터 저장 방식 변경은 하지 않습니다.
- UI와 시각화는 포함하지 않습니다.
```

### 컨텍스트 설계 5원칙

1. **목표 우선**: 작업 목적과 완료 기준을 먼저 고정합니다.
2. **최소 충분성**: 정답에 필요한 파일만 선택합니다 (3~5개).
3. **우선순위**: 최신 요구사항과 제약을 먼저 제공합니다.
4. **근거 연결**: 중요한 결정은 참조 파일 경로와 함께 제시합니다.
5. **오염 방지**: 오래된 규칙과 문서를 명시적으로 제외합니다.

## 7️⃣ 실습: 컨텍스트 패킷 작성 + 실행 [↑](#toc)

"TodoManager에 상태별 필터링 기능 추가"를 위한 패킷을 작성하고, 패킷 없이 요청했을 때와 비교합니다.

### 1단계: 패킷 없이 요청해보기 (비교용)

Copilot Chat에서 아래만 입력합니다.

```text
TodoManager에 상태별 필터링 기능을 추가해줘.
```

결과를 메모해둡니다. (범위가 예상보다 넓거나, 코딩 규칙이 반영되지 않을 수 있습니다.)

### 2단계: 컨텍스트 패킷 작성

새 대화 창을 열고 아래 패킷을 그대로 붙여 넣습니다.

```text
[작업 목표]
- TodoManager에 상태(pending/done) 기준으로 태스크를 필터링하는 메서드를 추가합니다.

[참조 파일]
- docs/PRODUCT.md (기능 요구사항)
- docs/CONTRIBUTING.md (코드 스타일)
- src/todo/manager.py (대상 클래스)
- tests/test_manager.py (테스트 기준)

[제약 조건]
- 공개 메서드에는 한국어 docstring을 반드시 포함합니다.
- 모든 파라미터와 반환값에 타입 힌트를 명시합니다.
- 예외 처리는 ValueError를 먼저 사용합니다.

[완료 기준]
1. 새로운 테스트가 추가되고 모두 통과합니다.
2. 기존 테스트에 회귀가 없습니다.
3. pending/done 모두의 필터링 동작이 검증됩니다.
4. docs/CONTRIBUTING.md의 모든 규칙을 따릅니다.

[제외 범위]
- CLI 명령어 추가는 이번 작업에서 하지 않습니다.
- 데이터 저장 방식 변경은 하지 않습니다.
- UI와 시각화는 포함하지 않습니다.

위 컨텍스트를 기준으로 구현 계획을 먼저 제시해주세요.
```

### 3단계: 결과 점검

| 점검 항목 | 확인 포인트 |
|----------|-----------|
| 수정 범위 적절성 | 작업 목표와 직접 관련된 파일만 수정되었는가? |
| 테스트 충실도 | 완료 기준을 검증하는 테스트가 포함되었는가? |
| 규칙 준수 | docstring, 타입 힌트, 예외 처리 규칙을 지켰는가? |
| 제외 범위 준수 | 제외한 항목이 실제로 변경되지 않았는가? |

> 🤔 **패킷을 사용했는데도 결과 변화가 거의 없으면?**
> - 참조 파일 경로가 실제 존재하는지 확인하세요.
> - 제약 조건이 너무 추상적이면 구체화하세요: "좋은 코드 작성" → "모든 공개 메서드에 한국어 docstring 포함"
> - 완료 기준에 테스트 함수명을 명시적으로 적어보세요.

> 🤔 **제외 범위를 무시하고 과도하게 수정하면?**
> 패킷 맨 끝에 한 줄을 추가합니다: "위 제외 범위를 반드시 지키고, 이 범위 밖의 파일이나 기능은 수정하지 마세요."
