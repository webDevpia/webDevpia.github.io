## 3. 다중 페르소나 분석

### 3.1 Dr. 이론 — 교육공학 연구자의 분석

#### 핵심 주장: "시퀀싱이 전부다"

학습 과학의 관점에서 AI-Native JavaScript 과정의 성패는 **AI 도입 시점과 사용 모드의 설계**에 달려 있다.

**근거 1: 인지 부하 이론 (Cognitive Load Theory)**

JavaScript를 처음 배우는 학습자에게는 세 가지 인지 부하가 동시에 작용한다:

```
내재적 부하: JavaScript 문법과 개념 자체의 복잡성
외재적 부하: AI 도구의 인터페이스, 프롬프트 작성법, 출력 해석
본유적 부하: 메타인지 — "AI가 맞는 답을 줬는가?" 판단
```

초보자에게 세 부하를 동시에 부과하면 **인지 과부하**가 발생한다. 따라서:

- **1단계 (주 1-3)**: AI는 **설명 모드(Explanation Mode)**로만 사용한다. 학습자가 직접 코드를 작성하되, 이해가 안 되는 부분을 AI에 질문한다. 이는 Anthropic 연구에서 가장 높은 학습 효과(65%+)를 보인 패턴이다.
- **2단계 (주 4-6)**: AI가 생성한 코드를 **읽고 수정하는** 훈련. "AI 출력 → 학습자 검증 → 수정"의 루프.
- **3단계 (주 7+)**: 학습자가 의도를 기술하고 AI가 생성하는 **완전한 AI 페어 프로그래밍** 모드.

**근거 2: 메타인지 연구**

ICER 2025 연구에서 상위 성과 학생은 Copilot을 **선택적으로** 사용(세밀한 인라인 제안만 수용)한 반면, 하위 성과 학생은 코드 블록을 **통째로 수용**했다. 이는 메타인지 능력이 AI 활용 효과를 결정한다는 것을 의미한다.

→ **교육적 함의**: 메타인지를 명시적으로 가르쳐야 한다. "AI가 준 코드를 설명해 보세요(Explain It Back)" 프로토콜이 이에 해당한다.

**근거 3: 전이 학습 (Transfer Learning)**

AI가 생성한 코드를 수동적으로 받아들이는 학습은 **표면적 성과**(과제 완료율, 점수)를 높이지만, **전이 능력**(새로운 문제에 학습한 개념을 적용하는 능력)은 향상시키지 못한다는 일관된 패턴이 35개 연구의 메타분석에서 확인된다.

#### 평가

| 항목 | 판정 |
|------|------|
| AI-Native JS 과정의 타당성 | **조건부 긍정** — 시퀀싱을 엄격히 설계해야 함 |
| 최대 위험 | 초보자에게 AI를 너무 일찍 코드 생성용으로 도입하면 기초 역량 형성 실패 |
| 핵심 설계 원칙 | "AI 사용 모드의 점진적 개방 (Explanation → Review → Generation)" |

### 3.2 박 시니어 — 현업 시니어 개발자의 분석

#### 핵심 주장: "2026년 현업이 요구하는 건 '둘 다'다"

채용 시장의 현실을 보면, "JS 기초만 아는 개발자"도, "AI만 잘 쓰는 개발자"도 원하지 않는다. 둘 다 할 줄 아는 사람을 원한다.

**현업 관점에서의 기존 JS 과정 평가**

```
[강점]
✓ 비유와 은유를 통한 개념 설명 (한국어 학습자에게 효과적)
✓ 기초→심화 선형 구조가 탄탄
✓ ToDo앱, 날씨앱 등 실전 프로젝트 포함

[치명적 결함]
✗ 테스트가 전혀 없다 — 현업에서 테스트 없는 코드는 코드가 아니다
✗ Git/버전관리 없음 — 협업 불가능
✗ 코드 리뷰 개념 없음
✗ AI 도구 언급 자체가 없음 — 2026년 기준으로 시대착오적
```

**현업 관점에서의 AI-Native 과정 평가**

```
[강점]
✓ TDD + AI 에이전트 워크플로우가 현업 프로세스와 일치
✓ Custom Instructions, Prompt Files — 팀 단위 AI 표준화 방법
✓ Context Engineering — 실무에서 AI 품질을 결정하는 핵심 역량

[한계]
✗ Python 전용 — JS 생태계에 적용하려면 전면 재작성 필요
✗ "이미 프로그래밍을 할 줄 아는 사람" 전제 — 초보자 적용 불가
✗ AI 출력이 틀렸을 때 어떻게 대응하는지 다루지 않음
```

**"현업이 원하는 AI-Native JS 개발자" 프로필**

1. `for` 루프, 함수, 클로저를 **AI 없이** 설명하고 작성할 수 있다
2. AI가 생성한 코드의 **버그를 찾고 수정**할 수 있다
3. JavaScript 프로젝트용 **Custom Instructions**를 작성할 수 있다
4. 테스트를 **먼저 작성**하고 AI에게 구현을 맡길 수 있다
5. AI가 생성한 코드의 **보안 취약점**을 인지한다

**실무에서 TDD + AI가 작동하는 이유**

> "TDD를 인간에게 고통스러운 작업으로 만드는 모든 것이, AI 에이전트에게는 완벽한 워크플로우를 만든다. AI는 명확하고 측정 가능한 목표에서 번성하며, 이진적인 테스트는 줄 수 있는 가장 명확한 목표 중 하나다."
> — codemanship.wordpress.com, 2026

Kent Beck도 AI 시대에서 TDD의 부활을 공개적으로 논의했으며, TDD + AI를 사용하는 팀은 결함 밀도 40-90% 감소를 보고했다.

#### 평가

| 항목 | 판정 |
|------|------|
| AI-Native JS 과정의 필요성 | **강력 긍정** — 현업 수요와 정확히 일치 |
| 기존 JS 과정의 현업 적합도 | **미흡** — 테스트, Git, AI 리터러시 전면 부재 |
| 핵심 추가 요소 | 테스트 프레임워크(Vitest), AI 출력 검증 훈련, 보안 인식 |

### 3.3 김 비평가 — 교육 비평가의 분석

#### 핵심 주장: "AI-Native 교육의 장밋빛 서사를 경계하라"

연구 데이터를 자세히 보면, AI 교육의 효과에 대한 낙관론에는 심각한 **방법론적 한계**와 **구조적 위험**이 존재한다.

**경고 1: 메타분석 효과 크기의 과대평가 가능성**

- Hedges' g = 1.10 (Tlili et al.)은 **비정상적으로 큰 효과 크기**다. 교육 연구에서 g > 0.8은 의심해야 한다.
- 원인: 소규모 연구, 비맹검 설계, 신기효과(novelty effect), 출판 편향(긍정 결과만 출판)
- ICER 2025 Copilot 연구의 n=10은 **통계적으로 의미 있는 결론을 내리기에 불충분**하다
- 대부분의 연구가 **단기 실험**(몇 주)이며, **장기 효과**(수년 후 역량)는 전혀 연구되지 않았다

**경고 2: "인지 위축(Cognitive Atrophy)" 증거**

| 연구/기관 | 발견 |
|----------|------|
| Microsoft + CMU (2025) | AI 도구 의존도↑ → 비판적 사고↓, 필요할 때 해당 능력 소환 더 어려워짐 |
| MIT Media Lab | AI 과의존 → "인지 위축"과 비판적 사고 능력 축소에 기여 |
| Anthropic RCT (2026) | 코드 생성 위임 시 이해도 테스트 **40% 미만** |
| Psychology Today (2026.3) | 성인은 기존 기술을 *잃고*, 아이들은 기술을 *처음부터 형성하지 못함* |

**경고 3: AI-Native 교육의 "세탁된 용어" 문제**

"AI-Native"라는 용어 자체가 마케팅 용어로 남용되고 있다:

- LEARN Academy가 "세계 최초 AI-Native 코딩 부트캠프"를 자칭하지만, **학습 성과 데이터가 공개되지 않았다**
- "AI-Native"의 실체적 정의가 기관마다 다르다 — 단순히 ChatGPT를 수업에 허용하는 것부터 전면적 커리큘럼 재설계까지
- Inside Higher Ed (2025.10): *"에드테크 기업들이 교육자 전문성보다 AI를 우선시하고 있다"* — 시장 주도의 교육학 왜곡

**경고 4: JavaScript 초보자에 대한 미검증 가정**

| 가정 | 검증 상태 | 위험 |
|------|----------|------|
| "초보자가 JS와 AI 워크플로우를 동시에 배울 수 있다" | **미검증** | 인지 과부하로 양쪽 모두 실패 가능 |
| "AI가 초보자 수준 JS 문제에 정확한 코드를 생성한다" | **부분 검증** | 고급 패턴(삼항 연쇄, 복잡한 구조분해) 사용으로 초보자 혼란 가능 |
| "학습자가 AI 출력을 맹목적으로 수용하지 않고 실제로 검증할 것이다" | **반증됨** | ICER 2025에서 하위 학생들이 코드 블록을 통째로 수용 |
| "AI 도구가 항상 사용 가능하다" | **불안정** | 구독 만료, 요금 제한, 서비스 장애 시 학습 중단 |

**경고 5: 비결정적 출력의 교육적 난제**

기존 JS 과정은 "예상 출력"을 보여줄 수 있다:
```javascript
console.log(sum(3, 5)); // 8
```

AI-Native 과정에서는 같은 프롬프트에 대해 AI가 **매번 다른 코드**를 생성한다. 이는:
- 교재의 "정답" 제시가 불가능
- 채점 기준 설계가 극도로 어려움
- "내 결과가 교재와 다른데 맞는 건가?"라는 학습자 불안 유발

#### 평가

| 항목 | 판정 |
|------|------|
| AI-Native JS 과정의 타당성 | **조건부 경고** — 위험 완화 장치 없이 진행하면 역효과 |
| 가장 우려되는 시나리오 | 기초 없이 AI 의존 → "코드를 읽을 수 없는 개발자" 양산 |
| 필수 완화 조치 | 수동 코딩 최소 기준, 버그 주입 연습, AI 없는 평가 |

### 3.4 정 설계자 — 커리큘럼 설계 전문가의 분석

#### 핵심 주장: "3단계 아크 설계가 모든 우려를 해소한다"

앞선 세 전문가의 분석을 종합하면, 핵심 쟁점은 **"AI를 언제, 어떤 모드로 도입하는가"**이다. 이를 해결하는 구조적 프레임워크를 제안한다.

**3단계 아크 (Three-Phase Arc)**

```
┌─────────────────────────────────────────────────────────┐
│  Phase 1: 기초 + AI 설명자    (주 1-4)                    │
│  "AI는 선생님이다 — 코드는 내가 쓴다"                       │
│  AI 모드: 설명(Explanation) 전용                          │
│  수동 코딩 비율: 90%                                      │
├─────────────────────────────────────────────────────────┤
│  Phase 2: 검증자 + AI 리뷰어  (주 5-8)                    │
│  "AI가 쓴 코드를 내가 검증한다"                             │
│  AI 모드: 생성(Generation) + 학습자 검증                   │
│  수동 코딩 비율: 50%                                      │
├─────────────────────────────────────────────────────────┤
│  Phase 3: AI-Native 개발      (주 9-12)                   │
│  "내가 설계하고 AI가 구현한다"                              │
│  AI 모드: 완전 페어 프로그래밍                              │
│  수동 코딩 비율: 20%                                      │
└─────────────────────────────────────────────────────────┘
```

**Phase 1: 기초 + AI 설명자 (주 1-4)**

| 주차 | 주제 | AI 사용 방식 | 기존 JS 과정 대응 |
|------|------|-------------|------------------|
| 1 | 환경 구축 + 변수/타입 | AI에게 개념 질문만 허용 | Ch01-02 압축 |
| 2 | 조건문 + 반복문 + 배열 | AI에게 "이 코드를 설명해줘" 질문 | Ch03-04 압축 |
| 3 | 함수 + 스코프 + 클로저 | AI 설명 + "Explain It Back" 연습 | Ch05-06 유지 |
| 4 | 객체 + ES6+ + 테스트 입문 | AI 설명 + Vitest 첫 테스트 작성 | Ch07-08 + 신규 |

**설계 근거**: Anthropic 연구에서 AI를 개념 질의용으로 사용한 그룹이 65%+ 이해도를 달성. 이 단계에서는 학습자가 모든 코드를 직접 작성하되, 막히는 부분을 AI에 질문하는 패턴.

**Phase 1 필수 관문(Gate)**: *학습자가 AI 없이 for 루프, 함수, 이벤트 핸들러를 종이/화이트보드에 작성할 수 있어야 Phase 2로 진행*

---

**Phase 2: 검증자 + AI 리뷰어 (주 5-8)**

| 주차 | 주제 | AI 사용 방식 | 기존 과정 대응 |
|------|------|-------------|--------------|
| 5 | DOM + 이벤트 | AI가 생성한 DOM 코드를 읽고 수정 | Ch09 재구성 |
| 6 | AI 출력 평가법 | 의도적 버그 주입 코드 디버깅 | **신규** |
| 7 | 콜백 + Promise + async/await | AI 생성 코드의 테스트 작성 | Ch11-12 재구성 |
| 8 | 미니 프로젝트: AI와 함께 ToDo 앱 | 스펙 작성 → AI 생성 → 테스트 → 리팩터 | Ch10 재설계 |

**설계 근거**: 버그 주입(Bug-Injection) 방법론은 Springer 체계적 리뷰에서 "비판적 사고, 반성적 사고, 논리적 사고, 적응적 사고를 촉진하는 이상적 맥락"으로 검증됨.

**Phase 2의 핵심 연습 유형 — "버그 헌트(Bug Hunt)"**

```javascript
// AI가 생성한 코드 (의도적 버그 2개 포함)
// 학습자는 버그를 찾고, 왜 잘못되었는지 설명하고, 수정해야 함

async function fetchUserData(userId) {
  const response = await fetch(`/api/users/${userId}`);
  const data = response.json();  // 버그 1: await 누락
  
  if (data.status = 'active') {  // 버그 2: == 또는 === 사용해야 함
    return data;
  }
  return null;
}
```

## 4. 쟁점별 페르소나 간 교차 분석

### 4.1 "AI를 언제 도입하는가?" — 합의와 쟁점

| 페르소나 | 입장 | 근거 |
|---------|------|------|
| Dr. 이론 | 주 3 이후 생성 모드 | 인지 부하 이론, 메타인지 선행 필요 |
| 박 시니어 | 주 1부터 설명 모드, 주 5부터 생성 모드 | 현업에서 AI 없는 개발은 비현실적 |
| 김 비평가 | 주 4 이후 생성 모드, 그 전까지 수동 코딩 100% | Anthropic RCT 결과, 인지 위축 위험 |
| 정 설계자 | 주 1부터 설명 모드, 주 5부터 생성+검증 모드 | 세 관점의 절충안 |

**합의점**: 코드 **생성** 모드는 기초 개념 확립 후 도입해야 한다 (최소 주 4 이후).  
**합의점**: AI를 **설명** 모드로 사용하는 것은 초기부터 긍정적이다.  
**쟁점**: 수동 코딩의 최소 비율에 대한 구체적 수치 (Dr.이론: 90%→50%→20%, 김 비평가: 100%→60%→30%)

### 4.2 "테스트를 어느 시점에 도입하는가?"

| 페르소나 | 입장 | 이유 |
|---------|------|------|
| Dr. 이론 | Phase 2 시작 (주 5) | 개념적 기초 없이 테스트 작성 불가 |
| 박 시니어 | Phase 1 중반 (주 3-4) | 실무에서 테스트는 코딩의 일부, 가능한 빨리 |
| 김 비평가 | Phase 1 후반 (주 4) | 테스트가 AI 출력 검증의 유일한 객관적 도구 |
| 정 설계자 | **주 4** (Phase 1 마지막) | 절충 — 기초 문법 습득 후 즉시 테스트 입문 |

**합의점**: 테스트는 기존 JS 과정처럼 빠뜨려서는 안 되며, Phase 2 시작 전에 도입되어야 한다.

### 4.3 "어떤 AI 도구를 표준으로 삼는가?"

| 도구 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| **GitHub Copilot** | 무료 티어 존재, VS Code 통합, 교육용 리소스 풍부 | 기능 변화가 빈번, UI 의존적 콘텐츠는 빠르게 구식화 | ★★★★ |
| **Cursor** | VS Code 포크로 친숙함, 멀티파일 컨텍스트 우수 | 유료, 한국어 지원 불안정 | ★★★ |
| **Claude Code** | 터미널 기반으로 도구 독립적, 에이전트 역량 우수 | CLI 환경 — 초보자에게 진입장벽 | ★★ |
| **ChatGPT** | 가장 친숙, 학습 모드 제공 | IDE 통합 부재, 코드 복사-붙여넣기 워크플로우 | ★★★ |

**정 설계자의 권고**: 원칙(Context Engineering, Role Separation)은 **도구 불가지론(Tool-Agnostic)**으로 가르치되, 실습 예제는 **GitHub Copilot Free**를 기본으로 한다. 각 실습에 "Cursor 사용 시", "ChatGPT 사용 시" 사이드바를 추가한다.

### 4.4 "한국어 프롬프트 vs 영어 프롬프트"

| 페르소나 | 입장 |
|---------|------|
| 박 시니어 | 영어 프롬프트가 출력 품질이 높다. 실무에서도 영어 사용. |
| 김 비평가 | 초보자에게 한국어+JS+AI+영어 프롬프트를 동시에 요구하면 인지 과부하. |
| 정 설계자 | Phase 1-2는 한국어 프롬프트 허용, Phase 3에서 영어 프롬프트 도입. |

## 6. 결론 — 페르소나 간 최종 합의

### 4인이 동의하는 것

1. AI-Native JavaScript 과정은 **시의적절하고 필요**하다
2. **기초 먼저, AI는 점진적으로** 도입해야 한다 (Anthropic RCT가 결정적 근거)
3. **테스트**는 반드시 포함되어야 한다 — AI 출력 검증의 유일한 객관적 도구
4. AI 사용 모드(설명/생성/완전 페어)를 **명시적으로 구분**하고 단계별로 개방해야 한다
5. **Bug Hunt** 연습은 비판적 사고를 보존하는 핵심 장치다
6. 기존 JS 과정의 14장은 12장으로 **압축 가능**하며, 빠진 핵심 역량(테스트, AI 워크플로우)을 추가해야 한다

### 4인이 동의하지 않는 것

| 쟁점 | 범위 |
|------|------|
| 수동 코딩 최소 비율 | 90%→20% (설계자) vs 100%→30% (비평가) |
| TypeScript 도입 여부 | 박 시니어(찬성) vs 김 비평가(반대 — 복잡성 과다) |
| 과정 형식 | 자기 주도형(Dr.이론) vs 집중 워크숍(박 시니어) |

## 7. 비전공자 대상 적용 — 커리큘럼 재설계

> **확정 사항**: 대상 학습자는 **비전공자** — 프로그래밍 경험이 전혀 없는 학습자

### 7.1 비전공자 특성과 설계 원칙

비전공자는 CS 전공자와 근본적으로 다른 출발점에서 시작한다:

| 차원 | CS 전공자 | 비전공자 |
|------|----------|---------|
| 사전 지식 | 알고리즘, 자료구조 기초 보유 | **전무** — 변수 개념부터 시작 |
| 추상적 사고 | 수학/논리 훈련 경험 | 구체적 결과를 먼저 보고 추상화 |
| 학습 동기 | 전공 필수 / 커리어 목표 | 실용적 목표 ("내 업무 자동화", "웹사이트 만들기") |
| 좌절 내성 | 상대적으로 높음 | **낮음** — 초기 3주 내 이탈 가능성 높음 |
| AI 도구 친숙도 | 개발 도구 사용 경험 있음 | ChatGPT 정도만 써봄 |
| 영어 프롬프트 | 기술 영어 독해 가능 | **한국어 우선** |

**비전공자를 위한 3대 설계 원칙**:

1. **즉각적 성취감 우선 (Immediate Gratification First)**
   - 첫 수업에서 "Hello, World!" 이상의 시각적 결과물을 만들어야 한다
   - AI를 설명 모드로 활용하여 "왜 이렇게 되는지" 즉시 답을 얻을 수 있는 경험 제공
   - **비전공자 이탈의 90%는 처음 3주 안에 발생** — 이 기간에 성취감이 생존을 결정

2. **구체→추상 순서 (Concrete Before Abstract)**
   - "함수란 재사용 가능한 코드 블록이다" (X) 
   - "버튼을 누르면 색이 바뀌는 걸 만들어 보자 → 이걸 함수라고 부른다" (O)
   - 기존 JS 과정의 비유/은유 접근법은 비전공자에게 **그대로 유지하거나 강화**

3. **인지 부하 최소화 (Minimal Cognitive Load)**
   - 한 수업에서 새로운 개념은 **최대 2개**
   - AI 도구 + JS 문법 + 개발 환경을 동시에 가르치지 않는다
   - Phase 1을 4주에서 **6주로 확장**

### 7.3 Phase 0 상세 — 비전공자 전용 워밍업

Phase 0은 기존 과정에 없던 **완전 새로운 단계**다. 프로그래밍 개념이 전혀 없는 학습자가 "나도 할 수 있겠다"는 확신을 갖게 하는 것이 목적이다.

**00장: 프로그래밍이란? + 환경 구축**

```
학습 목표:
- "프로그래밍"이 무엇인지 자신의 말로 설명할 수 있다
- VS Code를 열고 JavaScript 파일을 실행할 수 있다
- AI(ChatGPT)에게 프로그래밍 개념을 질문할 수 있다

핵심 비유:
"프로그래밍은 요리 레시피를 쓰는 것과 같습니다.
 재료(데이터)를 준비하고, 순서(알고리즘)대로 조리하면
 요리(프로그램)가 완성됩니다."

AI 사용 규칙:
- ChatGPT에게 "프로그래밍이 뭐야?", "변수가 뭐야?" 같은 
  개념 질문만 허용
- 코드 생성 요청 금지
- 목적: "AI는 무서운 게 아니라 친절한 선생님"이라는 인식 형성
```

**01장: 첫 번째 웹페이지 만들기**

```
학습 목표:
- HTML 파일을 만들고 브라우저에서 열 수 있다
- <script> 태그 안에 JavaScript 한 줄을 쓸 수 있다
- 버튼을 누르면 글자가 바뀌는 페이지를 만든다

30분 안에 만드는 첫 결과물:
┌─────────────────────────┐
│  나의 첫 웹페이지         │
│                         │
│  [인사하기 버튼]          │
│                         │
│  → 버튼 클릭 시:          │
│  "안녕하세요, 홍길동님!"   │
│  이 화면에 표시됨          │
└─────────────────────────┘

설계 근거: 
비전공자에게 console.log("Hello World")는 감동이 없다.
브라우저에서 시각적으로 변화가 보이는 결과물이 
"나도 할 수 있다"는 동기를 만든다.
```

### 7.4 비전공자를 위한 핵심 조정사항

#### 조정 1: Phase 1 확장 (4주 → 6주)

| 원래 | 변경 | 이유 |
|------|------|------|
| 1주에 변수+타입+조건문 압축 | 변수/타입(1주) + 조건문(1주) 분리 | 비전공자는 새 개념 2개/주가 한계 |
| 객체+ES6+테스트를 1주에 | 객체(1주) + 테스트 입문+AI설정(1주) 분리 | ES6는 Phase 2에서 자연 노출로 대체 |
| 스코프+클로저를 1주에 | 함수 기초(1주)에 집중, 클로저는 Phase 2에서 필요 시 | 클로저는 비전공자에게 Phase 1에서 과도 |

#### 조정 2: 클로저와 ES6를 Phase 1에서 제외

**Dr. 이론의 근거**: 클로저(Closure)는 추상적 개념으로, 함수와 스코프를 충분히 체험한 후에야 이해 가능하다. 비전공자에게 Phase 1에서 클로저를 가르치면 인지 과부하가 발생한다.

**대안**: Phase 2의 DOM/이벤트 실습에서 클로저가 자연스럽게 등장할 때 "여기서 사용된 패턴이 클로저입니다"로 소개한다.

```javascript
// Phase 2 - DOM 수업에서 자연스럽게 클로저를 만남
document.getElementById('counter-btn').addEventListener('click', () => {
  let count = 0;  // ← "이 변수가 아래 함수에서 계속 살아있는 이유가 뭘까?"
  return () => {   //    → AI에게 질문 → "이것이 클로저입니다" 
    count++;
    display.textContent = count;
  };
});
```

#### 조정 3: 한국어 프롬프트 전면 허용

| Phase | 프롬프트 언어 | 이유 |
|-------|-------------|------|
| Phase 0-1 (주 1-8) | **한국어 100%** | 한국어+JS+개발환경이 이미 인지 부하 최대 |
| Phase 2 (주 9-12) | **한국어 80% + 영어 키워드** 혼합 | "function을 만들어줘"처럼 기술 용어만 영어 |
| Phase 3 (주 13-16) | **영어 60% + 한국어 40%** | Custom Instructions는 영어가 AI 출력 품질 향상 |

#### 조정 4: TypeScript 제외 확정

**4인 합의**: 비전공자 대상에서 TypeScript는 **도입하지 않는다**.

| 찬성 논거 (박 시니어) | 반대 논거 (김 비평가 + Dr. 이론) | 판정 |
|---------------------|-------------------------------|------|
| AI 출력 품질 향상 | JS도 처음인 사람에게 타입 시스템은 과부하 | **제외** |
| 현업 표준 | 16주 안에 JS+AI+Testing+TS는 비현실적 | **제외** |
| 더 나은 에러 메시지 | JS의 에러 메시지도 아직 이해 못 하는 단계 | **제외** |

→ "TypeScript는 이 과정의 후속 과정에서 다룹니다" 안내로 대체

#### 조정 5: 수동 코딩 비율 상향

비전공자는 기초 형성 기간이 더 필요하므로, 수동 코딩 비율을 상향한다:

```
Phase 0: 100% 수동 (AI는 대화 상대로만)
Phase 1: 95% 수동 (AI는 설명 도구로만)
Phase 2: 60% 수동 (AI 생성 코드를 읽고/수정/검증)
Phase 3: 30% 수동 (AI가 구현, 학습자가 설계+검증)
```

#### 조정 6: 평가 방식 — 포트폴리오 기반

비전공자에게 전통적 시험은 부적절하다. 대신 **포트폴리오 기반 평가**:

| Phase | 평가 산출물 | 평가 기준 |
|-------|-----------|----------|
| Phase 0 | 첫 웹페이지 (HTML+JS) | 동작 여부만 확인 |
| Phase 1 | 간단한 계산기 프로그램 | AI 없이 직접 작성 가능 여부 |
| Phase 2 | AI 협업 ToDo 앱 + Bug Hunt 결과서 | AI 출력 검증 능력 |
| Phase 3 | 날씨 앱 + .github/ 폴더 | 전체 AI-Native 워크플로우 수행 능력 |

### 7.5 비전공자에게 AI가 특히 유용한 순간들

비전공자에게 AI는 **위험**이자 **기회**다. 올바르게 사용하면 비전공자의 가장 큰 장벽을 허물 수 있다:

| 비전공자의 장벽 | AI 활용 해법 | 사용 시점 |
|---------------|-------------|----------|
| "에러 메시지를 읽을 수 없다" | AI에게 에러 메시지를 한국어로 설명 요청 | Phase 1부터 |
| "문법을 외울 수 없다" | AI에게 "배열에서 특정 값을 찾으려면?" 질문 | Phase 1부터 |
| "영어 문서를 읽을 수 없다" | AI에게 MDN 문서 내용을 한국어로 요약 요청 | Phase 2부터 |
| "뭘 모르는지 모르겠다" | AI에게 "이 코드에서 내가 모르는 개념이 뭐야?" 질문 | Phase 1부터 |
| "혼자 공부하면 막혀서 포기" | AI가 24시간 질문 상대 역할 | 전 과정 |

이것이 바로 Anthropic 연구에서 가장 효과적이었던 **"개념 질의(Conceptual Inquiry)"** 패턴이며, 비전공자에게 이 패턴을 명시적으로 가르치는 것이 핵심이다.

### 7.9 대면 멘토링 운영 설계

자기 주도형 웹 문서만으로는 비전공자의 이탈을 막기 어렵다. 대면 멘토링을 병행하여 학습 효과를 극대화한다.

**멘토링 구조**

```
주간 사이클:
┌──────────────────────────────────────────────┐
│  월-금: 자기 주도 학습 (웹 문서 + AI 질의)      │
│  ──────────────────────────────────────────── │
│  주 1회: 대면 멘토링 세션 (2시간)               │
│   ├── 30분: 이번 주 학습 내용 확인 (Q&A)        │
│   ├── 60분: 실습 과제 함께 풀기 (페어 코딩)      │
│   └── 30분: 다음 주 예습 가이드 + 동기부여       │
└──────────────────────────────────────────────┘
```

**Phase별 멘토링 초점**

| Phase | 멘토링 초점 | 멘토 역할 |
|-------|-----------|----------|
| Phase 0 (주 1-2) | 환경 구축 트러블슈팅, "포기하지 마세요" | **기술 지원 + 동기부여자** |
| Phase 1 (주 3-8) | 문법 개념 확인, AI 질의 패턴 교정 | **개념 확인자 + AI 코치** |
| Phase 2 (주 9-12) | Bug Hunt 결과 리뷰, 테스트 작성 피드백 | **코드 리뷰어** |
| Phase 3 (주 13-16) | Custom Instructions 피드백, 프로젝트 아키텍처 조언 | **시니어 개발자** |

**Gate(관문) 통과 확인도 대면에서 진행**

- Phase 1 Gate: 멘토 앞에서 AI 없이 함수+반복문+조건문 작성
- Phase 2 Gate: 멘토에게 AI 생성 코드의 버그를 설명

### 7.11 API 연동 학습 설계 — 비전공자 친화적 접근

비전공자에게 API 연동은 "마법"처럼 느껴질 수 있다. 단계적으로 접근한다:

**사전 준비 (Phase 2, 10장 — 비동기 JavaScript)**

```
단계 1: "웹에서 데이터를 가져온다"는 개념 이해
  비유: "식당에서 음식을 주문하고 기다리는 것"
  - 주문(fetch) → 대기(await) → 수령(response)

단계 2: 무료 연습 API로 첫 경험
  JSONPlaceholder (https://jsonplaceholder.typicode.com)
  - /users → 사용자 목록
  - /posts → 게시물 목록
  → 복잡한 인증 없이 바로 데이터를 볼 수 있음

단계 3: AI에게 fetch 코드 생성 요청 → 한 줄씩 설명 → Vitest로 검증
```

**통합 프로젝트 (Phase 3, 15장 — 날씨 앱)**

```
사용 API: OpenWeatherMap (무료 티어)
  - API 키 발급 과정을 멘토링 시간에 함께 진행
  - .env 파일에 키 저장 (보안 습관)

프로젝트 진행 방식:
  1. 학습자가 요구사항을 한국어로 작성
  2. Custom Instructions로 AI에게 규칙 전달
  3. Copilot이 fetch + DOM 코드 생성
  4. 학습자가 Vitest로 API 응답 처리 로직 테스트
  5. 멘토와 함께 코드 리뷰
```
