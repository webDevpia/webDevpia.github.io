# 08장. 협업 기초 — 같이 문서 편집하기

> **같이 구글 문서를 편집하는 것과 비슷합니다.** 단, Git은 "동시에 같은 줄을 고치면" 알려줍니다. 각자 다른 부분을 고치면 자동으로 합쳐집니다.

---

<a id="part1"></a>

## 1️⃣ 협업 초대 [↑](#toc)

> "팀원에게 리포지토리에 접근할 수 있는 권한을 줍니다."

---

### GitHub에서 협업자 추가하기

**1단계**: 리포지토리 페이지 → 상단 **Settings** 탭

**2단계**: 왼쪽 메뉴 → **Collaborators** (또는 **Collaborators and teams**)

**3단계**: **Add people** 버튼 클릭

**4단계**: 팀원의 **GitHub 아이디** 또는 **이메일** 입력 → 선택 → **Add to repository**

**5단계**: 팀원이 이메일로 받은 초대를 수락하면 완료

---

### 협업자 권한

| 권한 | 읽기 | 쓰기 | 관리 |
|------|------|------|------|
| Read | ✅ | ❌ | ❌ |
| Write | ✅ | ✅ | ❌ |
| Maintain | ✅ | ✅ | 일부 |
| Admin | ✅ | ✅ | ✅ |

학습 팀 작업에는 **Write** 권한이 적합합니다.

---

### 팀원이 리포지토리 가져오기

초대를 수락한 팀원은 `git clone`으로 리포지토리를 내려받습니다:

```bash
git clone https://github.com/팀장_아이디/리포지토리명.git
```

이제 팀원도 `git push`를 사용할 수 있습니다.

<a id="part2"></a>

## 2️⃣ 팀 워크플로우 [↑](#toc)

> "각자 기능 브랜치에서 작업하고, 완성되면 main에 합칩니다."

이것을 **Feature Branch Workflow**라고 합니다. 실무에서 가장 기본적인 협업 방식입니다.

---

### 워크플로우 5단계

```bash
# 1단계: 항상 최신 main에서 시작
git switch main
git pull

# 2단계: 기능 브랜치 생성
git switch -c feature/my-feature

# 3단계: 작업, 스테이징, 커밋
git add .
git commit -m "Add: 로그인 페이지"

# 4단계: 원격에 푸시
git push -u origin feature/my-feature

# 5단계: GitHub에서 Pull Request 생성 (09장에서 자세히)
```

---

### 왜 이 순서를 지켜야 하나?

| 단계 | 이유 |
|------|------|
| `git pull` 먼저 | 팀원이 push한 최신 코드를 받아야 충돌 최소화 |
| 기능 브랜치 사용 | main을 항상 안정적으로 유지 |
| `git push`로 공유 | 팀원이 내 진행 상황을 볼 수 있음 |
| Pull Request | 코드 리뷰 후 main에 합침 |

---

### 실제 하루 작업 루틴

```bash
# 출근 (또는 작업 시작)
git switch main
git pull                           # 팀원 변경사항 받기

# 작업
git switch -c feature/add-navbar   # 기능 브랜치 생성
# ... 코드 작업 ...
git add .
git commit -m "Add: 상단 네비게이션 바 추가"

# 퇴근 (또는 작업 종료)
git push -u origin feature/add-navbar   # 원격에 백업
```

<a id="part3"></a>

## 3️⃣ git pull [↑](#toc)

> "`git pull` = 다른 사람의 변경사항을 내 컴퓨터로 내려받기"

```bash
git pull
```

실행 결과 (팀원이 push한 게 있을 때):
```
remote: Enumerating objects: 5, done.
remote: Counting objects: 100% (5/5), done.
remote: Total 4 (delta 1), reused 0 (delta 0)
Unpacking objects: 100% (4/4), done.
From https://github.com/username/my-first-repo
   a1b2c3d..7e8f9g0  main -> origin/main
Updating a1b2c3d..7e8f9g0
Fast-forward
 README.md | 3 +++
 1 file changed, 3 insertions(+)
```

실행 결과 (이미 최신 상태일 때):
```
Already up to date.
```

---

### git pull의 실제 동작

`git pull`은 두 명령어를 합친 것입니다:

```bash
git fetch   # 원격의 변경사항을 가져오기 (로컬 파일은 변경 안 함)
git merge   # 가져온 변경사항을 현재 브랜치에 합치기
```

---

### 특정 원격 브랜치에서 pull

```bash
# 특정 브랜치 pull
git pull origin feature/add-navbar

# 현재 추적 중인 브랜치에서만 pull (가장 일반적)
git pull
```

> 💡 **팁**: 팀 프로젝트에서는 아침마다 `git pull`로 시작하는 습관을 들이세요. 팀원이 어제 작업한 내용을 항상 최신 상태로 유지합니다.

---

<a id="part4"></a>

## 4️⃣ push 전에 pull! [↑](#toc)

> "push 하기 전에 반드시 pull! 안 하면 rejected 에러가 납니다."

---

### rejected 에러 발생 상황

내가 작업하는 동안 팀원이 같은 브랜치에 push했습니다.

```
원격:  A -- B -- C (팀원이 push함)
내 로컬: A -- B -- D (내가 커밋함)
```

이 상태에서 push하면:

```bash
git push
```

실행 결과:
```
To https://github.com/username/my-first-repo.git
 ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'https://github.com/username/my-first-repo.git'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. Integrate the remote changes (e.g.
hint: 'git pull ...') before pushing again.
```

`rejected` = "원격에 내가 모르는 커밋이 있습니다. 먼저 가져오세요."

---

### 해결 방법

```bash
# 1단계: 원격 변경사항 가져오기
git pull

# 2단계: 충돌이 없으면 자동 머지됨 (06장 참고)

# 3단계: 다시 push
git push
```

실행 후 상태:
```
원격:  A -- B -- C ------- M (머지 커밋)
                        /
내 로컬: A -- B -- D --
```

---

> ⚠️ **`git push -f`(force push)는 절대 하지 마세요.** 이 명령은 원격의 히스토리를 강제로 덮어씁니다. 팀원의 커밋이 영구적으로 사라질 수 있습니다. rejected 에러가 났을 때는 항상 `git pull` 후 다시 push합니다.

---

### push 전 체크리스트

```
□ 작업이 완료되었나?
□ git add 와 git commit 을 했나?
□ git pull 로 최신 상태를 받았나?
□ 충돌이 있다면 해결했나?
□ 이제 git push!
```

## 5️⃣ 브랜치 네이밍 규칙 [↑](#toc)

> "무엇을 하는 브랜치인지 이름만 봐도 알 수 있게."

---

### 표준 네이밍 패턴

| 패턴 | 용도 | 예시 |
|------|------|------|
| `feature/기능이름` | 새 기능 개발 | `feature/user-login`, `feature/dark-mode` |
| `fix/버그이름` | 버그 수정 | `fix/login-error`, `fix/broken-link` |
| `docs/문서이름` | 문서 작업 | `docs/api-guide`, `docs/readme-update` |
| `refactor/대상` | 코드 리팩토링 | `refactor/auth-module` |
| `hotfix/이슈` | 긴급 버그 수정 | `hotfix/payment-crash` |
| `chore/작업` | 설정, 빌드 등 | `chore/update-dependencies` |

---

### 좋은 이름 vs 나쁜 이름

| 나쁜 이름 | 좋은 이름 | 이유 |
|----------|----------|------|
| `test` | `feature/search-filter` | 무엇을 하는지 불명확 |
| `my-branch` | `feature/user-profile-page` | 누가 만들었는지는 중요하지 않음 |
| `branch1` | `fix/null-pointer-error` | 숫자는 의미가 없음 |
| `로그인` | `feature/login` | 한글보다 영문 권장 |

---

### 팀 규칙 정하기

브랜치 이름은 팀 내에서 사전에 합의합니다. 중요한 것은 **일관성**입니다. 아래처럼 팀 규칙 문서(보통 `CONTRIBUTING.md`)에 정리해두면 좋습니다:

```
브랜치 규칙:
- 기능: feature/기능명
- 수정: fix/버그명  
- 문서: docs/문서명
- 모두 소문자, 공백 대신 하이픈(-) 사용
```

## 6️⃣ 원격 브랜치 관리 [↑](#toc)

---

### 원격 브랜치에 push하기

```bash
# feature 브랜치를 GitHub에 올리기
git push -u origin feature/my-feature
```

실행 결과:
```
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
remote: 
remote: Create a pull request for 'feature/my-feature' on GitHub by visiting:
remote:   https://github.com/username/repo/pull/new/feature/my-feature
remote: 
To https://github.com/username/repo.git
 * [new branch]      feature/my-feature -> feature/my-feature
Branch 'feature/my-feature' set up to track remote branch 'feature/my-feature' from 'origin'.
```

GitHub가 자동으로 Pull Request 생성 링크를 안내합니다 (09장에서 자세히).

---

### 원격 브랜치 목록 확인

```bash
# 원격 브랜치 포함 전체 목록
git branch -a
```

실행 결과:
```
* feature/my-feature
  main
  remotes/origin/HEAD -> origin/main
  remotes/origin/feature/my-feature
  remotes/origin/main
```

`remotes/origin/`으로 시작하는 것이 원격 브랜치입니다.

---

### 원격 브랜치 삭제

```bash
# GitHub에서 브랜치 삭제 (Pull Request 머지 후)
git push origin --delete feature/my-feature
```

실행 결과:
```
To https://github.com/username/repo.git
 - [deleted]         feature/my-feature
```

> 💡 **팁**: Pull Request가 머지된 후 GitHub 웹에서 **"Delete branch"** 버튼을 클릭해도 됩니다. 로컬 브랜치는 따로 `git branch -d feature/my-feature`로 삭제합니다.

---

### 원격 브랜치 가져오기

팀원이 만든 브랜치를 내 로컬에서 작업하려면:

```bash
# 원격 정보 업데이트
git fetch

# 팀원 브랜치 로컬로 가져오기
git switch -c feature/teammate-work origin/feature/teammate-work
```

## 7️⃣ 실습: 2인 협업 시뮬레이션 [↑](#toc)

혼자서도 두 역할(팀장, 팀원)을 연습할 수 있습니다. 같은 컴퓨터의 두 폴더를 사용합니다.

---

### 기본 실습: 폴더 두 개로 협업 흉내내기

**준비:**

```bash
# 팀장 폴더 만들기
mkdir teamlead && cd teamlead
git init
echo "# 팀 프로젝트" > README.md
git add README.md
git commit -m "Init: 프로젝트 시작"
git remote add origin https://github.com/your-id/team-project.git
git push -u origin main
cd ..

# 팀원 폴더: clone으로 가져오기
git clone https://github.com/your-id/team-project.git teammate
cd teammate
```

---

**팀장: 기능 A 개발**

```bash
cd ../teamlead

git switch -c feature/feature-a
echo "기능 A 내용" > feature-a.txt
git add feature-a.txt
git commit -m "Add: 기능 A 추가"
git push -u origin feature/feature-a
```

---

**팀원: 기능 B 개발 (동시에)**

```bash
cd ../teammate

git switch main
git pull                               # 최신 상태 확인

git switch -c feature/feature-b
echo "기능 B 내용" > feature-b.txt
git add feature-b.txt
git commit -m "Add: 기능 B 추가"
git push -u origin feature/feature-b
```

---

**팀장: main에 기능 A 머지 후 push**

```bash
cd ../teamlead

git switch main
git merge feature/feature-a
git push
```

---

**팀원: pull 후 push 시도**

```bash
cd ../teammate

git switch main
git pull                               # 팀장이 push한 기능 A 받기
git merge feature/feature-b
git push                               # 이제 rejected 없음!
```

---

### 중급 실습: 충돌 상황 시뮬레이션

1. 팀장과 팀원이 모두 `README.md` 같은 줄을 수정합니다.
2. 팀장이 먼저 push합니다.
3. 팀원이 pull 없이 push를 시도하면 rejected 에러를 확인합니다.
4. `git pull` → 충돌 해결(06장 참고) → `git push` 순서로 해결합니다.

---

### 심화 실습

1. 실제 팀원(친구 또는 동기)을 초대해서 위 실습을 실제 환경에서 진행합니다.
2. 각자 다른 기능을 만들고 GitHub에서 두 브랜치의 상태를 확인합니다.
3. 09장(Pull Request)을 미리 예습하고 PR 생성까지 연결합니다.

---

## 🔟 정리 [↑](#toc)

| 명령어 | 역할 |
|--------|------|
| `git pull` | 원격 → 로컬 (fetch + merge) |
| `git fetch` | 원격 정보 업데이트 (파일 변경 없음) |
| `git push -u origin 브랜치` | 원격에 브랜치 push + upstream 설정 |
| `git push origin --delete 브랜치` | 원격 브랜치 삭제 |
| `git branch -a` | 로컬 + 원격 브랜치 전체 목록 |
| `git switch -c 이름 origin/원격브랜치` | 원격 브랜치를 로컬로 가져오기 |

---

### Feature Branch Workflow 한눈에 보기

```
main ─────────────────────────────────────► (항상 안정적)
      │                               │
      ▼                               ▲
feature/login ──커밋──커밋──커밋──► merge/PR
      │
      │  push
      ▼
origin/feature/login (GitHub)
```

1. `git switch main && git pull` — 최신 상태로 시작
2. `git switch -c feature/기능이름` — 기능 브랜치 만들기
3. 작업 → `git add` → `git commit` 반복
4. `git push -u origin feature/기능이름` — GitHub에 공유
5. GitHub에서 Pull Request 생성 → 리뷰 → 머지

---

### 협업 황금 규칙

```
push 하기 전에 반드시 pull!
```

이 한 줄을 지키면 rejected 에러의 90%를 예방할 수 있습니다.

---

### 다음 장 미리보기

함께 작업하는 기본을 배웠습니다. 하지만 아직 한 가지가 빠졌습니다: **코드 검토(리뷰)**입니다. 실무에서는 코드를 main에 직접 합치지 않고, **Pull Request**를 통해 팀원이 먼저 확인합니다. 다음 장에서 PR 생성, 리뷰, 머지의 전체 흐름을 배웁니다.
