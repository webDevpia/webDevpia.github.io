# 10장. 되돌리기 — 실수를 복구하는 방법들

<a id="overview"></a>

## 1️⃣ 되돌리기 종류 한눈에 [↑](#toc)

> "Git에는 여러 종류의 '되돌리기'가 있습니다. 연필 지우개(가벼운 수정)부터 타임머신(전체 복구)까지."

실수의 종류마다 적합한 명령어가 다릅니다. 아래 표를 상황판으로 활용하세요.

| 상황 | 명령어 | 위험도 |
|------|--------|--------|
| 작업 중인 파일 변경 취소 | `git restore 파일명` | 안전 |
| 스테이징 취소 (add 취소) | `git restore --staged 파일명` | 안전 |
| 마지막 커밋 메시지 수정 | `git commit --amend -m "새 메시지"` | 주의 — push 전에만 |
| 커밋 취소 (이력 유지) | `git revert 커밋ID` | 안전 |
| 커밋 되돌리기 (이력 변경) | `git reset` | ⚠️ 위험 |
| 임시 저장 | `git stash` | 안전 |
| 삭제된 커밋 복구 | `git reflog` | 안전 |

> 💡 기본 원칙: **이미 push한 커밋은 이력을 바꾸지 않습니다.** `git revert`처럼 새 커밋으로 취소하는 방법을 사용하세요.

<a id="restore"></a>

## 2️⃣ git restore — 작업 중인 파일 변경 취소 [↑](#toc)

> "파일을 수정했는데 다시 원래대로 돌리고 싶을 때 씁니다. 마지막 커밋 상태로 되돌아갑니다."

### 사용법

```bash
git restore 파일명
```

### 예시

```bash
# index.html을 실수로 수정했습니다
git status
```

실행 결과:
```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)

        modified:   index.html
```

```bash
# 변경을 취소하고 마지막 커밋 상태로 복원합니다
git restore index.html

git status
```

실행 결과:
```
On branch main
nothing to commit, working tree clean
```

### 여러 파일 한꺼번에 취소

```bash
# 모든 수정 취소
git restore .
```

> ⚠️ `git restore`는 **커밋하지 않은 변경을 영구 삭제**합니다. 실행 전에 저장할 내용이 없는지 꼭 확인하세요. 한 번 실행하면 되돌릴 수 없습니다.

### VS Code에서 사용하기

소스 제어 패널(Ctrl+Shift+G)에서 변경된 파일 옆 되돌리기 아이콘(↺)을 클릭하면 동일한 결과를 얻을 수 있습니다.

<a id="restore-staged"></a>

## 3️⃣ git restore --staged — 스테이징 취소 [↑](#toc)

> "포장 상자에 넣었다가 다시 꺼내기. `git add`를 했지만 아직 커밋하기 싫을 때 씁니다."

### 사용법

```bash
git restore --staged 파일명
```

### 예시

```bash
# 실수로 add했습니다
git add 실수로추가된파일.txt

git status
```

실행 결과:
```
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)

        new file:   실수로추가된파일.txt
```

```bash
# 스테이징만 취소합니다 (파일 내용은 그대로 유지됩니다)
git restore --staged 실수로추가된파일.txt

git status
```

실행 결과:
```
On branch main
Untracked files:
  (use "git add <file>..." to include in what will be committed)

        실수로추가된파일.txt
```

### 스테이징 전체 취소

```bash
git restore --staged .
```

> 💡 `git restore --staged`는 파일 내용을 변경하지 않습니다. 스테이징 영역에서 꺼내는 것뿐입니다. 파일은 Working Directory에 그대로 남아 있습니다.

### 4-Zone 모델로 이해하기

```
Remote ← [push/pull] → Local Repo ← [commit/restore] → Staging ← [add/restore --staged] → Working Dir
                                                                         ↑
                                                           여기서 Working Dir로 되돌림
```

<a id="amend"></a>

## 4️⃣ git commit --amend — 마지막 커밋 메시지 수정 [↑](#toc)

> "방금 커밋했는데 메시지에 오타가 있을 때 씁니다."

### 사용법

```bash
git commit --amend -m "수정된 커밋 메시지"
```

### 예시

```bash
# 오타가 있는 커밋을 했습니다
git commit -m "feat: 로그인 기능 추가" # "추가" 오타 — 실제로는 "수정"이었어야 합니다

# 메시지를 바로 수정합니다
git commit --amend -m "feat: 로그인 기능 수정"

git log --oneline
```

실행 결과:
```
a1b2c3d feat: 로그인 기능 수정
...
```

### 파일도 함께 수정하고 싶을 때

```bash
# 빠뜨린 파일을 추가 후 amend
git add 빠뜨린파일.txt
git commit --amend --no-edit  # 메시지는 그대로, 파일만 추가
```

> ⚠️ **이미 push한 커밋은 amend하지 마세요!** amend는 커밋 해시를 바꿉니다. push한 커밋을 amend한 뒤 다시 push하려면 강제 push(`git push -f`)가 필요한데, 이는 다른 사람의 이력과 충돌을 일으킵니다. amend는 **로컬에서, push 전에만** 사용하세요.

> ⚠️ **`git push -f`는 팀원의 작업을 파괴할 수 있습니다.** 공유 브랜치(main, develop)에서는 절대 사용하지 마세요.

<a id="revert"></a>

## 5️⃣ git revert — 커밋 취소 (안전, 이력 유지) [↑](#toc)

> "지나간 커밋의 변경을 취소하는 새 커밋을 만듭니다. 이력은 지우지 않으면서 결과만 되돌립니다."

### 사용법

```bash
git revert 커밋ID
```

### 예시

```bash
# 되돌려야 할 커밋 ID를 확인합니다
git log --oneline
```

실행 결과:
```
e5f6g7h feat: 결제 기능 추가 (버그가 있었습니다)
a1b2c3d feat: 로그인 기능 수정
...
```

```bash
# 버그가 있는 커밋을 취소하는 새 커밋을 만듭니다
git revert e5f6g7h
```

실행 결과:
```
[main h8i9j0k] Revert "feat: 결제 기능 추가"
 1 file changed, 5 insertions(+), 20 deletions(-)
```

```bash
git log --oneline
```

실행 결과:
```
h8i9j0k Revert "feat: 결제 기능 추가"
e5f6g7h feat: 결제 기능 추가
a1b2c3d feat: 로그인 기능 수정
```

원래 커밋(e5f6g7h)은 이력에 그대로 남고, 그것을 취소하는 새 커밋(h8i9j0k)이 추가됩니다.

### 여러 커밋 한꺼번에 revert

```bash
# 커밋을 바로 만들지 않고 스테이징만 합니다
git revert --no-commit 커밋ID1
git revert --no-commit 커밋ID2
# 한 번에 커밋합니다
git commit -m "revert: 결제 관련 커밋 전체 취소"
```

### revert 중 충돌이 생겼을 때

```bash
# 충돌 해결 후 계속 진행
git add 충돌해결된파일.txt
git revert --continue

# 취소하고 원래 상태로 돌아가기
git revert --abort
```

> 💡 이미 push한 커밋을 되돌릴 때는 항상 `git revert`를 사용하세요. 팀원의 이력을 건드리지 않아 안전합니다.

<a id="reset"></a>

## 6️⃣ git reset — 커밋 되돌리기 (주의 필요) [↑](#toc)

> "타임머신을 타고 과거로 돌아가는 것과 같습니다. 단, 그 사이의 이력이 사라집니다."

`git reset`은 **커밋 포인터(HEAD)를 과거로 이동**시킵니다. 세 가지 옵션이 있으며 각각 파일과 스테이징 영역을 어떻게 처리할지 결정합니다.

### --soft: 커밋만 취소, 스테이징 유지

![git reset --soft 다이어그램](./img/git/git_6_1.png)

```bash
git reset --soft 커밋ID
```

커밋은 취소되지만 변경 내용이 **스테이징 영역에 그대로 남습니다.** 바로 다시 커밋할 수 있습니다.

```bash
git log --oneline
```

실행 결과 (reset 전):
```
c3d4e5f feat: 기능 C 추가
b2c3d4e feat: 기능 B 추가
a1b2c3d feat: 기능 A 추가
```

```bash
# 기능 C 커밋을 취소하고 변경은 스테이징에 유지
git reset --soft b2c3d4e

git status
```

실행 결과:
```
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)

        modified:   feature_c.txt
```

---

### --mixed: 커밋 + 스테이징 취소, 파일은 유지 (기본값)

![git reset --mixed 다이어그램](./img/git/git_6_3.png)

```bash
git reset 커밋ID          # --mixed가 기본값입니다
git reset --mixed 커밋ID  # 동일합니다
```

커밋과 스테이징이 모두 취소되고, 변경 내용은 **Working Directory에 남습니다.**

```bash
git reset b2c3d4e

git status
```

실행 결과:
```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)

        modified:   feature_c.txt
```

---

### --hard: 모든 것 취소 (데이터 삭제!)

![git reset --hard 다이어그램](./img/git/git_6_5.png)

```bash
git reset --hard 커밋ID
```

커밋, 스테이징, 파일 변경 모두 사라집니다. 지정한 커밋 시점으로 완전히 돌아갑니다.

```bash
git reset --hard b2c3d4e
```

실행 결과:
```
HEAD is now at b2c3d4e feat: 기능 B 추가
```

> ⚠️ **`git reset --hard`는 작업 내용을 영구 삭제합니다.** 커밋하지 않은 변경과 취소된 커밋의 내용이 모두 사라집니다. 복구하려면 `git reflog`를 사용해야 합니다 (8️⃣ 섹션 참고).

> ⚠️ **push한 커밋에는 절대 `git reset`을 사용하지 마세요.** 로컬 이력이 원격과 달라져서 강제 push(`git push -f`)가 필요해지고, 팀원의 작업을 덮어씌울 수 있습니다.

> ⚠️ **`git push -f`는 팀원의 작업을 파괴할 수 있습니다.** 공유 브랜치에서는 절대 사용하지 마세요. 본인만 사용하는 개인 브랜치에서, 그리고 결과를 충분히 이해한 상태에서만 극히 제한적으로 사용하세요.

### 세 옵션 비교

![git reset 3가지 비교](./img/git/git_6_7.png)

| 옵션 | 커밋 취소 | 스테이징 | 파일 내용 | 용도 |
|------|-----------|----------|-----------|------|
| `--soft` | O | 유지 | 유지 | 커밋 메시지를 다시 작성할 때 |
| `--mixed` (기본값) | O | 취소 | 유지 | add를 다시 골라서 할 때 |
| `--hard` | O | 취소 | **삭제** | 완전히 되돌려야 할 때 |

## 7️⃣ git stash — 임시 저장 [↑](#toc)

> "작업 중인 내용을 서랍에 넣고 다른 일을 하다가, 나중에 꺼내는 것입니다."

급하게 다른 브랜치로 이동해야 하는데 현재 작업이 아직 커밋할 단계가 아닐 때 씁니다.

### 기본 사용법

```bash
# 현재 작업을 임시 저장합니다
git stash

# 임시 저장 목록 확인
git stash list

# 가장 최근 임시 저장을 꺼냅니다 (stash에서 삭제됨)
git stash pop

# 가장 최근 임시 저장을 적용합니다 (stash에서 삭제 안 됨)
git stash apply
```

### 예시

```bash
# 작업 중인 상태
git status
```

실행 결과:
```
On branch feature/login
Changes not staged for commit:
        modified:   login.html
        modified:   style.css
```

```bash
# 급하게 main으로 이동해야 합니다
git stash
```

실행 결과:
```
Saved working directory and index state WIP on feature/login: a1b2c3d feat: 로그인 기능 수정
```

```bash
# 이제 깨끗한 상태입니다
git status
```

실행 결과:
```
On branch feature/login
nothing to commit, working tree clean
```

```bash
# 다른 브랜치로 이동하고 작업합니다
git switch main
# ... main에서 급한 작업 처리 ...

# 다시 원래 브랜치로 돌아옵니다
git switch feature/login

# stash에서 작업을 꺼냅니다
git stash pop
```

실행 결과:
```
On branch feature/login
Changes not staged for commit:
        modified:   login.html
        modified:   style.css
Dropped refs/stash@{0} (3b8c9d1e...)
```

### stash 목록 관리

```bash
# 여러 개의 stash를 쌓았을 때 목록 확인
git stash list
```

실행 결과:
```
stash@{0}: WIP on feature/login: a1b2c3d feat: 로그인 기능 수정
stash@{1}: WIP on main: e5f6g7h feat: 결제 기능 추가
```

```bash
# 특정 stash 적용
git stash apply stash@{1}

# 특정 stash 삭제
git stash drop stash@{1}

# stash 전체 삭제
git stash clear
```

### 메시지를 붙여 stash 저장

```bash
# 나중에 알아보기 쉽게 설명을 붙입니다
git stash push -m "로그인 폼 스타일 작업 중"

git stash list
```

실행 결과:
```
stash@{0}: On feature/login: 로그인 폼 스타일 작업 중
```

> 💡 Untracked 파일(새로 만든 파일)까지 stash하려면 `-u` 옵션을 추가합니다.
> ```bash
> git stash -u
> ```

<a id="reflog"></a>

## 8️⃣ git reflog — Git의 블랙박스 [↑](#toc)

> "비행기 블랙박스처럼 Git의 모든 HEAD 이동 기록을 보존합니다. `git reset --hard`로 커밋을 삭제해도 reflog에서 복구할 수 있습니다."

`git log`는 커밋 이력만 보여주지만, `git reflog`는 **HEAD가 이동한 모든 기록**을 보여줍니다.

### 기본 사용법

```bash
git reflog
```

실행 결과:
```
a1b2c3d (HEAD -> main) HEAD@{0}: reset: moving to b2c3d4e
c3d4e5f HEAD@{1}: commit: feat: 기능 C 추가
b2c3d4e HEAD@{2}: commit: feat: 기능 B 추가
a1b2c3d HEAD@{3}: commit: feat: 기능 A 추가
```

### reset --hard 실수를 복구하는 방법

```bash
# 실수로 reset --hard를 했습니다
git reset --hard b2c3d4e

# 이제 기능 C 커밋(c3d4e5f)이 사라진 것처럼 보입니다
git log --oneline
```

실행 결과:
```
b2c3d4e feat: 기능 B 추가
a1b2c3d feat: 기능 A 추가
```

```bash
# reflog에서 삭제된 커밋을 찾습니다
git reflog
```

실행 결과:
```
b2c3d4e (HEAD -> main) HEAD@{0}: reset: moving to b2c3d4e
c3d4e5f HEAD@{1}: commit: feat: 기능 C 추가   ← 여기 있습니다!
b2c3d4e HEAD@{2}: commit: feat: 기능 B 추가
```

```bash
# 삭제된 커밋으로 복구합니다
git reset --hard c3d4e5f

git log --oneline
```

실행 결과:
```
c3d4e5f feat: 기능 C 추가   ← 복구되었습니다!
b2c3d4e feat: 기능 B 추가
a1b2c3d feat: 기능 A 추가
```

> 💡 reflog 기록은 기본적으로 **90일간 보존**됩니다. 90일이 지나면 Git의 가비지 컬렉션이 정리합니다. 실수를 발견하면 빨리 복구하세요.

> 💡 reflog는 **로컬에서만 동작**합니다. 다른 사람의 작업이나 원격 저장소의 이력은 reflog로 복구할 수 없습니다.

### 특정 브랜치의 reflog 확인

```bash
git reflog show main
git reflog show feature/login
```

## 9️⃣ 정리 [↑](#toc)

### 상황별 명령어 빠른 참조

| 상황 | 명령어 | 안전 여부 |
|------|--------|-----------|
| 작업 중인 파일을 마지막 커밋 상태로 복원 | `git restore 파일명` | 안전 (커밋 안 한 변경은 삭제됨) |
| add 취소 (스테이징 해제) | `git restore --staged 파일명` | 안전 |
| 마지막 커밋 메시지 수정 (push 전) | `git commit --amend -m "메시지"` | 주의 — push 전에만 |
| 커밋 안전하게 취소 (공유 브랜치) | `git revert 커밋ID` | 안전 |
| 커밋 취소, 변경은 스테이징에 보존 | `git reset --soft 커밋ID` | 로컬 전용 |
| 커밋 + 스테이징 취소, 파일은 보존 | `git reset --mixed 커밋ID` | 로컬 전용 |
| 커밋 + 파일 모두 삭제 | `git reset --hard 커밋ID` | ⚠️ 위험 |
| 작업 임시 저장 | `git stash` | 안전 |
| 임시 저장 꺼내기 | `git stash pop` | 안전 |
| 삭제된 커밋 찾기 | `git reflog` | 안전 |
| reflog로 커밋 복구 | `git reset --hard 복구할커밋ID` | 로컬 전용 |

### 한 줄 결정 트리

```
되돌려야 할 것이 있나요?
├── 커밋 안 한 파일 변경 → git restore 파일명
├── add를 취소하고 싶다 → git restore --staged 파일명
├── 방금 커밋 메시지가 잘못됐다 (push 전) → git commit --amend
├── 이미 push한 커밋을 취소해야 한다 → git revert 커밋ID
├── 로컬 커밋만 취소하고 싶다
│   ├── 변경 내용을 스테이징에 보존 → git reset --soft
│   ├── 변경 내용을 파일에 보존 → git reset --mixed
│   └── 모든 것을 삭제 → git reset --hard (⚠️ 위험)
├── 다른 브랜치로 잠시 이동해야 한다 → git stash
└── reset --hard로 삭제했는데 복구하고 싶다 → git reflog
```

### 위험도 정리

| 위험도 | 명령어 | 이유 |
|--------|--------|------|
| 안전 | `git restore`, `git revert`, `git stash` | 이력이 보존되거나 새 커밋으로 취소 |
| 주의 | `git commit --amend` | push 전에만 사용해야 합니다 |
| ⚠️ 위험 | `git reset --hard` | 커밋 안 한 변경이 영구 삭제됩니다 |
| ⚠️ 매우 위험 | `git push -f` (공유 브랜치) | 팀원의 작업을 파괴할 수 있습니다 |

### 이 장에서 배운 것

- `git restore`로 작업 중인 파일 변경을 안전하게 취소할 수 있습니다
- `git restore --staged`로 실수로 add한 파일을 스테이징에서 꺼낼 수 있습니다
- `git commit --amend`는 push 전 커밋에만 사용합니다
- `git revert`는 이력을 유지하면서 취소하므로 공유 브랜치에 안전합니다
- `git reset`의 세 옵션(--soft / --mixed / --hard)은 각각 다른 범위를 취소합니다
- `git stash`로 작업 내용을 임시 저장하고 나중에 꺼낼 수 있습니다
- `git reflog`는 최후의 안전망으로 90일 내 모든 HEAD 이동 기록을 보여줍니다
